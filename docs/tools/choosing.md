# ツールの選び方

「結局どれを使えばいいのか」——AI開発ツールの選定は、機能比較表を眺めても答えが出ません。なぜなら最適解は **あなたのタスク・制約・チーム** に依存するからです。本ページでは、製品名から入るのではなく、**選定基準の体系** から入ります。

前提として、[ツールエコシステムの全体像](/tools/landscape)で述べたとおり、これは「1つを選ぶ」問題ではなく「層をどう組むか」の問題です。多くの場合、補完層に1つ、エージェント層に1つ、という **複数選定** が現実的な答えになります。

::: tip このページの要点
選定は8つの軸で構造化できます。すべてを満点にするツールは存在しないので、**自分の制約のうち譲れないもの** を先に決め、そこから絞り込むのが実務的です。
:::

## 選定の8軸

| 軸 | 問い | 効いてくる場面 |
| --- | --- | --- |
| 1. タスク種別 | 補完中心か、タスク委任か、調査か | 必要な層が決まる |
| 2. コンテキスト規模 | 数行か、ファイルか、リポジトリ横断か | 検索型/エージェント型の要否 |
| 3. プライバシー/データ取扱い | 送信・学習利用・自己ホストの制約は | 使えるツールが絞られる |
| 4. IDE/CLI志向 | 既存IDE中心か、端末中心か | 形態(プラグイン vs 独立IDE vs CLI) |
| 5. コスト | 個人の月額か、組織のシート単価か | プラン選択・モデル選択 |
| 6. チーム統一 vs 個人裁量 | 全員同じツールか、各自最適か | ガバナンスの設計 |
| 7. ベンダーロックイン | 1ベンダー依存をどこまで許容するか | モデル/エディタの可搬性 |
| 8. モデル選択 | モデルを選べるか、固定か | 推論力とレイテンシの調整 |

以下、それぞれを掘り下げます。

### 1. タスク種別

最初に「自動化したい作業の粒度」を決めます。行を速く書きたいなら補完層、タスクを丸ごと委任したいならエージェント層、巨大なコードベースを理解したいなら検索型。粒度が決まれば候補は半分に絞れます([全体像の6分類](/tools/landscape)を参照)。

### 2. コンテキスト規模

扱う情報がファイル近傍で足りるなら補完・チャットで十分です。複数ファイルを横断する、あるいはリポジトリ全体の知識が要るなら、自分で探索できるエージェント型(Claude Code, Aider, Cline)や検索型(Cody, Amazon Q)が必要になります。コンテキストの限界については[コンテキストウィンドウ](/concepts/context-window)を参照してください。

### 3. プライバシー/データ取扱い

これは多くの組織で **最初に効く制約** です。

- コードがクラウドに送信されるか、自己ホスト/オンプレ運用が可能か。
- 入力が学習に使われるか(ビジネス向けプランでは学習除外が一般的だが要確認)。
- 保持期間・ログの扱い。

Tabnineのように自己ホスト/プライバシーを前面に出す製品もあれば、企業向けプランで学習除外やデータ境界を提供する製品もあります。**執筆時点の規約は変わりやすい** ため、必ず最新の利用規約とプラン条件を確認してください。

### 4. IDE/CLI志向

既存の開発体験を変えたくないなら、いま使っているIDEのプラグイン(Copilot, Codeium, JetBrains AI, Cline)。AIを中心に据えてよいなら独立IDE(Cursor, Windsurf)。端末中心のワークフローならCLI(Claude Code, Aider)。**移行コスト** が判断材料になります。

### 5. コスト

個人利用の月額サブスクリプションと、組織のシート単価+API従量課金では、コスト構造が異なります。エージェント型はモデル呼び出しが多くなりがちで、使い方次第でコストが膨らみます。

::: warning 価格は揮発的
各ツールの価格・無料枠・利用上限は頻繁に変わります。本ページでは具体的な金額を断定しません。**執筆時点の数字を鵜呑みにせず**、導入時点の公式情報で見積もってください。コスト最適化の一手は、軽い作業に安価/低レイテンシなモデル、難しい作業に高性能モデルを充てる **モデルの使い分け** です。
:::

### 6. チーム統一 vs 個人裁量

組織導入では「全員同じツールに揃える(ガバナンス・サポート・教育が容易)」か「各自が裁量で選ぶ(最適化しやすいが管理が散る)」かを決めます。多くの組織は **データ取り扱いとセキュリティだけ統一し、ツール選択は一定の許可リスト内で裁量** という折衷を採ります。

### 7. ベンダーロックイン

「エディタのロックイン」(独立IDEへ移ると戻りにくい)と「モデルのロックイン」(特定プロバイダのモデルしか使えない)は別物です。可搬性を重視するなら、既存IDEのプラグイン形態で、複数モデルを選べるツールが有利です。

### 8. モデル選択

モデルを選べるツールでは、タスクに応じて推論力とレイテンシを調整できます。Claudeを使う場合、難しい計画・設計にはOpus 4.8、日常的な実装・対話にはSonnet 4.6、軽量・高速な処理にはHaiku 4.5、といった使い分けが目安です(執筆時点)。モデル固定のツールは、その固定モデルの素性が自分の用途に合うかを確認します。

## 意思決定チェックリスト

導入前に、上から順に答えてください。上の問いほど候補を大きく絞ります。

1. ☐ **データ制約**: コードをクラウドに送れるか? 自己ホストは必須か?(必須なら候補が大きく絞られる)
2. ☐ **粒度**: 自動化したいのは 行 / 関数 / タスク / リポジトリ理解 のどれか?
3. ☐ **形態**: 既存IDE維持 / 独立IDE許容 / CLI のどれか?
4. ☐ **自律性**: 提案のみ / 承認制 / 自律実行 のどこまで許すか?(→[エージェント型ツール](/tools/agentic-tools)の承認フロー)
5. ☐ **検証**: 結果を検証する手段(テスト・レビュー)はあるか?
6. ☐ **コスト**: 予算は個人月額 / 組織シート単価 / API従量 のどれか?
7. ☐ **統一方針**: チームで揃えるか、個人裁量か?
8. ☐ **可搬性**: ベンダー/エディタ/モデルのロックインを許容できるか?

::: tip 1つに決めなくてよい
チェックリストの結論は多くの場合「補完層に1つ + エージェント層に1つ」という **組み合わせ** になります。最初から完璧を目指さず、まず補完層を導入し、慣れたらエージェント層を足す、という段階的導入が安全です。
:::

## ペルソナ別の推奨構成

以下はあくまで **出発点** です。製品名はカテゴリの代表であり、同カテゴリの他ツールで置き換え可能です。

| ペルソナ | 状況 | 構成の出発点 |
| --- | --- | --- |
| 個人開発者(既存IDE維持) | VS Code等を使い続けたい | 補完(Copilot/Codeium)+ CLIエージェント(Claude Code/Aider) |
| AI中心に振り切る個人 | ワークフローを刷新してよい | 統合IDE(Cursor/Windsurf)を主軸に |
| JetBrainsユーザー | IntelliJ/PyCharm中心 | JetBrains AI Assistant + 必要に応じてCLIエージェント |
| プライバシー重視の組織 | 送信・学習に厳しい制約 | 自己ホスト志向のツール(例: Tabnine)を中心に検討 |
| 大規模/レガシーを抱えるチーム | コード理解がボトルネック | 検索型(Cody/Amazon Q)+ エージェント型を併用 |
| エージェントを本格運用するチーム | タスク委任を増やしたい | CLIエージェント(Claude Code)+ AIコードレビューを必須化 |

::: warning 推奨は陳腐化する
ツールの機能・価格・モデルは速いペースで変化します。上表は「考え方の型」として使い、**導入時点では必ず各製品の最新情報で再評価** してください。重要なのは製品名そのものではなく、8軸に照らした判断です。
:::

## 選定後にやること

ツールを選んだら、それで終わりではありません。

- **権限と承認フローを設計する** — 特にエージェント型は[安全な権限設計](/tools/agentic-tools)が前提。
- **検証を整える** — テスト・linter・コードレビューが、生成物の品質を担保する。
- **指示の質を上げる** — 同じツールでも入力次第で出力が変わる。[プロンプトの基礎](/prompting/fundamentals)へ。
