# コスト最適化

LLMアプリケーションのコストは、放置すると静かに膨らみます。コンテキストにファイルを足し、リトライを重ね、全タスクをフラッグシップで回す——どれも単体では小さな判断ですが、トラフィックが増えると請求書で牙を剥きます。

本ページは、品質を落とさずにコストとレイテンシを下げるための実践的な手法を整理します。前提として[トークン経済](/concepts/how-llms-work)と[コンテキストウィンドウ](/concepts/context-window)、振り分けの基礎は[モデルの選び方](/models/choosing-models)を参照してください。

::: tip ひとことで言うと
コスト最適化の本質は **「同じ品質をより少ないトークン・より安いモデルで出す」** こと。トークンを減らし、安いモデルに回し、無駄な再生成を消す、の3本柱です。
:::

## トークン経済の基礎

LLMの課金は通常、処理した **トークン数** に比例します。トークンは単語より細かい単位で、おおむね英語で1単語=1〜2トークン、日本語はより多くなりがちです。重要なのは次の非対称性です。

| | 入力トークン(プロンプト) | 出力トークン(生成) |
| --- | --- | --- |
| 対象 | システムプロンプト+文脈+履歴+指示 | モデルが生成したテキスト |
| 単価 | 相対的に安い | **相対的に高い**(一般的傾向) |
| 制御性 | 設計で削れる(文脈選別) | 出力フォーマットや長さ指定で削れる |

::: warning 具体的な単価は揮発する
入力/出力の単価、モデル間の価格差は頻繁に改定されます。**執筆時点の各社一次情報(料金ページ)を必ず確認**してください。本ページは「どこにコストが効くか」の構造を扱い、絶対額は断定しません。
:::

1回のリクエストのコストは、おおむね次で見積もれます。

```text
コスト ≈ 入力トークン × 入力単価 + 出力トークン × 出力単価
```

エージェントやマルチターンでは、これが **ループの回数だけ累積** します。コスト最適化が効くのは、まさにこの累積部分です。

## コンテキスト肥大化のコスト

最も見落とされる無駄が **コンテキストの肥大化** です。エージェントやチャットでは、履歴・観測・ファイル内容が雪だるま式に積み上がり、毎ターンその全量が入力トークンとして課金されます。

```text
ターン1: [system + 質問]                         → 小
ターン2: [system + 質問 + 応答 + 新質問]          → 中
ターン3: [system + ...全履歴... + 新質問]         → 大
...
ターンN: [system + 巨大な全履歴 + 新質問]         → 毎回これを課金
```

つまり **会話が長いほど、1ターンあたりのコストが増え続けます**。しかも[lost in the middle](/concepts/context-window)により、肥大化は品質も下げます。対策は次の通りです。

- **要約圧縮** — 古い履歴を要約に畳む。詳細を捨て要点を残す。
- **文脈の選別** — 全ファイルでなく関連箇所だけを[RAG](/concepts/rag)で渡す。
- **不要な履歴の切り捨て** — 完了したサブタスクの観測を落とす。
- **システムプロンプトの精査** — 毎回送られる固定部分の冗長を削る(キャッシュとも相性が良い、後述)。

## プロンプトキャッシュ

同じプレフィックス(システムプロンプト、共通の文脈、ツール定義など)を何度も送る場合、**プロンプトキャッシュ**で大幅に節約できます。一度処理した固定部分を事業者側がキャッシュし、再送時の入力コストを下げる仕組みです。

```text
[キャッシュ可能なプレフィックス: system + ツール定義 + 共通文脈]  ← 固定・キャッシュ対象
[可変部分: ユーザーの今回の入力]                                ← 毎回変わる
```

効かせるコツは **「固定部分を前に、可変部分を後ろに」** 配置すること。プレフィックスが一致するほどキャッシュヒットします。エージェントのように長いシステムプロンプトを何度も使う用途では効果が大きいです。

::: info キャッシュの効き方は事業者ごとに違う
キャッシュの有効期間・最小サイズ・割引率・明示指定の要否は事業者やAPIによって異なります。**一次情報で仕様を確認**し、自分のトラフィック形状(同一プレフィックスの再利用率)で効果を見積もってください。
:::

## モデルルーティング(振り分け)

全タスクをフラッグシップで処理するのは、最も分かりやすい無駄です。**入力の難易度に応じて安いモデルと高いモデルを使い分ける**ことで、品質を保ったままコストを下げられます。

```text
入力 → [難易度・カテゴリの判定]
        ├─ 定型・易しい → 軽量モデル(安い・速い)
        ├─ 通常        → 中位モデル
        └─ 難所・複雑   → フラッグシップ(高い・賢い)
                         └─ 軽量で失敗 → 上位へエスカレーション
```

2つの代表的な戦略があります。

- **事前ルーティング** — 入力を見て最初から適切なモデルへ。判定は軽量モデルやヒューリスティックで。
- **エスカレーション** — まず安いモデルで試し、失敗(検証で落ちる、自信が低い)したら上位へ。コード生成のように **機械的に成否判定できるタスク**と相性が良い。

::: tip カスケードの考え方
「安いモデルで大半を片付け、難しい少数だけ高いモデルに回す」というカスケードは、トラフィックの難易度分布が偏っている(易しいものが多い)ほど効きます。多くの実トラフィックはこの形をしています。ただし振り分けロジックの精度と保守コストが新たに発生する点は織り込んでください。
:::

## バッチ処理

即時性が要らない大量処理(夜間のログ分類、データセットのラベル付けなど)では、**バッチAPI** が有効なことがあります。多数のリクエストをまとめて非同期で処理する代わりに、**割引価格**が適用される事業者が多いです。

- 向くもの: オフラインのバッチ生成・分類・要約・Eval実行。
- 向かないもの: 対話・低レイテンシが要る本番経路。

リアルタイム性と価格のトレードオフです。締切が緩いものはバッチへ寄せるだけで、まとまった節約になります。

## 不要な再生成の削減

意外に大きいのが「同じものを何度も生成する」無駄です。

- **結果キャッシュ** — 同一(または正規化して同一)の入力には、生成済みの結果を再利用する。FAQ的な問い合わせで特に効く。
- **冪等な設計** — リトライ時に丸ごと作り直すのでなく、失敗箇所だけ再生成する。
- **早期確定** — ストリーミングで十分な出力が得られたら打ち切る。
- **リトライの上限** — エージェントのループに **明示的な停止条件**(ステップ上限・予算上限)を必ず置く。停止条件の欠如はコスト暴走の最大要因。

```python
def run_with_budget(agent, goal, max_steps=20, max_tokens=200_000):
    used = 0
    for step in range(max_steps):
        resp = agent.step(goal)
        used += resp.token_usage
        if resp.done:
            return resp.result
        if used > max_tokens:                 # 予算ゲート
            return abort("予算超過で打ち切り")
    return abort("ステップ上限で打ち切り")     # 暴走防止
```

## レイテンシ最適化

コストと密接に絡むのがレイテンシです。速くする工夫の多くはコストにも効きます。

- **ストリーミング** — 体感レイテンシ(最初の文字までの時間)を下げる。総トークンは変わらないが、早期打ち切りが可能になる。
- **出力を短くする** — 出力トークンが減ればコストもレイテンシも下がる。フォーマットや最大長を指定する。
- **軽量モデル** — レイテンシが効く経路(対話・補完)では能力を多少譲って速いモデルを。
- **並列化** — 独立したサブタスクは並行実行する([マルチエージェント](/agents/multi-agent)のファンアウト)。ただし同時実行はコストの瞬間最大値を上げる点に注意。
- **投機的な先読み** — 高確率の次アクションを先回りで準備する(外れたら捨てる)。当たればレイテンシ短縮、外せばコスト増のトレードオフ。

## コスト見積もりの考え方

新機能を出す前に、桁感を見積もる習慣をつけます。完璧な精度は不要で、**桁が合っているか**を見ます。

```text
月間コスト ≈ リクエスト数/月
            × 1リクエストあたり平均ステップ数(エージェントなら)
            × (平均入力トークン × 入力単価 + 平均出力トークン × 出力単価)

調整係数:
  × (1 − キャッシュヒット率 × キャッシュ割引)
  × ルーティングによる平均単価の低下
```

この式から、効くレバーが見えます。**ステップ数(ループ)・入力トークン(文脈)・モデル単価(ルーティング)・キャッシュヒット率** の4つです。どこがボトルネックかは実測のトークン使用量を計測して特定します。

::: warning 計測なき最適化は思い込み
「ここが重いはず」という勘は外れがちです。まず **リクエスト別・経路別のトークン使用量をログに取り**、コストの大半を生んでいる少数の経路を特定してから最適化します。パレートの法則どおり、たいてい一部の経路が大半のコストを占めます。
:::

## 最適化の優先順位

労力対効果で並べると、おおむね次の順で着手するのが効率的です。

1. **停止条件・予算ゲートを入れる** — 暴走を止める。最優先かつ低コスト。
2. **コンテキストを削る** — 肥大化した文脈・履歴を選別/要約する。
3. **プロンプトキャッシュを効かせる** — 固定プレフィックスを前に。
4. **モデルをルーティングする** — 定型を軽量モデルへ。
5. **バッチ・並列を使い分ける** — 締切とレイテンシ要件で振り分け。
6. **再生成を削る** — 結果キャッシュ・冪等化。

## まとめ

- コストは **入力×出力トークン × ループ回数** で累積する。出力トークンは一般に割高。
- 最大の無駄は **コンテキスト肥大化** と **停止条件の欠如**。まずここを塞ぐ。
- **プロンプトキャッシュ**(固定を前に)、**ルーティング**(定型を軽量に)、**バッチ**(締切が緩いものを)で構造的に下げる。
- **計測してから最適化する**。コストの大半は少数の経路に集中している。単価は揮発するので一次情報で見積もる。
