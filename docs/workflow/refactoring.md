# リファクタリングと保守

開発時間の大半は新規開発ではなく**保守**に費やされます([AI駆動開発とは](/guide/what-is-aidd))。そして保守こそ、AI駆動開発が地味に、しかし確実に効く領域です。レガシーコードの読解、機械的な依存更新、退屈な移行作業——人間が嫌い、ミスしやすいタスクを、AIは厭わず大量にこなします。

ただし保守は**既存の動作を壊さない**ことが至上命題です。新規開発と違い、「動いているもの」を相手にするため、安全性の担保が何より重要になります。

::: tip 保守フェーズの鉄則
**変更する前に、現在の振る舞いをテストで固定する。** これがリファクタリングと保守のすべての出発点です。テストという安全網なしにAIへ変更を委ねるのは、命綱なしの綱渡りです。
:::

## 安全なリファクタリング: テストで囲ってから

リファクタリングの定義は「**外部から見た振る舞いを変えずに内部構造を改善する**」ことです。これを安全に行う唯一の方法は、変更前後で振る舞いが同じだと**証明できる**状態を作ることです。

手順は常にこの順序です。

```text
# Step 1: 現状の振る舞いをテストで固定(characterization test)
このレガシー関数 calculatePrice の現在の振る舞いを、
リファクタリング前に固定するための特性テストを書いてください。

- 「正しい仕様」ではなく「今そうなっている振る舞い」を記述する
- 代表的な入力、境界値、過去にバグが出たであろう入力を網羅
- このテストは現状のコードでパスすること
```

```text
# Step 2: テストが通る状態を確認してから、リファクタリング
先ほどの特性テストをすべてパスし続けることを絶対条件に、
calculatePrice を以下の方針でリファクタリングしてください。

- ネストした条件分岐をガード節に
- 公開シグネチャは変更しない
- 1コミット1目的の小さな差分に分けて提案
- テストを書き換えることは禁止
```

::: warning テストを書き換えさせない
リファクタリング中にテストが落ちたとき、AIに「テストを直す」を許すと、振る舞いの変化を見逃します。**テストは振る舞いの基準点**であり、リファクタリング中は不変です。落ちたら実装側を直します。
:::

## レガシーコードの理解

AIは**未知のコードを読み解く相棒**として優秀です。ドキュメントのない古いコードの仕様を復元させます。

```text
このモジュール(添付)が何をしているか解説してください。

- 全体の責務を1段落で
- 主要な関数ごとに「入力・出力・副作用」を表に
- 暗黙の前提・危うい箇所(グローバル状態、未処理の例外)を指摘
- 「この入力でどう動くか不明」な箇所があれば正直に挙げる

推測した部分は「推測」と明示してください。
```

::: warning 理解の"答え合わせ"を忘れない
AIの説明はもっともらしくても、**コードの実際の振る舞いと一致する保証はありません**。重要な部分は、AIの説明を仮説として扱い、特性テストや実行で答え合わせをします。
:::

## 依存更新・移行作業

機械的で退屈、かつ件数が多い——AIが最も得意とする保守タスクです。

```text
このコードを、deprecated になった oldLib の API から
newLib v3 へ移行してください。

- 移行マッピング(対応表)を添付(下記)
- 対応表にない API が使われていたら、推測せず「要確認」とマークする
- 1ファイルずつ diff で出力
- 各ファイル移行後、関連テストが通ることを前提に
```

::: tip 移行は「対応表」を渡して機械化する
ライブラリ移行やAPI変更は、**変換ルール(対応表)を明示**するとAIが安定して処理できます。ルールにない事例は推測させず「要確認」で人間に回す——この境界設定が事故を防ぎます。
:::

依存更新そのものは、CIと組み合わせれば高い自動化レベル(L4〜L5)に乗せやすいタスクです。テストが緑であることを合否基準にできるためです。

## 大規模変更の分割

「全モジュールのエラーハンドリングを統一する」のような横断的変更は、一度に行うと破綻します。**変更を独立した小単位に分割**します。

```text
このコードベース全体で、例外を投げる方式から Result 型を返す方式へ
移行する計画を立ててください(まだコードは変更しない)。

- 依存関係の少ない葉(leaf)モジュールから着手する順序を提示
- 各ステップが単独でマージ可能(他を壊さない)になるよう分割
- 各ステップのリスクと検証方法を添える
- 後方互換を保つための中間状態(両方式の併存)を考慮
```

::: tip 大規模変更は「マージ可能な小ステップ」に刻む
1つの巨大PRではなく、**それぞれ独立してレビュー・マージできる小さなステップ**の連なりにします。各ステップでテストが通り、いつ中断しても壊れない状態を保ちます。これは[実装フェーズ](/workflow/implementation)の差分主義と同じ思想です。
:::

## 得意なタスクと危険なタスクの線引き

保守タスクは、AIへの委ね方を見極めることが特に重要です。

| AIが得意(大胆に委任可) | AIが危険(慎重に) |
| --- | --- |
| 機械的なAPI移行(対応表あり) | 並行処理・ロック周りの変更 |
| 命名・整形・lint修正 | 暗黙の不変条件に依存する箇所 |
| 定型的な依存更新 | パフォーマンス critical なホットパス |
| ボイラープレートの整理 | 後方互換が必須の公開API |
| ドキュメント・コメント生成 | テストで囲えないレガシーの中核 |
| 特性テストの初稿作成 | セキュリティ・課金・認証の中枢 |

判断軸は一貫して**リスク × 可逆性 × 検証コスト**([ワークフロー全体像](/workflow/overview))です。

::: warning 「テストで囲えるか」が線引きの本質
危険なタスクに共通するのは、**変更の正しさをテストで機械的に確認しにくい**点です。並行処理のバグ、性能劣化、暗黙の不変条件の破壊は、テストをすり抜けます。こうした領域はAIに初稿を出させても、最終判断と検証は人間が深く関与します。
:::

## アンチパターン

::: warning リファクタリングと保守の失敗例
- **テストなしリファクタ**: 安全網なしでAIに構造を変えさせ、密かに振る舞いが変わる。
- **テストの書き換え許可**: リファクタ中にテストを直させ、退行を見逃す。
- **巨大PR**: 横断的変更を一括で行い、レビュー不能・切り戻し不能にする。
- **理解の鵜呑み**: AIのコード解説を検証せず正しいと信じる。
- **危険タスクの安易な委任**: 並行処理や課金の中枢を、テストで囲わず委ねる。
:::

## まとめ

- 保守の鉄則は「**変更前に特性テストで振る舞いを固定**」。
- リファクタ中、テストは不変の基準点。落ちたら実装を直す。
- レガシー理解・依存更新・移行はAIの得意領域。ただし答え合わせを怠らない。
- 大規模変更は**独立してマージ可能な小ステップ**に分割する。
- 線引きの本質は「**テストで囲えるか**」。囲えない領域は人間が深く関与する。
