# ガバナンスと倫理

AI駆動開発を組織で運用する以上、「何を入れてよく、何を入れてはいけないか」「生成物の責任は誰が負うか」を曖昧にしたままにはできません。ガバナンスが欠けた導入は、ある日の情報漏洩やライセンス事故をきっかけに、**「AI全面禁止」への揺り戻し** を招きます。

このページは、現場を萎縮させずに組織を守る、最小限で実効的なガバナンスの組み立て方を扱います。セキュリティの技術的詳細は[セキュリティ](/quality/security)に譲り、ここでは **ポリシー・責任・監査** の枠組みを中心に整理します。

::: tip ガバナンスの原則
良いガバナンスは「禁止リストを長くすること」ではありません。**「迷ったときに各自が正しく判断できる原則」** を与えることです。すべてを規定し尽くすことはできないので、判断基準を共有します。
:::

## 何を入れてよく、何を入れてはいけないか

最も基本的かつ事故が多いのが、AIサービスへの **入力(プロンプト)に何を含めてよいか** です。データを分類し、扱いを明文化します。

| データ区分 | 例 | 外部AIサービスへの投入 |
| --- | --- | --- |
| 公開情報 | OSSコード、公開ドキュメント | 原則OK |
| 社内一般 | 一般的な社内コード、設計メモ | 契約・設定を確認のうえ条件付きOK |
| 機密 | 認証情報、APIキー、顧客データ、個人情報 | **禁止** |
| 規制対象 | 医療・金融・個人データ等、法令が定めるもの | **禁止 / 専用環境のみ** |

::: warning 契約条件を必ず確認する
「外部に投入してよいか」は、利用するサービスの **契約形態** に決定的に依存します。一般に、データを学習に利用しない・保持しない設定や、エンタープライズ契約・ゼロデータ保持(zero data retention)オプションの有無で扱いが大きく変わります。**無料/個人プランと法人契約では前提が異なる** ため、調達・法務と連携して契約条件を確認し、その結論をポリシーに反映してください。本ハンドブックは特定サービスの契約内容を保証しません。
:::

## データ取扱いとコンプライアンス

- **データレジデンシー** — データがどの地理的範囲で処理・保存されるか。GDPRや各国の個人情報保護法、業界規制(医療・金融など)の要件に合致するかを確認する。
- **個人情報・顧客データ** — マスキング・匿名化なしに投入しない。ログやスタックトレースに個人情報が紛れ込みやすい点に注意。
- **保持期間とログ** — サービス側がプロンプトをどれだけ保持するか。自組織の監査ログとの整合を取る。
- **再委託先** — AIサービスが内部で利用する基盤(クラウド等)も含めて評価する。

規制の詳細は管轄・業界により異なります。**法務・コンプライアンス部門を早期に巻き込む** ことが、後戻りを防ぐ最大の近道です。

## 知的財産とライセンス

生成コードには固有のIPリスクがあります。

- **学習データ由来のライセンス汚染** — 生成物が特定ライセンス(例: コピーレフト)のコードに酷似する可能性は完全には排除できません。コードクローン検出やライセンススキャナをCIに組み込み、丸ごとのコピーを検知する。
- **成果物の権利帰属** — 生成コードの権利や利用条件はサービスの契約・利用規約に従う。商用利用や再配布の可否を契約で確認する。
- **第三者の機密** — 他社から預かったコードや、NDA配下の情報を外部AIに投入しない。
- **依存ライセンスの確認** — AIが提案したライブラリのライセンスを鵜呑みにしない。AIはライセンス情報を誤ることがある。

::: tip 「AIが書いたから権利は曖昧」ではない
生成物を取り込んだ瞬間、それは **あなたのコードベースの一部** であり、ライセンス適合性とIPの責任はあなたの組織が負います。「AI製だから免責される」という考えは通用しません。
:::

## 責任の所在

AIは責任主体になれません。生成物に起因する障害・脆弱性・ライセンス違反の責任は、**それを承認しマージした人間** にあります。これを制度として明確にします。

- **承認者責任の原則** — AI生成コードであっても、レビューしマージした人間が品質と適合性の責任を負う。「AIが書いた」は免責事由にならない。
- **トレーサビリティ** — どのコードがAI生成に由来するかを必要に応じて識別できるようにする(コミットメッセージの規約、PRラベル等)。インシデント調査と監査で効く。
- **エスカレーション経路** — 判断に迷う事案(機密の取扱い、ライセンスの疑義)の相談先を明示する。

## 監査ログ

ガバナンスは「規定する」だけでなく「事後に検証できる」ことで初めて機能します。

- **アクセスと利用の記録** — 誰が・いつ・どのAIサービスを業務利用したかを、組織が把握できる契約・設定にする(法人契約の監査機能を活用)。
- **生成由来の識別** — 前述のトレーサビリティと連動し、問題発生時に影響範囲を追える状態を保つ。
- **ポリシー違反の検知** — 機密投入を防ぐDLP(データ漏洩防止)やシークレットスキャンを併用する。技術的な実装は[セキュリティ](/quality/security)を参照。
- **定期レビュー** — ログを定期的に棚卸しし、ポリシーの実効性とリスクの変化を確認する。

## 許可リストと禁止事項

判断を速くするため、明示的な許可と禁止を用意します。

**許可リスト(例)**

- 承認済みAIサービス(法人契約・データ保護設定済みのもの)の業務利用。
- 公開情報・社内一般コードを対象とした、設計相談・実装補助・テスト生成・レビュー補助。
- 生成コードを **人間のレビューとテストを経て** 取り込むこと。

**禁止事項(例)**

- 認証情報・APIキー・顧客データ・個人情報を外部AIに投入すること。
- 未承認のAIサービス(個人アカウント等)を業務データで利用すること。
- 生成コードを **レビューなしで** 本番へマージすること。
- ライセンス・出典が不明な大量コードを検証せず取り込むこと。
- AIの出力を、事実確認なしに対外的な保証・回答として用いること。

## ポリシーテンプレート(雛形)

自組織の利用ポリシー文書に、最低限次の項目を含めることを推奨します。

- **目的とスコープ** — 何のためのポリシーか、対象者・対象システム。
- **承認済みツール** — 業務で使ってよいAIサービスの一覧と、その前提条件(契約・設定)。
- **データ分類と投入可否** — 公開/社内一般/機密/規制対象の定義と、各区分の外部投入ルール。
- **禁止事項** — 明示的にやってはいけないことの列挙。
- **責任と承認** — 承認者責任の原則、トレーサビリティの要件。
- **レビュー要件** — AI生成コードに対する必須レビュー・テスト基準([レビュー観点](/quality/security)と連動)。
- **IP・ライセンス** — ライセンススキャンの義務、権利帰属の扱い。
- **監査とログ** — 何を記録し、誰がどの頻度でレビューするか。
- **インシデント対応** — 機密投入や漏洩が疑われた場合の連絡先と初動手順。
- **相談窓口とエスカレーション** — 判断に迷ったときの問い合わせ先。
- **改訂プロセス** — 技術・契約・規制の変化に応じてポリシーを更新する責任者と頻度。

::: warning 「重すぎるポリシー」もアンチパターン
ガバナンスが過剰だと、現場は守らずに迂回します。守れないルールは、ないより危険です。**現場が現実的に守れる粒度** に保ち、技術的ガードレール(DLP・シークレットスキャン)で人間の判断ミスを補完するのが実効的です。
:::

## まとめ

- ガバナンスの本質は禁止の列挙ではなく、**迷ったときの判断原則** を与えること。
- データを分類し、外部AIへの投入可否を契約条件に基づいて明文化する。
- IP・ライセンスの責任、生成物の権利帰属、第三者機密の扱いを規定する。
- 責任は **承認しマージした人間** にある。AIは免責にならない。
- 監査ログとトレーサビリティで事後検証可能にし、技術的ガードレールで人間の判断を補完する。
- 重すぎるポリシーは迂回される。現場が守れる粒度に保つ。
