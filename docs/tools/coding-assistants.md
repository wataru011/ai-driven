# コーディングアシスタント

ここで言う「コーディングアシスタント」とは、IDE/エディタに統合され、**インライン補完** と **チャット** を主機能とするツールを指します。[ツールエコシステムの全体像](/tools/landscape)で言う(a)補完型と(b)チャット型の層です。最も導入障壁が低く、多くのエンジニアが最初に触れる層でもあります。

このカテゴリのツールは表面的には似ていますが、**設計思想** が異なります。「既存IDEへのプラグイン」なのか「IDEごと作り替える」のか、「補完の速度」を重視するのか「コードベース理解」を重視するのか——この違いを理解すると、使いどころが見えてきます。

::: tip このページの要点
補完とチャットは別の道具です。補完は「思考を止めずに書く」ため、チャットは「立ち止まって相談する」ためのもの。両者を混同すると、どちらも中途半端になります。
:::

## 主要ツールの思想の違い

| ツール | 形態 | 思想・特徴 | 強みが出る場面 |
| --- | --- | --- | --- |
| GitHub Copilot | VS Code等のプラグイン | 補完の元祖。広いエコシステム統合。チャット/PRレビューにも拡張 | 既存IDEを変えずに補完を足したい |
| Cursor | VS Codeフォークの独立IDE | エディタごと作り替え。補完・チャット・エージェント編集を深く統合 | AIをワークフローの中心に据えたい |
| Windsurf | 独立IDE | エージェント的編集を前面に。フロー重視のUX | エージェント編集をIDE内で多用 |
| Codeium | プラグイン | 補完を軸に幅広いIDE/言語をカバー | 多様な環境で補完を統一したい |
| JetBrains AI Assistant | JetBrains IDE統合 | IntelliJ/PyCharm等にネイティブ統合。IDEの解析情報を活用 | JetBrains IDEを使い続けたい |
| Tabnine | プラグイン | プライバシー/自己ホスト志向を打ち出す | データ取り扱いの制約が厳しい組織 |

::: warning 機能は急速に変化する
各ツールは補完・チャット・エージェント機能を相互に取り込み続けています。執筆時点(2026年)の機能差は、数か月で縮まることがあります。この表は **設計思想の傾向** として読み、具体的な機能・モデル・上限は必ず各製品の最新情報で確認してください。
:::

## インライン補完を効果的に使う

補完は「黙って良い続きを出してくれる」道具ですが、出力の質は **書き手のコンテキスト整備** に強く依存します。受理を急ぐより、AIが正しく推論できる状況を作ることが効きます。

- **シグナルを先に置く** — 関数名・型・docstring・コメントを先に書くと、補完の精度が跳ね上がります。空のファイルにいきなり期待するより、意図を1〜2行で示す。
- **近傍を整える** — 多くの補完は開いているファイルと近傍を主な文脈にします。関連する型定義やインポートを同じファイル/近くに置くと当たりやすい。
- **小さく受けて直す** — 長い提案を丸呑みせず、数行ずつ受理して検証する。受理は「採用」ではなく「下書きの取り込み」と捉える。
- **命名で誘導する** — `parseIso8601Date` のように具体的な名前を付けると、汎用的な名前より意図に沿った実装が出やすい。

::: tip 補完が向く作業・向かない作業
向く: 定型のループ・エラーハンドリング、テストの雛形、既存パターンの反復、データ変換。
向かない: アーキテクチャ判断、横断的な整合性が要る変更、正しさの検証が難しいロジック。後者はチャットやエージェント、あるいは人間の手に委ねる。
:::

## チャットを使い分ける

チャット型は「立ち止まって相談する」道具です。補完と違い、**何を渡すか(コンテキスト)を人間が決める** 点が肝心です。漫然と質問するのではなく、目的別に使い分けます。

| 目的 | 渡すべきコンテキスト | コツ |
| --- | --- | --- |
| 既存コードの説明 | 対象の関数/ファイル | 「何をしているか」より「なぜこうなっているか」を聞く |
| 局所リファクタ | 対象範囲+関連型 | 期待する形(不変条件)を明示する |
| エラー解決 | エラー全文+該当コード+再現手順 | スタックトレースを省略しない |
| 設計相談 | 制約・非機能要件・既存方針 | 選択肢を複数出させ、トレードオフを比較させる |

選択範囲を指定して指示できるツール(Cursor, Copilot Chat, Cody, JetBrains AI)では、**範囲を絞るほど精度が上がります**。ファイル全体を渡すより、対象の関数とその依存だけを渡すほうが、的を射た差分が返ります。

## 落とし穴

#### 過信(出力をレビューせず採用する)

補完もチャットも、**もっともらしいコード** を出すのであって、正しさは保証しません。特に補完は受理コストが低いぶん、検証せずに積み上がりやすい。「コンパイルが通る=正しい」ではありません。ハルシネーションの構造的な対策は別途扱いますが、原則は **生成物は常にレビューとテストの対象** です。

#### コンテキスト不足(AIが知らない前提で推論する)

補完が見ているのは基本的に近傍だけ、チャットが見ているのは渡した範囲だけです。リポジトリ全体の規約・既存ユーティリティ・ドメイン知識は、明示しなければ伝わりません。結果として「車輪の再発明」や「規約違反のコード」が出ます。リポジトリ規模の文脈が要るなら、補完層ではなく検索型やエージェント型([エージェント型ツール](/tools/agentic-tools))に切り替えるべきサインです。

#### モデルの素性を意識しない

多くのアシスタントは複数のモデルを選択できます。補完のような低レイテンシ重視の場面と、設計相談のような推論力重視の場面では、適切なモデルが異なります。Claudeを使う場合、軽量な補完にはHaiku系、込み入った相談にはSonnet系・Opus系といった使い分けが効きます(執筆時点の最新はOpus 4.8 / Sonnet 4.6 / Haiku 4.5)。

#### スタイルの平均化

補完はコーパスの平均的な書き方に引っ張られます。チームの規約やドメイン固有の流儀から外れた「無難だが浮くコード」が混ざりやすい。リンタ・フォーマッタ・コードレビューで矯正する前提を持つこと。

::: warning データの取り扱いを最初に確認する
コードはしばしば機密です。送信先・学習利用の有無・保持期間・自己ホスト可否は、ツールやプランによって大きく異なります。Tabnineのように自己ホスト志向を打ち出す製品もあります。導入前に必ず確認してください。詳細は[ツールの選び方](/tools/choosing)で扱います。
:::

## 比較: 補完 vs チャット vs エージェント

| 観点 | インライン補完 | IDEチャット | エージェント |
| --- | --- | --- | --- |
| 起動コスト | ほぼゼロ | 低 | 中 |
| 扱える粒度 | 行〜数行 | 関数〜ファイル | タスク〜機能 |
| コンテキスト供給 | 自動(近傍) | 人間が選ぶ | AIが探索 |
| マルチステップ実行 | 不可 | 限定的 | 可能 |
| 検証の負担 | 受理ごとに軽く | まとめて | 大きい |

補完とチャットで足りるうちはこの層に留まり、「複数ファイルを横断する」「失敗したら直す」を任せたくなったら、エージェント層へ移ります。
