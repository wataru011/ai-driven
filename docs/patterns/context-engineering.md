# コンテキストエンジニアリング

[プロンプトパターンカタログ](/patterns/prompt-catalog)が「何をどう指示するか」だとすれば、**コンテキストエンジニアリング**は「指示の周りにどんな情報を、どれだけ、どう配置するか」を設計する規律です。

経験を積むと、出力品質を分けるのはプロンプトの言い回しよりも**コンテキストの設計**だと分かってきます。同じ指示でも、適切な参照コードと規約が手元にあるモデルと、何も知らないモデルとでは、別人のように結果が変わります。

::: tip このページの立場
プロンプト = 1回の指示。コンテキスト = モデルが見ている文脈の総体(指示・参照コード・規約・履歴・取得結果)。本ページは後者を**設計対象**として扱います。トークン予算など下地の概念は[コンテキストウィンドウ](/concepts/context-window)を前提とします。
:::

## なぜ「エンジニアリング」なのか

コンテキストウィンドウは有限です。そこには厳しいトレードオフがあります。

- **少なすぎる** → モデルが前提を知らず、推測で埋めて的外れになる。
- **多すぎる** → 重要な情報が雑多な情報に埋もれ、精度が落ちる(lost in the middle)。コストとレイテンシも増える。

つまり「全部入れる」も「最小限にする」も最適ではなく、**限られた予算をどう配分するかという最適化問題**です。だからエンジニアリングなのです。

::: warning 詰め込みは無料ではない
「念のため全部貼る」は最も多い失敗です。関係ないコードは精度を下げ、コストを上げます。コンテキストは資産ではなく**予算**だと考えてください。
:::

## 何を入れ、何を捨てるか

判断軸はシンプルです。**「その情報がなければモデルが間違える」ものだけを入れる。**

| 入れるべき | 捨てるべき |
| --- | --- |
| 変更対象のコードそのもの | 変更に無関係なファイル |
| 直接呼ぶ関数・型の定義 | 間接的にしか関係しない実装の中身 |
| 守るべき規約・制約 | 一般的に知られている言語仕様 |
| 非自明なドメイン知識 | 推測可能な定型コード |
| 期待する入出力の例 | 冗長な重複サンプル |

迷ったら**インターフェースを入れて実装は省く**のが基本です。呼び出し側は型シグネチャを知れば十分で、中身まで要らないことが多いからです。

```text
# よくある過剰コンテキスト
- ファイル全体を10個貼る(うち9個は文脈用)

# 改善
- 変更対象1ファイルは全文
- 依存する2関数は「シグネチャ + 1行説明」のみ
- 規約は参照ファイル1つで代表させる
```

## ルールファイルの設計(CLAUDE.md など)

エージェント型ツールでは、リポジトリ直下のルールファイル(`CLAUDE.md` など)が**毎回自動で読まれる常設コンテキスト**になります。ここはプロジェクトの「憲法」であり、設計のしどころです。

::: tip 設計原則
ルールファイルは**短く・具体的・安定的**に。長文の理想論より、破られると困る**実務ルール**を箇条書きで。頻繁に変わる情報はここに書かない。
:::

良いルールファイルの構成例です。

```markdown
# プロジェクト規約(抜粋)

## 技術スタック
- TypeScript 5.x / Node 20 / Fastify
- テスト: vitest、E2E: Playwright

## コーディング規約
- 例外は投げず Result 型で返す(src/result.ts 参照)
- ログは logger 経由のみ。console.* 禁止
- 公開関数には必ず JSDoc

## ディレクトリ
- ドメインロジックは src/domain/(外部依存禁止)
- I/O は src/infra/ に隔離

## やってはいけないこと
- 新規依存の追加(必要なら相談)
- src/generated/ の手編集
```

::: warning ルールファイルのアンチパターン
- **長すぎる** — 数百行のルールは守られにくく、毎回のトークンも食う。
- **抽象的** — 「きれいなコードを書く」は無意味。検証できる規則を。
- **古い** — 実態と乖離したルールは有害。コードの真実と一致させる。
:::

階層化も有効です。リポジトリ全体の規約はルートに、特定ディレクトリ固有の作法はそのディレクトリのルールファイルに置くと、関連する場面でだけ効かせられます。

## リポジトリ規約の与え方

ルールを**言葉で説明する**より、**実物で示す**ほうが正確でトークン効率も良い、という原則([リファレンス駆動](/patterns/prompt-catalog#_9-リファレンス駆動))はここでも中心です。

| 与え方 | 精度 | トークン | 使いどころ |
| --- | --- | --- | --- |
| 言葉で規約を説明 | 中 | 中 | 抽象的な方針 |
| 代表ファイルを1つ見せる | 高 | 中 | 命名・構造の踏襲 |
| ルールファイルに常設 | 高 | 低(常設) | 普遍的な制約 |
| Lint/型で機械的に強制 | 最高 | ゼロ | 守らせたい規約 |

最良の規約伝達は、実は**プロンプトの外**にあります。Linter・フォーマッタ・型システム・CIで強制できる規約は、そちらに寄せるのが正解です。モデルに毎回お願いするより、機械的に弾くほうが確実だからです。

## コードの選別

変更対象に**関連するコードだけ**を選んで渡す作業は、コンテキストエンジニアリングの中で最も頻度が高く、効果も大きい工程です。

選別の手順例です。

1. **変更対象**を特定し、全文を入れる。
2. その対象が**直接呼ぶ**関数・型を辿り、シグネチャを入れる(実装は原則省略)。
3. **呼び出し元**のうち、影響を受けるものを入れる。
4. **規約の見本**を1つ入れる。
5. 残りは**入れない**。

::: tip 「広く浅く」より「狭く深く」
関連ファイルを10個浅く入れるより、本当に効く3個を全文で入れるほうが、判断の質は上がります。
:::

## 要約・圧縮でコンテキストを作る

長い情報をそのまま渡せないとき、[要約パターン](/patterns/prompt-catalog#_18-要約)で**圧縮したコンテキストを生成**します。これはコンテキストエンジニアリングの能動的な技法です。

代表的な圧縮対象です。

- **長い会話履歴** → 「ここまでの決定事項と未解決点」に要約して引き継ぐ。
- **巨大ログ** → 異常箇所だけ抽出した要約に。
- **大きなコードベース** → ファイルごとの責務を1行で並べた「地図」に。
- **過去のPR/設計** → 結論と理由だけを残す。

```text
# 会話が長くなったときの引き継ぎ要約テンプレート
これまでの作業を次の形式で要約してください。次のセッションへの
引き継ぎに使います。
- 確定した決定事項(箇条書き)
- 現在の実装状態(どこまで終わったか)
- 未解決の論点と保留理由
- 次にやるべきこと
数値・ファイルパス・固有名は省略しないこと。
```

::: warning 要約は劣化を伴う
圧縮は情報を捨てる行為です。「復元できない情報(数値・固有名・結論)は落とさない」と明示し、要約後に重要事項が消えていないか確認してください。
:::

## トークン予算の配分

コンテキストウィンドウを**予算**と捉え、用途ごとに配分を意識します。配分の目安です。

| 区分 | 内容 | 配分の考え方 |
| --- | --- | --- |
| 常設 | ルールファイル・システム指示 | 小さく固定。肥大化させない |
| タスク指示 | 今回の目的・制約・出力形式 | 明確に。冗長を避ける |
| 参照コード | 変更対象・依存・見本 | ここが主役。最も多く割く |
| 取得結果(RAG) | 検索で引いた断片 | 関連度の高いものだけ |
| 履歴 | これまでのやり取り | 長くなったら要約に置換 |
| 出力余地 | モデルが書く分 | 大きな生成には十分残す |

::: tip 出力分を忘れない
入力で予算を使い切ると、長い出力が途中で切れます。**生成に必要な余白**を残して入力を組むのも設計のうちです。
:::

## 長期メモリ

セッションを越えて保持したい情報(プロジェクトの決定事項、繰り返す指示)は、毎回プロンプトに書くのではなく**永続化された文脈**にします。

- **ルールファイル** — 普遍的な規約・制約の置き場。
- **ドキュメント/ADR** — 設計判断の経緯。必要時に参照させる。
- **要約された決定ログ** — 「なぜこうなっているか」を残す。

長期メモリの要点は、**全部を常時ロードしない**ことです。普遍的なものだけ常設し、状況依存のものは必要なときだけ引き込みます。これが次のRAGの話につながります。

## 検索(RAG)との接続

コードベースやドキュメントが大きく、何を渡すべきか事前に決められないとき、**検索で動的にコンテキストを組み立てる**のがRAG(Retrieval-Augmented Generation)です。仕組みの詳細は[埋め込みとRAG](/concepts/rag)を参照してください。

コンテキストエンジニアリングの視点でのRAGは、「選別を**自動化**する仕組み」です。

```text
[ユーザーの質問/タスク]
        │
        ▼
  関連断片を検索(意味的検索)
        │
        ▼
  上位N件を選別・要約
        │
        ▼
  タスク指示と合成してモデルへ
```

::: warning RAGも万能ではない
検索が的外れな断片を引けば、それはノイズとしてコンテキストを汚します。手で選んだほうが速く正確な場面も多く、**RAGは選別を自動化する道具**であって、選別の質を保証するものではありません。取得件数・関連度しきい値・要約の有無は、すべてチューニング対象です。
:::

手動選別とRAGの使い分けの目安です。

| 状況 | 推奨 |
| --- | --- |
| 対象が分かっている小〜中規模 | 手で選別 |
| 対象が不明な大規模コードベース | RAGで候補抽出 → 人/モデルで絞り込み |
| ドキュメント横断のQ&A | RAG |
| 厳密な正しさが要る変更 | 手で選別(RAGは補助) |

## 実例: 同じタスク、3つのコンテキスト設計

「`OrderService` に在庫チェックを追加する」という同一タスクで、コンテキスト設計だけを変えた比較です。

::: details パターンA(貧弱) — 指示だけ
```text
OrderService に在庫チェックを追加して。
```
結果: 既存の在庫管理の仕組み・規約を知らないため、独自の実装を勝手に作る。命名も構造も既存と合わず、ほぼ作り直しになる。
:::

::: details パターンB(過剰) — 全部貼る
```text
（service層10ファイル、model層8ファイル、テスト全部を貼る）
OrderService に在庫チェックを追加して。
```
結果: 関連の薄いコードがノイズになり、肝心の InventoryRepository を見落とす。コストとレイテンシも大きい。
:::

::: details パターンC(設計済み) — 選別 + 規約 + 見本
```text
あなたはこのリポジトリの規約に従うバックエンドエンジニアです。

## 目的
OrderService.place() に、確定前の在庫チェックを追加する。

## 関連コード(変更対象)
（OrderService.place() の全文）

## 直接使う依存(シグネチャのみ)
- InventoryRepository.getAvailable(sku): number
- DomainError(code, message)

## 規約(見本)
- エラーは例外でなく Result で返す（下記 PaymentService が見本）
（PaymentService の該当箇所だけ）

## 出力
変更概要 → 修正コード → 在庫不足ケースのテスト
```
結果: 既存規約に沿い、正しい依存を使い、検証可能なテスト付きで返る。トークンも過不足ない。
:::

この3つの違いは**プロンプトの文言ではなくコンテキストの設計**です。これがコンテキストエンジニアリングの核心です。

## まとめ

- 出力品質を分けるのは、しばしばプロンプトの言い回しより**コンテキストの設計**。
- コンテキストは資産でなく**予算**。「なければ間違える情報」だけを入れる。
- 規約は言葉より**実物**で、可能なら**機械的強制**(Lint/型)で。
- 長い情報は**要約して圧縮したコンテキスト**を作る。
- 大規模で選別できないときは**RAGで自動化**。ただし選別の質は保証されない。
