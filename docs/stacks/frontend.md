# フロントエンド開発でのAI活用

フロントエンドは、AIコード生成の効果が**最も体感しやすい**領域です。コンポーネントの骨格、フォームのバリデーション、テーブルのページネーション、ダイアログのフォーカストラップ——どれも「典型はあるが手で書くと退屈」というボイラープレートの宝庫で、まさにLLMが得意とするゾーンです。

しかし同時に、フロントエンドは**AIの弱点が表面化しやすい**領域でもあります。UIライブラリのAPIは破壊的変更が多く、モデルの学習データには古いバージョンが大量に混ざっています。「動くが古い」「動くが過剰」なコードを平然と出してくるため、レビューの目利きが品質を決めます。

::: tip この章の前提
プロンプトの基本作法は [プロンプトの基礎](/prompting/fundamentals) を、再利用できる雛形は [プロンプトパターンカタログ](/patterns/prompt-catalog) を前提とします。本章はフロントエンド固有の論点に集中します。
:::

## AI活用の勘所

### 1. 「画面の見た目」を入力に使う

フロントエンドの強みは、**成果物が視覚的**であることです。これはAIへの入力としても、検証としても使えます。

- **スクリーンショット → コード**: デザインカンプやFigmaの書き出し画像を渡し、マークアップとスタイルの初稿を生成させる。
- **既存画面のスクショ → リファクタ案**: 「このコンポーネントのレイアウトをCSS Gridで組み直して」と画像つきで依頼する。
- **エラー画面のスクショ → 原因推定**: 崩れたレイアウトの画像を渡し、CSSの当たり判定を推測させる。

ただし画像からの生成は**ピクセルパーフェクトにはならない**前提で使います。余白・フォントサイズ・色は必ずデザイントークン(変数)に置き換えさせること。直値が散らばると保守不能になります。

### 2. デザインシステムを「文脈」として固定する

素のLLMは、汎用的だが**あなたのプロジェクトとは無関係**なコンポーネントを書きます。既存のデザインシステムやコンポーネントライブラリを文脈として与えるかどうかで、出力の使い物になり方が一変します。

::: details 文脈として渡すべきもの(チェックリスト)
- 使用中のUIライブラリとバージョン(例: React 19, MUI 6, Tailwind v4)
- 既存の共通コンポーネント(`<Button>`, `<Card>`, `<Stack>` など)のpropsシグネチャ
- デザイントークン(色・余白・タイポグラフィのスケール)
- 状態管理の方針(後述)
- コーディング規約(命名、ファイル分割、import順)
:::

これらは毎回貼り付けるのではなく、リポジトリのルール定義ファイル(`CLAUDE.md`、エージェントツールのプロジェクト設定など)に一度書いておくのが定石です。詳しくは [コンテキスト設計](/prompting/context-design) を参照してください。

### 3. 状態管理は「方針」を先に決めてから依頼する

状態管理はフロントエンド設計の中核で、ここをAIに丸投げすると一貫性が崩壊します。**どこに状態を置くか**は人間が決め、AIには実装を任せます。

| 状態の種類 | 置き場所の例 | AIへの依頼の仕方 |
| --- | --- | --- |
| ローカルUI状態 | `useState` / `ref` | 「このコンポーネント内に閉じて」 |
| サーバー状態 | TanStack Query / SWR | 「キャッシュキーとinvalidate方針を明示して」 |
| 共有クライアント状態 | Zustand / Jotai / Pinia | 「ストアの形(state/actions)を先に提示する」 |
| フォーム状態 | React Hook Form / VeeValidate | 「スキーマ(zod等)から型を導出して」 |

::: warning 「とりあえずグローバルストア」を疑う
AIは迷うと安易にグローバルストアへ状態を逃がしがちです。本来ローカルで足りる状態がストアに漏れると、再レンダリング範囲が広がりバグの温床になります。「これは本当に共有が必要か」を必ずレビューしてください。
:::

### 4. アクセシビリティは「明示的に要求」しないと省かれる

LLMはアクセシビリティ(a11y)を**黙っていると省略**します。`<div onClick>` でボタンを作る、ラベルなしのアイコンボタンを置く、フォーカス管理を忘れる——これらは典型的な生成物の欠陥です。

プロンプトに以下を毎回含めるのが有効です。

> セマンティックなHTML要素を使い、対話要素には適切なrole/aria属性・キーボード操作・フォーカス可視化を付与してください。アイコンのみのボタンには `aria-label` を付けてください。WCAG 2.2 AA を満たすことを前提にしてください。

生成後は `eslint-plugin-jsx-a11y` や axe での自動チェックを通すこと。人手のレビューだけでは漏れます。

## 有効なプロンプト例

### コンポーネント生成(規約込み)

```text
React 19 + TypeScript + Tailwind v4 のプロジェクトです。
既存の <Button> と <Field> コンポーネント(下にprops定義を添付)を使い、
ユーザー編集用のフォームコンポーネントを作ってください。

要件:
- バリデーションは zod スキーマを単一の真実源とし、型はそこから infer する
- 送信中は二重送信を防ぎ、ボタンを disabled + ローディング表示にする
- エラーは各フィールド直下に aria-describedby で関連付けて表示
- 余白・色は直値を使わず、既存のデザイントークンのみ使用

添付:
<Button> の props: { variant: 'primary'|'ghost'; loading?: boolean; ... }
<Field> の props: { label: string; error?: string; children: ReactNode }
```

### スクリーンショットからの実装

```text
添付した画像のカード一覧UIを、Vue 3 (script setup) + Tailwind で実装してください。
- レスポンシブ: モバイル1列、タブレット2列、デスクトップ3列
- 余白・角丸・影は画像から推測した値ではなく、tailwind.config のスケールに丸めて
- 画像は遅延読み込み(loading="lazy")、alt は props で受け取る
- まずコンポーネントのprops設計だけ提示して。実装はその合意の後で。
```

最後の一文「props設計だけ先に」は、いきなり大量のコードを出させないための**段階分割**です。設計に同意してから実装させると手戻りが激減します。

## 落とし穴と対策

::: warning フロントエンド特有の典型的な失敗
- **古いAPIの提案** — クラスコンポーネント、`componentWillMount`、React Router v5 の書き方、Vue 2 Options API など。学習データの古さが直撃する。
- **過剰なライブラリ提案** — 数行で済む処理に moment.js / lodash / 巨大なUIキットを引き込む。バンドルサイズを無視しがち。
- **`useEffect` の濫用** — 派生状態を `useEffect` + `setState` で計算する反パターン。本来はレンダー中の算出で足りる。
- **key の不適切な指定** — リストの `key` に配列index を使い、並べ替えでバグる。
- **直値スタイルの散乱** — インラインスタイルにマジックナンバー(`marginTop: 13` などの直値)を直書きしがち。
- **依存配列の取りこぼし** — ESLintの `exhaustive-deps` 警告を無視した生成。
:::

対策は機械化が基本です。人間の注意力に頼らず、**ツールで弾く**仕組みを先に整えます。

| 落とし穴 | 機械的に弾く手段 |
| --- | --- |
| 古い/危険なAPI | ESLint(`react-hooks`, フレームワーク公式ルール) |
| a11y欠陥 | `eslint-plugin-jsx-a11y` + axe |
| バンドル肥大 | `size-limit` / bundle analyzer をCIに |
| 型の緩み | `tsc --noEmit` を必須化、`any` を禁止ルール化 |
| デザイン逸脱 | Tailwindの任意値禁止設定 / Stylelint |

## テストと検証

フロントエンドのテストは、AIが**得意な部分と任せてはいけない部分**がはっきり分かれます。

### Testing Library: 「振る舞い」をAIに書かせる

ユーザー視点の振る舞いテストは、AIが雛形を量産しやすい領域です。ただし**何をテストすべきか**は人間が決めます。実装の内部詳細(state名やクラス名)ではなく、ユーザーに見える結果を検証するよう指示します。

```text
このフォームコンポーネントに対する Testing Library のテストを書いてください。
- 実装詳細(state名やクラス名)ではなく、ロール・ラベル・表示テキストで要素を取得
- 「必須項目を空で送信 → エラーメッセージ表示 → 送信されない」を検証
- 非同期は findBy* / waitFor を使い、act 警告が出ないように
- ユーザー操作は userEvent を使う(fireEvent ではなく)
```

::: warning AIが書きがちな「無意味なテスト」
LLMは、テストを依頼すると**カバレッジは稼ぐが意味のない**テストを書くことがあります。「コンポーネントがクラッシュせずレンダーされる」だけのテスト、スナップショットの撮りっぱなし、モックを検証してアプリの挙動を検証していないテストなどです。生成されたテストは「これは壊れたら困る挙動を守っているか?」で取捨選択してください。
:::

### Playwright: E2Eのシナリオ化

E2Eは「重要な導線」に絞るのが鉄則で、シナリオの選定は人間が行います。AIには、合意したシナリオを Playwright のコードへ落とす作業を任せます。ロケータは `getByRole` / `getByLabel` を優先させ、CSSセレクタ依存を避けさせると壊れにくくなります。

### 型による検証

TypeScriptの型は、フロントエンドにおける**最も安いテスト**です。AIにコンポーネントを書かせる際は「propsとAPIレスポンスの型を厳格に。`any` 禁止」を徹底し、`tsc --noEmit` をCIの門番にします。型が通ること自体が、相当数のバグを未然に防ぎます。

テスト戦略全体の考え方は [テストとレビュー](/workflow/testing-review) を参照してください。

## 人間がやること / AIに任せること

| 人間が担う | AIに任せる |
| --- | --- |
| 状態の置き場所の決定 | コンポーネントの初稿生成 |
| デザインシステムとの整合判断 | ボイラープレート/フォーム実装 |
| テストすべき挙動の選定 | テストコードの雛形化 |
| バンドルサイズ・パフォーマンスの責任 | リファクタ案の提示 |
| アクセシビリティの最終確認 | aria属性の付与作業 |
