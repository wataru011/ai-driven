# 従来開発との違い

AI駆動開発を「速くコードを書ける従来開発」と捉えるのは、最も多い誤解です。実際には、**作業の経済構造そのものが組み替わります**。何にコストがかかり、どのスキルが効き、どこで品質を担保するか——その重心が移動するのです。

このページでは、感覚的な「便利になった」を超えて、**5つの軸** で構造的な差分を整理します。最後に「変わること/変わらないこと」を切り分け、過剰な期待と過剰な警戒の両方を避けます。

::: tip 結論を先に
従来開発のコストは主に **「正しく記述すること」** にありました。AI駆動開発では、記述コストが下がる代わりに **「生成物が正しいと確かめること(検証)」** にコストが移動します。生産性が上がるかは、この検証コストを設計で抑えられるかにかかっています。
:::

## 軸1: 成果物の性質 — 決定論的 vs 確率的

最も根本的な違いはここにあります。

| 観点 | 従来開発 | AI駆動開発 |
| --- | --- | --- |
| 生成過程 | 人間が **決定論的** に記述 | LLMが **確率的** に生成 |
| 再現性 | 同じ人が書けばほぼ同じ | 同じプロンプトでも揺れる |
| 誤りの種類 | ロジックの誤り、考慮漏れ | 上記 + **もっともらしい嘘**(ハルシネーション) |
| 信頼の根拠 | 「自分で書いたから分かる」 | 「読んで・テストして確かめたから」 |

従来のコードは、書いた本人の意図と1対1で対応します。バグがあっても、それは**意図の誤り**であり、追跡可能です。

一方、LLMの出力は確率分布からのサンプリングです。**構文的に完璧で、一見正しく、しかし微妙に間違っている**コードが出ることがあります。存在しないAPIを自信満々に呼ぶ、エッジケースを静かに無視する——こうした誤りは「もっともらしさ」に紛れるため、人間のバグより発見しにくい。

::: warning 「動いた」は「正しい」ではない
確率的な成果物では、最初の数ケースが通っても安心できません。生成物の正しさは、**実行とテストによる外部からの検証** に依存します。詳しくは[ハルシネーション対策](/quality/hallucination)を参照してください。
:::

## 軸2: コストの所在 — 記述 vs 検証

開発のコスト構造が移動します。これがAI駆動開発を理解する上で最も重要な視点です。

```text
従来開発:
  [設計] → [████████ 記述 ████████] → [レビュー] → [テスト]
            ↑ ここが重い

AI駆動開発:
  [設計+意図伝達] → [生成] → [██████ 検証・修正 ██████]
   ↑ 軽い            ↑ 一瞬   ↑ ここが重くなる
```

| コスト項目 | 従来開発 | AI駆動開発 |
| --- | --- | --- |
| 記述(タイピング) | 高 | **低**(生成が肩代わり) |
| 仕様・意図の言語化 | 暗黙でも進む | **明示が必須**(プロンプト) |
| 検証・レビュー | 中 | **高**(他人のコードを読む構造) |
| デバッグ | 自分の意図を辿る | 生成物の挙動を外から推定 |

つまり、AI駆動開発は **「書く仕事」を「読む・確かめる仕事」に交換** しています。タイピング量は減りますが、その分だけ **レビューと検証の負荷が増える**。この交換が割に合うのは、検証が記述より安いタスク——定型コード、テスト生成、ボイラープレートなど——です。逆に検証が極端に難しいタスク(微妙な並行処理など)では、交換が割に合わないこともあります。

## 軸3: スキルの重心 — 構文知識 vs 判断力

効くスキルの重心が、下流から上流へ移動します。

| スキル | 従来開発での重み | AI駆動開発での重み |
| --- | --- | --- |
| 構文・API暗記 | 高 | **低下**(生成が補う) |
| 高速なタイピング | 中 | 低 |
| **要求の言語化** | 中 | **上昇** |
| **設計・分割判断** | 高 | **さらに上昇** |
| **コードレビュー力** | 中 | **大きく上昇** |
| **検証可能性の設計** | 中 | **上昇** |

「どう書くか」の知識は価値が下がり、**「何を・なぜ作るか」「これは正しいか」を判断する力** の価値が上がります。経験豊富なエンジニアが有利なのはまさにこの点で、**設計の良し悪しを見抜く目** と **怪しいコードを嗅ぎ分ける勘** は、生成AIでは代替されにくく、むしろレバレッジが効きます。

::: info 初学者と熟練者で効果が逆転しうる
AIは「自分が検証できる範囲」では強力な加速装置ですが、「検証できない範囲」では危険な自動化です。熟練者は生成物の誤りを見抜けるため得をし、初学者は誤りに気づけず負債を蓄積しがち——この非対称性は[導入戦略](/adoption/strategy)で扱います。
:::

## 軸4: レビューの役割 — 最終関門 vs 中核工程

レビューの位置づけが変わります。

| 観点 | 従来開発 | AI駆動開発 |
| --- | --- | --- |
| 位置づけ | 品質の **最終確認** | 品質を作り込む **中核工程** |
| 対象 | 同僚が意図を持って書いたコード | 意図のない確率的生成物 |
| 前提 | 「書き手は理解している」 | 「書き手(AI)は理解していない」 |
| 頻度 | PR単位 | 生成のたび、繰り返し |

従来のレビューは「人間が考えて書いたものを、別の人間が確認する」工程でした。書き手には設計意図があり、レビューはその意図と実装のズレを探します。

AI駆動開発では、**書き手に意図がありません**。LLMは「もっともらしさ」を最適化しているだけで、なぜそう書いたかの一貫した理由を持ちません。したがってレビューは、後付けの確認ではなく **品質を担保する主工程** になります。「自分がこのコードの責任を引き受けられるか」を問う場です。

## 軸5: 生産性曲線 — 直線的 vs 非線形

生産性の伸び方が変わります。

```text
従来開発:        AI駆動開発:
生産性            生産性
 │      ／         │   ／‾‾‾‾  ← タスク次第で頭打ち/逆効果も
 │    ／           │  ／
 │  ／  (習熟で     │ ／ ＼____  ← 検証が破綻すると低下
 │／    緩やか上昇) │／
 └──────→ 経験     └──────→ タスク適合度
```

従来開発の生産性は、経験とともに比較的なだらかに上がります。AI駆動開発の生産性は **タスクへの適合度で大きく振れる** のが特徴です。

- **適合タスク**(定型・検証容易)では、数倍の加速が現実に起きる
- **不適合タスク**(検証困難・暗黙知依存)では、生成物の検証コストがかさみ、**むしろ遅くなる** ことがある

この非線形性ゆえに、「どのタスクをAIに任せるか」の見極めが生産性を決めます。万能の加速ではなく、**当たり外れのある加速** だと理解するのが実務的です。

## 変わること / 変わらないこと

過剰な期待と過剰な警戒を避けるため、切り分けておきます。

### 変わること

- **コストの所在** — 記述から検証へ移動する
- **効くスキル** — 構文知識より判断力・レビュー力
- **レビューの重み** — 最終関門から中核工程へ
- **作業の単位** — 行を書くより、タスクを委任して確かめる
- **試行のコスト** — プロトタイプや代替案の生成が安くなる

### 変わらないこと

- **何を作るべきかの判断** — 要求の解釈とトレードオフは人間の仕事
- **最終的な責任** — 出荷物の品質責任はAIに移譲できない
- **ソフトウェア工学の原則** — 良い設計、テスト、可観測性の価値はむしろ上がる
- **検証の必要性** — 「動くこと」を確かめる工程は省略できない
- **ドメイン理解** — 問題領域を分かっていない人は、正しい指示も検証もできない

::: warning AIは工学を不要にしない
テスト、CI、レビュー、設計といった既存の規律は、AI駆動開発で **不要になるどころか前提条件になります**。確率的な生成物を扱うほど、それを受け止める決定論的な仕組み(自動テスト、型、CI)の価値が高まるからです。基盤の弱いチームほど、AI導入の効果は出にくくなります。
:::

## まとめ

AI駆動開発は「速い従来開発」ではなく、**作業の経済構造が組み替わった別の開発スタイル** です。記述から検証へコストが移り、判断力とレビュー力が効き、タスク適合度で生産性が大きく振れます。
