# コンテキストウィンドウ

LLMには「一度に見渡せる範囲」に上限があります。これが**コンテキストウィンドウ(context window)** です。モデルへの入力(プロンプト・履歴・取得したドキュメント・ツール出力)と、モデルが生成する出力の**両方を合わせたトークンの上限**を指します。

経験者がまず捉えるべきは、コンテキストウィンドウを**有限の「作業机」**としてイメージすることです。机が広いほど多くの資料を広げられますが、広げすぎると目当ての資料が埋もれ、探す時間も机のレンタル代もかさむ——この比喩がそのまま実務の勘所になります。

::: tip ひとことで言うと
コンテキストウィンドウは**RAMでも保存領域でもなく「作業机の広さ」**です。広さには上限があり、机に載せた量に比例して**コスト・レイテンシ・気の散りやすさ**が増えます。だから「全部載せる」ではなく「**必要なものを厳選して載せる**」のが正解です。
:::

## トークン予算という考え方

コンテキストウィンドウを**予算(token budget)** として扱うと、設計がぐっと明瞭になります。1リクエストで使えるトークン総量は決まっており、その中で各要素にどう配分するかを意識的に決めます。

```text
コンテキストウィンドウ全体(例:総トークン上限)
┌────────────────────────────────────────────────────────┐
│ システムプロンプト / 指示         ← 固定費。簡潔に       │
│ ツール定義・スキーマ              ← 必要なものだけ       │
│ 取得した参照情報(RAG結果・コード)← ここが変動費の主役  │
│ 会話履歴                          ← 古いものは要約 or 破棄│
│ 直近のユーザー入力                ← 最優先で残す         │
│ ── 出力用の余白(これも予算内) ──                       │
└────────────────────────────────────────────────────────┘
```

::: warning 出力分も予算に含まれる
よくある事故は「入力でウィンドウをほぼ使い切り、出力が途中で切れる(truncation)」ことです。長い生成を期待するなら、**出力用の余白**を先に確保してから入力を組み立てます。
:::

## コンテキスト長の意味と、実効的な上限

「コンテキスト長が大きい=何でも入れてよい」ではありません。ここに2種類の上限があることを区別します。

| 種類 | 意味 |
| --- | --- |
| **物理上限(公称)** | モデルが受理できる最大トークン数。これを超えるとエラーになる |
| **実効上限** | 精度・コスト・レイテンシが許容範囲に収まる範囲。物理上限よりかなり手前 |

物理上限が大きいモデルでも、**実効上限は往々にしてその一部**です。ウィンドウを限界まで使うと、後述する精度劣化・コスト・レイテンシの問題が顕在化します。「入るから入れる」ではなく「**効くから入れる**」を基準にします。

## lost in the middle 問題

長いコンテキストにおける最重要の経験則が **"lost in the middle"** です。研究で繰り返し観察されている現象で、

> モデルは**入力の冒頭と末尾**に置かれた情報をよく利用するが、**中央付近**に埋もれた情報は見落としやすい。

注意が均等に行き渡るわけではない、ということです。これは[LLMの仕組み](/concepts/how-llms-work)で触れたアテンションの性質とも関係します。実務では次のように使います。

- **最重要の指示・制約・参照は冒頭か末尾**に置く。中央に埋めない。
- 直近のユーザー指示は**末尾**に近いほど効きやすい。
- 大量のドキュメントを並べる場合、**最も関連度の高い断片を端に寄せる**ように並び替える(reranking)。

```text
注意の効きやすさ(概念図)
冒頭  ████████░░░░░░░░████████  末尾
              中央=見落とされやすい
```

## 詰め込みすぎによる劣化(context rot)

ウィンドウに余裕があるからといって関連の薄い情報を大量に詰めると、**かえって精度が落ちます**。直感に反しますが、理由は明快です。

1. **信号対雑音比の低下** — 関連情報(信号)が無関係情報(雑音)に薄まり、モデルが要点を掴みにくくなる。
2. **lost in the middle の悪化** — 量が増えるほど中央が肥大し、見落としが増える。
3. **矛盾の混入** — 古い情報と新しい情報、相反する例が同居すると、モデルが混乱し一貫性を欠く。
4. **指示の希釈** — 長大なコンテキストの中で、肝心の指示が相対的に埋もれる。

::: tip 経験則
「念のため全部入れておく」は多くの場合**逆効果**です。コンテキストは**広いほど良いのではなく、適切に厳選されているほど良い**。少数の高関連断片 > 大量の中関連断片。
:::

## コスト・レイテンシとのトレードオフ

コンテキストは**タダではありません**。投入トークンに対して2つのコストがかかります。

| 軸 | 増えると何が起きるか |
| --- | --- |
| **金銭コスト** | 入力トークンに比例して課金。毎リクエスト全履歴を送ると累積で高額化 |
| **レイテンシ** | 入力が長いほど処理時間が伸びる。アテンションは入力長に対して重い |

特にエージェント(マルチステップ実行)では、各ステップで履歴が雪だるま式に膨らみ、**後半ほど遅く・高く**なりがちです。プロンプトキャッシュ(変化しない先頭部分の再利用)や履歴の要約が効きます。価格や具体的な上限値はモデルと提供時期で変わるため、ここでは断定せず**「トークン量に比例してコストとレイテンシが増える」という構造**だけ押さえてください。

## コンテキスト管理戦略

限られた予算を賢く使うための主要戦略を整理します。実務では複数を組み合わせます。

### 1. 選別(Selection / Retrieval)

全部を入れず、**いま必要な断片だけを取得して入れる**。コードベースなら関連ファイル・関数だけ、ドキュメントなら関連章だけ。意味的検索で関連断片を引く仕組みが[RAG](/concepts/rag)です。最も費用対効果が高い基本戦略です。

### 2. 要約(Summarization / Compaction)

長い会話履歴や大きなドキュメントを、**情報を保ちつつ短く圧縮**する。エージェントでは「ここまでの経緯を要約してコンテキストを作り直す(compaction)」運用が定番です。

```text
[古い履歴 8000トークン]
      ↓ 要約
[要約 600トークン] + [直近の重要なやり取りはそのまま保持]
```

### 3. チャンク(Chunking)

大きなドキュメントやコードを**意味のある単位に分割**し、関連チャンクだけを選んで入れる。分割の粒度設計は検索精度を左右します(詳細は[RAG](/concepts/rag))。

### 4. 構造化と優先順位付け

- **役割を明示**して並べる(指示 / 参照資料 / 履歴 / 入力)。
- **最重要を端(冒頭・末尾)**へ。lost in the middle を避ける。
- **不要になったツール出力・中間ログは捨てる**。机を片付ける。

## 実務での効果的なコンテキスト構成

タスクに依存しますが、汎用的に効く構成のテンプレートを示します。

```text
1. システム指示(役割・制約・出力形式)        — 簡潔・固定
2. 参照資料(RAGで選別した高関連の断片のみ)    — 関連順に、重要なものを端へ
3. Few-shotの例(必要なら1〜数件、多すぎない)
4. 会話履歴(古い部分は要約、直近は生で保持)
5. 直近のユーザー入力(末尾に置く=最も効く)
6. 出力用の余白を確保
```

チェックリスト:

- [ ] この断片は**本当にこのタスクに必要**か?(なければ削る)
- [ ] 最重要の指示・参照は**冒頭か末尾**にあるか?
- [ ] 矛盾する/古い情報が混ざっていないか?
- [ ] 出力分の**余白**は残っているか?
- [ ] 毎回送っている固定部分は**キャッシュ**できないか?

::: warning アンチパターン
- リポジトリ全体を丸ごと貼る(関連の薄いファイルが雑音になる)
- 会話履歴を無限に積み続ける(コスト・レイテンシ・lost in the middle が悪化)
- 「念のため」で巨大なドキュメントを毎回添付する
:::

## まとめ

- コンテキストウィンドウは**有限のトークン予算**。入力も出力も同じ予算から支払う。
- **lost in the middle** ——重要情報は中央に埋めず端へ。
- 詰め込みすぎは**精度を下げ、コストとレイテンシを上げる**。広さより**厳選**。
- 管理の基本は**選別・要約・チャンク・優先順位付け**の組み合わせ。
