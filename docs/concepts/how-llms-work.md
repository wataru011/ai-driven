# LLMの仕組み

大規模言語モデル(LLM)を「魔法の箱」として扱っているうちは、その挙動を予測も制御もできません。経験豊富なエンジニアがAIを道具として使いこなすには、内部で何が起きているのかを**最低限の解像度**で把握しておく必要があります。

このページのゴールは、LLMの数学を厳密に学ぶことではなく、現場で頻発する疑問——「**なぜもっともらしい嘘(ハルシネーション)をつくのか**」「**なぜ同じ入力でも出力が毎回違うのか**」「**なぜ長い文脈で精度が落ちるのか**」——に、仕組みの側から答えられるようになることです。

::: tip ひとことで言うと
LLMは「次に来そうなトークンを確率で予測し、1つずつ並べていく」装置です。**意味を理解しているのではなく、巨大な統計から尤もらしい続きを生成している**——この一点を握っておくと、ほとんどの挙動が説明できます。
:::

## LLMが実際にやっていること

LLMの動作は、究極的には次の1文に集約されます。

> これまでのトークン列を受け取り、**次に来るトークンの確率分布**を出力する。

そして、その確率分布から1つトークンを選び、列の末尾に足し、また次を予測する。この繰り返し(**自己回帰, autoregression**)が「文章生成」の正体です。チャットも、コード補完も、要約も、内部では同じこの操作が走っているだけです。

```text
入力:   "def add(a, b):\n    return "
        ↓ モデルが次トークンの確率分布を計算
予測:   "a"   (0.71)
        "a+b" (0.05)
        "sum" (0.03)
        ...
        ↓ サンプリングで1つ選ぶ → "a"
入力':  "def add(a, b):\n    return a"
        ↓ 再び次トークンを予測 ...
```

## トークン化:モデルは文字でも単語でも見ていない

人間は文章を「単語」で捉えますが、LLMが扱う最小単位は**トークン**です。トークンは「よく出現する文字の塊」で、単語より細かいことも粗いこともあります。

| 入力 | トークン分割の例 |
| --- | --- |
| `hello` | `hello`(1トークン) |
| `unbelievable` | `un` / `bel` / `ievable`(3トークン) |
| `日本語` | `日` / `本` / `語`(言語・モデルにより変動) |
| `getElementById` | `get` / `Element` / `By` / `Id`(4トークン) |

ここから実務的に重要な含意が導けます。

- **トークン数 ≠ 文字数 ≠ 単語数**。英語で大まかに「1トークン ≒ 4文字 ≒ 0.75単語」、日本語やコードはこれより**トークン効率が悪い**(同じ情報量でトークンを多く消費する)傾向があります。
- 課金もコンテキスト上限も**トークン単位**です。詳細は[コンテキストウィンドウ](/concepts/context-window)で扱います。
- モデルは文字を直接見ていないため、「`strawberry`の`r`の数を数えて」のような**文字レベルの操作が苦手**です。これはバグではなく、入力がトークンに圧縮されている構造的帰結です。

## 次トークン予測と確率分布

モデルの出力は、語彙(数万〜十数万トークン)全体にわたる**確率分布**です。各候補トークンにスコア(ロジット)が割り当てられ、softmaxで確率に変換されます。

```text
候補トークン   ロジット   確率
"return"        8.2       0.62
"yield"         5.1       0.11
"pass"          4.0       0.04
" "             3.8       0.03
...
```

この確率分布から実際にどのトークンを選ぶかが**サンプリング**であり、ここに出力のゆらぎの源があります。

### temperature と top-p

| パラメータ | 役割 | 低くすると | 高くすると |
| --- | --- | --- | --- |
| **temperature** | 確率分布の鋭さを調整 | 高確率トークンに集中(決定的・保守的) | 分布が平坦化(多様・冒険的) |
| **top-p (nucleus)** | 累積確率 p までの候補に絞る | 候補が少なく安定 | 候補が広がり多様 |
| **top-k** | 上位 k 個の候補に絞る | 同上 | 同上 |

- `temperature = 0`(に近い設定)では、ほぼ常に最尤トークンを選ぶため**決定的**に近づきます。コード生成・抽出・分類など「正解が1つ」のタスクに向きます。
- `temperature` を上げるとブレインストーミングや文章の多様性が増しますが、**逸脱・ハルシネーションのリスク**も上がります。

::: info 「同じ入力でも出力が揺れる」のはなぜか
理由は2つあります。(1) サンプリングが確率的であること(temperature > 0なら選択は確率的)。(2) 浮動小数点演算の非決定性やバッチ処理の都合で、`temperature = 0` でも**完全な再現性は保証されない**こと。再現性が要るテストでは、temperatureを下げつつ「出力は揺れうる」前提でアサーションを組むのが安全です。
:::

## Transformer とアテンションの直感

現代のLLMは **Transformer** アーキテクチャを土台とし、その核が **アテンション(注意機構)** です。厳密な行列演算は脇に置き、エンジニアが持つべき直感だけを述べます。

- 各トークンを処理する際、モデルは入力中の**他のすべてのトークンを見渡し**、「いま予測に関係が深いのはどれか」に重み(注意)を割り振ります。
- この仕組みにより、`function foo(`...数十行後...`return foo` のような**長距離の依存関係**を捉えられます。変数のスコープ、対応する括弧、文脈上の主語などを「思い出せる」のはこのためです。
- ただしアテンションは**入力長の二乗に比例**する計算コストを持つため、コンテキストを無制限に伸ばせない一因になっています。

```text
"The function reads a file and returns its <?>"
                ▲                ▲          予測対象
                └──強い注意──────┘
   "returns" を予測する際、"function" "file" に強く注意が向く
```

この「全トークンを見渡して関連度で重み付けする」性質が、LLMの文脈理解力の源泉であり、同時に[コンテキスト設計](/prompting/context-design)が効く理由でもあります。

## 学習の3段階:モデルはどう作られるか

LLMの振る舞いは、学習プロセスを知ると腑に落ちます。一般的に3段階を経ます。

| 段階 | 何をするか | 何が身につくか |
| --- | --- | --- |
| **① 事前学習(Pre-training)** | 大量のテキスト/コードで「次トークン予測」をひたすら学習 | 言語・コード・世界知識の統計的下地。ただし"指示に従う"概念はまだ薄い |
| **② 指示チューニング(Instruction Tuning / SFT)** | 「指示→望ましい応答」のペアで教師あり学習 | 質問に答え、タスクを遂行する振る舞い |
| **③ 人間フィードバック強化学習(RLHF等)** | 人間の選好に沿うよう報酬モデルで微調整 | 有用・無害・正直さ、トーン、安全性 |

ここから重要な事実が見えます。

- **事前学習はある時点で打ち切られる**ため、モデルには**知識のカットオフ**があります。最新ライブラリのAPIや直近のニュースを「知らない」のはこのためで、これを補うのが[RAG](/concepts/rag)です。
- ②③で「もっともらしく、自信ありげに答える」振る舞いが強化されるため、**知らないことでも自信満々に答える**傾向が生まれます。これがハルシネーションの温床の1つです。

## なぜハルシネーションが起きるのか

[ハルシネーション](/quality/hallucination)を「バグ」と捉えると対策を誤ります。これはLLMの設計目的の**裏返し**として必然的に生じます。

1. **目的が「尤もらしさ」だから** — モデルは「真実か」ではなく「次に来そうか」を最適化しています。事実と虚構が統計的に同じくらい尤もらしいとき、平然と虚構を選びます。
2. **知識は確率的に保持されている** — 事実はテーブルに格納されているのではなく、重みに分散圧縮されています。曖昧な記憶は**それらしい捏造で補間**されます。
3. **「分からない」と言うインセンティブが弱い** — 学習データには断定的な文章が多く、空白を埋めるよう訓練されています。
4. **カットオフ外・低頻度の知識** — 学習時に見ていない、あるいは稀にしか見ていない情報ほど捏造が増えます。

::: warning 実務上の帰結
ハルシネーションは「良いプロンプト」で**ゼロにはできません**。減らすことはできても、消すことはできない。したがって対策の本丸は入力側ではなく**出力側の検証**です——テスト、型チェック、コンパイル、根拠の提示(RAG)。この発想は[AI駆動開発とは](/guide/what-is-aidd)の「検証可能性を設計に織り込む」と一貫しています。
:::

## コード補完への含意

以上を踏まえると、エディタ上のコード補完(L1)の挙動も説明できます。

- **補完は「このコードの続きとして尤もらしいトークン列」**を生成しているだけで、あなたの意図やプロジェクトの正しさを保証しません。周囲のコードが綺麗なら尤もらしい続きも綺麗になり、汚ければ汚いコードを"自然に"続けます(GIGO)。
- **既存コードのスタイルや命名に強く引きずられる**のはアテンションの帰結です。良い補完を引き出したいなら、近傍のコード・型注釈・docstringを整えるのが最も効きます。
- **存在しないAPIをそれらしく補完する**ことがあります。カットオフ外のライブラリや、稀なAPIほど顕著です。補完を**コンパイル・テストで必ず裏取り**する習慣が前提になります。
- **temperatureが低めに設定された補完**は決定的に近く、ブレインストーミング用途とは別物だと理解しておきましょう。

## まとめ:仕組みから導かれる実務原則

- LLMは**確率的な次トークン予測器**である。理解ではなく統計。
- 出力の**ゆらぎはサンプリングの帰結**であり、決定性が要るなら temperature を下げる。
- **ハルシネーションは構造的**に生じる。入力で減らし、**出力で検証**する。
- **知識にはカットオフがある**。最新性が要るなら[RAG](/concepts/rag)で外部知識を注入する。
- 文脈理解は**アテンション**が支える。だから[コンテキスト設計](/prompting/context-design)が効く。
