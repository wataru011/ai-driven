# マルチエージェント

単一のエージェントで解けない、あるいは解きづらいタスクに対し、**複数のエージェントを協調させる**のがマルチエージェントの考え方です。本ページでは代表的な協調パターン、その利点と難点、そして「いつ単一で十分か」の判断基準を示します。

::: tip 出発点
本ページは[AIエージェントとは](/agents/what-is-agent)と[ツール使用とMCP](/agents/tools-and-mcp)を前提とします。マルチエージェントは「ツールを持つエージェント」を部品として組み合わせる設計です。
:::

## なぜ複数にするのか

単一エージェントの限界は主に3つです。

1. **文脈の飽和** — 大きなタスクを1つの文脈に詰め込むと、コンテキストが溢れ精度が落ちる。
2. **役割の混線** — 「実装しながら自分をレビューする」のは、人間同様、見落としが生じやすい。
3. **逐次のボトルネック** — 独立に進められる作業も、単一エージェントでは直列になる。

マルチエージェントは、これを **分業(役割分離)・並列化・多視点検証** で乗り越えようとします。ただしタダではなく、調整コストとトークン消費という代償を伴います。

## パターン1: オーケストレーター/ワーカー

最も基本的な構成。**オーケストレーター(指揮役)** がタスクを分解し、各サブタスクを **ワーカー(実行役)** に割り当て、結果を統合します。

```text
                   ┌──────────────────┐
                   │  オーケストレーター │  タスク分解・割当・統合・最終判断
                   └─────────┬────────┘
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
       ┌────────┐      ┌────────┐      ┌────────┐
       │ワーカーA│      │ワーカーB│      │ワーカーC│   各々が独立した文脈で作業
       └────────┘      └────────┘      └────────┘
```

- 各ワーカーは **自分専用の文脈** を持つため、文脈の飽和を避けられる。
- オーケストレーターは全体方針と統合に専念し、細部の文脈を抱えなくてよい。
- 「コードベース全体を調査する」ようなタスクで、領域ごとにワーカーを割り当てる用途に向く。

## パターン2: 役割分担(設計者/実装者/レビュアー)

人間の開発チームを模した構成。各エージェントに**専門の役割とプロンプト**を与えます。

| 役割 | 責務 | 入力 | 出力 |
| --- | --- | --- | --- |
| 設計者(Architect) | 要件を設計に落とす | 要求仕様 | 設計方針・インターフェース |
| 実装者(Implementer) | 設計に沿って書く | 設計 | コード・テスト |
| レビュアー(Reviewer) | 品質・規約・脆弱性を点検 | コード | 指摘・修正要求 |

役割を分けると、それぞれのプロンプトを役割に特化でき、**関心の分離** が効きます。レビュアーは「実装者が書いた」という事実を知らずに批判的に読めるため、自己レビューより指摘の質が上がる傾向があります。

```text
設計者 → 設計案 → 実装者 → コード → レビュアー → 指摘
                    ▲                          │
                    └──────── 修正要求 ◀───────┘   (合格まで反復)
```

## パターン3: 並列ファンアウト(fan-out / fan-in)

独立なサブタスクを**同時に**複数ワーカーへ展開(fan-out)し、結果を集約(fan-in)します。

```text
        ┌─→ ワーカー1: モジュールAのテスト生成 ─┐
依頼 ───┼─→ ワーカー2: モジュールBのテスト生成 ─┼──→ 集約 → 統合結果
        └─→ ワーカー3: モジュールCのテスト生成 ─┘
```

- **壁時計時間(レイテンシ)を短縮** できる。直列なら3倍かかる作業が並列なら1回分。
- 同じ問題を異なる温度/方針で複数回解かせ、**多数決や最良選択** を取る用途(self-consistency)にも使える。
- 前提は **サブタスクが互いに独立** であること。依存があると並列化は破綻する。

## パターン4: 批評(critic)ループ

生成役(actor)と批評役(critic)を分け、**生成→批評→改稿** を反復します。役割分担パターンのレビュアーを、より一般化したものです。

```text
Actor:  ドラフトを生成
Critic: 「テストのエッジケースが不足」「この関数は古いAPIを使用」と指摘
Actor:  指摘を反映して改稿
Critic: 「OK、ただしエラーハンドリングを追加すべき」
Actor:  反映
Critic: 合格 → 終了
```

::: warning critic ループは無限反復に注意
批評役が常に何か指摘を返すと、ループが止まりません。**反復回数の上限**と、「これ以上の指摘は軽微」という**収束判定**を必ず設けます。改善が頭打ちになったら打ち切るのが鉄則です。
:::

## 利点と難点

マルチエージェントは銀の弾丸ではありません。利点と難点を天秤にかけて採用を判断します。

| | 利点 | 難点 |
| --- | --- | --- |
| 文脈 | 役割ごとに文脈を分離し飽和を回避 | 一貫性の維持が難しい(各々が別の前提を持つ) |
| 速度 | 独立タスクを並列化し壁時計時間を短縮 | 調整・統合のオーバーヘッドが増える |
| 品質 | 多視点(生成/批評)で見落としを減らす | 反復で**トークン消費が数倍**に膨らむ |
| 設計 | 関心の分離で各プロンプトを特化できる | エージェント間通信の設計・デバッグが複雑 |

特に **トークン消費** は見落とされがちです。エージェント間でやり取りする文脈は重複しがちで、単一エージェントの数倍のコストになることも珍しくありません。**速度や品質の向上がコスト増に見合うか**を都度評価してください。

::: info 一貫性の問題
複数エージェントが別々の文脈で判断すると、命名規則・設計方針・前提が食い違うことがあります。共有すべき制約(コーディング規約、アーキテクチャ決定)は、**全エージェントに共通のシステムプロンプトやResourceとして注入** し、ブレを抑えます。
:::

## いつ単一エージェントで十分か

**多くのタスクは単一エージェントで十分です。** マルチエージェント化は、明確な理由があるときだけ採用すべき複雑化です。

単一で十分なサイン:

- タスクが直線的で、サブタスク間に強い依存がある(並列化の旨味がない)。
- 1つの文脈に収まり、文脈飽和が起きない。
- 速度よりコスト・単純さを優先したい。

マルチエージェントを検討すべきサイン:

- **独立に並列実行できる**サブタスクが明確に分かれる(ファンアウトの旨味)。
- 探索範囲が広く、**単一文脈では飽和する**(オーケストレーター/ワーカー)。
- **生成と検証を分離**したい品質要件がある(critic / レビュアー)。

::: tip 設計の指針
**まず単一エージェントで作り、ボトルネック(文脈飽和・直列の遅さ・自己レビューの甘さ)が実際に観測されてから**、その箇所だけをマルチ化する。最初から多エージェントで組むと、調整コストとデバッグ難度に見合わないことが多いです。
:::

## まとめ

- 主要パターンは **オーケストレーター/ワーカー、役割分担、並列ファンアウト、批評ループ** の4つ。
- 利点は **分業・並列・多視点検証**。難点は **調整コスト・トークン消費・一貫性**。
- critic系は**反復上限と収束判定**を必ず設ける。共有制約は全エージェントに注入して一貫性を保つ。
- **既定は単一エージェント。** ボトルネックが観測されてから、その箇所だけマルチ化するのが堅実。
