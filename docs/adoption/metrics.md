# 生産性の測定

「AIで開発はどれだけ速くなったのか」——経営も現場も知りたい問いですが、これは見かけよりはるかに難しい問題です。素朴な指標は容易に誤導し、間違った最適化を引き起こします。

このページは、AI駆動開発の効果を **誤らずに測る** ための考え方を扱います。結論を先に言えば、単一の魔法の指標は存在せず、**定量と定性の組み合わせ・ベースラインとの比較・複数指標のバランス** で立体的に捉えるしかありません。

::: warning 測らないこともアンチパターン、測りすぎもアンチパターン
「感覚で速くなった気がする」では投資を正当化も改善もできません。一方で、人を単一指標で評価し始めると、その指標は即座にゲーム化されます([グッドハートの法則](/reference/glossary))。目的は **学習と意思決定** であって、人の査定ではありません。
:::

## なぜ測定が難しいのか

- **ソフトウェア開発の生産性はもともと測りにくい** — 「アウトプット」が均質でない。難しい1行と簡単な1000行は比較できません。AI以前から未解決の問題です。
- **価値は下流に現れる** — 速く書けても、レビュー・修正・本番障害で相殺されることがある。局所最適(コーディング速度)が全体最適(リードタイム)を保証しない。
- **交絡が多い** — 学習効果、メンバーの入れ替わり、タスクの難易度、他の施策が同時に動く。AI単独の寄与を切り分けるのは困難。
- **自己申告のバイアス** — 「速くなった感」は実測と乖離することがある。新しい道具は主観的満足を押し上げやすい。

だからこそ、測定は **慎重に設計し、複数の角度から** 行う必要があります。

## 見かけの指標の罠

直感的で集めやすい指標ほど、危険です。

| 見かけの指標 | なぜ罠か |
| --- | --- |
| コード行数(LOC) | 多く書くことは価値ではない。むしろ良い設計は行数を減らす。AIは冗長なコードを大量生成しがちで、LOCを膨らませるほど「生産的」に見える倒錯が起きる。 |
| コミット数 / PR数 | 粒度を細かくすれば増える。質を全く反映しない。 |
| AI受容率(提案の採用割合) | ツールベンダー指標。採用が多い=良いとは限らない。安易に受け入れれば後で手戻りが増える。 |
| 「タイピングが減った」 | 入力は開発時間のごく一部。理解・設計・検証が本体。 |

::: warning これらを目標(KPI)にしてはいけない
LOCやPR数を目標にした瞬間、人はそれを増やす行動を取り、コードベースは肥大化し品質は下がります。これらは **診断のための補助情報** としてのみ、慎重に見ます。
:::

## DORAとSPACE: 確立されたレンズ

AI固有の指標を発明する前に、ソフトウェアデリバリーで確立された2つのフレームワークを使います。

### DORA(デリバリーのアウトカム)

DORA(DevOps Research and Assessment)は、デリバリー性能を **アウトカム** で捉える4つの主要指標で知られます。AIの効果は、最終的にここに現れるべきです。

| DORA指標 | 意味 | AI導入で見るべきこと |
| --- | --- | --- |
| デプロイ頻度 | どれだけ頻繁に本番リリースできるか | 速くなったか、それとも変わらないか |
| 変更のリードタイム | コミットから本番までの時間 | コーディングの高速化が下流まで波及しているか |
| 変更失敗率 | リリースが障害を引き起こす割合 | **上がっていないか**(品質の犠牲を検知) |
| 復旧時間(MTTR) | 障害からの回復時間 | AI支援が運用にも効いているか |

DORAの要点は、**スループット(頻度・リードタイム)と安定性(失敗率・復旧)を同時に見る** ことです。片方だけ追うと必ず崩れます。詳細は[DORA公式](/reference/resources)を参照。

### SPACE(多次元の生産性)

SPACEフレームワークは、生産性を単一指標で測ることの誤りを正面から扱い、**5つの次元** から多面的に捉えるよう提案します。

| 次元 | 内容 | AI文脈での例 |
| --- | --- | --- |
| Satisfaction(満足度・ウェルビーイング) | 開発者の満足・燃え尽き | AIで雑務が減り集中時間が増えたか |
| Performance(成果) | アウトカムの質 | 障害率、顧客影響 |
| Activity(活動量) | 完了した作業量 | 完了タスク数(単独で評価しない) |
| Communication & Collaboration | 連携・知識共有 | レビュー負荷の変化、知識の偏り |
| Efficiency & Flow(効率・フロー) | 中断のない進行 | フロー状態の確保、待ち時間の削減 |

SPACEの教訓は明快です。**生産性は1次元ではない。複数の次元を、必ず複数組み合わせて見る。** 1つの次元だけを最大化すると他が犠牲になります。

## ベースラインを取る

効果測定の前提は、**比較対象** です。これが最も見落とされます。

- **導入前に測る** — AIを入れる前のDORA指標・サイクルタイム・障害率を記録する。後から「Before」は取れません。
- **同種で比較する** — 難易度やドメインが異なるタスク同士を比較しない。可能なら似た性質のチーム・期間で対照を取る。
- **十分な期間** — 単発のスプリントでは学習効果と区別できない。数か月単位のトレンドで見る。
- **交絡を記録する** — 同時に走った他施策、人員変動をメモしておき、解釈時に考慮する。

::: tip ベースラインがないなら、まずそこから
すでにAIを使い始めていてBeforeがない場合でも、**今から定点観測を始める** 価値はあります。完璧な対照実験は望めなくとも、トレンドの変化は意思決定に十分役立ちます。
:::

## 定量と定性のバランス

数字だけでは、なぜそうなったかが分かりません。定性情報が解釈を支えます。

- **定量** — DORA指標、サイクルタイム、レビューにかかる時間、障害率。トレンドと異常の検知に使う。
- **定性** — 開発者サーベイ(満足度・どのタスクで効いたか・どこで手戻りが出たか)、振り返りでの生の声、具体的な成功/失敗事例。
- **三角測量** — 定量の変化を定性で裏取りする。「リードタイムが縮んだ」が「テスト生成が効いた」のか「レビューを手抜きした」のかは、定性でしか分からない。

開発者の主観的な体験(DevEx)は、それ自体が重要なアウトカムです。燃え尽きを増やして数字を上げる施策は、長期的には失敗です。

## 何を・どう測るか(実践テーブル)

| 測りたいこと | 推奨指標(定量) | 補完する定性 | 注意 |
| --- | --- | --- | --- |
| デリバリーが速くなったか | 変更リードタイム、デプロイ頻度(DORA) | 「どの工程が短縮されたか」の振り返り | コーディング速度ではなく全体を見る |
| 品質を犠牲にしていないか | 変更失敗率、本番障害数、欠陥流出率 | インシデントの根本原因にAI起因があるか | スループットと必ず対で見る |
| レビュー負荷の変化 | PRレビュー所要時間、レビュー差し戻し率 | レビュアーの負担感 | 生成量増でレビューがボトルベ化していないか |
| 開発者体験 | 開発者サーベイ(満足度・フロー) | 自由記述の声 | 単独で人事評価に使わない |
| どのタスクで効くか | タスク類型別のサイクルタイム | 効いた/効かなかった事例の収集 | [導入戦略](/adoption/strategy)の知見蓄積に直結 |
| 採用・定着 | 実利用者率の推移 | 使わない理由のヒアリング | 利用率自体はアウトカムではない |

## 過剰最適化の弊害

::: warning 指標は地図であって領土ではない

- **グッドハートの法則** — 「測定が目標になると、それは良い測定でなくなる」。LOCを目標にすればコードは膨らみ、PR数を目標にすれば粒度が崩れる。
- **個人のランキングに使わない** — 開発生産性指標を個人査定に使うと、即座にゲーム化され、協働・知識共有・レビューの質といった「測りにくいが重要なもの」が犠牲になる。指標は **チーム・システムの改善** に使う。
- **単一指標の最大化を避ける** — DORAもSPACEも「複数を同時に見る」ことが設計思想。1つを突出させると他が壊れる。
- **測定のコストに見合うか** — 計測自体が重く現場を疲弊させるなら本末転倒。軽量に、意思決定に効く分だけ測る。
:::

## まとめ

- 単一の魔法の指標は存在しない。**定量×定性、複数指標のバランス、ベースライン比較** で立体的に捉える。
- LOC・PR数・受容率などの見かけの指標を **目標にしない**。
- DORA(スループットと安定性)とSPACE(生産性の多次元)を確立されたレンズとして使う。
- 導入前にベースラインを取り、交絡を記録し、トレンドで判断する。
- 指標は人の査定ではなく **システムの改善** のため。目標化はゲーム化を招く(グッドハートの法則)。
