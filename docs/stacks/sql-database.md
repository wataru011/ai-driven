# SQL・データベースでのAI活用

SQLは、AIが**最も流暢に書ける言語の一つ**です。複雑なJOIN、ウィンドウ関数、CTE、集計——人間が組み立てると時間のかかるクエリを、AIは一瞬で形にします。生産性の効果は絶大です。

ところがSQLは、**間違いの代償が最も大きい**領域でもあります。`WHERE` を一行書き忘れた `UPDATE` は全レコードを書き換え、誤ったJOINは件数を静かに水増しし、文字列連結で組んだクエリはSQLインジェクションの入口になります。しかもこれらは**構文エラーにならない**——SQLは喜んで実行してしまいます。本章は、AIのSQL生成力を活かしつつ、この破壊力をどう封じるかに焦点を当てます。

::: warning 大原則: 本番の書き込みSQLは「人間の二重確認」を経る
AI生成の `UPDATE`/`DELETE`/`DROP`/マイグレーションを本番に直接流すのは厳禁です。**読み取り専用での確認、トランザクション内での件数検証、ステージングでの試行**を経てから本番に適用する——この順序を崩さないことが、データを守る唯一の確実な方法です。
:::

## AI活用の勘所

### 1. クエリ生成は「スキーマ」を文脈に与える

スキーマを渡さずにクエリを頼むと、AIは**存在しない列名やテーブル名を推測**します(典型的なハルシネーション)。`CREATE TABLE` 文や `\d` の出力、ERの説明を文脈として与えるのが第一歩です。

> 以下のスキーマ(DDLを添付)に対して、直近30日の顧客別売上トップ10を出すクエリを書いてください。
> - 返金(refunds)は売上から差し引く
> - 売上ゼロの顧客は除外
> - PostgreSQL 16。ウィンドウ関数を使ってよい
> まず、どのテーブルをどう結合するかを日本語で説明してから、クエリを書いて。

「結合の説明を先に」と求めると、**誤ったJOINや粒度の取り違え**を早期に発見できます。

### 2. 最適化は「実行計画」で語らせる

「このクエリを速くして」と頼むと、AIは**推測で**インデックスを足したりサブクエリを書き換えたりします。当たることもありますが、根拠がありません。実測の `EXPLAIN (ANALYZE, BUFFERS)` を渡して判断させるのが正攻法です。

> このクエリの `EXPLAIN (ANALYZE, BUFFERS)` 出力を貼ります。
> - ボトルネック(Seq Scan、ソート、Nested Loopの繰り返し等)を特定
> - 追加すべきインデックスを、対象列と理由つきで提案
> - インデックス追加の副作用(書き込みコスト、サイズ)も述べて
> 推測ではなく、この実行計画の数値に基づいて説明してください。

::: tip インデックス提案は鵜呑みにしない
AIは「とりあえずインデックスを足す」を好みますが、過剰なインデックスは書き込みを遅くし、ストレージを食います。**選択性が低い列**(性別など)へのインデックスは効きません。提案は実行計画と実データの分布で裏取りしてください。
:::

### 3. スキーマ設計は「制約」をAIに守らせる

スキーマ設計でAIに任せられるのは雛形までで、**正規化の判断・整合性制約**は人間が握ります。AIは制約(外部キー、UNIQUE、CHECK、NOT NULL)を省きがちなので、明示的に要求します。

> このテーブル設計に、データ整合性を守る制約を追加してください。
> - 参照整合性(外部キー)と、適切なON DELETE挙動
> - ビジネスルールをCHECK制約で(例: amount >= 0、status は列挙値のみ)
> - 一意であるべき組み合わせにUNIQUE制約
> - NULLを許してよい列/許さない列を、業務的根拠とともに判断
制約は「最後の砦」です。アプリ側のバリデーションをAIが書き忘れても、DB制約があれば不正データを防げます。

### 4. マイグレーションは「前進と後退」をセットで

マイグレーションはデータベース操作の中でも特にリスクが高く、AIに任せる際は**ロールバックとデータ保全**を必ず要求します。

::: warning マイグレーションの危険な生成パターン
- **NOT NULL列の無防備な追加** — 既存行が違反して失敗、または不適切なデフォルトで埋まる。
- **列のリネーム/削除** — デプロイ中の旧コードが壊れる(後方互換の欠如)。
- **巨大テーブルへのロック** — `ALTER TABLE` が長時間ロックし、本番を止める。
- **down マイグレーションの欠如/不整合** — 戻せない、または戻すとデータが消える。
:::

> このマイグレーションを安全に書き直してください(本番テーブルは数千万行)。
> - 長時間ロックを避ける(必要なら複数ステップに分割、オンラインDDL/CONCURRENTLY)
> - 後方互換: 旧コードと新コードが同時に動く期間を考慮(expand/contractパターン)
> - down(ロールバック)を用意し、データ損失が起きないことを確認
> - NOT NULL列追加はデフォルト設定→バックフィル→制約付与の順に分ける

### 5. アプリからのクエリは「パラメータ化」を死守

アプリコードにSQLを書かせるとき、AIが**文字列連結でクエリを組む**ことがあります。これはSQLインジェクションの直接的な原因です。

> SQLは必ずパラメータ化(プレースホルダ/バインド変数)で書いてください。
> ユーザー入力を文字列連結でクエリに埋め込むことは禁止。
> 動的な列名・テーブル名が必要な場合は、許可リストで検証してから使う。

詳細な脅威モデルと対策は [セキュリティリスク](/quality/security) を参照してください。

## 落とし穴と対策

::: warning SQL・DB特有の典型的な失敗
- **WHERE句なしのUPDATE/DELETE** — 全レコードを破壊。構文エラーにならない。
- **誤JOINによる件数膨張** — 多対多の結合で行が重複し、集計が水増しされる。
- **SQLインジェクション** — 文字列連結によるクエリ構築。
- **暗黙の型変換** — インデックスが効かない、または予期せぬ比較結果。
- **NULLの罠** — `NOT IN` にNULLが混じると0件、`= NULL` は常に偽。
- **集計の粒度ミス** — JOIN後に `SUM` して二重カウント。
- **タイムゾーン/日付境界** — `BETWEEN` の端点取りこぼし、tz未指定。
:::

| 落とし穴 | 対策 |
| --- | --- |
| WHERE忘れのUPDATE/DELETE | トランザクションで `SELECT` 件数確認 → 想定どおりなら `UPDATE` → commit。`autocommit`を切る |
| 誤JOINの件数膨張 | JOIN前後の件数を比較。`COUNT(*)` と `COUNT(DISTINCT pk)` を照合 |
| インジェクション | パラメータ化を強制、文字列連結を静的解析/レビューで禁止 |
| NULLの罠 | `NOT EXISTS` を `NOT IN` の代わりに、NULL挙動を明示的にテスト |
| 集計の二重計上 | 既知の小データで検算、サブクエリで先に集約してからJOIN |

## 検証: 本番での安全フロー

AI生成SQL、とくに**書き込み系**は、決まった検証フローを通します。

1. **読み取りで影響範囲を見る** — `UPDATE`/`DELETE` を実行する前に、同じ `WHERE` で `SELECT COUNT(*)` を実行。**何件に影響するか**を確認する。意図より多ければ止める。
2. **トランザクションで試す** — `BEGIN; ... ` の中で実行し、件数や結果を確認してから `COMMIT`。違えば `ROLLBACK`。
3. **EXPLAINで重さを見る** — 大きなテーブルへの操作は、ロックや全表走査の有無を `EXPLAIN` で確認。
4. **ステージングで本番相当データに対し試行** — 件数・実行時間・ロック挙動を本番前に観察。
5. **マイグレーションはバックアップ前提** — 不可逆な変更の前にバックアップ/スナップショット。down手順を確認。
6. **最小権限の接続を使う** — アプリの実行ユーザーに `DROP` や不要な `DELETE` 権限を与えない。AI生成コードが暴走しても権限で止める。

::: details 書き込みSQLレビュー チェックリスト
- `UPDATE`/`DELETE` に意図した `WHERE` があるか(全件操作になっていないか)
- 同じ条件の `SELECT COUNT(*)` で影響件数を確認したか
- JOIN後に集計している場合、粒度の二重計上はないか
- ユーザー入力はパラメータ化されているか
- マイグレーションに down と後方互換があるか
- 巨大テーブルで長時間ロックしないか
- 不可逆操作の前にバックアップがあるか
:::

::: tip LLMは「実データを見ていない」
AIはスキーマからクエリを書けますが、**実際のデータ分布・件数・カーディナリティは知りません**。「影響は数件のはず」というAIの言明は推測です。件数は必ず自分で `SELECT COUNT(*)` して確かめてください。これは [ハルシネーション対策](/quality/hallucination) の実践そのものです。
:::
