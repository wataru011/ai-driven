# 実装フェーズ

設計が固まったら実装です。ここでのAI駆動開発の要諦は、**「速く大量に書く」ではなく「検証可能な単位で確実に積む」**ことに尽きます。[ワークフロー全体像](/workflow/overview)で述べた検証ファーストの原則が、最も具体的に効いてくるフェーズです。

::: tip このフェーズの心構え
生成の速さは手段であって目的ではありません。**レビューと統合のコストを最小化する単位**で生成することが、結果的に最速になります。
:::

## 1. タスク分解

設計を、**1つずつ検証できる小さなタスク**に割ります。この分解の質が、実装フェーズの成否をほぼ決めます。

```text
承認フロー機能(設計は添付)を実装します。
まずコードは書かず、実装タスクを分解してください。

各タスクは:
- 単独でテスト可能な単位にする
- 依存関係(どのタスクが先か)を示す
- 1タスクの目安は「1つのPRでレビューできる大きさ」

依存順に並べたタスクリストを出してください。
```

::: tip 分解はAIにやらせて人間が承認する
分解自体をAIに提案させると抜けが減りますが、**粒度と順序の最終決定は人間**が行います。「このタスクは大きすぎる」「この順序だと検証できない」という判断が要点です。
:::

## 2. スキャフォールド生成

骨組み(ディレクトリ構成、空の関数、型定義、インターフェース)を先に生成します。**全体構造を固めてから中身を埋める**ほうが、手戻りが少なくなります。

```text
タスク1「承認エンティティと repository 層」のスキャフォールドを生成。

- 型定義(Approval, ApprovalStatus)
- repository インターフェースとメソッドシグネチャ
- 実装は中身を空にし、// TODO で意図をコメント

既存の src/repository/user.ts のスタイルに厳密に合わせること。
中身の実装はまだ書かないこと。
```

## 3. テスト先行(TDD的アプローチ)

検証ファーストを最も自然に実現するのが、**テストを先に書く**進め方です。期待する振る舞いをテストとして固定し、それを満たす実装を生成させます。

```text
タスク1の repository に対し、まずテストを書いてください。

仕様:
- save(approval): 保存して採番されたIDを返す
- findById(id): 存在すれば Approval、なければ null
- 同一申請の二重承認は ConflictError を投げる

境界・異常系を必ず含めること。実装はまだ書かないこと。
テストは現時点で「失敗する」状態でよい。
```

テストを人間がレビューして合意したうえで、実装を生成させます。

```text
先ほど合意したテストをすべてパスする実装を書いてください。
- テストを書き換えて通すことは禁止(仕様はテスト側が正)
- 既存規約(Result 型、JSDoc)に従う
```

::: warning テストを正とする
「テストが通らないからテストを直す」をAIに許すと、検証の意味が消えます。**テストは仕様であり、実装が従う**という原則をプロンプトで明示します。生成テスト自体の妥当性は[テストとレビュー](/workflow/testing-review)で別途検証します。
:::

## 4. 差分単位の生成とレビュー

一度に大きなコードを生成させると、レビューが破綻します。**diff(差分)単位**で生成・レビュー・統合を回します。

```text
このタスクの変更を unified diff 形式で出力してください。
- 変更点の概要を先に箇条書き
- そのあと diff
- 想定されるリスクを最後に列挙

無関係なファイルやフォーマット変更を含めないこと。
```

差分が小さいほど、レビューで誤りを見つけやすく、問題があっても切り戻しが容易です。

::: tip 差分を小さく保つ
「ついでに周辺もリファクタした」という大きな差分は危険です。**1差分1目的**を徹底し、リファクタリングは別タスクに分離します(→[リファクタリングと保守](/workflow/refactoring))。
:::

## 5. 既存規約への適合

生成コードが既存コードベースに馴染むことは、可読性と保守性に直結します。適合を担保する手段は3つです。

1. **規約ファイル** — `CLAUDE.md` 等に命名・構造・エラー処理方針を集約([コンテキスト設計](/prompting/context-design))。
2. **実例の提示** — 「このファイルのスタイルに合わせて」と既存ファイルを1つ渡す。最も確実。
3. **機械的検証** — lint / formatter / 型チェックを生成後に必ず通し、その結果をフィードバックする。

```text
生成後、`npm run lint && npm run typecheck` を通したところ
以下の警告が出ました。規約に沿って修正してください。
（lint 結果を貼る）
```

## エージェントへの委任の粒度

ツールを実行できるエージェント([エージェントとは](/agents/what-is-agent))を使う場合、**どこまで一気に任せるか**が判断のしどころです。

| 委任の粒度 | 向くタスク | リスク |
| --- | --- | --- |
| 1関数・1差分 | 高リスク・複雑なロジック | 低(レビュー容易) |
| 1タスク(数ファイル) | 明確な仕様・テストあり | 中 |
| 1機能(複数タスク) | 定型・低リスク・テスト充実 | 高(レビュー困難) |

::: warning 委任の粒度はテストの厚さで決める
**テストで囲える範囲までしか委任しない**のが鉄則です。エージェントが「テストが通った」と報告しても、そのテスト自体が妥当かは人間が確認します。検証できない範囲を委任すると、もっともらしいが誤ったコードを見抜けません。
:::

判断軸は[ワークフロー全体像](/workflow/overview)と同じ、**リスク × 可逆性 × 検証コスト**です。可逆で検証が安いタスクは大胆に委任し、不可逆で検証が高いタスクは差分単位に刻みます。

## アンチパターン

::: warning 実装フェーズの失敗例
- **一括生成**: 機能全体を一度に生成させ、レビューが破綻する。
- **テストの後付け**: 実装してからテストを生成し、実装に都合よく合わせてしまう。
- **規約の未共有**: 既存スタイルを渡さず、浮いたコードが量産される。
- **ついでリファクタ**: 実装差分に無関係な変更を混ぜ、レビューを困難にする。
- **検証なき委任**: テストで囲えない範囲をエージェントに丸投げする。
:::

## まとめ

- 実装は「速く大量」ではなく「**検証可能な小さい単位で確実に**」。
- タスク分解 → スキャフォールド → テスト先行 → 差分生成、の順で積む。
- **テストを仕様の正**とし、実装を従わせる。
- 既存規約は規約ファイル + 実例 + 機械的検証で担保する。
- 委任の粒度は**テストで囲える範囲**まで。
