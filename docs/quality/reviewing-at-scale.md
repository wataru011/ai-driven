# 大量生成物のレビュー戦略

AIは生成を桁違いに速くします。しかし**レビューは速くなりません。** その結果、レビューがスループットの上限になります。本ページは、生成量が増えてレビューがボトルネック化する問題への、実務的な対処戦略を扱います。

::: tip 前提
1件ずつの**評価のやり方**は[生成物を人が評価する方法](/quality/evaluating-output)で扱いました。本ページはその評価を、**量が増えた状況でどう回すか**という戦略の話です。両方そろって初めて持続可能なレビューになります。
:::

## なぜ問題か: 生成とレビューの非対称

生成とレビューの速度差が、AI駆動開発の最大の構造的課題です。

```text
従来:    [遅い実装] → [レビュー]          レビューは全体の一部
AI駆動:  [速い生成] → [   レビュー   ]     レビューが支配的なボトルネック
                       ▲ ここが詰まる
```

- **生成は速い** — 数分で大量の差分が出る。供給に上限がない。
- **レビューは遅い** — 人間の注意力・時間は有限で、増やせない。
- **量はレビュー品質を非線形に下げる** — 量が増えると[automation bias](/quality/evaluating-output)とレビュー疲れが効き、1件あたりの精度まで落ちる。

### レビュー崩壊から無検証マージへ

放置すると、次の悪循環に陥ります。

```text
生成量↑ → レビュー待ち行列↑ → 「捌くために雑に通す」 → 無検証マージ
   ▲                                                      │
   └──────────── 品質低下が後で大量の手戻りを生む ◀────────┘
```

::: warning 最悪のシナリオは「ラバースタンプ」
レビュー待ちが積み上がると、人は中身を見ずに承認(ラバースタンプ)し始めます。これは**レビューがあるように見えて実質ない**最悪の状態で、automation bias と相まって欠陥を素通りさせます。レビューが崩壊する前に、戦略で**レビュー総量を制御**する必要があります。
:::

## 戦略の全体像

対処は2方向です。**(1) レビュー対象の量を減らす**、**(2) 残ったものを効率よく見る**。以下の戦略を組み合わせます。

| 戦略 | 効果 | 主眼 |
| --- | --- | --- |
| 小さなPRに分割 | 1件を読める大きさに | 量を減らす・精度を上げる |
| 差分の構造化レビュー | 見る順序と観点を固定 | 効率を上げる |
| リスクベースの濃淡 | 重要箇所に注意を集中 | 効率を上げる |
| 機械的ふるい(一次選別) | 機械で弾けるものを弾く | 量を減らす |
| サンプリングレビュー | 低リスク群は抽出検査 | 量を減らす |
| AIで一次パス → 人が最終 | 機械的な指摘を前段で処理 | 効率を上げる |
| 検証可能性を設計に織り込む | そもそも評価しやすく生成 | 量・精度の両方 |

### 1. 生成を小さなPRに分割する

レビュー精度は差分の大きさに反比例します。**1つの関心事=1つの小さなPR**を原則にします。

- [ ] エージェントに「巨大な変更を一度に出さない」ことを指示する。
- [ ] 1つのPRで評価が完結する粒度に保つ(他ファイルを大量に追わないと判断できないものは分割)。
- [ ] 大きすぎる差分は内容以前に**評価不能=差し戻し**とする([評価できない生成物は不合格](/quality/evaluating-output))。

分割の具体的な粒度・運用基準は[PRレビューの基準](/ready/pr-review-standards)を参照してください。

### 2. 差分の構造化レビュー

見る順序と観点を毎回固定すると、速くブレなく評価できます。

```text
差分を見る順序:
  1. 説明 / 意図   — 何を・なぜ変えるか(要件と照合)
  2. 公開API / 契約 — 外向きの変更(不可逆性が高い)
  3. コアロジック   — 正しさ・エッジ・エラー処理
  4. テスト         — 妥当性(実装に合わせただけでないか)
  5. 周辺 / 雑多     — 設定・整形など(機械に任せた残り)
```

観点フレーム(正しさ・妥当性・安全性ほか)は[生成物を人が評価する方法](/quality/evaluating-output)の8観点をそのまま使います。

### 3. リスクベースで濃淡をつける

全差分を均一に厚くレビューするのは不可能で、不要です。**不可逆・高リスクな箇所を厚く、影響の小さい箇所を薄く**します。

| リスク区分 | 例 | レビューの濃さ |
| --- | --- | --- |
| 高(不可逆・広範囲) | 認証・認可・課金・データ削除・外部公開API・暗号 | 全観点を複数人で厚く。機械+人を必須 |
| 中 | 内部ロジック・データ変換・新機能 | 8観点を1人で確認。テストで裏取り |
| 低 | 整形・コメント・文言・自動生成物 | 機械チェック+サンプリングで足りる |

::: tip 注意は希少資源、高リスクに配分する
レビュアーの注意は有限です。低リスクな差分に注意を均等配分すると、高リスク箇所が手薄になります。**「壊れたとき取り返しがつくか」**で濃淡を決め、注意を不可逆な場所に集中させてください。
:::

### 4. 機械的ふるい(一次選別)

人間が見る前に、**機械で判定できることは機械に弾かせ**ます。これがレビュー総量を最も効果的に減らします。

```text
生成 → フォーマッタ → リンタ → 型 → テスト → 静的解析(SAST) → シークレットスキャン → 依存監査
                                                                                          │
                                                          すべて緑のものだけ ▼
                                                                  人間レビュー(設計・意図・妥当性)
```

- [ ] フォーマット・リント・型・テストを**必須CIゲート**にし、赤はマージ不可。
- [ ] 静的解析(SAST)・シークレットスキャン・依存脆弱性スキャンをパイプラインに組み込む([セキュリティリスク](/quality/security))。
- [ ] **生成物に例外を設けない**。「AIが書いたから」はスキップの理由にしない。
- [ ] 機械が緑にしたものだけを人間レビューに回し、人は**機械が見られないこと**に集中する。

### 5. サンプリングレビュー

低リスクで量が多い生成物(定型コード・大量の文言・自動マイグレーションなど)は、**全数ではなく抽出検査**で品質を監視します。

- 低リスク群から一定割合を抽出して厚くレビューし、品質の傾向を測る。
- 抽出で欠陥率が上がったら、その生成元(プロンプト・モデル・設定)を是正する。
- 高リスク区分には**サンプリングを使わない**(全数を見る)。

::: info サンプリングは品質の代用監視であって保証ではない
サンプリングは「この生成プロセスがおおむね健全か」を測る手段です。個々の高リスク変更の正しさを保証するものではありません。**重要・不可逆な箇所は必ず全数レビュー**してください。
:::

### 6. AIで一次パス → 人が最終

AIレビューを**前段フィルタ**として使い、機械的な指摘(規約違反・明白なバグ・抜けたテスト)を人間の前で処理させます。

- AIレビューボットが一次パスし、機械的な指摘を出す([コードレビューBot](/practice/code-review-bot))。
- 人間は**AIが指摘しきれない設計・意図・妥当性・責任判断**に集中する。
- AIレビューは人間レビューの**代替ではなく削減**。最終承認と責任は人間が負う。

::: warning AIレビューでAI生成物を「自己採点」させない閉ループに注意
AIが生成し、AIがレビューし、人が中身を見ずに通すと、**検証の実体がない閉ループ**になります。AIレビューはあくまで人間の負荷を減らす一次パスであり、最終判断を肩代わりさせてはいけません。
:::

### 7. 検証可能性を設計に織り込む

最も効くのは、**そもそも評価しやすい形で生成させる**ことです。

- テスト容易な構造で生成させ、正しさを機械判定しやすくする。
- 変更を小さく・独立に保ち、差分単独で評価できるようにする。
- 受け入れ基準(DoD)を**生成前に**固定し、評価を基準への照合だけにする([評価の前提](/quality/evaluating-output))。

## レビュー負荷を増やさない生成運用

戦略以前の運用ルールとして、**生成の速度をレビューの速度に合わせる**ことが本質的です。供給を絞ることが、最も確実なボトルネック対策です。

- [ ] まとめて大量生成しない。**レビューが追いつく速度で生成**する。
- [ ] 仕掛り(レビュー待ち)に上限(WIP制限)を設け、超えたら生成を止める。
- [ ] 1つを通してから次を生成する。未レビューの山を積み上げない。
- [ ] 「速く大量に出せる」を**美徳にしない**。価値はマージされ検証された変更の量で測る。

::: tip ボトルネックは消えず、移動する
AIは「書く」ボトルネックを解消しますが、それは**ボトルネックをレビュー・検証・統合へ移しただけ**です。生成を無制限に増やすと、移動先のボトルネックが破綻します。AI駆動開発の習熟とは、この新しいボトルネックを**意図的に律速**する技術でもあります。
:::

## アンチパターン

避けるべき典型を挙げます。詳しくは[アンチパターン](/patterns/anti-patterns)を参照してください。

| アンチパターン | 何が起きるか |
| --- | --- |
| 生成し放題 → レビュー後回し | レビュー待ちが崩壊し、ラバースタンプ・無検証マージに至る |
| 巨大PR | 評価不能。automation bias とレビュー疲れで欠陥が素通りする |
| 全差分を均一レビュー | 高リスク箇所が手薄になり、注意が枯渇する |
| AI生成だから機械チェック免除 | 最も安いふるいを捨て、人間に全負荷をかける |
| AI生成 → AIレビュー → 人は素通り | 検証の実体がない閉ループ。形だけのレビュー |
| サンプリングを高リスクに適用 | 不可逆な欠陥を抽出漏れで見逃す |
| 速さ・生成量を成果指標にする | 未検証コードの山を「成果」と誤認し、手戻りが増える |

## 全体フロー

ここまでの戦略を1枚にまとめます。

```text
[生成] ── WIP制限で律速 ──┐
                          ▼
              [機械的ふるい] フォーマット/リント/型/テスト/SAST/シークレット/依存
                          │ 赤 → 自動差し戻し(人を介さない)
                          ▼ 緑のみ
              [リスク判定] 高 / 中 / 低
              ┌───────────┼───────────────┐
              ▼           ▼               ▼
         高: 全数厚く   中: 8観点を1人   低: サンプリング
         複数人 + テスト  + テストで裏取り   + 機械チェック
              │           │               │
              └─── AI一次パス(機械的指摘)を前段に ───┘
                          ▼
              [人間が最終承認 = 品質に責任を負う]
                          ▼
                      [マージ]
```

## まとめ

- 生成は速くレビューは遅い。放置すると**レビュー崩壊 → 無検証マージ**に至る。
- 量を減らす(分割・機械的ふるい・サンプリング)と、効率を上げる(構造化・濃淡・AI一次パス)の両面で対処する。
- **リスクベースの濃淡**が要。不可逆・高リスクを厚く、低リスクは機械+抽出で足りる。
- 最も確実な対策は、**生成速度をレビュー速度に合わせる**運用(WIP制限)。
- AIレビューは人間の代替でなく**前段フィルタ**。最終承認と責任は人間が負う。
