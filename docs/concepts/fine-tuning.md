# ファインチューニングと拡張

「自社の業務にモデルを合わせたい」と思ったとき、多くのエンジニアが真っ先に**ファインチューニング**を連想します。しかしそれは、たいていの場合**最後に検討すべき手段**です。

モデルを業務に適応させる手段は1つではなく、**コストと侵襲性の異なる階層**をなしています。下の段ほど手軽で可逆的、上の段ほど強力ですが高コストで不可逆。この階層を理解し、**下から順に尽くす**のが熟練者の戦略です。

::: tip 結論を先に
**まずプロンプトとRAGを尽くせ。ファインチューニングは最後の手段。**
ほとんどの業務適応は、適切なプロンプト設計と[RAG](/concepts/rag)で達成できます。ファインチューニングが本当に効くのは、「知識の不足」ではなく「**振る舞い・形式・スタイルが安定しない**」問題が残ったときだけです。
:::

## 適応手段の階層

下から上へ、コストと強度が上がります。

```text
強  ┌─────────────────────────────┐
力  │ ⑤ 継続事前学習(Continued Pre-training) │  ドメイン丸ごと注入・最重量
↑   ├─────────────────────────────┤
    │ ④ ファインチューニング(SFT / LoRA など) │  振る舞い・形式を焼き込む
    ├─────────────────────────────┤
    │ ③ RAG(検索拡張生成)                    │  外部知識を実行時に注入
    ├─────────────────────────────┤
    │ ② Few-shot(プロンプト内の例示)         │  数件の例で形式を示す
    ├─────────────────────────────┤
手  │ ① プロンプト / 指示(Zero-shot)          │  最軽量・即時・可逆
軽  └─────────────────────────────┘
```

### ① プロンプト / 指示(Zero-shot)

役割・制約・出力形式を**自然言語で明示**するだけ。コードもデータも訓練も不要、即時に試せて完全に可逆。多くの課題はここで解決します。詳細は[コンテキスト設計](/prompting/context-design)。

### ② Few-shot(例示)

プロンプトの中に**入力→望ましい出力の例を数件**入れ、形式やトーンを示す。「こう書いてほしい」を言葉で説明しきれないときに有効。コストは例の分のトークンだけ。

### ③ RAG(検索拡張生成)

**知識の不足**を、外部から関連情報を取得して注入することで補う。最新情報・社内固有情報・出典提示が必要なときの第一選択。詳細は[埋め込みとRAG](/concepts/rag)。

### ④ ファインチューニング(SFT / LoRA など)

**追加の訓練データでモデルの重みを調整**し、特定の振る舞い・形式・スタイルを焼き込む。代表的な手法:

| 手法 | 概要 | 特徴 |
| --- | --- | --- |
| **フルファインチューニング(SFT)** | 全パラメータを更新 | 強力だが計算・データ・運用コストが最大 |
| **LoRA / QLoRA(PEFT)** | 少数の追加パラメータのみ学習 | 軽量・安価・差分管理しやすい。実務の主流 |
| **選好最適化(DPOなど)** | 望ましい/望ましくない出力の対で調整 | トーンや安全性、好みの微調整 |

LoRAなどの**PEFT(パラメータ効率的微調整)** により、かつてより敷居は下がりましたが、それでも③までとは別次元の手間がかかります。

### ⑤ 継続事前学習(Continued Pre-training)

未学習の**ドメイン(専門言語・独自フレームワーク・大量の社内コード)** を、大規模な追加学習で取り込む。最も強力かつ最も高コストで、対象も限られます。基盤を持つ一部の組織向けです。

## コスト・効果・適用条件の比較

| 手段 | 初期コスト | 更新の容易さ | 知識の追加 | 振る舞いの矯正 | 可逆性 | 向いている課題 |
| --- | --- | --- | --- | --- | --- | --- |
| ① プロンプト | 極小 | 即時 | △ | △ | ◎ | まず全部ここで試す |
| ② Few-shot | 小 | 即時 | △ | ○ | ◎ | 出力**形式**を例で示したい |
| ③ RAG | 中 | 容易(再インデックス) | ◎ | △ | ○ | **最新/固有知識**が要る、出典が要る |
| ④ ファインチューニング | 大 | 困難(再学習) | △〜○ | ◎ | △ | **振る舞い/形式が安定しない**、毎回同じ指示を繰り返している |
| ⑤ 継続事前学習 | 極大 | 極めて困難 | ◎ | ○ | × | 独自言語・巨大ドメインの根本的習得 |

::: warning ファインチューニングは「知識追加」に向かないことが多い
直感に反しますが、**事実知識を覚えさせる目的でのファインチューニングは費用対効果が悪い**ことが多いです。データが古くなれば再学習が必要で、覚えた事実も[ハルシネーション](/quality/hallucination)同様に曖昧化・捏造されうる。**新しい事実を確実に・更新可能に与えたいなら、ファインチューニングよりRAG**です。ファインチューニングが本領を発揮するのは「知識」ではなく「**振る舞い・形式・スタイル**」の安定化です。
:::

## どう選ぶか:意思決定フロー

```text
業務にモデルを合わせたい
        │
  まずプロンプトで試したか? ──No──→ ① プロンプトを設計して試す
        │ Yes(まだ不足)
  出力の「形式・例」の問題か? ──Yes─→ ② Few-shot を足す
        │ No
  「最新・社内固有の知識」の不足か? ─Yes→ ③ RAG を構築する
        │ No(知識ではなく振る舞いが不安定)
  形式/トーン/手順が指示しても安定しないか?
        │ Yes、かつ十分な品質の訓練データがある
        ▼
  ④ ファインチューニング(まず LoRA から小さく)
        │ さらに、独自言語・巨大ドメインを根本から
        ▼
  ⑤ 継続事前学習(限られたケース)
```

### ④を選ぶ前のチェックリスト

ファインチューニングに進む前に、次がすべて埋まっているか確認します。1つでも欠けるなら、たいてい下の段に戻るべきです。

- [ ] プロンプトとFew-shotを**本気で**作り込んだか?(指示が曖昧なだけのことが多い)
- [ ] 課題は「知識の不足」ではなく「**振る舞い/形式の不安定さ**」か?(知識ならRAG)
- [ ] **質の高い訓練データ**(数百〜数千件規模、一貫性のある入出力対)を用意できるか?
- [ ] 評価セットと**回帰評価の仕組み**があるか?(改善したかを測れないと改善できない)
- [ ] モデル更新時の**再学習・運用コスト**を引き受けられるか?

::: info RAGとファインチューニングは排他ではない
両者は競合ではなく**補完**です。実務では「**ファインチューニングで振る舞いを固め、RAGで最新知識を注入**する」併用が強力です。「どちらか」ではなく「役割分担」で考えます。
:::

## アンチパターン

::: warning よくある失敗
- **いきなりファインチューニング** — プロンプトを詰める前に重い手段へ飛び、コストだけかかって効果が薄い。
- **知識注入目的のファインチューニング** — 古くなる・捏造する。RAGにすべき。
- **少量・低品質データでの微調整** — かえって性能が劣化(過学習・スタイル崩壊)する。
- **評価なしの微調整** — 良くなったか測れず、回帰に気づけない。
- **再現性・更新計画の欠如** — ベースモデルが更新されたとき、作り直せない。
:::

## まとめ

- 適応手段は**プロンプト → Few-shot → RAG → ファインチューニング → 継続事前学習**の階層。
- **下から順に尽くす**。多くの課題はプロンプトとRAGで解ける。
- ファインチューニングは「**知識**」ではなく「**振る舞い・形式**」の安定化に効く。最後の手段。
- RAGとファインチューニングは**補完関係**——固めるのは振る舞い、注入するのは知識。
- どの段でも**評価の仕組み**が前提。測れないものは改善できない。
