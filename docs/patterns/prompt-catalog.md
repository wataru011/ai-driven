# プロンプトパターンカタログ

[プロンプトの基礎](/prompting/fundamentals)と[実践テクニック](/prompting/techniques)で原理を押さえたら、次に欲しくなるのは**手を動かすときにそのまま引ける引き出し**です。このページは、実務で繰り返し登場するプロンプトを「デザインパターン」として整理したカタログです。

各パターンは **名前 / 目的 / いつ使う / テンプレート / 例 / 注意** の統一フォーマットで記述しています。テンプレートはコピーして埋めればそのまま使える形にしてあります。GoFのデザインパターンと同じく、**名前で会話できる**ようにすることがこのカタログの裏の目的です。「ここは自己レビューパターンを挟もう」と言えるチームは、プロンプトを設計対象として扱えています。

::: tip 使い方
パターンは排他的ではありません。実務では「役割付与 + 仕様明確化 + 出力フォーマット指定 + テスト先行」のように複数を**重ねて**使います。1つのパターンを覚えるより、組み合わせの勘所を掴んでください。
:::

## 1. 役割付与 (Role Priming)

**目的** — モデルの広大な能力空間から、タスクに関連する領域へ注意を集中させる。専門知識が増えるのではなく、既にある知識の**重み付け**が変わる。

**いつ使う** — ほぼすべてのプロンプトの前置きとして。特に品質基準やドメインの作法を効かせたいとき。

**テンプレート**

```text
あなたは{ドメイン}に精通した{役割}です。
{重視してほしい価値観・観点}を特に重視します。
```

**例**

```text
あなたはセキュリティに厳格なバックエンドエンジニアです。
入力検証、最小権限、機密情報の取り扱いを特に重視します。
以下のログイン処理をレビューしてください。
```

::: warning 注意
役割は「効く方向づけ」であって「正しさの保証」ではありません。「あなたは世界一の…」のような誇張は精度に寄与せず、トークンの無駄です。重視させたい**具体的な観点**を書くほうが効果的です。
:::

## 2. 仕様明確化 (Spec First)

**目的** — 「何が問題で、どうなったら成功か」を曖昧さなく言語化する。最も省略されがちで最も効果が大きい。

**いつ使う** — 常に。特に「直して」「良くして」と書きたくなったとき。

**テンプレート**

```text
## 目的
{達成したいこと}

## 現状
{今どうなっているか}

## 成功条件
- {検証可能な条件1}
- {検証可能な条件2}
```

**例**

```text
## 目的
このリスト処理を、要素数100万件でもメモリ使用量が一定になるよう変更する。

## 現状
全件をメモリに読み込んでからフィルタしている(下記)。

## 成功条件
- ピークメモリが入力サイズに依存しない(ストリーミング)
- 出力順序は入力順を維持する
- 既存のテストが全て通る
```

::: tip
「きれいに」「効率的に」「ちゃんと」は検証不能なので、必ず**測定可能な基準**へ翻訳します。詳しくは[基礎](/prompting/fundamentals)を参照。
:::

## 3. Few-shot (例示)

**目的** — 期待する品質や形式を、言葉で説明する代わりに**入出力ペアの見本**で示す。

**いつ使う** — 出力の形式・スタイル・粒度を揃えたいとき。分類やフォーマット変換のような定型タスク。

**テンプレート**

```text
以下の例にならって変換してください。

入力: {例1の入力}
出力: {例1の出力}

入力: {例2の入力}
出力: {例2の出力}

入力: {本番の入力}
出力:
```

**例**

```text
コミットメッセージを Conventional Commits 形式に直してください。

入力: ログイン画面のボタン色を変えた
出力: style(login): adjust sign-in button color

入力: ユーザー検索が遅いのを直した
出力: perf(search): speed up user lookup with index

入力: 課金APIにリトライを入れた
出力:
```

::: warning 注意
例は2〜5個で十分なことが多く、増やすほどトークンを食います。**境界やエッジを含む代表例**を選ぶのがコツ。悪い例を混ぜたいときは「これはNG例」と明示しないと、模倣されます。
:::

## 4. Chain-of-Thought (思考の明示)

**目的** — 結論の前に推論過程を出させ、論理の飛躍やバグを減らす。複雑な判断ほど効果が大きい。

**いつ使う** — アルゴリズム設計、複雑な条件分岐、トレードオフのある判断。

**テンプレート**

```text
{問題}

最終的な実装を書く前に、次の順で考えを明示してください。
1. 制約と前提の整理
2. 考えられるアプローチと却下理由
3. 選んだアプローチの根拠
その上で実装を示してください。
```

**例**

```text
1000万件のイベントログから、直近1時間のユニークユーザー数を
低メモリで概算したい。

実装の前に、(1)正確に数える場合の問題点、(2)候補となる確率的
データ構造とその誤差特性、(3)採用案の根拠 を順に述べてから、
実装を示してください。
```

::: tip
「考えてから書いて」だけでも効きますが、**考える軸を指定**するとブレが減ります。自動処理パイプラインでは思考が邪魔になるため、最後に「思考は出力に含めず結論のみ」と切り替えることもあります。
:::

## 5. ステップ分解 (Decomposition)

**目的** — 大きなタスクを、検証可能な小さい順序付きステップに割る。一気に巨大な成果物を出させない。

**いつ使う** — 複数ファイルにまたがる実装、見通しの悪い機能追加。

**テンプレート**

```text
{ゴール}を実装したい。

まず実装を**ステップに分解した計画**だけを提示してください。
各ステップは独立してレビュー・テストできる粒度にすること。
私が計画を承認したら、ステップ1から順に実装します。
```

**例**

```text
記事にコメント機能を追加したい。

まず計画だけを出してください。各ステップは
「DBスキーマ」「API」「バリデーション」「テスト」のように
独立してレビューできる単位に分けること。コードはまだ書かないでください。
```

::: warning 注意
計画段階で**コードを書かせない**ことが肝心です。承認なしに突き進ませると、後半の手戻りが大きくなります。委任の粒度は[実装フェーズ](/workflow/implementation)で詳述します。
:::

## 6. 自己レビュー/批評

**目的** — 生成直後の出力を、別の観点でモデル自身に検証させ、ミスを自己修正させる。

**いつ使う** — 一発生成の品質に不安があるとき。レビュー前の足切りとして。

**テンプレート**

```text
{先に生成させた成果物}

上記を、次の観点で批判的にレビューしてください。
- 仕様の取りこぼし
- エッジケース(null/空/境界値/並行性)
- セキュリティ上の懸念
見つかった問題を列挙し、修正版を出してください。
```

**例**

```text
先ほど書いた decode() 関数を、不正な入力(空文字、想定外の
エンコーディング、巨大入力)の観点で自己レビューし、
問題があれば修正版を出してください。
```

::: tip
自己レビューは**別ターン**に分けると効きやすい傾向があります。生成と検証を1メッセージに詰め込むより、いったん出力させてから「これをレビューして」と返すほうが、批判的な視点が立ちやすいです。ただし自己レビューは人間レビューの代替ではありません。
:::

## 7. 制約付き生成

**目的** — 使ってよいもの・触ってよい範囲を明示し、解の探索空間を縛って事故を防ぐ。

**いつ使う** — 依存を勝手に増やされたくない、特定ファイルだけ変えたい、互換性を保ちたいとき。

**テンプレート**

```text
## 制約
- 使用可能: {言語/バージョン/ライブラリ}
- 追加禁止: 新規依存ライブラリ
- 変更可: {ファイル/関数}
- 変更不可: 公開API・既存のシグネチャ
これらを破る必要がある場合は、実装せず理由を述べて確認すること。
```

**例**

```text
## 制約
- Python 3.12 / 標準ライブラリのみ
- 変更してよいのは utils/date.py のみ
- parse_date() の引数と戻り値の型は変えない
上記が満たせない場合は、コードを書かずに相談してください。
```

::: warning 注意
最後の一文「破る必要があれば相談」が効きます。これがないと、制約を満たせないとき**黙って制約を破る**ことがあります。
:::

## 8. 出力フォーマット指定

**目的** — 成果物の形を固定し、後工程(適用・パース・自動処理)を機械化しやすくする。

**いつ使う** — 差分をそのまま適用したい、JSONをパースしたい、CIに食わせたいとき。

**テンプレート(diff)**

```text
変更は unified diff 形式のみで出力してください。
説明文・コードフェンス外のテキストは付けないこと。
git apply できる正確なパスとハンクヘッダを含めること。
```

**テンプレート(JSON)**

```text
次のスキーマの JSON のみを出力してください(前後の文章なし)。
{
  "summary": string,
  "risks": string[],
  "confidence": number   // 0.0〜1.0
}
```

**例**

```json
{
  "summary": "認証ミドルウェアの順序バグを修正",
  "risks": ["既存セッションの再ログインが必要になる可能性"],
  "confidence": 0.8
}
```

::: tip
JSONを確実に取りたいときは「**前後の文章を付けない**」と明示し、可能なら実際の構造を例示します。それでも崩れることはあるため、パースは失敗前提でガードを入れます。
:::

## 9. リファレンス駆動

**目的** — 既存の実物(ファイル・関数・PR)を見本として渡し、命名・構造・作法を踏襲させる。

**いつ使う** — リポジトリの規約に揃えたいとき。「このファイルに合わせて」が最も確実。

**テンプレート**

```text
下記の既存ファイル {path} のスタイル(命名・エラー処理・
テストの書き方・ディレクトリ構成)に厳密に合わせて、
{新機能} を実装してください。

--- 参照: {path} ---
{既存コードを貼る}
```

**例**

```text
下記の handlers/user.go と同じ構造・命名・エラーハンドリングで、
handlers/order.go を新規作成してください。ロガーの使い方も踏襲すること。

--- 参照: handlers/user.go ---
（ここに既存コードを貼る）
```

::: tip
規約を**言葉で長々説明するより実物1つ**のほうが、トークン効率も精度も上です。何を見本に選ぶかは[コンテキスト設計](/prompting/context-design)の核心です。
:::

## 10. テスト先行 (Test First)

**目的** — 実装より先にテストを書かせ、仕様を実行可能な形で固定する。実装はテストを満たす作業に変わる。

**いつ使う** — 仕様が固まっている機能。回帰を防ぎたいバグ修正。

**テンプレート**

```text
{機能}について、まず**失敗するテスト**だけを書いてください。
正常系・異常系・境界値を含めること。実装はまだ書かないでください。
テストを承認したら実装に進みます。
```

**例**

```text
slugify(text) の仕様:
- 英数字とハイフンのみ、小文字化、連続空白は1ハイフン
- 先頭末尾のハイフンは除去、空入力は空文字
この仕様を満たすテストを先に書いてください。実装は後で。
```

::: tip
テストが**仕様の合意**になります。テストの段階で人間がレビューすれば、実装の正しさはテストで機械的に確認できます。詳細は[実装フェーズ](/workflow/implementation)へ。
:::

## 11. リファクタリング指示

**目的** — 外部から見た振る舞いを変えずに、内部構造を改善させる。

**いつ使う** — 重複除去、関数分割、可読性向上、命名整理。

**テンプレート**

```text
下記コードを、**振る舞いを一切変えずに**リファクタしてください。
目的: {重複除去 / 責務分割 / 可読性 など}
不変条件:
- 公開APIのシグネチャは維持
- 既存テストは変更せず全て通ること
変更は構造のみ。機能追加・仕様変更はしないこと。
```

**例**

```text
このネストの深い validate() を、早期リターンとヘルパー関数で
平坦化してください。振る舞いとシグネチャは不変。
既存のテストはそのまま通ること。
```

::: warning 注意
リファクタリングでは「機能追加しない」を明示しないと、ついでに仕様を変えられることがあります。**テストを固定**しておくのが安全装置になります。
:::

## 12. バグ修正(再現付き)

**目的** — 症状だけでなく**再現条件と期待動作**を渡し、推測でなく根拠ある修正を引き出す。

**いつ使う** — バグ報告から修正させるとき。

**テンプレート**

```text
## 症状
{エラーメッセージ / 観測された誤動作}

## 再現手順
{入力・操作・前提}

## 期待動作
{本来こうなるべき}

## 対象コード
{該当箇所}

## 出力
原因の説明 → 修正 → 再発防止のテスト
```

**例**

```text
## 症状
getUser() で API がユーザーを返さないと
`Cannot read property 'id' of undefined` が出る。

## 再現手順
存在しない id を渡す。

## 期待動作
存在しなければ null を返す。例外は投げない。戻り値型は User | null に変更可。

## 対象コード
（該当箇所を貼る）

## 出力
原因 → 修正コード → null ケースを検証するテスト
```

::: tip
「原因の説明を先に」させると、的外れな修正を早期に検知できます。原因が腑に落ちなければ、その修正は採用しないでください。
:::

## 13. コードレビュー依頼

**目的** — 観点を明示してレビューさせ、漫然とした「良いと思います」を防ぐ。

**いつ使う** — 自分の書いた/生成されたコードの一次チェック。

**テンプレート**

```text
下記の差分を、次の観点でレビューしてください。
- 正しさ(ロジック・エッジケース)
- 並行性・リソースリーク
- セキュリティ(入力検証・権限・機密)
- 可読性・命名
各指摘に「重大度(高/中/低)」と「具体的な修正案」を付けること。
問題がなければ「問題なし」と明記すること。
```

**例**

```text
このPRの差分をレビューしてください。特に SQL 組み立て箇所の
インジェクション耐性と、トランザクション境界の正しさを重点的に。
指摘は重大度付きで、修正案も添えてください。
```

::: warning 注意
重大度と修正案を要求しないと、些末な指摘と重大な指摘が同列に並びます。「問題なければ問題なしと書く」を入れないと、無理に何か指摘を作ることがあります。
:::

## 14. 説明生成

**目的** — コードや概念を、対象読者のレベルに合わせて解説させる。

**いつ使う** — 他人のコードの把握、引き継ぎ資料、学習。

**テンプレート**

```text
下記コードを {読者像} 向けに説明してください。
- 全体の責務を1段落で
- 重要な処理を上から順に
- 非自明な箇所・落とし穴を明示
推測が混じる場合は「推測」と明記すること。
```

**例**

```text
このレートリミッタの実装を、チームに新しく入った中堅エンジニア
向けに説明してください。トークンバケットの考え方と、なぜこの
ロック範囲なのかを重点的に。確証がない箇所は推測と明記すること。
```

::: tip
「推測は推測と明記」が重要です。コードに書かれていない意図(なぜこうしたか)は、モデルが**もっともらしく捏造**しがちです。
:::

## 15. 比較検討(複数案)

**目的** — 単一の答えでなく、複数の選択肢とトレードオフを並べさせ、意思決定の材料にする。

**いつ使う** — 設計判断、ライブラリ選定、アーキテクチャの分岐点。

**テンプレート**

```text
{課題}に対するアプローチを2〜3案、出してください。
各案について:
- 概要
- 長所 / 短所
- 向くケース / 向かないケース
最後に、{前提・制約}を踏まえた推奨案と理由を述べること。
```

**例**

```text
セッション管理を Cookie + サーバセッション にするか JWT にするか、
両案を長所短所つきで比較し、「スケールアウト前提・ログアウト即時
無効化が必要」という制約での推奨を述べてください。
```

::: tip
「推奨を最後に」させると、選択肢を公平に並べた上での結論になります。前提・制約を渡さないと、一般論しか返りません。
:::

## 16. 命名相談

**目的** — 変数・関数・型の命名候補を、根拠付きで複数出させる。

**いつ使う** — 名前が決まらないとき。ドメイン用語を揃えたいとき。

**テンプレート**

```text
{役割の説明}を表す {変数/関数/型} の名前を5案、出してください。
各案に「ニュアンス」と「避けたい誤解」を添えること。
命名規約: {camelCase / snake_case など}、ドメイン用語: {あれば}
```

**例**

```text
「未処理だが受理済みの注文」を表すステータス値の名前を5案。
各案のニュアンスと誤解の余地を添えて。snake_case、ECドメイン。
```

::: tip
候補だけでなく**ニュアンスの違い**を出させると、自分の意図に最も近いものを選べます。最終決定は人間が行う前提のパターンです。
:::

## 17. 移行/変換

**目的** — コードを別の言語・形式・APIバージョンへ機械的かつ忠実に移し替えさせる。

**いつ使う** — 言語移植、設定形式変換(YAML↔JSON)、非推奨APIの置換。

**テンプレート**

```text
下記を {変換元} から {変換先} へ移行してください。
- 振る舞いを変えないこと(等価変換)
- {変換先}の慣用的な書き方に従うこと
- 自明でない判断をした箇所はコメントで明示
変換できない/意味が変わる箇所は、勝手に補わず指摘すること。
```

**例**

```text
この Express(JS)のルータを Fastify に移行してください。
ミドルウェアの順序とエラーハンドリングの意味を保つこと。
等価でない箇所が出たら、補完せず指摘してください。
```

::: warning 注意
「等価でない箇所は指摘」を入れないと、辻褄合わせのために**意味を変える補完**が混入します。移行は差分が大きくなりがちなので、小さく分割すると安全です。
:::

## 18. 要約

**目的** — 長い文書・ログ・差分を圧縮し、後続の判断やコンテキストとして使えるようにする。

**いつ使う** — 長い会話の引き継ぎ、巨大ログの調査、PR概要の生成。

**テンプレート**

```text
下記を、{目的}のために要約してください。
- {観点}に絞ること
- {字数 / 箇条書きN個}以内
- 数値・固有名・結論など、復元不能な情報は落とさないこと
```

**例**

```text
このエラーログ(2000行)を、原因調査の観点で要約してください。
最初に出た異常、繰り返し出るエラー、最後の致命的例外を、
タイムスタンプ付きで5項目以内に。
```

::: tip
要約は**何のための要約か**で取捨選択が変わります。「コンテキスト節約のための要約」は[コンテキストエンジニアリング](/patterns/context-engineering)の中核技法です。
:::

## 19. エラー解決

**目的** — スタックトレースやエラー出力から、原因の仮説と検証手順を引き出す。

**いつ使う** — 原因が不明なエラー。環境依存のトラブル。

**テンプレート**

```text
## エラー
{スタックトレース / エラー出力をそのまま}

## 環境
{言語/バージョン/OS/関連ライブラリ}

## 直前にやったこと
{変更や操作}

考えられる原因を**可能性の高い順**に挙げ、各原因の
切り分け方法(確認コマンドやログの見方)を示してください。
```

**例**

```text
## エラー
（スタックトレースを貼る）
## 環境
Node 20 / Docker / pg 8.x
## 直前にやったこと
接続プールのサイズを上げた

可能性の高い順に原因と、それぞれの切り分け方法を示してください。
```

::: warning 注意
1つの「これが原因です」を鵜呑みにしないこと。**複数仮説 + 切り分け手順**を出させ、自分で確認するのが安全です。無限リトライ(同じ修正を投げ続ける)に陥らないよう、原因が分かるまでコードを変えないのも手です。
:::

## 20. 段階的詳細化 (Progressive)

**目的** — いきなり完成形を求めず、骨子→肉付け→詳細と反復し、各段階でレビューしながら育てる。

**いつ使う** — 設計が固まっていない機能。探索的な実装。

**テンプレート**

```text
{テーマ}について、まず**骨子(インターフェースとデータ構造)だけ**を
提示してください。実装の中身は空(TODO)で構いません。
私がレビューしたら、合意した部分から中身を埋めていきます。
```

**例**

```text
通知システムの設計をしたい。まずは型定義と公開インターフェース、
主要コンポーネントの責務だけを示してください。実装は TODO で。
そこから一緒に詰めていきます。
```

::: tip
ステップ分解(#5)が「タスクを横に割る」のに対し、段階的詳細化は「同じ対象を縦に深める」パターンです。設計の方向性を早期に合意できるため、手戻りが小さくなります。
:::

## パターンの組み合わせ例

実務では単独で使うことは少なく、重ねて使います。代表的なスタックを挙げます。

| 場面 | パターンの重ね方 |
| --- | --- |
| 規約に沿った新規実装 | 役割付与 + 仕様明確化 + リファレンス駆動 + テスト先行 |
| 信頼できるバグ修正 | バグ修正(再現付き) + 制約付き生成 + 自己レビュー |
| 設計判断 | 比較検討 + Chain-of-Thought + 段階的詳細化 |
| 自動処理パイプライン | 仕様明確化 + 出力フォーマット指定(JSON) + 制約付き生成 |
| 大きな機能追加 | ステップ分解 → 各ステップで(リファレンス駆動 + テスト先行) |

::: warning やりすぎ注意
パターンを盛りすぎると、プロンプトが長大化してかえって焦点がぼやけます。**必要な最小セット**から始め、出力が不満なら1つずつ足すのが定石です。長文プロンプトを磨き続ける「プロンプト沼」は[アンチパターン集](/patterns/anti-patterns)で扱います。
:::

## まとめ

- プロンプトは**名前のある再利用部品**として扱える。チームで名前を共有すると会話が速くなる。
- 各パターンは「目的・適用条件・テンプレート・例・注意」で押さえる。
- 単独でなく**重ねて**使うのが実務。必要最小から始める。
- どのパターンも、人間による**検証**を置き換えるものではない。
