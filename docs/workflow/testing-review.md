# テストとレビュー

テストとレビューは、AI駆動開発における**検証ファースト**([ワークフロー全体像](/workflow/overview))の中核です。皮肉なことに、ここはAIが大いに役立つと同時に、**最も過信が危険**なフェーズでもあります。「AIがテストを書いて、AIがレビューして、全部通った」——この状態は安心材料ではなく、警戒すべき状況です。

::: tip このフェーズの逆説
AIはテストを書ける。だが**自分が書いたコードのテストを書かせると、同じ盲点を共有する**ため、見るべき所を見ないテストになりがちです。検証する側とされる側を意識的に分離するのが鍵です。
:::

## テスト生成

### ユニットテスト

最も基本的な用途です。ただし「テスト書いて」だけでは正常系に偏ります。**観点を指定**します。

```text
この関数のユニットテストを生成してください。

必ず含める観点:
- 正常系(代表的な入力)
- 境界値(空、1件、上限)
- 異常系(null、型不一致、範囲外)
- 副作用の検証(呼ばれるべきものが呼ばれたか)

テストは「何を検証しているか」が名前で分かるようにすること。
```

### エッジケースの洗い出し

実装より先に、**起こりうる異常系を列挙させる**使い方が有効です。網羅性の確認に向きます。

```text
この決済処理関数について、考えられるエッジケース・障害シナリオを
網羅的に列挙してください(まだテストコードは不要)。

観点: 金額(0、負、極大)、通貨、同時実行、ネットワーク失敗、
冪等性、部分失敗時の状態、タイムアウト。
列挙後、各ケースの危険度(高/中/低)を付けてください。
```

人間が列挙を確認・取捨選択してから、テストコードを生成させます。

### プロパティベーステスト

個別の入出力ではなく、**常に成り立つべき性質(不変条件)**を検証します。AIは性質の言語化を手伝えます。

```text
この sort 関数について、プロパティベーステストを提案してください。
例: 出力は昇順 / 出力は入力の順列 / 冪等(2回適用しても不変)。

性質を列挙し、fast-check を使ったテストコードに落としてください。
```

## カバレッジの罠

カバレッジ100%は**テストが十分な証拠ではありません**。これはAI生成テストで特に深刻になります。

::: warning カバレッジは「実行された行」しか語らない
カバレッジは「コードが実行されたか」を測るだけで、「**正しさを検証したか**」は測りません。アサーションが弱い、または無いテストでもカバレッジは上がります。
:::

```ts
// カバレッジは上がるが、何も検証していないアンチパターン
test('processes order', () => {
  processOrder(sampleOrder);  // 例外が出ないことしか確認していない
  // アサーションが無い!
});
```

AIにテストを大量生成させると、カバレッジは稼げても**アサーションが空虚**なテストが紛れ込みます。確認すべきは行数ではなく、「このテストは**何が壊れたら落ちるか**」です。

::: tip ミューテーションテストで検証する
テストの質を機械的に測るなら、ミューテーションテスト(コードをわざと壊してテストが落ちるか確認する手法)が有効です。「壊しても落ちないテスト」は無意味なテストです。
:::

## AIによるコードレビュー

### 活用できる領域

AIレビューは、**機械的・定型的な観点**で高速かつ網羅的に働きます。

- 入力検証・null安全・例外時のリソースリーク
- 明白なロジックミス、オフバイワン
- 規約逸脱、命名の一貫性
- 典型的なセキュリティ問題(SQLインジェクション、ハードコードされた秘密情報)

```text
この diff をシニアレビュアーとしてレビューしてください。

観点ごとに指摘を分類:
- 🔴 バグ・セキュリティ(必ず直す)
- 🟡 設計・保守性(検討)
- 🔵 スタイル・好み(任意)

各指摘に「なぜ問題か」と「修正案」を添えること。
問題がなければ正直に「問題なし」と述べること(無理に指摘を作らない)。
```

### 限界

::: warning AIレビューが見落とすもの
- **設計・アーキテクチャの妥当性** — 「この機能は本当に必要か」「この抽象は適切か」という上位の判断。
- **ビジネス文脈** — 仕様そのものの正しさ、要件との整合。
- **暗黙の制約** — チームの運用事情、過去の経緯、政治的判断。
- **自分の盲点** — AIが書いたコードをAIがレビューすると、同種の誤りを見逃す。
:::

「指摘を作れ」と圧をかけると、**些末な指摘を捏造**することがあります。「問題がなければ問題なしと言え」と明示し、指摘の質を保ちます。

## 人間レビューとの役割分担

AIレビューと人間レビューは代替ではなく**補完**です。役割を分けます。

| レビュー観点 | 主担当 | 理由 |
| --- | --- | --- |
| 構文・規約・明白なバグ | AI(一次) | 高速・網羅的・退屈 |
| null安全・例外処理 | AI(一次)→人間確認 | 機械的に拾える |
| 設計・アーキテクチャ | 人間 | 文脈と将来を見る判断 |
| 要件との整合 | 人間 | AIは仕様の正しさを知らない |
| トレードオフの妥当性 | 人間 | 責任を負う者が判断 |

::: tip AIは「一次レビュー」、人間は「最終承認」
AIに退屈で機械的な一次レビューを任せ、**人間は浮いた時間を設計・要件レベルのレビューに使う**。これが役割分担の理想形です。AIがレビューしたから人間は見なくてよい、ではありません。
:::

## 生成テストを信用しすぎない

最後に、このフェーズで最も重要な原則です。

::: warning 「テストが通った」は終わりではなく始まり
AIが生成したテストがパスしても、それは「**AIが想定した範囲では動く**」ことしか意味しません。テストの観点自体が間違っていれば、誤った実装が"検証済み"の顔をして通過します。
:::

防御策:

1. **テストを人間がレビューする** — 何を検証しているか、観点に抜けはないかを確認する。実装より先にテストをレビューするのが理想([実装フェーズ](/workflow/implementation))。
2. **生成元を分ける** — 可能なら、実装とテストを別の文脈/別のセッションで生成し、盲点の共有を避ける。
3. **アサーションの質を見る** — カバレッジではなく「何が壊れたら落ちるか」で評価する。
4. **重要パスは手で1本書く** — 最重要の振る舞いは、人間が自分の理解でテストを1本書き、AIの理解と突き合わせる。

## アンチパターン

::: warning テストとレビューの失敗例
- **カバレッジ至上主義**: 数値を追い、空虚なアサーションを量産する。
- **自己採点**: 実装したAIに、そのテストとレビューを丸ごと任せる。
- **指摘の捏造を許す**: 「必ず3つ指摘して」と圧をかけ、些末な指摘を生む。
- **AIレビュー = 人間不要**: 設計・要件レビューまでAIに委ねる。
- **通った=正しい**: テストパスを最終ゴールと誤認する。
:::

## まとめ

- テスト生成は**観点を指定**し、エッジケースを先に洗い出す。
- カバレッジは正しさを保証しない。**何が壊れたら落ちるか**で評価する。
- AIレビューは一次レビュー、人間は設計・要件・最終承認を担う。
- **生成テストを過信しない**。レビューし、生成元を分け、重要パスは手で書く。
