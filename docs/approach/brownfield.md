# 既存システム改修のアプローチ

新規開発(グリーンフィールド)では、AIは「まだ無いものを生み出す」加速装置として働きます。既存システムの改修(ブラウンフィールド)では話が変わります。**すでに動いているコードが存在し、それが唯一の真実**です。あなたがやろうとしているのは「作る」ことではなく、「**動いている何かを、壊さずに変える**」ことです。

この違いは決定的です。ブラウンフィールドでの最大の作業は、コードを書くことではなく、**既存コードを理解すること**だからです。そして、AIが最も効くのはまさにこの「読解の加速」です。

::: tip このページの位置づけ
ブラウンフィールドとグリーンフィールドの性質の違いそのものは[新規開発 vs 既存改修](/approach/greenfield-vs-brownfield)で扱います。本ページは「既存改修を、AIを使ってどう進めるか」という実務に踏み込みます。リバースエンジニアリングの具体的な手順と成果物は[リバースエンジニアリングと作るべきドキュメント](/approach/reverse-engineering)、大規模な置換・移行は[レガシー移行・大規模変更](/practice/legacy-migration)に続きます。
:::

## 本質: コードが「正」であり、最大の作業は理解

既存システムには、長い時間をかけて積み上がった「現実」が詰まっています。仕様書には書かれていない条件分岐、誰かが障害対応で入れた一行、もう退職した人が想定していた前提。これらはすべて**コードの中にだけ正確に存在**します。

ここから、ブラウンフィールド改修の第一原則が導かれます。

::: warning 第一原則: ドキュメントは仮説、コードが事実
仕様書・設計書・コメント・Wiki — これらはすべて「**過去のある時点での意図**」にすぎません。コードはその後も変更され続けています。両者が食い違っているとき、**正しいのは常にコード(と、それが実際に示す挙動)**です。ドキュメントは「こうなっているはず」という仮説として扱い、コードと実行結果で検証します。
:::

この原則は、AIの使い方をそのまま規定します。AIに既存システムを理解させるとは、**「コードという事実」を、人間とAIが扱いやすい形に翻訳させる**ことです。新しい事実を作り出させることではありません。

| 観点 | グリーンフィールド | ブラウンフィールド |
| --- | --- | --- |
| 真実の源 | これから決める設計 | 既存のコードと実挙動 |
| 最大の作業 | 設計と実装 | **既存の理解** |
| AIの主な効きどころ | 生成の加速 | **読解の加速** |
| 最大のリスク | 過剰設計・方向の誤り | 既存挙動の破壊・副作用の見落とし |
| 安全網 | これから書くテスト | 現挙動を固定する特性化テスト |

### AIの効きどころは「読解の加速」

数万行のコードを人間が読むには何日もかかります。AIは、その読解を桁違いに速くします。具体的には次のような作業です。

- このモジュールが「何をしているか」を数段落で要約させる
- ある関数がどこから呼ばれ、何に影響するかを追わせる
- 散らばった条件分岐から、暗黙のビジネスルールを言語化させる
- 「この変更をしたら、どこに影響が及びそうか」を当たりをつけさせる

ただし、ここには常に但し書きがつきます。**AIの理解は推論であり、誤り(ハルシネーション)を含みうる**ということです。AIが「このコードはユーザーIDで重複排除しています」と言っても、それは仮説です。実コードと実挙動で裏を取るまで、事実として扱ってはいけません([ハルシネーション対策](/quality/hallucination))。

::: tip 「読ませて要約させる」は最高のコスパ、ただし裏取り必須
AIによる読解の加速は、ブラウンフィールドで最もリターンの大きいAI活用です。一方で、要約の流暢さと正確さは別物です。重要な判断に関わる理解は、必ず実コード・テスト・実行ログで確認します([出力の評価](/quality/evaluating-output))。
:::

## 既存ドキュメントをどれだけ活かすか

既存システムには、多くの場合なんらかのドキュメントが付いています。仕様書、設計書、コードコメント、README、Wiki、コミット履歴、ADR、課題管理チケット。これらをAIへのコンテキストとして与えれば、理解は加速します。

問題は、**それらをどこまで信じるか**です。

### ドキュメントの信頼度を評価する

既存ドキュメントを使う前に、まずその信頼度を見積もります。評価軸は3つです。

1. **最新か** — いつ更新されたか。直近のコード変更より古ければ、その分だけ陳腐化している可能性がある。
2. **コードと一致するか** — サンプルでいいので、記述とコードを突き合わせる。1箇所でも明確な食い違いがあれば、文書全体の信頼度を下げる。
3. **陳腐化していないか** — 「もう存在しないモジュール」「廃止されたフラグ」への言及がないか。死んだ記述が混じる文書は、生きた記述も疑わしい。

::: warning 古い仕様書は「無いより危険」なことがある
何もないところから読むなら、人は慎重にコードを追います。しかし「もっともらしい古い仕様書」があると、人もAIも**それを前提に読んでしまい、現実とのズレを見落とします**。陳腐化したドキュメントは、単に役に立たないのではなく、**積極的に誤誘導する**ことがあります。鵜呑みは禁物です。
:::

### 信頼度の高い順に使い、矛盾はコードで検証する

すべてのドキュメントを同格に扱ってはいけません。**コードに近く、改ざんされにくいものほど信頼度が高い**という序列があります。

| ソース | 信頼度の傾向 | 理由・使い方 |
| --- | --- | --- |
| 実コード・実挙動 | 最高(=事実) | これが正。すべての検証の基準 |
| テストコード | 高 | 「こう動くべき」を実行可能な形で固定している。ただしテスト自体が古い可能性は残る |
| 型定義・スキーマ・OpenAPI | 高 | コードと連動して陳腐化しにくい |
| コミット履歴・PR | 中〜高 | 「いつ・なぜ変えたか」の一次情報。`git blame`で意図を追える |
| ADR | 中〜高 | 「なぜこの作りか」の意図。ただし覆された決定が残っていることもある |
| コードコメント | 中 | 近くのコードが変わるとズレやすい。`// TODO`や`// FIXME`は地雷の在り処 |
| 課題チケット・障害記録 | 中 | 「なぜこの一行があるか」の背景。地雷マップの材料 |
| 仕様書・設計書・Wiki | 低〜中 | 更新されにくく陳腐化しやすい。仮説として使う |

使い方の原則は明快です。

- **信頼度の高いものから読み、低いものは「仮説」として補助に使う。**
- **矛盾を見つけたら、必ずコード(と実挙動)で裁定する。**
- AIに渡すときは、何が事実で何が仮説かを明示する。

### AIへのコンテキストとしての与え方

集めた材料は、そのまま雑に貼るのではなく、**信頼度のラベルを付けて**AIに渡します。これは[コンテキストエンジニアリング](/patterns/context-engineering)の実践そのものです。

```text
これから既存システムの一部を改修します。あなたに渡す情報には
信頼度の差があるので、その前提で扱ってください。

## 事実(最優先・これが正)
- 対象コード: [貼る]
- 既存テスト: [貼る]

## 仮説(参考。コードと矛盾したら無視してよい)
- 設計書の該当箇所(最終更新: 2年前): [貼る]
- 関連ADR: [貼る]

## 背景情報
- このコードに関連する過去の障害チケット: [貼る]

タスク:
1. まず「事実」だけから、このコードが実際に何をしているかを要約
2. 次に「仮説」と照合し、コードと食い違う記述があれば指摘
3. 食い違いはどちらが正しいか断定せず、「コードはこうだが文書はこう」
   という形で並べて提示してください(裁定は私がします)
```

::: tip コンテキストの量と質を両立させる
大規模システムでは、関連コードをすべて貼るとコンテキスト長を超えます([コンテキストウィンドウ](/concepts/context-window))。「触る範囲+その直接の依存」に絞って渡すのが基本です。範囲外の全体像が必要なときは、後述のリバースエンジニアリングで作った要約や索引([RAG](/concepts/rag))を使います。生のコードを全部渡すのではなく、**事前に作った要約を渡す**ことで、質と量を両立させます。
:::

## リバースエンジニアリングは必要か

「既存改修だから、まずリバースエンジニアリング(リバエン)で全部ドキュメント化しよう」——これはよくある過剰反応です。**全部をリバエンする必要はありません**。要否は、状況に応じて判断します。

### 判断の基準

リバエンの必要度は、おおむね次の5つで決まります。

| 基準 | リバエン不要に傾く | リバエン必要に傾く |
| --- | --- | --- |
| 既存ドキュメントの有無・信頼度 | 最新で信頼できる文書がある | 文書が無い/古い/コードと食い違う |
| 改修範囲 | 局所的・閉じている | 広範・複数モジュールに波及 |
| 変更のリスク | 壊れても影響小・戻せる | 基幹機能・課金・データ整合に関わる |
| チームの既存理解 | 書いた本人がいる・熟知している | 誰も全体を把握していない・属人化 |
| コードの見通し | 素直で読めば分かる | 複雑・密結合・暗黙の前提が多い |

### 判断フローチャート

```text
                  ┌─────────────────────────────┐
                  │ これから既存コードを改修する │
                  └──────────────┬──────────────┘
                                 │
              ┌──────────────────▼──────────────────┐
              │ 触る範囲のコードを、自分(チーム)は │
              │ すでに正確に理解しているか?         │
              └────────┬───────────────────┬────────┘
                  はい  │                   │ いいえ
                        │                   │
        ┌───────────────▼──────┐   ┌────────▼─────────────────────┐
        │ 信頼できる最新文書が │   │ 触る範囲を「読めば」短時間で │
        │ あるか?             │   │ 理解できそうか?             │
        └───┬─────────────┬────┘   └────┬──────────────────┬──────┘
       はい │             │ いいえ  はい │                  │ いいえ
            │             │             │                  │
   ┌────────▼───┐  ┌──────▼──────┐ ┌────▼─────┐  ┌─────────▼─────────┐
   │ リバエン   │  │ 軽いリバエン│ │AIに読解  │  │ 範囲を絞った       │
   │ 不要。     │  │(該当範囲のみ│ │を補助させ│  │ リバエン+特性化   │

```

::: warning 全部リバエンではなく「触る範囲を必要十分に」
リバエンの目的は「システム全体を文書化すること」ではなく、「**安全に改修するために必要な理解を得ること**」です。触らないモジュールを丁寧に文書化しても、改修の助けにはなりません(し、すぐ腐ります)。リバエンの範囲は「**今回触る範囲+その境界(直接依存している周辺)**」に絞ります。線引きの詳しい原則は[リバースエンジニアリングと作るべきドキュメント](/approach/reverse-engineering)で扱います。
:::

## 既存改修の進め方

リバエンの要否を判断したら、改修そのものに入ります。ブラウンフィールド改修は、次の4ステップで進めるのが安全です。AIは各ステップを補助しますが、**ステップを飛ばしたり大きくしたりしてはいけません**。

```text
影響範囲の特定 → 安全網で囲う → 小さく変更 → 検証
     │              │              │          │
   AIで当たり    特性化テスト    最小の差分   テスト+実挙動
   をつける      を生成・固定    で1つだけ    で裏取り
```

### ステップ1: 影響範囲の特定

変更したい箇所が「どこに影響するか」を先に把握します。隠れた依存・副作用を見落とすと、ここで事故ります。AIに依存関係を追わせて、当たりをつけます。

```text
このリポジトリで、関数 [calculateBillingAmount] を変更しようとしています。
変更が影響しうる範囲を洗い出してください。

- この関数を呼んでいる箇所(直接・間接)
- この関数が依存している箇所(読んでいるグローバル状態、外部I/O、DB)
- 副作用(ログ、メトリクス、キャッシュ、イベント発火など)
- 戻り値の形・例外の種類に依存していそうな呼び出し側

## 重要
- 推測が混じる箇所は「要確認」と明記してください
- 「影響なし」と断言できない箇所は、グレーとして残してください
（対象コードと、関連しそうなファイルを貼る)
```

AIが挙げた範囲は出発点です。`grep`や参照検索、`git grep`で**機械的に裏を取る**ことを忘れないでください。

### ステップ2: 安全網で囲う(特性化テスト)

変更する前に、**現在の挙動を固定するテスト**(characterization test / 特性化テスト)を張ります。ポイントは「正しい挙動」ではなく「**今の挙動**」を記録することです。改修で挙動が変わったら、それを検出する網になります。

```text
関数 [calculateBillingAmount] の「今の挙動」を固定する
特性化テスト(characterization test)を生成してください。

## 方針
- 正しさは判断しない。現状の実際の出力をそのまま期待値にする
- 入力は境界を網羅(ゼロ、負、上限、空、null、月末、通貨の端数 等)
- 副作用がある場合は、それも観測・固定する

## 重要
- まだ実装は変えません。今の実装に対して緑になるテストだけ作る
- 期待値は、あなたの推測ではなく「実際に実行した結果」を入れること。
  あなたが値を推測した箇所は [要実行確認] と印を付けてください
（対象コードを貼る)
```

::: tip 特性化テストはリバエン成果物の「裏取り装置」でもある
特性化テストは安全網であると同時に、**AIの理解が正しいかを実行で確かめる装置**でもあります。「AIはこう動くと言った」を「実際にこう動いた」に変えるのが特性化テストです。詳しくは[リバースエンジニアリングと作るべきドキュメント](/approach/reverse-engineering)の特性化テストの節を参照してください。
:::

### ステップ3: 小さく変更する

安全網ができたら、変更を入れます。**一度に1つの理由でしか変わらない**よう、差分を小さく保ちます。AIに大きな一括変更を提案されても、人間が単位を小さく割ります。

```text
[calculateBillingAmount] に、以下の変更を加えてください。

変更内容: [税率の計算を、固定値から設定テーブル参照に変える]

## 厳守
- 既存の特性化テストは緑のまま通ること(挙動を変えない箇所は変えない)
- 今回の変更で「意図的に変わる挙動」だけを明示し、それに対応する
  テストの修正案を別途示すこと
- リファクタリングは混ぜない。今回の変更だけに集中する
- 出力は unified diff で

（対象コードと既存テストを貼る)
```

### ステップ4: 検証する

変更後、AIに既存挙動と変更後の挙動を説明させ、理解を確認します。ただし、**説明はあくまで仮説**。最終的な裏取りはテストと実挙動で行います。

```text
今回の差分について説明してください。
- 何が変わり、何が変わらないか
- 既存の呼び出し側に影響が出る箇所はあるか
- 特性化テストのうち、期待値を変える必要があったものとその理由

（差分とテスト結果を貼る)
```

```bash
# AIの説明を鵜呑みにせず、必ず実行で裏を取る
npm test -- billing
git grep -n "calculateBillingAmount"   # 取りこぼした呼び出し側がないか
```

::: warning AIに既存挙動を説明させるのは「理解の確認」、裏取りは別
AIに「このコードは何をしているか説明して」と尋ねるのは有効ですが、それで理解が確定したわけではありません。説明はコードからの推論であり、**自信たっぷりに間違える**ことがあります。重要な判断は、必ずテストと実行ログで裏を取ります。
:::

## 落とし穴

ブラウンフィールド改修でAIが引き起こしがちな事故には、明確なパターンがあります。

### 1. AIが古いパターンを「改善」と称して壊す

AIは「より新しい・よりきれいな書き方」を好みます。改修を頼むと、頼んでいない箇所まで**「ついでに改善」してしまう**ことがあります。その「古いパターン」には理由があったかもしれません。

- 一見冗長な分岐 → 実は特定環境のバグ回避だった
- 非効率なループ → 実は順序保証のためだった
- 奇妙なnullチェック → 実は実データに混じる異常値への対処だった

::: warning 「ついでの改善」を禁止する
改修プロンプトには「**今回の変更だけに集中し、リファクタリングや最適化を混ぜない**」と明示します。AIが良かれと思って変えた一行が、誰も覚えていない地雷を踏み抜くことがあります。改善は別タスク・別PRに分けます。
:::

### 2. 隠れた依存・副作用の見落とし

既存コードは、表面に見えない形で外部とつながっています。グローバル状態、シングルトン、環境変数、暗黙のキャッシュ、ログを前提にした監視、イベント発火。AIに渡したコンテキストにこれらが含まれていないと、AIは**それらが存在しないかのように**変更を提案します。

```text
この変更を入れる前に、見落としがちな「隠れた結合」を点検してください。
- グローバル状態・シングルトン・モジュールスコープの可変変数への依存
- 環境変数・設定フラグによる分岐
- ログ出力やメトリクスを「外部が前提にしている」可能性
- イベント/フックの発火に依存している箇所
- 実行順序・初期化順序への暗黙の依存

該当しそうな箇所を挙げ、確認方法も添えてください。
（対象コードと、設定・初期化まわりを貼る)
```

### 3. コンテキスト不足での的外れな提案

AIに局所的なコードだけ渡すと、システム全体の制約を知らないまま「一般論として正しいが、このシステムでは間違っている」提案をします。チームの規約、既存の設計判断、規制要件を知らないからです。

対策は、必要なコンテキストを意図的に渡すこと([コンテキストエンジニアリング](/patterns/context-engineering))。特に関連するADRを渡すと、過去の判断に反する提案を抑えられます。

### 落とし穴の対処まとめ

| 落とし穴 | 何が起きるか | 対処 |
| --- | --- | --- |
| ついでの「改善」 | 理由があった古いコードを壊す | 変更を1つに絞る指示・リファクタは別PR |
| 隠れた依存の見落とし | グローバル状態・副作用が壊れる | 隠れた結合の点検プロンプト・特性化テスト |
| コンテキスト不足 | 一般論で的外れな提案 | 関連コード・ADR・規約を渡す |
| 理解の鵜呑み | AIの推測を事実扱いし誤改修 | コード・テスト・実挙動で裏取り |
| 大きな一括変更 | 切り分け・レビュー不能 | 小さく割る・1コミット1理由 |

## まとめ

- ブラウンフィールドでは**コードが正**であり、最大の作業は**既存の理解**。AIの効きどころは**読解の加速**。
- **ドキュメントは仮説、コードが事実**。既存文書は信頼度を評価し、高い順に使い、矛盾はコードで裁定する。
- リバエンは**全部ではなく要否を判断**。「触る範囲+境界」を必要十分に。判断は文書の信頼度・範囲・リスク・既存理解で。
- 進め方は**影響範囲の特定 → 特性化テストで囲う → 小さく変更 → 検証**。ステップを飛ばさない。
- AIの理解は誤りうる。**特性化テストと実挙動で必ず裏を取る**。
- 落とし穴は「ついでの改善」「隠れた依存」「コンテキスト不足」。指示で封じ、テストで守る。
