# 主要ツールの使用感

カタログスペックや機能表を眺めても、実際に使ったときの「手触り」はなかなか伝わりません。同じ「補完とチャットができる」という説明でも、待ち時間の長さ、提案が割り込んでくる頻度、間違ったときの直しやすさ、リポジトリ全体をどこまで見てくれるか——こうした体験の差が、日々の生産性と「気持ちよさ」を決めます。

このページは、代表的なツールを実務で触ったときの **体験軸** での印象を、できるだけ誇張せずにまとめたものです。機能の網羅は[ツールエコシステムの全体像](/tools/landscape)・[コーディングアシスタント](/tools/coding-assistants)・[エージェント型ツール](/tools/agentic-tools)に譲り、ここでは「こんな時に気持ちよく、こんな時に詰まる」という観点に絞ります。

::: warning これは主観であり、執筆時点のスナップショットです
以下の印象は筆者・チームの実務経験に基づく **主観** です。各ツールは数か月単位で機能・モデル・UXを更新しており、ここで「詰まる」と書いた点が次の版で解消されていることも珍しくありません。導入判断の前には、必ず自分のコードベースで短期間試し、公式ドキュメントで最新仕様を確認してください。価格・上限・対応モデルは特に変化が速いため、本ページでは断定しません。
:::

::: tip このページの読み方
ツールに優劣の絶対順位はありません。あるのは **作業との相性** です。「補完で思考を止めずに書きたい」のか「タスクを丸ごと委任したい」のか、目的が変われば気持ちよく感じるツールも変わります。自分の典型作業を思い浮かべながら読んでください。
:::

## 体験を比べるための観点

ツールごとの印象に入る前に、本ページで使う観点を揃えておきます。

- **導入の手軽さ** — インストールから最初の有用な出力までの距離。既存環境を変えずに入るか。
- **補完の体験** — インライン補完のテンポ。割り込みの鬱陶しさと、当たったときの快感のバランス。
- **チャットの体験** — 立ち止まって相談するときの応答の質と、コンテキストの渡しやすさ。
- **エージェント的な自律作業** — 複数ステップを任せたときの、ループの粘り強さと脱線のしにくさ。
- **コンテキスト把握** — リポジトリ全体・周辺ファイルをどこまで自分で集めてくれるか。
- **レビューのしやすさ** — 出てきた変更を差分として確認・取捨選択しやすいか。
- **向き不向き** — 気持ちよく回る作業と、詰まりやすい作業。

## Claude Code

ターミナル中心のエージェント型ツール。CLIで対話しながら、ファイルの読み書き・コマンド実行・複数ステップの自律作業を任せられます。IDE拡張・デスクトップ・Web版もありますが、性格を最もよく表すのはCLIでの使用感です。

- **導入の手軽さ** — CLIを入れてプロジェクトのルートで起動するだけ、という素直さ。IDEを乗り換える必要がないのが心理的に軽い。一方で「補完がエディタに出る」体験を期待すると肩透かしになる。あくまでエージェントです。
- **補完の体験** — インライン補完を主機能とするツールではないため、ここを目当てにすると物足りない。タイピングの傍らで黙って続きを出してほしい人には別の層が要る。
- **チャットの体験** — 「相談」というより「依頼」に近い。質問にも答えるが、真価は「このバグを直して」「この機能を実装して」と投げたときに、自分でファイルを探し、関連箇所を読み、手を動かしてくれる点にある。
- **エージェント的な自律作業** — ここが本領。テストを走らせて失敗を読み、直してまた走らせる、というループを粘り強く回す。タスク単位の委任で気持ちよさが際立つ。
- **コンテキスト把握** — リポジトリを自分で検索・読解して必要な文脈を集める。人間が逐一ファイルを貼る手間が少ない。プロジェクト規約をファイル(ルールファイル)で渡しておくと一貫性が上がる。
- **レビューのしやすさ** — 変更を差分として提示し、コマンド実行前に承認を挟める。git管理下で使えば、いつでも戻せる安心感がある。
- **モデルとFastモード** — Claudeモデル(執筆時点でOpus 4.8 / Sonnet 4.6 / Haiku 4.5、Fable 5)で動作。**Fastモードは小型モデルへの切替ではなく、Claude Opusを高速に出力させるモード**である点に注意。重い計画にOpus、日常実装にSonnet、軽い処理にHaiku、といった使い分けが効く。

::: tip 気持ちよく回る / 詰まる
気持ちよい: 複数ファイルにまたがる実装、テスト駆動の修正、調査と修正の反復、定型的な移行作業。「タスクを言葉で投げて、差分で受け取る」流れがはまる。
詰まる: タイピング中の細かな行補完が欲しいとき、仕様が曖昧で判断が割れるとき。後者は人間が方針を固めてから渡すと急に滑らかになる。
:::

## GitHub Copilot

GitHubのAIペアプログラマー。**IDE統合**(インライン補完+チャット+エージェントモード)が中心で、Copilot CLIやコードレビュー機能などもあります。多くのエンジニアが最初に触れる定番です。

- **導入の手軽さ** — VS Codeなどに拡張を入れるだけ。既存IDEをそのまま使えるので、乗り換えコストがほぼない。チーム導入のハードルが低い。
- **補完の体験** — 補完の体験は完成度が高く、テンポが良い。関数名やコメントを先に書くと精度が上がる、という補完の基本作法がそのまま効く。割り込みの頻度も実用域。
- **チャットの体験** — IDE内のチャットで、選択範囲を指定して相談できる。エディタの文脈に密着しているので、「いま見ている関数」についての会話が自然。
- **エージェント的な自律作業** — エージェントモードが追加され、複数ステップの作業も任せられるようになってきた。ただし性格としては「IDEに根ざした補助」が強く、ターミナル/リポジトリ全体志向のエージェントとは手触りが異なる。
- **コンテキスト把握** — 開いているファイルや選択範囲を中心に文脈を取るのが基本。リポジトリ全体の規約や既存ユーティリティは、明示しないと拾われないことがある。
- **レビューのしやすさ** — 補完は受理単位、チャットは差分提案単位で確認しやすい。GitHubのコードレビュー機能との連携も持つ。
- **モデル選択** — **モデルは選択可能で、Claude系・GPT系・Gemini系などから選べる**(執筆時点。提供内容は変化)。同じCopilotでも、選ぶモデルで応答の質感が変わる点は意識したい。

::: tip 気持ちよく回る / 詰まる
気持ちよい: IDEから離れずに書き続けたいとき、補完のテンポを重視するとき、既存パターンの反復。
詰まる: リポジトリ全体を横断する大きな変更を「丸ごと任せたい」とき。範囲を絞って渡すか、ターミナル志向のエージェントに切り替えると楽になる。
:::

## Cursor

VS Codeをフォークした独立IDE。エディタごと作り替え、補完・チャット・エージェント編集を深く統合しています。

- **導入の手軽さ** — IDEを乗り換える前提なので、VS Code資産(設定・拡張)を引き継ぎつつ移行できる一方、「いつものIDE」を変えること自体に抵抗を感じる人もいる。
- **補完の体験** — 補完が滑らかで、複数行・複数箇所の編集提案が前に出る。次に直したい場所を先回りで示す挙動が、はまると快適。
- **チャットの体験** — 範囲指定・ファイル参照を交えた相談がIDEに密着していて自然。コードベースを参照させながら会話できる。
- **エージェント的な自律作業** — エージェント編集をIDE内で完結させられるのが持ち味。差分をその場で見ながら承認できるので、IDEを見ている安心感のままエージェントを使える。
- **コンテキスト把握** — リポジトリのインデックスを活用し、関連ファイルを引いてくる。Copilotより「全体を見ている」感覚が出やすい場面がある。
- **レビューのしやすさ** — 変更がエディタ上の差分として見えるため、視覚的に追いやすい。
- **向き不向き** — AIをワークフローの中心に据えたい人に向く。逆に「IDEは変えたくないが補完は欲しい」だけなら、拡張型のほうが軽い。

## Windsurf

エージェント的編集を前面に出した独立IDE。フロー重視のUXが特徴です。

- **導入の手軽さ** — こちらも独立IDE。乗り換え判断が要る。
- **補完・チャットの体験** — 補完とチャットは一通り揃う。性格としてはエージェント的な編集体験を「流れ」として滑らかに見せることに重心がある。
- **エージェント的な自律作業** — IDE内でエージェント編集を多用する人に向く。提案された一連の変更を、流れに乗ったまま受け入れていける感覚。
- **コンテキスト把握 / レビュー** — IDE統合型らしく、開いている文脈とインデックスを併用。差分は視覚的に確認できる。
- **向き不向き** — エージェント編集をIDEの中で日常的に回したい人向け。ターミナルで完結させたい人にはClaude Code等のほうが素直。

## Cline

VS Code拡張として動くエージェント。エディタ内でエージェントを動かし、ステップ承認のUIを持ちます。

- **導入の手軽さ** — VS Codeに拡張として入るので、IDEを変えずにエージェントを試せる。既存環境のままエージェント体験に踏み込みたいときの入口になりやすい。
- **エージェント的な自律作業** — アクションごとに承認を挟むUIで、「どこで何をしようとしているか」が見える。自律度を抑えめにして、手綱を握りながら使いたいときに安心。
- **コンテキスト把握** — リポジトリ内のファイルを読み、必要に応じてコマンドを実行する。IDE内で完結する分、エディタの文脈と地続きで使える。
- **レビューのしやすさ** — ステップ承認のおかげで、変更とコマンドを一つずつ確認できる。慣れていないコードベースで効く。
- **モデル** — 接続するモデルを選べる構成が一般的。組み合わせるモデルで手触りが変わる(具体は公式・設定で確認)。
- **向き不向き** — 「IDEから離れずに、承認しながらエージェントを使いたい」人に向く。完全自律で長時間回したいワークフローには、承認の手数が逆に負担になることもある。

## Aider

ターミナルで動く、gitと密に統合されたエージェント。変更を自動コミットし、差分で管理するのが思想の中心です。

- **導入の手軽さ** — CLIを入れてリポジトリで起動。gitリポジトリであることが前提として効いてくる。
- **エージェント的な自律作業** — 編集を行い、その変更をgitコミットとして積んでいく。各ステップが履歴に残るので、後から追いやすい。
- **コンテキスト把握** — 対象ファイルを明示して会話する運用がはまりやすい。何を見せるかを人間が決める感覚が強い。
- **レビューのしやすさ** — **自動コミット**により、いつでも差分で確認でき、戻すのもgitの作法そのまま。バージョン管理を安全網にする設計が好相性。
- **モデル** — 接続するモデルを選べる。込み入った変更には強いモデル、定型処理には軽いモデル、と使い分けられる(具体は公式・設定で確認)。
- **向き不向き** — gitを軸に変更を厳密に管理したい人に向く。git管理外の作業や、対話的にファイルを探させたいワークフローでは、別のエージェントのほうが楽な場面もある。

## 比較サマリ

各観点での **傾向** をまとめます。あくまで執筆時点の主観的な印象であり、絶対評価ではありません。

| ツール | 形態 | 補完の体験 | チャット/相談 | 自律作業 | コンテキスト把握 | 主に気持ちよい場面 |
| --- | --- | --- | --- | --- | --- | --- |
| Claude Code | CLI(IDE/デスクトップ/Web版も) | 主機能でない | 依頼型が得意 | 強い | 自分で探索 | タスク丸ごとの委任 |
| GitHub Copilot | IDE統合 | 完成度が高い | IDE密着で自然 | エージェントモードあり | 近傍中心 | IDEで書き続ける |
| Cursor | 独立IDE | 滑らか・先回り | IDE密着 | IDE内で完結 | インデックス活用 | AIを中心に据える |
| Windsurf | 独立IDE | 一通り揃う | 一通り揃う | フロー重視 | 統合+インデックス | IDE内エージェント編集 |
| Cline | IDE拡張 | 主眼でない | エージェント寄り | ステップ承認 | IDE内で探索 | 承認しながら委任 |
| Aider | CLI | 主眼でない | 対象明示型 | git中心 | 人間が明示 | gitで厳密に管理 |

::: info モデルが同じでも体験は変わる
上の表で「同じモデルを選べるツール」同士でも、補完の出方・コンテキストの集め方・承認の挟み方が違えば、手触りは大きく変わります。これは「モデルではなくハーネス(足回り)が体験を決める」という論点で、[Claude Code vs GitHub Copilot](/tools/claude-code-vs-copilot)で詳しく掘り下げます。
:::

## 使い分けの実感

実務では、これらを **排他的に選ぶ** より、作業に応じて使い分けるのが自然です。

- **書き続けたい時間帯** — IDE統合の補完(Copilot, Cursor等)で思考を止めずに進める。
- **タスクを丸投げしたい時** — ターミナル志向のエージェント(Claude Code, Aider)に、テスト駆動で委任する。
- **慣れないコードベース** — ステップ承認のあるエージェント(Cline等)で、手綱を握りながら進める。

「補完で足りるうちは補完層に留まり、横断的・反復的な作業になったらエージェント層へ移る」という移動の感覚は、[コーディングアシスタント](/tools/coding-assistants)・[エージェント型ツール](/tools/agentic-tools)で述べた切り替えのサインと同じです。

::: warning 取り扱いと権限は最初に確認する
どのツールも、コードの送信先・学習利用の有無・保持期間・自己ホスト可否はプランによって異なります。エージェント型では加えて、コマンド実行の権限設計(許可リスト・承認フロー)が安全運用の前提です。詳細は[ツールの選び方](/tools/choosing)と[エージェント型ツール](/tools/agentic-tools)を参照してください。
:::
