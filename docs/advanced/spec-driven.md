# 仕様駆動開発(Spec-Driven Development)

従来、ソースコードは「真実の源(source of truth)」でした。仕様書は書かれても、実装が進むうちに乖離し、やがて誰も読まなくなる——多くの現場が経験してきた光景です。

**仕様駆動開発(Spec-Driven Development, SDD)** は、この関係を反転させます。**自然言語で書かれた仕様を一次成果物**とし、コードとテストは **そこから生成・同期される派生物** と位置づけるのです。LLMが自然言語を高精度にコードへ変換できるようになったことで、この反転が現実的になりました。

::: tip ひとことで言うと
仕様駆動開発とは、**「コードでなく仕様を真実の源にする」** 開発スタイルです。人間は"何を・なぜ"を仕様として書き、AIが"どう書くか"=コードとテストを生成・更新する。レビューの主戦場もコードから仕様へ移ります。
:::

## なぜ仕様を一次成果物にするのか

AI駆動開発では、コードは「安く・速く・大量に」生成できる消耗品に近づきます。すると価値の重心が、**「正しく書く」から「正しく指示する」** へ移ります。

| | コード中心(従来) | 仕様中心(SDD) |
| --- | --- | --- |
| 真実の源 | ソースコード | 自然言語仕様 |
| 人間の主作業 | コードを書く | 仕様を書く・レビューする |
| AIの役割 | 補完・断片生成 | 仕様からコード/テストを生成 |
| 再生成 | しにくい(手書き資産) | しやすい(仕様から作り直せる) |
| レビュー対象 | 実装の差分 | 意図(仕様)の差分 |

::: info コードが"使い捨て"に近づくという発想
仕様が真実なら、実装はそこから再生成できる派生物です。フレームワーク移行や言語変更も、「仕様は据え置き、実装だけ作り直す」と捉えられます。これは[AI駆動開発とは](/guide/what-is-aidd)で述べた「人間が"何を・なぜ"、AIが"どう書くか"」の分業を、成果物の構造にまで徹底した形です。
:::

## 仕様の粒度

仕様駆動開発の成否は **仕様の粒度** で決まります。粗すぎれば曖昧で誤生成を招き、細かすぎればコードを日本語で書いているだけになり、保守コストが二重化します。

| 粒度 | 内容 | 問題 |
| --- | --- | --- |
| 粗すぎ | 「ログイン機能を作って」 | 解釈の幅が広すぎ、生成が安定しない |
| 適切 | 入力・出力・制約・例外・受入条件を明記 | — |
| 細かすぎ | 変数名・行単位の手順まで指定 | コードの二重メンテ。SDDの利点が消える |

適切な仕様が含むべき要素は、おおむね次の通りです。

- **目的(なぜ)** — その機能が解く問題。判断の根拠になる。
- **入力と出力** — 何を受け取り、何を返すか。
- **制約・不変条件** — 守るべきルール、性能・セキュリティ要件。
- **例外・エッジケース** — 異常系の振る舞い。
- **受入条件(acceptance criteria)** — 「完成」を判定する観測可能な条件。

::: tip 粒度の指針
**「実装の詳細は書かず、観測可能な振る舞いを書く」**。`どうやって`(アルゴリズム・変数名)はAIと実装に委ね、`何が成り立つべきか`(入出力・制約・受入条件)を仕様に書きます。受入条件が[テストに変換できる](/models/evaluation)レベルまで具体的であることが理想です。
:::

## 実行可能仕様(Executable Specifications)

仕様駆動開発の核心は、仕様を **「読むだけの文書」から「検証できる契約」** へ引き上げることです。これを **実行可能仕様** と呼びます。

考え方はこうです。仕様に書いた受入条件を、**テスト・型・スキーマといった機械検証可能な形へ落とし込み**、コードがそれを満たすかを自動で確認できるようにします。

```text
自然言語仕様
  ├─→ 受入条件 ──→ 受入テスト(自動実行)
  ├─→ 入出力契約 ──→ 型定義 / スキーマ
  └─→ 不変条件 ──→ プロパティテスト / アサーション

コード生成 → 上記の検証を通る → 仕様を満たすと言える
```

この構造により、**仕様とコードの整合を機械的に確認**できます。エージェントにとっても、受入テストという「機械判定できる成功条件」があることが、[安定した自律実行](/agents/what-is-agent)の前提になります。

## 仕様とコードの乖離(ドリフト)管理

仕様駆動開発でも、放置すれば仕様とコードはずれていきます。緊急修正でコードだけ直す、仕様を更新し忘れる——従来と同じ罠です。ただしSDDには乖離を **検出・抑制する仕組み** を組み込めます。

| 乖離の防ぎ方 | 仕組み |
| --- | --- |
| 機械的検出 | 受入テストの失敗=仕様とコードの乖離として表面化させる |
| 一方向の流れを固定 | 「仕様を変える→生成/修正→テスト通過」の順を運用ルールにする |
| 緊急パッチの還元 | コードを直接直した場合、仕様への反映を必須タスクにする |
| 定期監査 | 仕様とコードの整合をLLMにレビューさせる(差分の説明を求める) |

::: warning 仕様だけ更新してコードを忘れる/その逆
SDDで最も多い失敗は **「二重の真実」** が生まれることです。仕様とコードのどちらが正なのかが曖昧になると、最悪の状態(両方信用できない)に陥ります。**どちらが真実かを明確に1つに決め**、もう一方は必ずそこから同期する、という規律が要ります。受入テストを「仕様側の代理人」として常時走らせるのが現実的な歯止めです。
:::

## レビューを"仕様"に引き上げる利点

仕様駆動開発の最大の効用は、**レビューの対象がコード差分から意図(仕様)の差分へ上がる**ことです。

- **意図のレビューは速い** — 何百行のdiffを追うより、「この仕様変更は正しいか」を問うほうが本質的で速い。
- **誤りを上流で捕まえる** — 要件の誤解は仕様レビューで止まる。実装まで進んでから気づくより安い。
- **AIの誤生成を検出しやすい** — 「仕様にない振る舞いをコードが持っていないか」という観点でレビューできる。
- **属人性が下がる** — 意図が文書化されるため、実装者以外もレビュー・引き継ぎができる。

::: info コードレビューが不要になるわけではない
仕様レビューが主戦場になっても、生成コードのレビューは依然必要です。LLMは[もっともらしいが誤ったコード](/quality/hallucination)を生むため、最終的な[コードレビュー](/quality/code-quality)と[セキュリティ確認](/quality/security)は残ります。SDDは"何を最初に・最も重くレビューするか"の重心を仕様へ移すものであって、下流の検証を消すものではありません。
:::

## 実例ワークフロー

仕様駆動開発の一回転は、おおむね次のように進みます。

```text
1. 仕様を書く / 更新する(人間)
   └─ 目的・入出力・制約・例外・受入条件を自然言語で

2. 仕様をレビューする(人間)
   └─ ここが主レビュー。意図の正しさ・抜け漏れを問う

3. 受入条件をテストに変換する(AI支援)
   └─ 仕様 → 受入テスト / 型 / スキーマ

4. コードを生成する(AI)
   └─ 仕様とテストを文脈に、実装をエージェントが生成

5. 検証ループ(AI)
   └─ テスト・型・リンタを通すまで自己修正

6. 生成コードをレビューする(人間)
   └─ 仕様に対する逸脱・セキュリティ・可読性

7. 変更要求 → 仕様へ戻る(1へ)
   └─ コードでなく仕様を起点に修正する規律を守る
```

擬似コードで表すと、仕様を起点とした再生成可能なパイプラインになります。

```python
def spec_driven_cycle(spec):
    review_spec(spec)                       # 1-2: 人間が意図をレビュー
    tests = generate_acceptance_tests(spec) # 3: 仕様→受入テスト
    code = generate_code(spec, tests)       # 4: 仕様+テストから実装

    while not run_tests(code, tests).all_passed:  # 5: 検証ループ
        code = repair(code, failures=run_tests(code, tests).failures)

    human_review(code, against=spec)        # 6: 仕様に対する逸脱を確認
    return code

# 変更は spec を起点に。コード直し放しを禁じる(7)
```

::: tip 既存コードベースへの導入
最初から全面SDDにする必要はありません。**新規モジュールや変更頻度の高い領域**から始め、仕様→受入テスト→生成のループを部分導入します。レガシー部分は「仕様を後付けで起こす(仕様復元)」ところから着手すると現実的です。
:::

## 向く場面・向かない場面

| 向く | 向かない |
| --- | --- |
| 振る舞いを明確に書ける機能(API・業務ロジック) | 探索的・要件が固まらない試作(PoC) |
| 受入条件がテスト化できるもの | 主観品質が支配的なもの(UIの感性・デザイン) |
| 長く保守する・引き継ぐコード | 一度きりの使い捨てスクリプト |
| 仕様の合意がチームに必要な領域 | 個人の手元作業 |

要件が流動的な初期探索では、仕様を固める作業がかえって足枷になります。**振る舞いが安定し、受入条件を観測可能に書ける**段階で効果を発揮します。

## まとめ

- 仕様駆動開発は **自然言語仕様を真実の源**とし、コード/テストをそこからの派生物にする。
- 仕様の **粒度** が鍵。「観測可能な振る舞い」を書き、実装詳細は委ねる。
- 受入条件を **実行可能仕様**(テスト・型・スキーマ)に落とし、仕様とコードの整合を機械検証する。
- **レビューの重心を仕様へ**上げると、誤りを上流で安く捕まえられる。ただし下流の検証は残る。
- **真実を1つに決め**、もう一方は必ず同期する規律で乖離を防ぐ。
