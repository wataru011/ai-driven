# データ・ML/データエンジニアリングでのAI活用

データとMLの領域は、AIコード生成の恩恵が大きい一方で、**最も危険な落とし穴**を抱えています。理由は単純で、この領域の成果物は「コードが動くか」ではなく「**数字が正しいか**」で評価されるからです。エラーも例外も出さずに完走したパイプラインが、静かに誤った集計をしている——これがデータ領域における最悪のシナリオです。

さらにやっかいなのは、LLMにデータ探索を任せると、**実際に集計していないのに集計結果らしき数字を語る**ことがある点です。コード生成と「データそのものについての主張」は、まったく信頼度が違います。この区別が本章の背骨です。

::: warning 最重要の原則
LLMの「このデータはおよそ平均◯◯です」「外れ値は約◯件です」という**数値の主張は、コードを実行して得たものでない限り信用しない**。LLMは数字をもっともらしく捏造します。信頼できるのは「集計を行うコード」であって、「集計結果の言明」ではありません。
:::

## AI活用の勘所

### 1. パイプラインは「変換の意図」を渡す

ETL/ELTの変換ロジックは、AIが雛形を書きやすい領域です。ただし「このCSVをきれいにして」のような曖昧な依頼は、**勝手な前提**(欠損を0で埋める、重複を黙って消す)で危険な変換を生みます。

変換の意図を明示します。

> 以下の変換を pandas で実装してください。各ステップで件数の変化をログ出力すること。
> - 欠損: `amount` の欠損は除外(0埋めはしない)。除外件数をログ
> - 重複: `order_id` の重複は最新の `updated_at` を採用
> - 型: `date` 列はUTCとして parse。タイムゾーン未指定はエラーにする
> - 各ステップ後の行数を assert で前後比較し、想定外の激減を検知

「件数の変化をログ・assertする」のがデータ領域の防御線です。**データが静かに消える/増える**事故を早期発見できます。

### 2. 前処理の「暗黙の前提」を明文化させる

前処理はバグが結果に直結します。AIに前処理を書かせたら、**どんな前提を置いたかを列挙させる**と漏れを発見できます。

> 上記の前処理コードについて、置いた前提を箇条書きで列挙してください。
> (例: 文字コード、区切り文字、日付フォーマット、外れ値の扱い、単位)
> それぞれ、前提が崩れたときに何が起きるかも添えて。

### 3. ノートブックは「実行順序」の罠に注意

Jupyter等のノートブックは、**セルの実行順序**が状態を作るため、AIが生成したセルをバラバラに実行すると再現不能になります。

::: warning ノートブック特有の事故
- **隠れた状態依存** — 上のセルで作った変数に依存するのに、上を再実行していない。
- **out-of-order実行** — セルを飛ばして実行し、結果が「たまたま正しく見える」。
- **再現不能** — カーネルを再起動して上から流すと別の結果になる。
:::

対策は「**Restart & Run All で通ることを必須**」とし、確定したロジックは早めに`.py`モジュールへ切り出すこと。AIに「このノートブックを上から順に実行して再現可能な形に整理し、関数として切り出して」と依頼するのは有効です。

### 4. 特徴量エンジニアリングは「リーク」を最優先で疑う

特徴量生成でAIがやりがちな最悪のミスが**データリーク(leakage)**です。未来の情報や目的変数の情報が特徴量に混入し、検証では高精度なのに本番でまるで当たらないモデルを作ります。

- **時系列リーク** — 集計に未来データを含める。`rolling`の窓が未来にはみ出す。
- **目的変数リーク** — ターゲットから派生した列を特徴量に混ぜる。
- **fit/transformの混在** — テストデータでスケーラを`fit`してしまう(統計量が漏れる)。

> 特徴量生成について、データリークの観点で監査してください。
> - 各特徴量が「予測時点で実際に入手可能か」を1つずつ判定
> - スケーリング/エンコーディングは train でのみ fit し、valid/test に transform しているか
> - 時系列の集計窓が未来を含んでいないか
> 疑わしい箇所は「なぜリークか」を具体的に説明して。

### 5. 実験管理と再現性は「仕組み」で担保する

AIに実験コードを量産させると、パラメータと結果の対応が**たちまち追えなくなります**。再現性は気合いではなく仕組みで守ります。

- 乱数シードを固定し、データバージョン・コードバージョン・パラメータを記録(MLflow / W&B 等)。
- 「この学習スクリプトに、シード固定と実験トラッキング(params/metrics/artifacts記録)を追加して」とAIに依頼。
- 環境(ライブラリのバージョン)を固定(lockファイル、コンテナ)。

## 有効なプロンプト例

### 安全なデータ探索(集計はコードで)

```text
このDataFrame(df)の品質を調べたいです。あなたの推測ではなく、必ず実行可能な
コードを書いてください。私がそれを実行して結果を確認します。
- 各列の欠損率、ユニーク数、型
- 数値列の基本統計と外れ値候補(IQR法)
- カテゴリ列の上位カテゴリと、想定外カテゴリの有無
出力は表とコードのみ。「おそらく◯◯です」という数値の断定はしないで。
```

::: tip なぜ「コードを書かせて自分で実行」なのか
LLMにデータを要約させると、見ていない数字をそれらしく語ります。**集計はコードで行い、数字は実行環境から取る**。LLMの役割は「正しい集計コードを書くこと」に限定するのが安全です。
:::

### 前処理の検証コード生成

```text
この前処理関数に対する検証を書いてください(pytest + 小さな合成データ)。
- 入力の行数と出力の行数の関係をアサート(意図しない激減を検知)
- 欠損処理が仕様どおりか(0埋めしていないこと)
- 重複排除後に order_id がユニークであること
- 型と値域(amount >= 0、date が tz-aware)
合成データには境界ケース(欠損、重複、外れ値)を含めて。
```

## 落とし穴と対策

::: warning データ・ML特有の典型的な失敗
- **数値の捏造** — LLMが集計せずに「平均は約◯◯」と断定する。
- **誤集計** — `groupby` のキー違い、`join` の重複による件数水増し、平均の平均。
- **データリーク** — 検証では高精度、本番で当たらない。
- **静かなデータ欠損** — フィルタやjoinで行が消えても気づかない。
- **不均衡の無視** — 精度(accuracy)だけ見て、少数クラスをまったく当てていない。
- **再現不能** — シード未固定、out-of-order実行、環境差。
- **単位・スケールの取り違え** — 円とドル、秒とミリ秒、対数変換の戻し忘れ。
:::

| 落とし穴 | 対策 |
| --- | --- |
| 数値の捏造 | 集計は必ずコード実行。LLMの言明は検証対象であって根拠ではない |
| 誤集計 | join後の件数assert、`groupby`キーの明示、結果を既知の小データで検算 |
| データリーク | 時間軸での分割、fit/transform分離、特徴量の「入手可能性」監査 |
| 静かな欠損 | 各ステップで件数ログ + 前後assert |
| 不均衡 | 適切な指標(PR-AUC、F1、混同行列)を明示して評価させる |
| 再現不能 | シード固定、Restart & Run All必須、環境・データのバージョン管理 |

## 検証

データ・MLでは「テストが通る」だけでは不十分で、**結果の妥当性**を確かめる検証が要ります。

1. **既知データでの検算** — 答えが手計算でわかる小さなデータで集計コードを通し、期待値と一致するか確認する。AIの集計ロジックを信じる前の必須ステップ。
2. **件数の連続性チェック** — パイプラインの各段で行数を記録し、説明できない増減を検知する。
3. **分布のサニティチェック** — 出力の分布が常識的か(年齢が負、確率が1超、など)をassertする。
4. **ベースラインとの比較** — モデルが「常に多数派を予測」より本当に良いかを確認する。改善が誤差範囲でないか。
5. **ハルシネーション対策の徹底** — LLMの主張は出典(=実行結果)で裏取りする。詳しくは [ハルシネーション対策](/quality/hallucination) を参照。

::: details データ領域のレビュー観点チェックリスト
- 集計結果はコード実行で得たものか(LLMの言明ではないか)
- join/groupby の後で件数は説明可能か
- 前処理で消えた/増えた行は意図どおりか
- 特徴量は予測時点で入手可能か(リークなし)
- スケーラ/エンコーダは train でのみ fit したか
- シードは固定され、Restart & Run All で再現するか
- 評価指標は問題設定(不均衡など)に適切か
:::
