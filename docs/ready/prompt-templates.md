# プロンプトテンプレート集

このページは、日々の開発タスクで**そのままコピペして使える**プロンプトの全文集です。各テンプレートは角括弧 `[...]` のプレースホルダを埋めるだけで動きます。説明は各テンプレートの「使い方」一言にとどめます。

::: info このページと [プロンプトカタログ](/patterns/prompt-catalog) の違い
[プロンプトカタログ](/patterns/prompt-catalog) は「なぜそう書くと効くのか」を**解説**するページです。一方こちらは**即コピペ**に振り切った成果物集です。仕組みを理解したいときはカタログを、手を動かしたいときはこのページを開いてください。設計原則の土台は [プロンプトの基礎](/prompting/fundamentals) にあります。
:::

::: tip 共通の前提
以下のテンプレートは、[ルールファイル](/ready/rule-files)(`CLAUDE.md` など)で技術スタック・規約を既に渡してある前提で書いています。ルールファイルがない場合は、各プロンプトの冒頭に言語・フレームワーク・バージョンを足してください。

各テンプレート内の ` ``` ` で囲まれた箇所は、あなたのコードや差分を貼り込む場所です。コピー後、角括弧プレースホルダと合わせて埋めてください。
:::

## 1. 機能実装

**使い方:** 目的・要件・制約・出力形式を埋める。要件は箇条書きで、境界条件まで書くほど精度が上がる。

````text
あなたは [言語/フレームワーク、例: TypeScript + NestJS] に精通したエンジニアです。

## 目的
[何を実現したいか1〜2文。例: 記事一覧APIにカーソルベースのページネーションを追加する]

## 既存の状況
[関連するエンドポイント・関数・データ構造。下に実コードを貼る]

```
[既存コードを貼る]
```

## 要件
- [要件1。例: クエリ limit はデフォルト20、最大100、cursor は任意]
- [要件2。例: レスポンスは items と nextCursor を持つ形式]
- [境界条件。例: 最終ページでは nextCursor を null にする]

## 制約
- [例: 既存の repository 層のクエリビルダを踏襲する]
- [例: 新規の依存ライブラリは追加しない]

## 出力
1. 変更点の概要を箇条書き
2. 修正後のコード(変更したファイルごと)
3. 境界値を含むテスト
````

## 2. バグ修正(再現付き)

**使い方:** 「再現手順」と「期待 vs 実際」を必ず埋める。これがないとAIは推測で直し、的外れになりやすい。

````text
あなたは原因を特定してから直す、慎重なデバッグ担当エンジニアです。
まず原因を説明し、私が同意してから修正を出してください。

## 症状
[何が起きるか。例: ユーザー削除後に一覧を開くと500エラー]

## 再現手順
1. [手順1]
2. [手順2]
3. [手順3 → ここでエラー]

## 期待する動作
[本来どうあるべきか]

## 実際の動作
[実際に何が起きるか。エラーメッセージ全文・ステータスコードを含める]

## エラー / スタックトレース
```
[エラー全文を貼る]
```

## 関連コード
```
[疑わしい箇所のコードを貼る]
```

## 進め方
1. 原因の仮説を提示(複数あれば優先度付き)
2. それを確認する最小の方法
3. 合意後、修正と「失敗を再現する回帰テスト」を出す
````

## 3. テスト生成

**使い方:** 対象コードと「重視する観点(境界値・異常系など)」を指定する。テストフレームワークはルールファイル任せでよい。

````text
以下の [関数/クラス/モジュール] に対する単体テストを書いてください。

## 対象
```
[テスト対象コードを貼る]
```

## 重視する観点
- 正常系: [例: 典型的な入力で期待値を返すこと]
- 境界値: [例: 空配列、最大件数、0、負値]
- 異常系: [例: 不正入力で適切な例外を投げること]
- [例: 副作用(DB書き込み)が想定どおり起きること]

## 方針
- テストは [フレームワーク、例: Vitest]。命名は「対象_条件_期待」の形式で。
- 1テスト1アサーションを基本にし、何を検証しているか名前で分かるようにする。
- モックは外部I/Oのみ。ロジックはモックしない。

## 出力
テストコードのみ。各テストの意図が分かるよう、説明的なテスト名を付ける。
````

## 4. リファクタリング

**使い方:** 「振る舞いを変えない」ことを最優先で明示する。改善のゴール(可読性/重複排除/分割など)を1つに絞ると暴走しない。

````text
以下のコードを、外部から見た振る舞いを一切変えずにリファクタリングしてください。

## 対象
```
[対象コードを貼る]
```

## 改善したい点(優先順)
1. [例: 30行を超える関数を、責務ごとに分割したい]
2. [例: 重複している分岐をまとめたい]

## 厳守
- 公開インターフェース(関数シグネチャ・戻り値の型)は変えない。
- 既存テストが全て通る状態を維持する。テストがない箇所は先に特性化テストを提案する。
- 一度に大きく変えず、安全な小ステップに分けて、各ステップの意図を説明する。

## 出力
1. 変更方針の要約
2. リファクタ後のコード
3. 振る舞いが変わっていないことをどう確認できるか
````

## 5. コードレビュー依頼

**使い方:** 自分の書いた差分をAIにレビューさせる。観点を指定すると指摘が散らからない。`git diff` の出力をそのまま貼れる。

````text
以下の変更を、厳しめのレビュアーとしてレビューしてください。
褒める必要はありません。問題と改善点だけを挙げてください。

## 変更の意図
[このPRで何をしたかったか1〜2文]

## 差分
```diff
[git diff の出力を貼る]
```

## 観点
- バグ・エッジケースの見落とし(null、空、境界、並行性)
- エラーハンドリングの漏れ
- 命名・可読性
- テストの不足
- [このプロジェクト固有の観点。例: N+1クエリになっていないか]

## 出力形式
指摘ごとに次を書く:
- 重要度(高/中/低)
- 該当箇所(ファイル:行)
- 何が問題か / なぜ問題か
- 修正案(コード片で)
重要度の高い順に並べる。
````

## 6. 設計レビュー

**使い方:** 実装前に方針の妥当性を検証する。代替案とトレードオフを出させると、見落としに気づける。

````text
これから実装する設計をレビューしてください。実装はまだしないでください。

## 解決したい課題
[ビジネス/技術上の課題]

## 制約・前提
- [例: 既存DBはPostgreSQL、スキーマ変更は最小にしたい]
- [例: 1日100万リクエストを捌く必要がある]
- [例: チームはGoに習熟、新言語は導入しない]

## 検討中の設計
[現時点の案を文章 or 図 or 擬似コードで]

## 見てほしい点
- この設計の弱点・破綻しうる条件
- もっと単純にできないか
- 代替案が複数あれば、それぞれのトレードオフ

## 出力
1. 案の評価(良い点・懸念点)
2. 代替案(あれば)とトレードオフ表
3. 推奨と、その理由
````

## 7. PR説明文生成

**使い方:** 差分を渡してPRの説明文を生成。レビュアーが読む順(背景→変更→確認方法)に整える。

````text
以下の差分から、レビュアーが理解しやすいPR説明文を書いてください。

## 差分
```diff
[git diff の出力を貼る]
```

## 補足(分かる範囲で)
- 関連Issue/チケット: [例: #123]
- 背景: [なぜこの変更が必要か]

## 出力フォーマット(このまま Markdown で)
- 概要: 何を・なぜ。2〜3文
- 変更内容: 箇条書き
- 動作確認: レビュアーが確認すべき手順 or 自動テストの場所
- 影響範囲 / 注意点: 後方互換性、マイグレーション、ロールバック手順など
````

## 8. コミットメッセージ生成

**使い方:** ステージ済みの差分から Conventional Commits 形式のメッセージを作る。1コミットの粒度が適切か点検にも使える。

````text
以下のステージ済み差分から、Conventional Commits 形式のコミットメッセージを
1つ書いてください。

## 差分
```diff
[git diff --staged の出力を貼る]
```

## ルール
- 形式: type(scope): subject
  type は feat / fix / refactor / test / docs / chore / perf のいずれか。
- subject は命令形・現在形で50字以内。末尾にピリオドを付けない。
- 本文(必要なら)に「なぜ」を書く。1行72字で折り返す。
- この差分が複数の論点を含む場合は、その旨を指摘し、分割案を示す。

## 出力
コミットメッセージのみ(コードブロックで)。
````

## 9. エラー解決

**使い方:** エラー全文・環境・直前の操作を渡す。「推測ではなく原因の切り分け手順から」と書くと、闇雲な変更を防げる。

````text
以下のエラーの原因を切り分けたいです。いきなり修正コードを出さず、
まず原因の仮説と、それを確認する手順を提示してください。

## エラー全文
```
[エラー・スタックトレースを貼る]
```

## 環境
- [言語/ランタイム/バージョン。例: Node.js 20]
- [OS、フレームワーク、関連パッケージのバージョン]

## 直前にしたこと
[例: 依存を更新した / この関数を追加した / デプロイ後に発生]

## すでに試したこと
- [例: キャッシュ削除 → 変化なし]

## 欲しいもの
1. 最も可能性の高い原因(理由つき)
2. それを確認するコマンド/手順
3. 確認できた場合の修正方針
````

## 10. コード説明

**使い方:** 読めないコードを理解する。「どのレベルで説明してほしいか」を指定すると冗長さを制御できる。

````text
以下のコードが何をしているか説明してください。

## 対象
```
[対象コードを貼る]
```

## 知りたいこと
- このコードの目的(一言で)
- 処理の流れ(ステップごとに)
- 非自明な箇所・トリッキーな部分の意図
- [必要なら] 想定される入力と出力の例
- [必要なら] 潜在的なバグや注意点

## 説明のレベル
[例: この言語は分かるが、このライブラリの作法を知らない前提で]
専門用語は使ってよいが、初出時は一言補足する。
````

## 11. パフォーマンス改善

**使い方:** 「現状の計測値」と「目標」を数値で渡す。憶測の最適化を避けるため、まず原因の特定を求める。

````text
以下の処理が遅いです。推測で書き換える前に、ボトルネックの特定から始めてください。

## 対象
```
[対象コードを貼る]
```

## 現状(計測値)
- [例: 1万件の入力で 4.2秒。計算量は O(n^2) と推測]
- [プロファイル結果があれば貼る]

## 目標
- [例: 1万件で 200ms 以下にしたい]

## 許容できる手段 / できない手段
- 許容: [例: ハッシュマップでメモリと引き換えに高速化してよい]
- 不可: [例: 外部キャッシュ(Redis等)の新規導入は不可]

## 出力
1. ボトルネックの特定(なぜ遅いか)
2. 改善案(複数あれば計算量・メモリのトレードオフ付き)
3. 改善後のコードと、効果を測る方法
````

## 12. セキュリティレビュー

**使い方:** 入力境界・認証・機密データを扱うコードに。観点を列挙すると見落としが減る。指摘は深刻度付きで出させる。

````text
以下のコードをセキュリティ観点でレビューしてください。
攻撃者の視点で、悪用できる箇所を探してください。

## 対象
```
[対象コードを貼る]
```

## 文脈
- このコードが扱うデータ: [例: ユーザーのメールとパスワード]
- 信頼境界: [例: リクエストボディは外部からの未検証入力]

## チェックしてほしい観点
- 入力検証(インジェクション: SQL / コマンド / パストラバーサル)
- 認証・認可の漏れ(権限チェックの抜け)
- 機密情報の漏洩(ログ出力、エラーメッセージ、レスポンス)
- 安全でない既定値、暗号・乱数の誤用
- [Web の場合] XSS / CSRF / SSRF / オープンリダイレクト

## 出力
発見ごとに:
- 深刻度(Critical / High / Medium / Low)
- 該当箇所と、どう悪用されるか(攻撃シナリオ)
- 修正方法(コード片で)
問題がない項目も「確認済み」と明記する。
````

::: warning AIのセキュリティレビューは補助
AIによるレビューは見落としの網を増やす補助であり、専門家のレビューや専用ツール(SAST/DAST)の代替にはなりません。重大な領域では必ず人とツールで多重化してください。詳細は [コード品質](/quality/code-quality) を参照。
:::

## 13. 命名相談

**使い方:** 役割が分かりにくい変数・関数・型の名前を相談する。文脈(何を表すか)を渡すのがコツ。候補を複数出させて選ぶ。

````text
以下の [変数/関数/クラス/型] の名前の候補を複数提案してください。

## 対象
```
[該当コードと、その周辺(呼び出し箇所)を貼る]
```

## これが表す概念
[何を保持/実行するものか。例: ユーザーが過去30日に閲覧した商品IDのリスト]

## 命名規約
- [例: 変数は camelCase、boolean は is/has で始める]
- [例: ドメイン用語は英語の業界標準に合わせる]

## 出力
- 候補を3〜5個、それぞれ「ニュアンスの違い」を一言添えて
- 最も推奨するものと理由
- 避けるべき名前があれば、その理由も
````

## 14. 移行・変換

**使い方:** 言語・フレームワーク・APIの移行に。「等価性の保証」と「変えてよい/いけない点」を明示する。

````text
以下のコードを [変換元] から [変換先] へ移行してください。
例: JavaScript から TypeScript / class から関数コンポーネント / moment から date-fns

## 対象
```
[対象コードを貼る]
```

## 移行の方針
- 振る舞いは等価に保つ。出力が変わる箇所があれば必ず指摘する。
- [例: 命名やフォーマットは変換先の慣習に合わせてよい]
- [例: 非推奨APIは現行の推奨APIに置き換える]

## 制約
- [例: 公開インターフェースは維持する]
- [例: 依存追加は date-fns のみ許可。それ以外は不可]

## 出力
1. 移行後のコード
2. 意味が変わりうる箇所の注意書き(あれば)
3. 移行が正しいか確認するためのチェックポイント
````

## 15. 仕様の明確化

**使い方:** 曖昧なタスクを実装前に詰める。**AIに質問させる**のがポイント。いきなり実装させず、不明点を洗い出す。

````text
これから [機能/タスク] を実装したいですが、仕様に曖昧な点が残っています。
実装はまだしないでください。代わりに、実装前に決めるべき不明点を質問してください。

## 今分かっていること
[例: 「ユーザーがファイルをアップロードできるようにする」とだけ決まっている]

## やってほしいこと
1. この仕様で曖昧・未決定な点を洗い出す
   (例: 対応形式、サイズ上限、保存先、重複時の挙動、権限、失敗時の扱い)
2. それぞれについて、現実的な選択肢と推奨を提示する
3. 私が回答したら、確定仕様を箇条書きでまとめ直す

まずは「決めるべきこと」のリストだけを出してください。
````

## 16. ログ・スタックトレース解析

**使い方:** 大量のログから原因の手がかりを絞る。時系列と「正常時との差」を渡すと精度が上がる。

````text
以下のログから、障害の原因を絞り込んでください。

## 状況
- 何が起きたか: [例: 14:03頃から一部リクエストが断続的にタイムアウト]
- 影響範囲: [例: /checkout のみ。他のエンドポイントは正常]

## ログ
```
[関連するログを時系列で貼る。タイムスタンプは残す]
```

## 欲しいもの
1. ログから読み取れる事実(推測と分けて)
2. 原因の仮説(可能性の高い順、根拠つき)
3. 次に確認すべきログ・メトリクス・コマンド
4. 一時的な緩和策と、恒久対策の方向性
````

## 17. 依存アップグレード

**使い方:** メジャーバージョン更新の影響を事前に把握する。破壊的変更と移行手順を整理させる。

````text
[ライブラリ名] を [現在のバージョン] から [目標バージョン] に上げたいです。
影響を事前に把握したいので、移行計画を立ててください。

## 現状の使い方
```
[このライブラリを使っている主要箇所を貼る、または import 一覧]
```

## やってほしいこと
1. このバージョン間の破壊的変更のうち、上記コードに影響するもの
2. 各影響箇所の修正方法(コード片で before / after)
3. 非推奨化された機能と、その置き換え先
4. 移行を安全に進める手順(小さなステップに分割)
5. 移行後に確認すべきテスト・動作

不確かな点は「要確認」と明示し、推測で断定しないでください。
````

::: tip 最新情報の確認
ライブラリの破壊的変更は学習データより新しいことがあります。重要な移行では、AIの回答を公式の移行ガイド(CHANGELOG / migration guide)と必ず突き合わせてください。
:::

## 18. ドキュメント生成

**使い方:** 関数・モジュール・READMEの下書きを作る。読者像と粒度を指定すると過不足が減る。

````text
以下の [関数/モジュール/プロジェクト] のドキュメントを書いてください。

## 対象
```
[対象コードを貼る]
```

## 読者
[例: このコードを初めて使う他チームの開発者]

## 含めてほしい内容
- 何をするものか(目的)
- 使い方(最小の例つき)
- 引数・戻り値・例外([関数の場合])
- 注意点・制約・既知の落とし穴
- [必要なら] 内部の仕組みの概要

## 形式
- [例: TSDoc コメント形式 / Markdown の README 形式]
- 実在しないAPIや挙動を創作しない。コードから読み取れないことは
  「[要確認]」と明示する。
````

---

## カスタマイズの指針

これらは出発点です。自分のチームに合わせて育ててください。

- **よく使うものは保存する。** ルールファイル([rule-files](/ready/rule-files))や、エディタのスニペット、`.github/PULL_REQUEST_TEMPLATE.md` などに落とし込むと、毎回コピペする手間が消えます。
- **「出力形式」を固定する。** 後工程(レビュー・適用・自動処理)が決まっているなら、出力形式を強く指定するほど扱いやすくなります。
- **検証をセットにする。** 実装系のテンプレートは必ず「テスト」「確認方法」を要求しています。これは [プロンプトの基礎](/prompting/fundamentals) の検証可能性の原則そのものです。

::: warning コピペ前に1点だけ確認
角括弧 `[...]` のプレースホルダを**埋め忘れたまま**送らないこと。特に「コードを貼る」箇所を空のまま送ると、AIは存在しないコードを想像で書きます。
:::
