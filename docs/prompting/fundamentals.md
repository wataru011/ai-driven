# プロンプトエンジニアリングの基礎

プロンプトは、LLMという**確率的なテキスト生成器**に対する仕様書(spec)です。曖昧な仕様からは曖昧な実装が生まれます。これはAIに限った話ではなく、人間の同僚にチケットを渡すときとまったく同じ構造です。違いは、AIが行間を読まず、文脈を共有せず、不明点を聞き返さない(ことが多い)という点にあります。

つまりプロンプトエンジニアリングとは「魔法の呪文」を探す作業ではなく、**意図を曖昧さなく言語化し、検証可能な形に落とし込む**エンジニアリング作業です。

::: tip このページのゴール
良いプロンプトを構成する要素を分解し、「なんとなく書く」から「設計して書く」へ移行することを目指します。テクニックの引き出し([実践テクニック](/prompting/techniques))は次ページで扱います。
:::

## 良いプロンプトの6要素

実務で再利用できるプロンプトは、おおむね次の6つの要素から構成されます。すべてを毎回書く必要はありませんが、**意図的に省略する**のと**書き忘れる**のは別物です。

| 要素 | 役割 | 例 |
| --- | --- | --- |
| 役割(Role) | 振る舞いと前提知識を方向づける | 「あなたはGoの並行処理に詳しいエンジニアです」 |
| 目的(Goal) | 何を達成したいか | 「このハンドラのレース条件を解消したい」 |
| 制約(Constraints) | やってよいこと・いけないこと | 「外部ライブラリは追加しない」 |
| コンテキスト(Context) | 判断に必要な背景 | 既存コード、規約、データ構造 |
| 出力形式(Format) | 成果物の形 | 「差分(unified diff)で出力」 |
| 例(Examples) | 期待する品質の見本 | 既存テストのスタイル |

### 役割(Role)

役割の指定は、モデルの広大な能力空間から**関連する領域に焦点を当てる**ためのものです。「あなたはセキュリティに厳格なバックエンドエンジニアです」と書くと、入力検証やエラーハンドリングへの注意が増す傾向があります。

ただし過信は禁物です。役割を与えても専門知識が"増える"わけではなく、既に持っている知識の**重み付けが変わる**だけです。

### 目的(Goal)

最も省略されがちで、最も重要な要素です。「このコードを直して」ではなく、**何が問題で、どうなったら成功か**を書きます。

```text
悪い: このコードを良くして
良い: このリスト処理を、要素数100万件でもメモリ使用量が一定に
      なるようストリーミング処理に変更したい
```

### 制約(Constraints)

制約は「解の探索空間」を絞ります。特にコード生成では以下が効果的です。

- 使用してよい/いけない言語・ライブラリ・バージョン
- 変更してよい/いけないファイルや関数
- パフォーマンス・互換性の要件
- 守るべき既存規約(命名、エラー処理方針)

### コンテキスト(Context)

LLMはあなたのリポジトリを知りません。判断に必要な情報は**明示的に渡す**必要があります。何をどう渡すかは奥が深いため、[コンテキスト設計](/prompting/context-design)で独立して扱います。

### 出力形式(Format)

出力の形を指定すると、後工程(レビュー・適用)が楽になります。

- `unified diff` 形式 → そのまま `git apply` しやすい
- ファイル全体 → 短いファイルの置き換えに向く
- 「コードのみ、説明不要」→ 自動処理パイプライン向け
- 「変更点を箇条書きで先に、その後コード」→ 人間のレビュー向け

### 例(Examples)

「期待する品質」を言葉で説明するより、**見本を1つ見せる**ほうが正確です。これがFew-shotの原理で、[実践テクニック](/prompting/techniques)で詳述します。

## 3つの設計原則

### 1. 明確さ(Clarity)

代名詞や指示語を避け、対象を一意に特定します。「それ」「この関数」が何を指すか、プロンプト単体で確定できる状態にします。

### 2. 具体性(Specificity)

抽象的な形容詞(「きれいに」「効率的に」「ちゃんと」)は、検証不能なので避けます。**測定可能な基準**に翻訳します。

```text
抽象: パフォーマンスを良くして
具体: 現在O(n^2)のこのループを、O(n log n)以下にしたい。
      ハッシュマップの使用は許容する
```

### 3. 検証可能性(Verifiability)

良いプロンプトは、出力が正しいかを**あなた自身が判定できる**形を要求します。「正しい実装をして」ではなく「この入力に対しこの出力を返すこと。テストも併せて生成すること」と書けば、成否を機械的に確認できます。

::: warning プロンプトの巧拙より検証の設計
プロンプトをどれだけ磨いても、LLMの出力が正しい保証は得られません。重要なのは**生成物を検証できる仕組み**(テスト、型、レビュー)を先に用意することです。これは[コード品質](/quality/code-quality)の中心テーマでもあります。
:::

## ビフォーアフター

### 例A: バグ修正

```text
# ❌ Before
このエラー直して
TypeError: Cannot read property 'id' of undefined
```

問題点: エラーメッセージだけで、コード・再現条件・期待動作がない。モデルは推測でコードを書くしかなく、的外れになりやすい。

```text
# ✅ After
あなたはTypeScriptに精通したフロントエンドエンジニアです。

## 状況
下記の getUser() で、API がユーザーを返さなかった場合に
`Cannot read property 'id' of undefined` が発生します。

## 期待する動作
- ユーザーが存在しない場合は null を返す
- 例外は投げない
- 呼び出し側のシグネチャ(戻り値 User)は変えてよいので、
  User | null に変更し、型の整合性も保つ

## 制約
- 既存の fetchJson ヘルパーを使う(新規ライブラリ追加不可)

## 対象コード
```ts
async function getUser(id: string): Promise<User> {
  const res = await fetchJson(`/users/${id}`);
  return res.data.user;   // res.data が undefined のことがある
}
```

## 出力
修正後のコード全体と、null ケースを検証するテストを1つ。
```

違い: 役割・目的・期待動作・制約・対象コード・出力形式が揃い、**出力が正しいかを呼び出し側が判定できる**。

### 例B: 機能追加

```text
# ❌ Before
ページネーション機能を追加して
```

```text
# ✅ After
## 目的
記事一覧 API にカーソルベースのページネーションを追加する。

## 既存
- エンドポイント: GET /articles
- 現在は全件を配列で返す(下記コード参照)
- DB は PostgreSQL、created_at に降順インデックスあり

## 要件
- クエリパラメータ: limit(デフォルト20, 最大100), cursor(任意)
- cursor は最後の記事の created_at を base64url エンコードした値
- レスポンス: { items: Article[], nextCursor: string | null }
- nextCursor は次ページが存在しない場合 null

## 制約
- オフセットページネーションは使わない(大量データで遅いため)
- 既存の repository 層のクエリビルダ(下記)を踏襲する

（ここに既存コードを貼る）

## 出力
1. 変更点の概要を箇条書き
2. 修正後のコード
3. 境界値(cursor 不正、最終ページ)を含むテスト
```

## コード生成特有のコツ

自然言語タスクと違い、コード生成では**実行環境との整合**が品質を決めます。

- **言語とバージョンを明示** — 「Python 3.12」「Node 20」。構文や標準ライブラリの差で動かないことがある。
- **フレームワークと流儀を明示** — 「React(関数コンポーネント + hooks)」「Spring Boot 3」。同じ言語でも書き方が大きく変わる。
- **既存規約を見本で渡す** — 命名・ディレクトリ構成・エラー処理・ログの作法を、既存ファイル1つを例示して伝える。「このファイルのスタイルに合わせて」が最も確実。
- **依存の制約を明示** — 「標準ライブラリのみ」「lodash は使用可」など。勝手に依存を増やされる事故を防ぐ。
- **テストとセットで要求** — 実装とテストを同時に出させると、モデル自身が仕様を二重化するため、ズレに気づきやすい。

::: details なぜ「このファイルに合わせて」が効くのか
モデルは与えられたコードの**統計的パターン**(命名、インデント、構造)を強く参照します。規約を言葉で長々と説明するより、実物を1つ見せるほうが、トークン効率も精度も上です。これはコンテキスト設計の核心でもあります。
:::

## まとめ

- プロンプトは**仕様書**。曖昧さは出力の曖昧さに直結する。
- 役割・目的・制約・コンテキスト・出力形式・例の6要素を意識する。
- 抽象的形容詞を**測定可能な基準**に翻訳する。
- 最重要は**検証可能性**。プロンプトの巧拙は検証の仕組みを代替しない。
