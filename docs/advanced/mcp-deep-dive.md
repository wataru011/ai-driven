# MCP 実装ディープダイブ

[ツール使用とMCP](/agents/tools-and-mcp)では、エージェントがツールを使う仕組みと、それを標準化する **MCP(Model Context Protocol)** の概要を扱いました。本ページはその発展編として、**MCPをどう実装し、どう安全に運用するか**を設計レベルで掘り下げます。

MCPは、Anthropicが提唱した **オープンな標準プロトコル**です。狙いは単純で、「LLMアプリケーションに外部のデータやツールをつなぐ口を標準化する」こと。これにより、各ツール提供者が一度MCPサーバーを実装すれば、対応するあらゆるホストから使えるようになります。

::: tip ひとことで言うと
MCPとは、**「LLMアプリと外部能力をつなぐUSB-Cのような共通規格」** です。M個のアプリ × N個のツールを個別に配線する代わりに、双方がMCPに準拠すれば疎結合につながります。
:::

## なぜMCPが必要か — M×N問題

MCPがない世界では、各LLMアプリが各ツール・各データソースに **個別の接続コード** を書く必要があります。アプリがM個、ツールがN個あれば、最悪 M×N 通りの実装が要ります。

```text
[MCP以前]                      [MCP以後]
アプリA ─┬─ DB                 アプリA ─┐
        ├─ GitHub                       ├─ (MCPクライアント) ─┬─ DBサーバー
        └─ ファイル            アプリB ─┤                     ├─ GitHubサーバー
アプリB ─┬─ DB                 アプリC ─┘                     └─ ファイルサーバー
        ├─ GitHub
        └─ ファイル           各ツールは1回だけMCPサーバーを実装すれば全アプリから使える
（M×N の個別配線）             （M+N に削減）
```

標準化の利点は、コードの再利用だけではありません。**権限・認証・エラー処理のパターンを共通化**でき、エコシステムとしてツールが流通するようになります。

## アーキテクチャ: ホスト / クライアント / サーバー

MCPは3つの役割で構成されます。

| 役割 | 実体 | 責務 |
| --- | --- | --- |
| ホスト(Host) | LLMアプリ本体(IDE、チャット、エージェント等) | ユーザーとLLMを抱え、全体を統括。どのサーバーに接続するか決める |
| クライアント(Client) | ホスト内のMCP接続コンポーネント | 1つのサーバーと1対1で接続し、プロトコルを話す |
| サーバー(Server) | ツール/データを公開するプロセス | tools/resources/promptsを提供する。ホストとは別プロセス |

重要な点は、**サーバーがホストとは独立したプロセス**であることです。これが後述する **信頼境界(trust boundary)** の基盤になります。サーバーはローカルのプロセスでも、リモートのサービスでもかまいません。

```text
┌─────────────── Host(LLMアプリ)───────────────┐
│  LLM  ←→  オーケストレーション                  │
│            │                                   │
│       ┌────┴────┐  ┌─────────┐                 │
│       │Client A │  │Client B │  ... 各サーバーに1対1
│       └────┬────┘  └────┬────┘                 │
└────────────┼────────────┼──────────────────────┘
             │ (トランスポート)                  │
        ┌────┴────┐  ┌────┴────┐
        │Server A │  │Server B │  ← 別プロセス。信頼境界
        │(GitHub) │  │ (DB)    │
        └─────────┘  └─────────┘
```

## サーバーが提供する3要素: tools / resources / prompts

MCPサーバーは、ホストに対して3種類の能力を公開できます。この区別の理解が実装の肝です。

| 要素 | 何か | 制御の主体 | たとえ |
| --- | --- | --- | --- |
| Tools | 副作用を伴う/結果を返す **関数** | モデルが呼ぶか判断(model-controlled) | POST的な操作 |
| Resources | 読み取り可能な **データ**(ファイル、レコード等) | アプリ/ユーザーが選んで渡す(app-controlled) | GET的な参照 |
| Prompts | 再利用可能な **プロンプトテンプレート** | ユーザーが選ぶ(user-controlled) | スラッシュコマンド的 |

- **Tools** — モデルが「これを呼びたい」と判断して実行するもの。検索、コード実行、API操作など。副作用を持ちうるため最も注意が要ります。
- **Resources** — モデルに読ませたいコンテキスト。ファイル内容、DBレコード、ドキュメントなど。**誰が・いつ渡すかをアプリ側が制御**できる点がツールと異なります。
- **Prompts** — 定型のプロンプトを部品化したもの。ユーザーが明示的に呼び出す想定。

::: info この3分割が安全設計につながる
「モデルが勝手に呼べるもの(tools)」「アプリが選んで渡すもの(resources)」「ユーザーが起動するもの(prompts)」と **制御主体が分かれている**ことが重要です。副作用のある操作をtoolsに、機微なデータの提示をresources(アプリ制御)に置くことで、権限の粒度を設計できます。
:::

## トランスポートの概念

ホストのクライアントとサーバーは、**トランスポート**を介してメッセージをやり取りします。MCPはメッセージ形式(JSON-RPCベースの構造化メッセージ)を規定し、それを運ぶ経路として代表的に2系統があります。

| トランスポート | 接続形態 | 主な用途 |
| --- | --- | --- |
| 標準入出力(stdio) | ローカルの子プロセスと標準入出力で通信 | ローカルツール、同一マシン |
| HTTPベース | ネットワーク越しのリモートサーバー | リモートサービス、共有サーバー |

ローカルツール(ファイルアクセス等)は子プロセスとしてstdioで、リモートの共有サービスはHTTPで、と使い分けます。**トランスポートが変わってもツール/リソースの定義は同じ**——これが標準化の恩恵です。具体的な仕様(エンドポイント、認証方式)は更新されるため、実装時は **MCPの一次仕様を確認**してください。

## MCPサーバー実装の流れ

サーバー実装は、概念的には次の手順です。擬似コードで骨格を示します。

```python
# 1. サーバーを初期化し、トランスポートを選ぶ
server = MCPServer(name="orders-server")

# 2. 提供する tools を登録する(名前・説明・入力スキーマ)
@server.tool(
    name="search_orders",
    description="注文IDまたはメールで注文を検索する",
    input_schema={
        "type": "object",
        "properties": {
            "order_id": {"type": "string"},
            "email": {"type": "string"},
        },
    },
)
def search_orders(order_id=None, email=None):
    # 入力を検証してから実処理(権限チェックはここで)
    require_authorized(current_caller())
    return db.search(order_id=order_id, email=email)

# 3. 読み取り可能な resources を公開する
@server.resource(uri="orders://recent")
def recent_orders():
    return db.recent(limit=50)

# 4. 再利用プロンプトを登録する
@server.prompt(name="refund_flow")
def refund_flow(order_id):
    return f"注文 {order_id} の返金可否を、返金ポリシーに照らして判断してください。"

# 5. ハンドシェイクして待ち受ける(capability のネゴシエーション)
server.run(transport="stdio")
```

実装の流れを要約すると次の通りです。

1. **初期化とハンドシェイク** — ホストのクライアントと接続し、互いの対応機能(capability)を交換する。
2. **tools/resources/promptsの登録** — それぞれを定義し、説明と入力スキーマを明示する。説明の質が[ツールの使われ方](/agents/tools-and-mcp)を左右する。
3. **リクエスト処理** — `list`(一覧)と `call`(実行)に応じ、**入力検証と権限チェックを通してから**実処理する。
4. **結果の返却とエラー処理** — 構造化した結果/エラーを返す。

## セキュリティ: 権限・信頼境界・サンドボックス

MCPサーバーは外部プロセスであり、しばしば **強い権限(ファイル、DB、API)** を持ちます。ここはセキュリティ設計の最重要点です。詳細な脅威は[セキュリティ ディープダイブ](/advanced/security-deep-dive)で扱いますが、MCP固有の勘所を挙げます。

::: warning MCPサーバーは攻撃面そのもの
サーバーの説明文(ツールのdescription)やリソースの中身は、**LLMの文脈に入る**ため、[間接プロンプトインジェクション](/advanced/security-deep-dive)の経路になりえます。「信頼できないサーバーを安易に接続しない」「サーバーが返すデータを命令として実行しない」が原則です。
:::

| 観点 | 設計指針 |
| --- | --- |
| 最小権限 | サーバーには必要最小限の権限だけを与える。読み取りで足りるなら書き込み権限を渡さない |
| 信頼境界 | ホストとサーバーはプロセス分離。サーバーの出力を無検証で信頼しない |
| サンドボックス | コード実行・ファイル操作系は隔離環境(コンテナ等)で。ネットワーク・ファイルアクセスを制限 |
| 認証・認可 | リモートサーバーは認証必須。tool呼び出しごとに呼び出し元の権限を確認 |
| 人間の承認 | 破壊的/不可逆な操作(削除・課金・公開)は[Human-in-the-loop](/advanced/agent-design-patterns)で承認ゲート |
| サプライチェーン | 出所不明のMCPサーバーは[供給網リスク](/advanced/security-deep-dive)。検証してから導入 |
| 監査ログ | どのツールがどの引数で呼ばれたかを記録し、追跡可能にする |

```text
信頼境界の考え方:
  Host(信頼) ── 境界 ── Server(準信頼/非信頼)
                          └─ さらに外部API/DB(非信頼の入力源)

  原則: 境界を越えて入ってくるデータは「データ」として扱い、
        「命令」として実行しない。権限は境界ごとに最小化する。
```

## 実務での使いどころ

MCPが特に効く場面と、注意が要る場面を整理します。

**効く場面**

- **社内システムへの接続を標準化したい** — DB、チケット、社内APIを一度MCPサーバー化すれば、複数のAIツールから再利用できる。
- **ツールを組織内で流通させたい** — チームが作ったツールを、対応ホスト間で共有できる。
- **ホストとツールを疎結合に保ちたい** — ホスト(AIクライアント)を乗り換えても、MCPサーバー資産が活きる。

**注意が要る場面**

- **信頼できないサーバーの導入** — 第三者製サーバーは権限とプロンプトインジェクションの観点で審査する。
- **過剰なツール公開** — 多すぎるツールはモデルの誤用と[コンテキスト肥大化](/models/cost-optimization)を招く。必要なものに絞る。
- **機微データの露出** — resourcesで何を読ませるかは慎重に。最小権限とアクセス制御を徹底する。

::: tip まずは読み取り中心の小さなサーバーから
最初から書き込み・実行権限を持つサーバーを作ると、リスク評価が重くなります。**読み取り(resources)中心の小さなサーバー**から始め、副作用のあるtoolsは権限チェックと承認ゲートを固めてから追加するのが安全です。
:::

## まとめ

- MCPはAnthropic提唱の **オープン標準**。LLMアプリと外部能力をつなぐ口を標準化し、M×N問題をM+Nに縮める。
- アーキテクチャは **ホスト / クライアント / サーバー**。サーバーは別プロセスで、これが信頼境界になる。
- サーバーは **tools(モデル制御)/ resources(アプリ制御)/ prompts(ユーザー制御)** を提供し、制御主体の違いが安全設計の鍵。
- 実装は **登録 → ハンドシェイク → 入力検証・権限チェック → 実行**。
- セキュリティは **最小権限・信頼境界・サンドボックス・承認ゲート・監査**。サーバーの出力を命令として実行しない。
