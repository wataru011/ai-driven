# プロンプト/CLIチートシート

すぐ手元で引ける早見表です。テンプレはコピーして文脈を埋めるだけで使えます。原理の解説は[プロンプトの基礎](/prompting/fundamentals)に譲り、ここでは**「使える形」**に絞ります。

::: tip 前提
ここに挙げる型は出発点です。プロジェクトの文脈(言語・規約・制約)を必ず差し込んでください。空欄`{ }`は埋めるべき箇所です。
:::

## プロンプトの基本骨格

良いプロンプトはおおむね次の5要素で構成されます。

```
# 役割
あなたは {言語/領域} に精通したエンジニアです。

# 文脈
{プロジェクト・既存コード・制約・前提}

# 指示
{何を・どうしてほしいか を具体的に}

# 出力形式
{コードのみ / 差分 / 箇条書き / 表 など}

# 制約
{使ってよい/ダメな依存・スタイル・禁止事項}
```

## 効くプロンプトの型(短文テンプレ)

### 実装

```
{関数/クラス} を実装して。入力は {型}、出力は {型}。
エッジケース({空/null/境界値}) も処理し、{言語} の慣用に従って。
```

```
この要求を満たす最小の実装を書いて。過剰な抽象化はしないで:
{要求}
```

### 理解・読解

```
このコードが何をするか、3段階(要約 / 詳細な流れ / 注意点)で説明して:
{コード}
```

```
このコードベースで {機能} はどこで実装されている? 関連ファイルと
呼び出し関係を挙げて。推測なら推測と明記して。
```

### リファクタリング

```
振る舞いを変えずに {可読性/重複削減/命名} を改善して。
変更点を差分で示し、各変更の理由を1行で添えて。
```

```
このコードの問題点を、優先度順(高/中/低)に列挙して。
直す前に、まず指摘だけ出して。
```

### テスト

```
この関数のテストを書いて。正常系・異常系・境界値を網羅し、
各テストが「何を保証するか」をテスト名で表して。
```

```
このコードで未カバーになりやすいエッジケースを列挙して。
コードはまだ書かないで、観点だけ出して。
```

### デバッグ

```
この症状の原因仮説を、可能性の高い順に挙げて。
各仮説の確認方法も添えて。まだ修正はしないで:
症状: {現象} / 期待: {期待} / 再現: {手順}
```

### レビュー

```
このPRを、正確性・セキュリティ・パフォーマンス・可読性の観点でレビューして。
ブロッカーと nit を分けて示し、根拠を添えて。
```

### 設計

```
{要件} を満たす設計案を2〜3個出して。それぞれ長所・短所・適する状況を
表で比較して。結論は出さず、選択軸を示して。
```

::: warning 効きにくい指示
「いい感じにして」「最適化して」「ちゃんと書いて」のような曖昧語は出力を不安定にします。**何を・どの基準で・どの形式で**を必ず添えてください。
:::

## プロンプトを安定させるコツ

- [ ] タスクを小さく割る(1プロンプト1目的)
- [ ] 必要な文脈だけ渡す(巨大すぎる入力は精度を下げる)
- [ ] 出力形式を指定する(差分/表/コードのみ)
- [ ] 「推測なら推測と明記して」と一言添える(ハルシネーション対策)
- [ ] 例を1〜2個示す(Few-shot で形式が安定)
- [ ] 「まず計画 → 承認後に実装」と段階を分ける(大きな変更時)

## レビュー観点の早見

AI生成コードは**人間のコードと同じ基準**でレビューします。特に下記は機械任せにせず人間が見ます。

| 観点 | 見るポイント |
| --- | --- |
| 正確性 | 要求どおりか / エッジケース / オフバイワン / 例外処理 |
| セキュリティ | 入力検証 / インジェクション / 認証・認可 / シークレット混入 |
| 依存 | 追加パッケージは実在し信頼できるか([slopsquatting](/reference/glossary)に注意) |
| 整合性 | 既存の規約・命名・アーキテクチャに沿うか |
| パフォーマンス | 不要なループ / N+1 / 過剰なメモリ |
| 可読性 | 命名 / 過剰な抽象化がないか |
| テスト | テストが本質を保証しているか(通るだけでないか) |
| 幻覚 | 実在しないAPI・関数・設定を使っていないか |

セキュリティの詳細は[セキュリティ](/quality/security)を参照。

## タスク別おすすめの進め方

| タスク | 進め方の勘所 | 任せる度合い |
| --- | --- | --- |
| ボイラープレート生成 | 形式と制約を明示。そのまま使えることが多い | 高 |
| 定型リファクタリング | 「振る舞いを変えるな」を明記。テストで担保 | 高 |
| テスト追加 | 観点は人間が決め、雛形をAIに | 中〜高 |
| 新機能実装 | 仕様を先に固める。計画→実装に分ける | 中 |
| 込み入ったバグ修正 | 原因仮説を出させ、修正は人間が判断 | 低〜中 |
| アーキテクチャ設計 | 案出しに使い、決定は人間 | 低 |
| 本番に影響する操作 | AIに任せず人間が承認 | 不可 |

仕様を起点にする進め方は[仕様駆動開発](/advanced/spec-driven)、機能開発の流れは[機能開発の実践](/practice/feature-development)。

## エージェントCLIの操作の考え方

::: info 製品固有のフラグは断定しません
コーディングエージェントCLIは製品ごとにコマンド・オプションが異なり、変化も速いです。ここでは**どの製品にもおおむね通じる「操作の概念」**を示します。正確なフラグは各製品の公式ドキュメントを確認してください([リソース集](/reference/resources))。
:::

| 概念操作 | 目的 | 勘所 |
| --- | --- | --- |
| 作業ディレクトリ/スコープの指定 | AIが触れる範囲を限定 | 関係ないディレクトリを巻き込ませない |
| コンテキスト/ファイルの明示的な追加 | 必要な文脈だけ渡す | 大量投入より関連ファイルを厳選 |
| プロジェクト指示ファイルの用意 | 規約・前提を毎回伝えなくて済む | 命名・スタイル・禁止事項を記述 |
| 計画モード/確認モード | 実行前に意図を確認 | 大きな変更はまず計画を出させる |
| 差分の提示と承認 | 適用前に人間がレビュー | 自動適用はリスクに応じて絞る |
| 権限/許可リストの設定 | 危険操作をブロック | 最小権限を既定にする |
| 履歴/コンテキストのリセット | 文脈の汚染を断つ | タスクが変わったらリセット |
| モデルの切り替え | 難度に応じてコスト最適化 | 難所だけ上位モデル |

### 運用の作法

- [ ] バージョン管理下で作業し、変更はコミット単位で確認できるようにする
- [ ] 重要ブランチで直接作業させず、作業ブランチを切る
- [ ] 実行前に差分をレビューしてから取り込む
- [ ] エージェントの権限は最小から始め、必要に応じて広げる
- [ ] 信頼できない入力(外部Issue・Web・ファイル)を特権操作に直結させない

## 危険操作の禁止リスト

AIやエージェントに**任せてはいけない/必ず人間の承認を挟む**操作です。エージェントの許可リストにも反映してください。

::: danger 自動実行させない操作
- 本番環境へのデプロイ・リリース
- データベースの破壊的操作(`DROP` / `DELETE` / `TRUNCATE` / 一括 `UPDATE`)
- ファイル/ディレクトリの再帰的・強制削除(`rm -rf` 相当)
- Git履歴の破壊的書き換え・強制プッシュ(`push --force` 相当)
- 認証・認可・権限・課金まわりの変更
- シークレット・認証情報の生成/取得/出力/送信
- 外部への任意ネットワーク送信(情報持ち出しの経路)
- パッケージの無確認な追加・グローバルインストール
- インフラ/権限を変更するコマンドの無確認実行
:::

::: warning 「無確認」が危険の核
上記の多くは操作自体ではなく**人間の確認を飛ばすこと**が危険です。可逆で検証容易なら任せ、不可逆・高リスクなら承認ゲートを必ず挟む——この線引きは[エージェントとは](/agents/what-is-agent)の委任の考え方に沿います。
:::

## ハルシネーションを踏まないチェック

- [ ] 提案された関数・API・設定キーが実在するか確認した
- [ ] 追加された依存パッケージが実在し、信頼できるソースか確認した
- [ ] 引用された数値・仕様は一次情報で裏取りした
- [ ] 「自信ありげな断定」を鵜呑みにせず検証した

背景は[ハルシネーション](/quality/hallucination)を参照。
