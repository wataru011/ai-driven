# 埋め込みとRAG

[LLMの仕組み](/concepts/how-llms-work)で見たとおり、モデルの知識には**カットオフ**があり、社内コードや最新仕様といった**学習時に存在しなかった情報**は知りません。また[コンテキストウィンドウ](/concepts/context-window)は有限なので、関連資料を全部貼ることもできません。

この2つの制約を同時に解くのが **RAG(Retrieval-Augmented Generation, 検索拡張生成)** です。要点は単純で、

> 質問に**関連する情報を外部から検索して取得**し、それをコンテキストに**注入してから**生成させる。

これにより、モデルを再学習せずに「最新かつ自社固有の知識」を扱え、しかも**根拠(出典)を提示**できます。

::: tip ひとことで言うと
RAGとは、**LLMに「カンニングペーパー」を渡してから答えさせる**仕組みです。記憶に頼らせる代わりに、**手元の根拠を見て答えさせる**ことで、ハルシネーションを減らし最新情報に対応します。
:::

## 埋め込みベクトルとは

RAGの心臓部は**埋め込み(embedding)** です。テキスト(単語・文・コード片)を、**意味を表す数百〜数千次元の数値ベクトル**に変換したものです。

重要な性質はただ1つ。

> **意味が近いテキストほど、ベクトル空間上で近くに配置される。**

```text
"猫"        → [0.21, -0.88,  0.05, ... ]
"ねこ"      → [0.20, -0.85,  0.07, ... ]   ← "猫" に近い(意味が近い)
"自動車"    → [-0.74, 0.12,  0.66, ... ]   ← 遠い(意味が違う)
```

近さは通常 **コサイン類似度**で測ります。キーワード一致(文字列が同じか)ではなく、**意味の一致**を捉えられるのがポイントです。「ファイルを読む」と「read a file」、「ログイン処理」と「authentication handler」が、表記が違っても近いベクトルになります。

## 意味的検索(Semantic Search)

埋め込みを使うと、従来のキーワード検索を超えた**意味的検索**が可能になります。

| | キーワード検索(全文検索) | 意味的検索(埋め込み) |
| --- | --- | --- |
| マッチの基準 | 文字列の一致 | 意味の近さ |
| 言い換えへの強さ | 弱い(同義語に弱い) | 強い |
| 表記ゆれ・多言語 | 弱い | 強い |
| 「authとloginは関連」 | 捉えられない | 捉えられる |
| 厳密な識別子(`getUserById`) | 強い | やや弱いことも |

実務では**両者を併用するハイブリッド検索**(キーワード検索 + 意味的検索を統合)が、コード検索を含め最も堅実です。

## ベクトルDB

埋め込みを大量に保存し、「クエリベクトルに近いベクトルを高速に探す(**近傍探索, ANN**)」役割を担うのが**ベクトルDB**です。

- 専用DB:Pinecone, Weaviate, Qdrant, Milvus など
- 既存DBの拡張:PostgreSQLの `pgvector`、Elasticsearch など
- ローカル/軽量:FAISS, Chroma など

数千件程度なら総当たりでも十分ですが、件数が増えると近似最近傍(ANN)インデックスで高速化します。選定はスケール・運用・既存スタックとの相性で決めます。

## RAGのパイプライン:取得 → 拡張 → 生成

RAGは大きく**事前のインデックス構築**と**実行時の取得・生成**に分かれます。

### フェーズ1:インデックス構築(事前・オフライン)

```text
ドキュメント/コード
  → チャンク分割(意味のある単位に切る)
  → 各チャンクを埋め込みベクトルに変換
  → ベクトルDBに格納(本文・メタデータ付き)
```

### フェーズ2:取得 → 拡張 → 生成(実行時)

```text
ユーザーの質問
  ① 取得(Retrieve)    : 質問を埋め込み、ベクトルDBで近いチャンクを上位k件取得
  ② 拡張(Augment)     : 取得チャンクをプロンプトに差し込み、コンテキストを組み立て
  ③ 生成(Generate)    : LLMが「渡された根拠」を基に回答(出典付きで)
```

疑似コードで示すと次のようになります。

```python
# --- 事前:インデックス構築 ---
for doc in documents:
    for chunk in split_into_chunks(doc):          # チャンク戦略が効く
        vec = embed(chunk.text)                    # 埋め込みモデルで変換
        vector_db.upsert(id=chunk.id, vector=vec, payload=chunk)

# --- 実行時:取得→拡張→生成 ---
def answer(question: str) -> str:
    q_vec   = embed(question)                      # ① 質問を埋め込む
    hits    = vector_db.search(q_vec, top_k=5)     # ① 近いチャンクを取得
    hits    = rerank(question, hits)               # 任意:再ランクで精度向上
    context = "\n\n".join(h.payload.text for h in hits)  # ② 拡張

    prompt = f"""以下の参照情報のみに基づいて回答し、根拠の出典を示すこと。
参照情報:
{context}

質問: {question}"""
    return llm.generate(prompt)                     # ③ 生成
```

::: info なぜ「参照情報のみに基づいて」と指示するのか
RAGの価値は**根拠に基づかせること**にあります。指示で「渡した情報の外を勝手に補完するな/分からなければ分からないと言え」と縛ることで、[ハルシネーション](/quality/hallucination)を抑えられます。取得が外れたときに堂々と捏造させない歯止めです。
:::

## コードベースへのRAG適用

RAGはドキュメントQAだけのものではありません。**大規模コードベースの横断理解**にこそ強力です。AIコーディングツールやエージェントは、内部でコード版RAGを使ってリポジトリを"読みこなして"います。

典型的なユースケース:

- **コード検索** — 「ユーザー認証はどこで実装されている?」に、ファイル名を知らなくても意味で到達。
- **横断理解** — 機能改修時に、関連する呼び出し元・型定義・テストを集めてコンテキストに載せる。
- **仕様復元・オンボーディング** — レガシーコードの該当箇所を集約し、挙動を説明させる。
- **既存パターンの踏襲** — 「このリポジトリの流儀でエンドポイントを追加して」を、近傍の実例を取得して実現。

コード特有の工夫:

- **構文を意識した分割**(関数・クラス・メソッド単位)。行数で機械的に切ると意味が壊れる。
- **メタデータの付与**(ファイルパス、言語、シンボル名、依存関係)で絞り込みと出典提示を可能に。
- **記号検索の併用** — 厳密な識別子(`getUserById`)は意味検索よりキーワード/シンボル検索が強い。ハイブリッドが効く。

これらをツールとして呼び出す仕組みは[ツールとMCP](/agents/tools-and-mcp)で扱います。

## チャンク戦略

RAGの品質は**チャンクの切り方で大きく変わります**。代表的な戦略とトレードオフ:

| 戦略 | 内容 | 長所 | 短所 |
| --- | --- | --- | --- |
| **固定長** | N トークンごとに機械的に分割 | 実装が単純 | 文・関数の途中で切れ意味が壊れる |
| **オーバーラップ付き** | 隣接チャンクを一部重複させる | 境界の文脈断絶を緩和 | 冗長・トークン増 |
| **構造ベース** | 見出し/段落、関数/クラス境界で分割 | 意味的にまとまる | パーサ・言語ごとの実装が要る |
| **意味ベース** | 話題の切れ目で分割 | 高品質 | 計算コストが高い |

実務指針:

- **小さすぎる**チャンク → 文脈が足りず、断片だけでは意味が通らない。
- **大きすぎる**チャンク → 1チャンクに無関係な内容が混ざり、検索精度と[コンテキスト効率](/concepts/context-window)が落ちる。
- コードは**関数/クラス単位 + 適度なオーバーラップ**を起点に、対象に合わせて調整するのが堅実。

## RAGの限界と落とし穴

RAGは万能ではありません。失敗の多くは「取得」の失敗に起因します。

::: warning よくある落とし穴
- **取得が外れると生成も外れる(garbage in, garbage out)** — 関連チャンクを引けなければ、いくら賢いモデルでも正答できない。RAGの精度は**検索精度に律速**される。
- **取得したのに無視される** — チャンクが[lost in the middle](/concepts/context-window#lost-in-the-middle-問題)で見落とされる。関連度順の並べ替えと再ランクが効く。
- **チャンク分断による文脈喪失** — 答えが複数チャンクにまたがると拾いきれない。
- **インデックスの陳腐化** — 元データが変わったのに再インデックスしていない。コードベースなら更新パイプラインが必須。
- **意味は近いが用途は違う** — 似た語のコードを誤って引く。メタデータ絞り込みやハイブリッド検索で緩和。
- **マルチホップな問い** — 「AがBを呼び、BがCに依存する」のような連鎖推論は単純RAGでは弱い。
:::

RAGとファインチューニングはしばしば対比されますが、**目的が違います**。RAGは**知識の注入(何を知っているか)**、ファインチューニングは主に**振る舞いの調整(どう答えるか)**に効きます。使い分けは[ファインチューニングと拡張](/concepts/fine-tuning)で体系的に整理します。

## まとめ

- **埋め込み**はテキストを意味ベクトルに変換し、**意味的検索**を可能にする。
- **RAG = 取得 → 拡張 → 生成**。根拠を渡してから答えさせ、最新性と出典を確保する。
- コードベースでは**横断理解・コード検索**に強く、**構文を意識したチャンク**と**ハイブリッド検索**が鍵。
- RAGの精度は**検索精度に律速**される。落とし穴の多くは「取得の失敗」。
