# AI駆動開発とは

**AI駆動開発(AI-Driven Development, AIDD)** とは、生成AI、とりわけ大規模言語モデル(LLM)を、設計・実装・テスト・レビュー・運用といったソフトウェア開発ライフサイクルの中核的な"作業主体"として組み込み、開発の速度・品質・スケールを引き上げる開発手法の総称です。

重要なのは、これが単なる「賢い入力補完」ではないという点です。補完(autocomplete)は1980年代から存在する機能の延長ですが、AI駆動開発が扱うのは **意図(intent)からの生成** です。自然言語で表現された要求・制約・コンテキストを入力に、コード・テスト・ドキュメント・設計案といった成果物を生成し、人間がそれをレビュー・統合するという、新しい作業分担を前提とします。

::: tip ひとことで言うと
AI駆動開発とは、**「人間が"何を・なぜ"を担い、AIが"どう書くか"の初稿を担う」** ように開発の分業を組み替える試みです。最終的な責任と判断は依然として人間にあります。
:::

## なぜ今、注目されているのか

LLMによるコード生成は概念としては新しくありませんが、2022年以降に実用段階へ移行しました。背景には3つの変化があります。

1. **モデル能力の閾値突破** — コードの文法的正しさだけでなく、複数ファイルにまたがる文脈理解、ライブラリのAPIの正確な利用、テストの生成までが実用水準に達した。
2. **コンテキスト長の拡大** — 数千トークンから数十万トークンへとコンテキストウィンドウが拡大し、リポジトリ規模の情報を一度に扱えるようになった。
3. **エージェント化** — モデルが「ファイルを読む」「コマンドを実行する」「結果を見て次の行動を決める」といったツール使用を行えるようになり、単発の生成からマルチステップの自律実行へと進化した。

この3つが揃ったことで、AIは「スニペットを提案する補助輪」から「タスクを任せられる協働者」へと役割を変えつつあります。

## AI駆動開発の成熟度モデル

組織やチームのAI活用は、一足飛びには進みません。以下の5段階で自分の現在地を把握すると、次の打ち手が見えやすくなります。

| レベル | 名称 | 主な使い方 | 人間の関与 |
| --- | --- | --- | --- |
| L0 | 不使用 | AIを使わない / 禁止 | 100% |
| L1 | 補完 | インライン補完、ボイラープレート生成 | 生成のたびに確認 |
| L2 | 対話 | チャットで設計相談・コード片生成・エラー解決 | 抜粋を取捨選択 |
| L3 | 委任 | タスク単位でAIに実装させ、人間はレビュー | PR単位でレビュー |
| L4 | 協調 | エージェントが複数ステップを自律実行、人間は要所で介入 | 目標設定と承認 |
| L5 | 自律 | 定型タスクをエージェントがエンドツーエンドで完遂 | 例外時のみ介入 |

多くの先進的なチームは現在 **L2〜L3** におり、L4の「エージェント協調」へ移行しようとしている段階です。L5は限定的なドメイン(テスト生成、依存更新、定型リファクタリングなど)で部分的に実現しています。

::: warning レベルを上げること自体は目的ではない
高いレベルが常に正しいわけではありません。タスクのリスク・可逆性・検証コストによって最適なレベルは変わります。重要なバグ修正は L2 で慎重に、依存パッケージの更新は L4 で自動化、というように **タスクごとに使い分ける** のが熟練者の戦略です。
:::

## AI駆動開発が扱う範囲

「コードを書く」だけがAI駆動開発ではありません。本サイトでは以下の範囲を扱います。

- **理解(Comprehend)** — 既存コードの読解、レガシーコードの仕様復元、ドキュメント生成
- **設計(Design)** — アーキテクチャ案の比較、API設計、データモデリングのたたき台
- **実装(Implement)** — 機能実装、ボイラープレート、定型コードの生成
- **検証(Verify)** — テスト生成、エッジケース洗い出し、レビュー観点の提示
- **保守(Maintain)** — リファクタリング、依存更新、移行作業、バグ修正
- **運用(Operate)** — ログ解析、インシデント調査支援、ランブック生成

開発時間の多くは「新規コードを書く」ことではなく、**理解・検証・保守**に費やされます。AI駆動開発の真価も、実はこの地味な領域で発揮されます。

## よくある誤解

#### 誤解1: AIがプログラマーを置き換える

現実には、AIは"書く"作業の一部を加速しますが、**何を作るべきかの判断、トレードオフの評価、責任の引き受け**は人間に残ります。むしろAIによって、上流の設計判断やレビュー能力の価値が相対的に高まります。

#### 誤解2: 生成されたコードはそのまま使える

LLMは**もっともらしいテキスト**を生成するのであって、**正しいコード**を保証しません。生成物は常にレビューとテストの対象です。これは[ハルシネーション対策](/quality/hallucination)で詳述します。

#### 誤解3: プロンプトが上手ければ品質は保証される

プロンプトは入力の質を左右しますが、出力の正しさを保証しません。**検証可能性(testability)を設計に織り込む**ことが、プロンプトの巧拙よりも重要です。
