# ツールエコシステムの全体像

AI開発ツールは、2022年以降に爆発的に増えました。GitHub Copilot, Cursor, Claude Code, Cline, Aider, Windsurf, Codeium, JetBrains AI Assistant, Tabnine, Amazon Q Developer, Devin, Sourcegraph Cody——名前だけでも目眩がします。しかも各ツールが「補完もできる」「チャットもできる」「エージェントもできる」と機能を拡張し続けるため、製品名で覚えようとすると、すぐに陳腐化します。

熟練者が採るべき視点は、**製品ではなく役割(role)で分類する**ことです。ツールは「どの作業を、どの粒度で、どの程度の自律性で担うか」によって性格が決まります。本ページでは、AI開発ツールを6つの役割に分類するフレームを提示し、それらを **層(layer)として組み合わせて使う** という実務的な見方を示します。

::: tip このページの要点
ツール選定は「1つの最強ツールを選ぶ」問題ではなく、「複数の層をどう組み合わせるか」という問題です。インライン補完・IDEチャット・CLIエージェントは競合ではなく **共存** します。
:::

## 6つの役割分類

| 分類 | 役割 | 粒度 | 自律性 | 代表例 |
| --- | --- | --- | --- | --- |
| (a) インライン補完型 | カーソル位置の続きを提案 | 行〜数行 | 最小 | GitHub Copilot, Codeium, Tabnine, JetBrains AI |
| (b) IDE統合チャット型 | エディタ内で対話・選択範囲編集 | 関数〜ファイル | 低 | Cursor, Copilot Chat, Cody, JetBrains AI |
| (c) CLI/エージェント型 | 端末で複数ファイル編集・コマンド実行 | タスク〜機能 | 中〜高 | Claude Code, Aider, Cline |
| (d) コードベース検索型 | リポジトリ横断の検索・質問応答 | リポジトリ | 低 | Sourcegraph Cody, Amazon Q Developer |
| (e) 自律エージェント型 | 課題を受け取り端から端まで実行 | 課題〜PR | 高 | Devin |
| (f) レビュー/品質型 | 差分のレビュー・脆弱性指摘 | PR/差分 | 低〜中 | Copilot (PRレビュー), Amazon Q, Cody |

::: warning 分類は性格であって境界ではない
多くの製品は複数の役割を兼ねます。たとえばCursorは(a)補完と(b)チャットと(c)エージェント的編集を1つのIDEに統合しています。表の「代表例」は **その役割を象徴する** ツールであり、「その役割しかできない」という意味ではありません。
:::

## (a) インライン補完型

エディタでコードを書いている最中に、カーソル位置の続きをグレーアウト表示で提案する方式です。Tabキーで受理、無視すれば消える、という低摩擦な体験が特徴です。

- **適する場面**: ボイラープレート、定型的なループやエラーハンドリング、既存パターンの反復、テストの雛形。
- **強み**: 思考を中断しない。受理コストがゼロに近い。
- **弱み**: 文脈は基本的に「開いているファイル+近傍」に限られ、設計判断やリポジトリ全体の整合性は守れない。
- **代表**: GitHub Copilot, Codeium, Tabnine, JetBrains AI Assistant。

詳細は[コーディングアシスタント](/tools/coding-assistants)で扱います。

## (b) IDE統合チャット型

エディタの中で対話し、選択範囲やファイル単位で生成・修正させる方式です。「この関数をリファクタして」「このエラーを直して」といった指示を、コードのコンテキストとともに送れます。

- **適する場面**: 既存コードの説明、局所的なリファクタリング、エラー解決、小〜中規模の機能追加。
- **強み**: 該当箇所を選んで指示できるため、補完より大きな単位を扱える。差分プレビューが効く。
- **弱み**: 人間がコンテキスト(関連ファイル)を手動で渡す必要があり、規模が大きいと破綻しやすい。
- **代表**: Cursor, GitHub Copilot Chat, Sourcegraph Cody, JetBrains AI Assistant。

## (c) CLI/エージェント型

端末から起動し、複数ファイルの編集・コマンド実行・テスト実行を **マルチステップで** こなす方式です。「失敗したら直す」ループを自分で回せる点が、補完やチャットと決定的に異なります。

- **適する場面**: 複数ファイルにまたがる実装、リファクタリング、テスト駆動の修正、移行作業。
- **強み**: ファイルシステムと端末への直接アクセス。人間が逐一コンテキストを渡さなくても、自分で探索する。
- **弱み**: 自律性が高いぶん、権限設計と承認フローを誤ると破壊的。検証が追いつかないと品質が崩れる。
- **代表**: Claude Code, Aider, Cline。

詳細は[エージェント型ツール](/tools/agentic-tools)で扱います。

## (d) コードベース検索型

リポジトリ全体(あるいは複数リポジトリ)をインデックス化し、「この機能はどこで実装されているか」「この関数の呼び出し元は」といった質問に、コードを根拠として答える方式です。

- **適する場面**: 大規模・レガシーなコードベースの理解、オンボーディング、影響範囲調査。
- **強み**: リポジトリ規模の文脈を扱える。根拠箇所を提示できる。
- **弱み**: 編集や実行は主目的ではない。検索精度はインデックスの質に依存する。
- **代表**: Sourcegraph Cody, Amazon Q Developer。

## (e) 自律エージェント型

課題(issue)やタスクの説明を受け取り、計画立案・実装・テスト・PR作成までを **可能な限り人間の介入なしに** こなすことを目指す方式です。

- **適する場面**: 仕様が明確で可逆な定型タスク、テスト整備された環境での独立した機能追加。
- **強み**: 委任の粒度が最も大きい。並行して複数タスクを走らせられる。
- **弱み**: 仕様の曖昧さに弱い。検証コストが高く、ブラックボックス化しやすい。執筆時点では適用範囲が限定的。
- **代表**: Devin。

::: info (c)と(e)の違い
CLI/エージェント型(c)は **人間が同席して対話的に進める** のが基本で、自律エージェント型(e)は **タスクを投げて結果を受け取る** 運用に寄っています。両者は連続的で、Claude Codeのようなツールも自律度を上げれば(e)に近づきます。詳しくは[エージェントとは何か](/agents/what-is-agent)を参照してください。
:::

## (f) レビュー/品質型

人間またはAIが書いた差分をレビューし、バグ・スタイル違反・脆弱性・テスト不足を指摘する方式です。AI駆動開発では「生成が速くなるほどレビューがボトルネック」になるため、この層の重要性は今後高まります。

- **適する場面**: PRの一次レビュー、脆弱性スキャン、レビュー観点の洗い出し。
- **強み**: 生成とは独立した視点で品質を検査できる。見落としを減らす。
- **弱み**: 指摘の精度にばらつきがあり、ノイズが多いと無視される。最終判断は人間に残る。
- **代表**: GitHub Copilotのコードレビュー機能, Amazon Q Developer, Sourcegraph Cody。

## ツールは「層」として組み合わせる

最も重要な視点がこれです。上の6分類は **互いに排他ではなく、積み重ねて使う層** です。熟練したエンジニアは、1つのツールに統一するのではなく、作業の粒度に応じて層を使い分けます。

| 作業の粒度 | 主に使う層 | 具体例 |
| --- | --- | --- |
| 1行〜数行を書く | (a) インライン補完 | Copilotで定型処理を流す |
| 関数を直す・説明させる | (b) IDEチャット | Cursorで選択範囲をリファクタ |
| 複数ファイルの機能実装 | (c) CLIエージェント | Claude Codeにタスクを委任 |
| 全体構造を理解する | (d) コードベース検索 | Codyで影響範囲を調べる |
| 差分を検査する | (f) レビュー/品質 | PRにAIレビューを通す |

たとえば典型的な一日は、こうなります。

1. 新機能の調査を **(d)検索型** で行い、関連実装と影響範囲を把握する。
2. 細かいコードは **(a)補完型** を効かせながら手で書く。
3. まとまった実装は **(c)エージェント型** にタスクとして委任する。
4. 局所的な修正や説明は **(b)チャット型** で素早く済ませる。
5. 出来上がった差分を **(f)品質型** に通してからPRを出す。

::: tip 統合製品でも「層」で考える
CursorやWindsurfのように複数層を1製品に統合したツールでも、頭の中では「いま自分は補完層を使っているのか、エージェント層を使っているのか」を意識すると、過信や誤用を防げます。機能の名前ではなく、**いま委任している作業の粒度と自律性** を基準に判断してください。
:::

## 分類を選定にどう使うか

このフレームの実用的な使い方は、次の問いに落とし込むことです。

- いま自動化したいのは、**どの粒度** の作業か(行・関数・タスク・リポジトリ)。
- その粒度に対し、**どの自律性** を許容できるか(提案のみ・承認制・自律実行)。
- 既存のワークフローは **IDE中心か、CLI中心か**。
- 結果の **検証** は誰が・どうやって担うか。

これらの問いに沿った体系的な選定手順は[ツールの選び方](/tools/choosing)で扱います。
