# PR・レビュー基準テンプレート

AI支援で生成量が増えると、ボトルネックはコードを書くことからレビューへ移ります。**「AI生成だから軽くレビュー」は最も危険なアンチパターン** です。ハルシネーション(存在しないAPI、もっともらしい誤り)や微妙なセキュリティ欠陥が、人間が書いたコードより紛れ込みやすいからです。

このページは、AI生成コードを含むPRを安全に回すための運用基準を、**そのままリポジトリに置けるテンプレート** として提供します。品質の考え方の詳細は[コード品質](/quality/code-quality)を参照してください。

::: tip 原則: AI生成こそ検証を厚く
レビューの目的は「責める」ことではなく「コードベースを守る」ことです。AI生成コードは速く大量に出るぶん、**レビューと検証の比重を上げて** バランスを取ります。速度の利得は、検証を省くのではなく、検証を効率化することで得てください。
:::

## `.github/pull_request_template.md`(コピペ用)

以下を `.github/pull_request_template.md` として保存すると、PR作成時に自動で本文に展開されます。チームの実情に合わせて項目を増減してください。

```markdown
## 概要

<!-- この変更は何を・なぜ行うか。関連Issue: #___ -->

## 変更内容

<!-- 主な変更点を箇条書きで -->
-

## AI利用の有無

- [ ] このPRにはAI支援で生成・補助されたコードが含まれる
- [ ] 含まれる場合、生成コードを自分で読んで理解した
- [ ] 使用ツール: [ツール名 / なし]

## レビュー観点(自己チェック)

- [ ] 要件・受け入れ基準を満たしている
- [ ] 存在しないAPI・ライブラリ・関数を使っていない(ハルシネーション確認)
- [ ] エッジケース・エラー処理・境界条件を考慮した
- [ ] 既存のコード規約・設計方針に沿っている
- [ ] 冗長・不要な抽象・コピペ重複がない

## テスト

- [ ] 変更に対応するテストを追加・更新した
- [ ] 正常系だけでなく異常系・境界値もカバーしている
- [ ] テストが仕様・振る舞いを検証している(実装の写像ではない)
- [ ] ローカルおよびCIでテストが通ることを確認した

## セキュリティ

- [ ] 認証情報・APIキー・トークンをハードコードしていない
- [ ] 入力検証・出力エスケープが適切で、インジェクション耐性がある
- [ ] 認可・アクセス制御が要件どおり実装されている
- [ ] 機密がログ・エラーメッセージに漏れていない
- [ ] 追加依存のライセンス・脆弱性・保守状況を確認した

## 動作確認

<!-- 確認手順・スクリーンショット・ログなど -->

## レビュアーへの依頼事項

<!-- 特に見てほしい点、判断に迷った点、未解決の懸念 -->
```

::: details テンプレートを軽量化したいとき
小規模チームでは項目が多すぎると形骸化します。最低限「AI利用の有無」「ハルシネーション確認」「テスト」「セキュリティの要点」を残し、残りはCIの自動チェックに委ねるのも一案です。**埋めるだけで思考停止する長大なテンプレートより、考える短いテンプレート** を優先してください。
:::

## AI生成コードのレビュー観点チェックリスト(レビュアー用)

レビュアーが生成コードを含むPRを見るときの網羅的な点検リストです。通常のレビューに上乗せして使います。

**理解と正しさ**

- [ ] レビュアー自身がコードを読んで理解できる(理解できないコードは承認しない)
- [ ] 要件・受け入れ基準を満たしている
- [ ] ロジックが正しく、仕様の解釈にズレがない
- [ ] 「動くから良い」ではなく「正しく・保守可能か」で判断している

**ハルシネーション検知**

- [ ] 存在しないAPI・メソッド・ライブラリ・設定キーを使っていない
- [ ] 参照しているライブラリのバージョン・引数・戻り値が実在し正しい
- [ ] コメントやドキュメント文字列が実装と一致している(もっともらしい嘘がない)

**エラー処理・エッジケース**

- [ ] null/空/境界値・異常入力を適切に処理している
- [ ] 例外・失敗時の挙動が妥当(握りつぶし・過剰なtry-catchがない)
- [ ] 並行性・競合・タイムアウトなど見落としやすい条件を考慮している

**設計・保守性**

- [ ] 既存のコード規約・命名・設計方針に沿っている
- [ ] 不要な抽象・過剰な汎用化・コピペ重複がない
- [ ] 変更が局所的で、関係ない箇所を巻き込んでいない

**セキュリティ**

- [ ] 認証情報・シークレットがハードコードされていない
- [ ] 入力検証・出力エスケープが適切でインジェクション耐性がある
- [ ] 認可・アクセス制御が要件どおり
- [ ] AIが提案した暗号・認証実装を専門知識で検証した
- [ ] 外部入力をプロンプトに渡す箇所でプロンプトインジェクションを考慮した

**テスト**

- [ ] 変更に対応するテストがあり、意味のある検証をしている
- [ ] 常に通るだけのテスト・実装の写像になっていないか確認した
- [ ] 異常系・境界値をカバーしている

**依存・ライセンス**

- [ ] 追加された依存の必要性・保守状況・ライセンスを確認した
- [ ] ライセンススキャン・依存脆弱性スキャンを通過している

::: warning レビューがボトルネック化していないか観察する
生成量が増えると、レビュアーの負荷が急増し、結果として「ざっと見て承認」が常態化します。これは検証を厚くする原則の正反対です。PRを小さく保つ、CIで機械的チェックを自動化する、重要度に応じてレビュー強度を変えるなどで、**レビュアーが本当に見るべき箇所に集中** できるようにしてください。
:::

## コミットメッセージ規約とAI生成物の表示方針

[Conventional Commits](https://www.conventionalcommits.org/) 風の規約を基本とし、AI生成由来のトレーサビリティを足します。

**フォーマット**

```text
<type>(<scope>): <要約(命令形・50字以内目安)>

<本文: 何を・なぜ。how ではなく why を中心に>

<フッター: 関連Issue、破壊的変更、AI支援の表示など>
```

**type の例**

| type | 用途 |
| --- | --- |
| feat | 新機能 |
| fix | バグ修正 |
| refactor | 振る舞いを変えないリファクタ |
| test | テストの追加・修正 |
| docs | ドキュメントのみの変更 |
| chore | ビルド・依存・雑務 |

**AI生成物の表示方針**

トレーサビリティ(どのコードがAI由来か事後に追える状態)は、インシデント調査と監査で効きます。チームで方針を統一してください。

- 生成コードを含むコミットは、フッターに支援ツールを明記する(例: `Assisted-by: [ツール名]`)。
- 表示の粒度はチームで合意する。全コミットに付けるか、相当量をAI生成した場合に限るかを決める。
- PRラベル(例: `ai-assisted`)を併用すると、PR単位での識別と集計がしやすい。
- 重要なのは様式の統一。バラバラに付けると検索・集計できず意味が薄れる。

::: tip トレーサビリティは免責のためではない
AI由来の表示は「AIのせいにする」ためではありません。承認者責任の原則(レビューしマージした人間が責任を負う)は変わりません。表示は **影響範囲を追える状態を保つため** の運用です。詳細は[ガバナンスと倫理](/adoption/governance)を参照。
:::

## マージ基準

共有ブランチ・本番へマージするための最低条件を、ブランチ保護ルール等で機械的に強制します。

**必須(マージブロック条件)**

- [ ] CIが緑(ビルド・テスト・Lint・型チェックがすべて成功)
- [ ] 静的解析(SAST)・シークレットスキャン・依存脆弱性スキャンを通過
- [ ] 変更に対応するテストが存在し、意味のある検証をしている
- [ ] 第三者の人間によるレビュー承認が1件以上ある
- [ ] PRテンプレートの該当チェック項目が埋まっている
- [ ] 未解決のレビューコメント(変更要求)が残っていない

**推奨**

- [ ] PRが十分小さく、レビュー可能な粒度に収まっている
- [ ] 重要・不可逆な変更は、テストに加えて人間が動作を確認した
- [ ] 破壊的変更がある場合、影響範囲と移行手順が記載されている

::: warning AI生成コードに「レビュー省略」の例外を作らない
「AIが書いた小さな変更だから」とレビューやCIをスキップする運用は作らないでください。むしろ生成コードはハルシネーションのリスクがあるため、**人間が書いたコードと同等以上** の検証を通します。マージ基準にAI由来の抜け道を設けないことが、品質を保つ要です。CIへの組み込み方は[CIインテグレーション](/ready/ci-integration)を参照。
:::

## 30分で導入するなら

1. `.github/pull_request_template.md` を上記テンプレートで作成し、コミットする。
2. レビュー観点チェックリストを社内Wikiに掲載し、レビュアーに周知する。
3. コミット規約とAI生成物の表示方針をチームで合意し、ドキュメント化する。
4. ブランチ保護でマージ基準(CI緑・レビュー必須)を機械的に強制する。
5. 1スプリント運用し、形骸化している項目・足りない項目を改訂する。

すぐに着手したい場合は[30分クイックスタート](/ready/quickstart-30min)も参照してください。

## まとめ

- 「AI生成だから軽くレビュー」は危険。**むしろ検証を厚く** するのが原則。
- `pull_request_template.md` でAI利用の有無・レビュー観点・テスト・セキュリティを可視化する。
- レビュアーはハルシネーション検知・セキュリティ・テストの質を重点的に点検する。
- Conventional Commits風の規約にAI生成物の表示を足し、トレーサビリティを保つ。
- マージ基準(CI緑・テスト・レビュー必須)を機械的に強制し、AI由来の抜け道を作らない。
