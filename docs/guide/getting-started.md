# はじめての一歩

理屈は十分です。ここからは手を動かします。このページのゴールは、**今日のうちに最初の生成→レビュー→反復のループを1周回す** ことです。

経験者にとっての難所は、ツールの使い方ではありません。**「どのタスクを、どの粒度で任せ、どう検証するか」という運用の勘** をつかむことです。これは読むだけでは身につかないので、小さく安全なタスクで実地に獲得します。

::: tip 最初に握っておくこと
完璧なツール選定や完璧なプロンプトを目指さないでください。**安く失敗できる環境で、速く反復する** ことが最優先です。最初の1周は質より「ループを体感する」ことに価値があります。
:::

## ステップ0: ツールを1つだけ選ぶ

比較に時間を溶かさないでください。**今の環境で最も摩擦が少ないもの** を1つ選び、まず使い込みます。ツールの全体像は[ツールの全体像](/tools/landscape)で扱いますが、最初の選択は下表で十分です。

| あなたの状況 | 最初の一手 | 形態 |
| --- | --- | --- |
| VS Code / JetBrainsを使っている | GitHub Copilot または JetBrains AI | IDE統合・補完+チャット |
| エディタを問わず軽く試したい | Cursor または Windsurf | AI特化エディタ |
| ターミナル中心で作業する | Claude Code / Aider / Cline | CLI・エージェント型 |
| まず対話だけで感触を見たい | チャットUI(ブラウザ) | 対話のみ |

::: info 補完型から始めるかエージェント型から始めるか
**補完型**(Copilot等)は摩擦が小さく日常に溶け込むので、ループの感覚をつかむ入口に向きます。**エージェント型**(Claude Code等)は「タスクを委任する」体験を直接得られますが、最初は手綱を強めに——小さなタスクで挙動を観察してください。どちらでもよいので、**1つに絞って** 始めることが肝心です。
:::

## 最初のループ: 5つの工程

AI駆動開発の基本サイクルはこれだけです。最初はあえて遅く、各工程を意識して回します。

```text
  ① タスク選定 ──→ ② プロンプト ──→ ③ 生成
      ↑                                  │
      │                                  ▼
  ⑤ 反復 ←────────── ④ レビュー・検証
```

### ① 小さく安全なタスクを選ぶ

最初の題材は、**間違っても害がなく、正しさを自分で確認できる** ものに限ります。

良い最初のタスク:
- 既存関数のユニットテストを書かせる(正しさをテスト実行で確認できる)
- 既存コードの **読解・要約** をさせる(あなたが答えを知っている)
- 純粋関数(入出力が明確)の実装
- ログ整形・小さなスクリプトなど **使い捨て** のコード
- 正規表現やSQLの下書き(手元で実行して確認できる)

避けるべき最初のタスク:
- 本番に直接触れる変更、認証・課金・セキュリティ周り
- 検証手段が用意できない、暗黙知だらけの中核ロジック
- 「なんとなく全体をいい感じに」という曖昧で大きな依頼

### ② 意図を言語化してプロンプトを書く

ここが従来開発と最も違う工程です。**頭の中の暗黙の前提を、明示的な指示に変換** します。最初は欲張らず、4要素を意識するだけで十分です。

```text
[文脈]   このコードは決済前のバリデーション層です
[タスク] validateOrder の単体テストを書いて
[制約]   既存の vitest を使う / モックは最小限 / 異常系を重視
[期待]   境界値と空入力のケースを必ず含めて
```

詳しい技法は[プロンプトの基礎](/prompting/fundamentals)に譲りますが、初回は **「文脈・タスク・制約・期待」を一言ずつ添える** だけで出力が大きく変わることを体感してください。

### ③ 生成させる

出力を受け取ります。ここで **そのまま受け入れない** こと。生成物は「同僚が出してきたドラフトPR」だと思ってください——意図のない、確率的なドラフトです。

### ④ レビューと検証 — ここが本番

最も時間をかけるべき工程です。最低限、次を通します。

- [ ] **読んで理解できるか** — 説明できないコードはマージしない
- [ ] **実行・テスト** — 動かして確かめる。生成されたテスト自体も疑う
- [ ] **存在確認** — 呼ばれているAPI・ライブラリが実在するか([ハルシネーション対策](/quality/hallucination))
- [ ] **エッジケース** — 空入力・境界値・異常系が抜けていないか
- [ ] **責任を引き受けられるか** — 「自分が書いた」と言える品質か

### ⑤ 反復する

レビューで見つけた問題を、**次のプロンプトに反映** します。「ここでnullが来たら落ちる。そのケースを追加して」のように、具体的なフィードバックで往復します。一発で完璧を狙わず、**短いサイクルで磨く** のがコツです。

## やってはいけないこと

最初に踏みやすい地雷を、明示的に避けてください。

::: warning これだけは守る
- **機密を投入しない** — APIキー、認証情報、個人情報、未公開のソースを、利用規約やデータ保持ポリシーを確認せずに送らない。社用なら会社のAI利用ポリシーを先に確認する。
- **無検証でマージしない** — 「動いたっぽい」でmainに入れない。確率的生成物は必ず外部検証を通す。
- **理解できないコードを採用しない** — 説明できないコードは、あなたの負債であってAIの手柄ではない。
- **大きく曖昧なタスクを丸投げしない** — 「アプリ作って」ではなく、検証可能な小タスクに割る。
- **生成テストを盲信しない** — テストもAIが書いたなら、テスト自体の妥当性を疑う。
:::

| アンチパターン | なぜ危険か | 代わりに |
| --- | --- | --- |
| 機密の貼り付け | 情報漏洩・規約違反 | ダミー値・最小再現で尋ねる |
| 無検証マージ | 確率的誤りが本番へ | テスト・実行で必ず確認 |
| ブラックボックス採用 | 保守不能な負債化 | 理解できる粒度に分けて生成 |
| 巨大プロンプト丸投げ | 検証不能な大物が出る | タスクを小さく分割 |

## 最初の30分プラン

今すぐ回せる具体的な手順です。

1. **(5分)** 上表からツールを1つ選び、インストール/サインインする。
2. **(5分)** 安全な題材を1つ決める。おすすめは「既存の純粋関数にテストを書かせる」。
3. **(10分)** 「文脈・タスク・制約・期待」を添えてプロンプトを書き、生成させる。
4. **(10分)** 生成物を読み、実際にテストを **実行** する。落ちたら原因を1つ指摘して再生成する。
5. ループを1周回せたら、今日の目標は達成です。

::: tip 1周回したら振り返る
「どこで時間がかかったか」を意識してください。多くの場合、ボトルネックは生成ではなく **検証** にあります。これがAI駆動開発の本質([従来開発との違い](/guide/vs-traditional))を体で理解する瞬間です。
:::

## 最初の1週間プラン

30分の単発で終わらせず、1週間で「使い分けの勘」まで育てます。

| 日 | テーマ | 具体アクション |
| --- | --- | --- |
| 1日目 | ループを回す | 上記30分プランを実行。テスト生成で1周 |
| 2日目 | 対話で使う | 設計相談・エラー解決を **チャット** で往復してみる |
| 3日目 | 読解に使う | 不慣れな既存コードを要約・図解させ、答え合わせ |
| 4日目 | 反復を磨く | わざと曖昧な指示→具体化、で出力差を観察 |
| 5日目 | 限界を知る | **検証が難しいタスク** をあえて試し、なぜ遅くなるか体感 |
| 週末 | 振り返り | 「速くなったタスク/遅くなったタスク」を仕分けする |

1週間後に得たいのは「AIは万能」でも「使えない」でもなく、**「このタスクは任せる/これは自分でやる」という線引きの感覚** です。この線引きこそが、熟練者がAIから最も価値を引き出すための核心です。

## つまずいたら

| 症状 | ありがちな原因 | 対処 |
| --- | --- | --- |
| 出力が的外れ | 文脈・制約が不足 | 関連コードや前提を明示して添える |
| 嘘のAPIを使う | ハルシネーション | 公式ドキュメントで確認、[対策](/quality/hallucination)参照 |
| 大きすぎて検証不能 | タスクが粗い | 小さく分割して1ステップずつ |
| かえって遅い | タスク不適合 | そのタスクは自分で書く判断も正解 |
