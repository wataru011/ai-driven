# レガシー移行・大規模変更

レガシー移行や大規模な置換は、AI駆動開発の中で**最もリスクが高く、最も価値が出る**領域です。価値が出るのは、人間が手作業でやると数週間かかる退屈で機械的な作業を加速できるから。リスクが高いのは、変更の規模が大きく、**1つの見落としが何百箇所にも伝播**し、回帰の検出も困難だからです。

このページの主題は「速く変換する方法」ではありません。**安全に、検証可能に、後戻りできる形で大規模変更を進める方法**です。

::: warning 大原則: 動いているものを止めない
レガシー移行の最大の失敗は「全部を一気に書き換えて、全部が同時に壊れる」ことです。**段階移行(strangler fig)** — 古いものと新しいものを共存させ、少しずつ置き換える — が基本戦略です。AIは各ステップを速くしますが、**ステップを大きくしてはいけません**。
:::

## 題材: 日付ライブラリの移行

題材は「Moment.js から date-fns への移行」とします。よくある実在の移行で、論点が揃っています。

- 数百ファイルにまたがる広範な使用
- APIの設計思想が違う(Momentは可変オブジェクト、date-fnsは純粋関数)
- 機械的に変換できる部分と、人間/AIの判断が要る部分が混在
- タイムゾーンやロケールの扱いで**微妙な挙動差**が事故になりやすい

## 全体戦略

大規模移行は次の5フェーズで進めます。順序が重要です。

1. **仕様復元**: 現在のコードが「何をしているか」を把握する
2. **安全網の構築**: 変更前に振る舞いを固定するテストを張る
3. **計画の分割**: 変換を機械的/AI/手動に仕分け、小さな単位に割る
4. **段階変換**: 単位ごとに変換 → 検証 → マージを繰り返す
5. **検証と切り戻し設計**: 全体の整合と、問題時の戻し方を確保

::: tip AIは「全部やる」のではなく「各フェーズを補助する」
移行をまるごとAIに投げると、見えないところで挙動が変わります。AIは**仕様の復元**、**変換ルールの設計**、**個別変換**、**差分のレビュー**をそれぞれ補助する道具として使い、全体の制御は人間が握ります。
:::

## フェーズ1: 仕様復元

レガシーコードの怖さは「なぜそう書いてあるか」が失われていることです。移行前に、AIに**現状の挙動を言語化**させます。

```text
あなたはコード読解と仕様復元に長けたエンジニアです。
以下は Moment.js を使った日付処理ユーティリティです。
このコードが「実際に何をしているか」を、入出力の観点で仕様化してください。

- 各関数の入力・出力・境界条件(月末、うるう年、タイムゾーン、無効日付)
- 暗黙の前提(ローカルタイムゾーン依存か、UTC前提か)
- date-fns へ移行する際に「挙動が変わりうる危険ポイント」を特に強調

推測が混じる箇所は「要確認」と明記してください。
（対象コードを貼る）
```

応答イメージ(抜粋):

```text
formatBusinessDate(date):
  入力: Moment互換の値 or ISO文字列
  出力: "YYYY/MM/DD" 形式の文字列
  危険ポイント:
  - moment(undefined) は「現在時刻」になる。date-fns には等価がなく、
    undefined を渡すと Invalid Date。呼び出し側に undefined 流入の
    可能性があれば挙動が変わる → 要確認
  - moment はローカルタイムゾーンで解釈。date-fns/format も同様だが、
    ISO文字列に Z が無い場合の解釈差に注意 → 要確認

addBusinessDays(date, n):
  危険ポイント:
  - moment の add は元オブジェクトを破壊的に変更(mutate)。
    呼び出し側がそれに依存している場合、date-fns の純粋関数に
    置き換えると「元の変数が変わらない」ことで壊れる → 要確認(最重要)
```

::: warning 仕様復元はAIの推論であって真実ではない
AIが復元した仕様は「コードからの推測」です。特に「要確認」項目は、**テストや実行で裏を取る**必要があります([ハルシネーション対策](/quality/hallucination))。ここで `moment` の破壊的変更(mutation)への依存が浮かんだのは大きい — これは機械変換が最も事故りやすい箇所です。
:::

## フェーズ2: 安全網(characterization test)

変更する前に、**現在の挙動を固定するテスト**(characterization test / ゴールデンテスト)を張ります。「正しい挙動」ではなく「**今の挙動**」を記録するのがポイント。移行で挙動が変わったら、それを検出するための網です。

```text
formatBusinessDate と addBusinessDays について、現在の Moment 実装の
「今の挙動」を固定する characterization test を vitest で生成してください。
正しさの判断はせず、現状の出力をそのまま期待値にする。
入力は境界を網羅: 月末/月初、うるう年2/29、12/31→1/1、無効入力、
タイムゾーン依存が出そうな深夜帯、undefined。
テストはタイムゾーンを固定(TZ=Asia/Tokyo)した前提で。
```

```ts
// 現在のMoment実装の挙動を「写し取る」テスト
describe('characterization: formatBusinessDate (Moment)', () => {
  it.each([
    ['2024-02-29T00:00:00+09:00', '2024/02/29'], // うるう年
    ['2023-12-31T23:59:00+09:00', '2023/12/31'],
    ['2024-01-31T15:00:00Z',       '2024/02/01'], // ← TZでズレる代表例
  ])('format(%s) === %s', (input, expected) => {
    expect(formatBusinessDate(input)).toBe(expected)
  })
})
```

::: tip 移行のゴールはテストが「緑のまま」であること
理想は「Moment実装でこのテストを緑にし、date-fns実装に差し替えても**同じテストが緑のまま**」です。挙動を変えずに実装だけ替えられたことの証明になります。意図的に挙動を変える箇所だけ、テストを明示的に書き換えます。
:::

## フェーズ3: 計画の分割 — 機械 vs AI vs 手動

ここが移行の設計の肝です。すべてをAIにやらせるのは非効率かつ危険。**変換を3種類に仕分け**ます。

| 種別 | 向く対象 | 手段 |
| --- | --- | --- |
| 機械的変換(codemod) | 構文が決まった単純置換 | jscodeshift / ts-morph / sed |
| AI変換 | 文脈判断が要る、定型化しにくい | LLMにファイル単位で変換 |
| 手動 | 破壊的変更への依存、複雑なロジック | 人間が書き換え |

AIに**この仕分け自体**を手伝わせます。

```text
Moment → date-fns 移行を、変換の性質で3分類してください。
1. codemod(機械的置換)で安全に変換できるパターン
2. 文脈判断が必要でAI変換が向くパターン
3. 機械変換が危険で手動レビュー必須のパターン
各パターンに、変換前→変換後の例を1つずつ。
特に「破壊的変更(mutate)への依存」を含むコードは必ず3に分類して。
```

応答イメージ:

```text
1. codemod 可:
   moment(x).format('YYYY-MM-DD')  →  format(new Date(x), 'yyyy-MM-dd')
   ※ フォーマットトークンが違う(YYYY→yyyy, DD→dd)。一律置換可。

2. AI変換が向く:
   moment(x).isBefore(moment(y))  →  isBefore(new Date(x), new Date(y))
   ※ 比較対象の型(Moment/Date/文字列)を文脈で判断する必要あり。

3. 手動必須:
   const d = moment(x); d.add(1, 'day'); use(d); use(x追従の何か)
   ※ d を破壊的に変更し、その後 d を使い回している。date-fns では
     addDays(d, 1) が新しい値を返すため、代入先の付け替えが必要で、
     周辺ロジックの読み替えが要る。機械変換は事故る。
```

::: warning codemod のトークン差は地雷
Moment(`YYYY/MM/DD`)と date-fns(`yyyy/MM/dd`)は**フォーマットトークンが異なります**。`DD`(日)を `dd` に、`YYYY`(年)を `yyyy` に変える必要があり、見落とすと**サイレントに間違った日付**を出力します(エラーにならないのが最悪)。codemodのルールはAIに作らせ、**characterization testで必ず裏を取る**。
:::

## フェーズ4: 段階変換

仕分けに沿って変換します。**1ディレクトリ/1機能単位**でコミットし、各単位でテストを通します。

### codemod 部分

機械的変換はAIに**codemodスクリプト**を書かせ、人間が差分をレビューしてから一括適用します。「AIに各ファイルを直接書き換えさせる」より、「**変換ルールを書かせて機械適用**」のほうが、再現性があり差分が読みやすく安全です。

```text
ts-morph を使い、以下のルールで Moment → date-fns に変換する
codemod スクリプトを書いてください。
- moment(x).format('TOKENS') を format(new Date(x), '変換後TOKENS') へ
- フォーマットトークンの対応表(YYYY→yyyy, DD→dd, ...)を明示し、適用
- import 文も追加(date-fns の format / 既存の moment import は削除)
- 変換できないパターンに遭遇したら「// TODO: manual review」を挿入し、
  そのファイルと行をレポートに出力(自動で壊れた変換をしない)
```

::: tip 「変換できないものはマークして残す」が安全装置
codemodに「判断できないものは触らず TODO を残せ」と指示するのが鍵です。AIやcodemodが**自信のないまま変換する**のが事故の元。残したTODOは手動(フェーズ3の分類3)に回します。
:::

### AI変換 部分

文脈判断が要る部分は、ファイル単位でAIに変換させます。**変換前後でテストが緑**であることを各ファイルの受け入れ条件にします。

```text
このファイルの Moment 使用箇所を date-fns に変換してください。
## 厳守
- 挙動を変えない(characterization test が緑のまま通ること)
- 破壊的変更(.add/.subtract で元を変えている)に依存している箇所が
  あれば、変換せず "// NEEDS MANUAL: mutation" とコメントして報告
- import の整理も行う
## 出力: unified diff
（ファイルとそのテストを貼る)
```

変換後は必ずテストを実行します。

```bash
TZ=Asia/Tokyo npx vitest run path/to/file.test.ts
```

## フェーズ5: 大規模置換の検証戦略

数百ファイルの変換が「本当に挙動を変えていないか」をどう保証するか。検証戦略を多層に組みます。

| 層 | 手段 | 何を担保 |
| --- | --- | --- |
| 単体 | characterization test | 関数単位の挙動不変 |
| 横断 | 型チェック(tsc) | API誤用・型不整合 |
| 横断 | 旧APIの残存検出 | `moment` の取り残しゼロ |
| 全体 | 差分のサンプリング目視 | codemodの体系的ミス検出 |
| 全体 | 本番影響の少ない段階リリース | 想定外の挙動を小さく検出 |

旧APIの取り残しは機械的に検出できます。

```bash
# moment の import や呼び出しが残っていないか(取り残し検出)
grep -rn "from 'moment'" src/ || echo "クリーン"
```

::: tip A/B 検証(shadow comparison)
最も強力な検証は、**新旧両方を一定期間並走させて出力を突き合わせる**ことです。重要な日付処理なら、移行期間中だけ「Momentの結果」と「date-fnsの結果」を両方計算し、**不一致をログに出す**(挙動は旧のまま使う)。本番トラフィックで差分が出た箇所だけを潰せば、テストで網羅しきれない実データのエッジケースを捕まえられます。AIにこの shadow comparison のラッパー実装を書かせるのは有効な使い方です。
:::

## 切り戻しの設計

大規模変更は**戻せること**を最初に設計します。

- 変更は**小さなコミット/PR**に分け、問題が出たら部分的に revert できるように。
- フィーチャーフラグで新実装をオンオフできるようにし、本番で問題が出たら旧実装に即切り戻し。
- 「一度に1つの理由でしか壊れない」状態を保つ(同時に複数の大変更を混ぜない)。

::: warning AIに巨大な一括変換をさせない
「全ファイルを変換して1つのPRで」は、レビュー不能で、問題時に切り分け不能です。AIが「全部やりましょうか?」と提案しても、**人間が単位を小さく保つ**判断をしてください。大きな差分は、それ自体がリスクです。
:::

## アンチパターン集

| アンチパターン | 何が起きるか | 対処 |
| --- | --- | --- |
| 一括全置換 | 全部同時に壊れ、切り分け不能 | 段階移行・小さな単位 |
| 安全網なしで着手 | 挙動が変わっても気づけない | characterization test を先に |
| codemodのトークン差無視 | サイレントに誤った出力 | 対応表をAIに作らせ、テストで裏取り |
| 破壊的変更依存の機械変換 | 純粋関数化で周辺が壊れる | 手動分類に回す |
| 仕様復元を鵜呑み | AIの推測を真実扱い | 「要確認」をテストで検証 |
| 切り戻し未設計 | 問題時に戻せず長時間障害 | フラグ・小コミット・shadow比較 |

## まとめ

- 大規模移行は **段階移行(strangler fig)** が基本。一気にやらない。
- 順序が命: **仕様復元 → 安全網 → 仕分け → 段階変換 → 多層検証**。
- 変換は **機械(codemod)/AI/手動** に仕分ける。判断できないものは触らずマーク。
- characterization test で「今の挙動」を固定し、移行後も**緑のまま**を目標に。
- 実データの取りこぼしは **shadow comparison(新旧並走)** で捕まえる。
- **切り戻し**を最初に設計する。大きな差分はそれ自体がリスク。
