# エージェント型ツール

エージェント型ツールとは、**ファイルを編集し、コマンドを実行し、結果を見て次の行動を決める** という一連のループを、人間が逐一指示しなくても回せるツールを指します。[ツールエコシステムの全体像](/tools/landscape)で言う(c)CLI/エージェント型と(e)自律エージェント型の層です。

補完やチャットとの本質的な違いは、**マルチステップ実行** にあります。補完は「次の数行」を、チャットは「この差分」を返して終わりますが、エージェントは「テストを書く→走らせる→失敗を読む→修正する→また走らせる」というループを、自分で回します。これは委任の粒度を「タスク」や「機能」へと一段引き上げる変化です。

::: tip このページの要点
エージェント型の強みは自律性ですが、その自律性こそがリスクの源でもあります。鍵は **能力(何ができるか)と権限(何を許すか)を分けて設計する** こと。両者を混同すると、便利すぎて危険な道具になります。
:::

## 何が「能力差」を生むのか

エージェント型ツールを評価するときは、次の4つの能力軸で見ると差が明確になります。

| 能力軸 | 説明 | 影響 |
| --- | --- | --- |
| ファイル編集 | 複数ファイルを横断して読み書きできるか | 扱える変更の規模 |
| コマンド実行 | テスト・ビルド・linterを自分で走らせるか | 自己検証できるか |
| マルチステップ実行 | 失敗を読んで自律的に直すループを回せるか | 委任できる粒度 |
| コンテキスト探索 | リポジトリを自分で検索・読解できるか | 人間が渡す手間の少なさ |

補完型はこのうち1つも持たず、チャット型は限定的にしか持ちません。エージェント型はこれらを **組み合わせて** 持つことで、タスク単位の委任を可能にします。

## 代表的なツール

| ツール | 形態 | 特徴 | 性格 |
| --- | --- | --- | --- |
| Claude Code | CLI | 端末で対話的に動作。ファイル編集・コマンド実行・探索を統合。権限制御を備える | 対話的エージェント |
| Aider | CLI | gitと密に統合。変更を自動コミットし差分で管理 | git中心の編集エージェント |
| Cline | IDE拡張(VS Code) | エディタ内でエージェントを動かす。ステップ承認のUI | IDE内エージェント |
| Devin | 自律エージェント | 課題を受け取り端から端まで実行することを目指す | 自律実行寄り |

::: warning 機能は流動的
ここに挙げた性格づけは執筆時点(2026年)の傾向です。各ツールはモデル更新や機能追加で性格が変わります。Claudeを使うツールでは、込み入った計画にOpus 4.8、日常的な実装にSonnet 4.6、軽量な処理にHaiku 4.5、といったモデル選択が効くこともあります。最新の対応モデル・上限は各製品で確認してください。
:::

## 人間の関与モデル(承認フロー)

エージェント型ツールを安全に使う上で最重要なのが、**人間がどこで介入するか** の設計です。自律性は連続的なスペクトラムであり、タスクのリスクに応じて調整します。

| モード | 人間の関与 | 適するタスク |
| --- | --- | --- |
| 提案のみ | 全変更を人間がレビューしてから適用 | 重要・不可逆な変更、未知の領域 |
| ステップ承認 | アクションごとに承認(特にコマンド実行) | 通常の機能実装、慣れていないコードベース |
| 一括承認 | 計画を承認し、実行は任せて後でまとめてレビュー | 可逆で定型的なタスク |
| 自律実行 | 例外時のみ介入 | テスト整備済みの環境での定型作業 |

::: tip 可逆性で承認レベルを決める
熟練者の判断基準は **可逆性とリスク** です。gitで簡単に戻せる・テストが守ってくれる変更は自律度を上げ、本番設定・マイグレーション・外部APIへの書き込みのように戻しにくい操作は提案のみに落とす。「速いから自律」ではなく「戻せるから自律」です。
:::

## 安全に使うための権限設計

エージェントが端末とファイルシステムにアクセスする以上、権限設計は機能ではなく **前提** です。次の4点を最低限おさえます。

#### 1. サンドボックス / 隔離

可能なら、エージェントを隔離環境で動かします。コンテナ、専用の作業ディレクトリ、ネットワーク制限などで、**最悪の場合の被害範囲を限定** します。本番認証情報を環境に置かない、書き込み範囲を作業ツリーに限る、といった基本が効きます。

#### 2. コマンド実行の制御

最も危険なのは任意コマンド実行です。多くのツールは「許可リスト/拒否リスト」「実行前の承認」を備えています。`rm -rf`・`git push --force`・パッケージのグローバルインストール・本番への接続といった操作は、承認必須または禁止に倒します。

#### 3. バージョン管理を安全網にする

エージェントの変更は **常にgit管理下** で行い、こまめにコミット・ブランチ分離します。Aiderのようにコミットを自動化するツールもあります。差分が常に見え、いつでも戻せる状態を保つことが、自律度を上げる前提条件です。

#### 4. 検証の自動化

エージェントの自律ループが価値を持つのは、**自己検証できるとき** です。テスト・型チェック・linterが整っているコードベースでは、エージェントは失敗を読んで自分で直せます。逆に検証手段がないと、もっともらしいが壊れた変更が積み上がります。「エージェントを入れる前にテストを整える」が正しい順序です。

::: warning 自律度とコンテキスト窓のトレードオフ
マルチステップ実行は、各ステップの出力(ファイル内容、コマンド結果、エラー)をコンテキストに積み上げます。長く走るほど文脈が膨らみ、初期の指示や重要な制約が埋もれることがあります。タスクを小さく区切る、要所で状況を要約させる、といった運用が品質を保ちます。背景は[コンテキストウィンドウ](/concepts/context-window)を参照してください。
:::

## エージェント型が向く作業・向かない作業

| 向く | 向かない |
| --- | --- |
| 複数ファイルにまたがる機能実装 | 仕様が曖昧で判断が割れるタスク |
| テスト駆動の修正・リファクタ | 検証手段がない領域 |
| 定型的な移行・依存更新 | 不可逆で高リスクな本番操作 |
| パターンの一括適用 | 設計の根幹に関わる意思決定 |
| 調査と修正の反復 | レビューが追いつかない大量生成 |

エージェントは「書く」を加速しますが、「何を作るべきか」「このトレードオフでよいか」の判断は人間に残ります。これは[AI駆動開発とは](/guide/what-is-aidd)で述べた分業原則の延長です。
