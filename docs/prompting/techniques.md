# 実践テクニック

[プロンプトの基礎](/prompting/fundamentals)で扱った6要素は「何を書くか」でした。ここでは「どう構造化するか」、すなわち**出力の精度を引き上げる技法**を扱います。各技法には適した場面があり、万能薬ではありません。「どんなときに効くか」をセットで覚えてください。

::: tip 技法は積み重ねられる
これらは排他的ではありません。「Few-shot + 出力フォーマット指定 + 自己レビュー」のように組み合わせるのが実務です。
:::

## 1. Few-shot(例示)

期待する入出力のペアを数個示してから本題を与える技法です。**スタイルや形式を言葉で説明しづらいとき**に絶大な効果があります。

```text
以下の規約でコミットメッセージを生成してください。

例1:
変更: ログイン時のトークン有効期限チェックを追加
出力: feat(auth): validate token expiry on login

例2:
変更: README の誤字修正
出力: docs: fix typo in README

では、次の変更のメッセージを:
変更: ユーザー削除時に関連セッションも削除するよう修正
```

::: warning Few-shotのアンチパターン
- **例が偏っている**: 正常系の例ばかり渡すと、異常系の出力がブレる。エッジケースの例も混ぜる。
- **例が多すぎる**: 5〜6個を超えると効果は逓減し、トークンを浪費する。
- **例と本題の形式が不一致**: 例の出力形式と、本題で求める形式は揃える。
:::

## 2. Chain-of-Thought(思考の連鎖)

「結論の前に推論過程を書かせる」技法です。**ロジックが複雑なタスク**(アルゴリズム設計、バグの原因特定、トレードオフ判断)で精度が上がります。

```text
この関数がテストケース [3, 1, 2] で誤った結果を返す原因を特定したい。

まず処理を1行ずつ追い、各ステップでの変数の値を表にしてください。
どのステップで期待と実際がズレるかを特定してから、修正案を出してください。
いきなり修正コードを書かないこと。
```

最新のClaude(Opus 4.8など)は内部で推論を行うため、単純なタスクでは明示的なCoTが不要なこともあります。ただし**推論過程を人間がレビューしたい**場合は、明示的に書かせる価値があります。

::: details 「ステップごとに考えて」だけでは弱い
漠然と「ステップバイステップで」と書くより、**何の軸で分解するか**(変数の遷移、入力の場合分け、制約の充足順)を指定するほうが効果的です。
:::

## 3. 段階的分解(タスク分割)

大きなタスクを一度に投げると、品質が落ち、レビューも困難になります。**1プロンプト1責務**に分けます。

```text
# Step 1 のプロンプト
新しい通知機能を実装する前に、まず設計だけ進めます。
必要なテーブル、API エンドポイント、責務の境界を提案してください。
コードはまだ書かないこと。
```

設計が固まってから、Step 2で実装、Step 3でテスト、と進めます。これは[実装フェーズ](/workflow/implementation)のタスク分解と直結します。

## 4. 自己レビュー(self-critique)

生成直後に、モデル自身へ批判的レビューをさせる技法です。**初稿の品質を一段引き上げる**安価な方法です。

```text
（実装を生成させた後、続けて）

いま書いたコードを、シニアレビュアーの視点で批判的にレビューしてください。
観点:
- 競合状態・例外時のリソースリーク
- 入力検証の漏れ
- 既存規約からの逸脱
問題があれば指摘し、修正版を出してください。なければ「問題なし」と明言してください。
```

::: warning 自己レビューの限界
モデルは自分の誤りに気づけないことがあります(同じ盲点を持つため)。自己レビューは**人間レビューの代替ではなく前処理**です。詳しくは[テストとレビュー](/workflow/testing-review)で扱います。
:::

## 5. 制約付き生成

出力の探索空間を明示的に縛ります。コード生成では「禁止事項」を書くのが特に有効です。

```text
この関数をリファクタリングしてください。

絶対制約(違反したら無効):
- 公開シグネチャ(引数・戻り値の型)を変更しない
- 新規の外部依存を追加しない
- 既存テストをすべてパスし続ける
- 1関数50行以内
```

「やってはいけないこと」を列挙すると、モデルが暴走して周辺コードまで書き換える事故を防げます。

## 6. 出力フォーマット指定

後工程を機械化するため、出力構造を固定します。

```text
出力は次のJSON形式のみ。前置きや説明は一切不要:
{
  "summary": "変更の1行要約",
  "files": [
    { "path": "...", "action": "modify" | "create", "diff": "..." }
  ],
  "risks": ["想定されるリスク"]
}
```

JSONやdiffを要求するとパースが容易になり、CI/自動適用パイプラインに乗せられます。

## 7. リファレンス提示(grounding)

モデルの「知識」ではなく、**あなたが渡した資料**に基づいて答えさせる技法です。ハルシネーション(もっともらしい嘘)の抑制に効きます。

```text
添付の API ドキュメント(下記)のみを根拠にクライアントコードを書いてください。
ドキュメントに記載のないエンドポイントやパラメータは使用しないこと。
不明な点があれば、推測せず「ドキュメントに記載なし」と書いてください。

--- API ドキュメント ---
（ここに公式ドキュメントを貼る）
```

「推測せず、なければ"分からない"と言え」という一文が、捏造を大きく減らします。

## 8. 反復改善ループ

一発で完璧を狙わず、**フィードバックを与えて収束させる**運用です。

```text
（1回目の出力に対し)
テストを実行したら、以下が失敗しました:

  FAIL: should return null for missing user
  Expected: null, Received: undefined

この失敗を踏まえて修正してください。他のテストは壊さないこと。
```

実行結果・エラーログ・lint結果といった**客観的フィードバック**を渡すループは、エージェント型開発の基本動作でもあります([エージェントとは](/agents/what-is-agent))。

::: tip フィードバックは具体的に
「まだダメ」ではなく、**何が・どう・なぜダメか**(失敗したテスト名、期待値と実際値)を渡すこと。曖昧なフィードバックは曖昧な修正を生みます。
:::

## 技法の使い分け早見表

| 状況 | 第一選択の技法 |
| --- | --- |
| 出力形式を揃えたい | Few-shot / フォーマット指定 |
| 複雑なロジック・原因特定 | Chain-of-Thought |
| 大きすぎるタスク | 段階的分解 |
| 初稿の品質を上げたい | 自己レビュー |
| 暴走・過剰変更を防ぐ | 制約付き生成 |
| 捏造を抑えたい | リファレンス提示 |
| 一発で決まらない | 反復改善ループ |

## アンチパターン総覧

::: warning やりがちな失敗
- **呪文信仰**: 「最高のコードを書いて」のような無意味な強調語。測定可能な基準に翻訳する。
- **詰め込みすぎ**: 1プロンプトに設計・実装・テスト・ドキュメントを全部要求。分解する。
- **コンテキスト過多**: 関係ないファイルまで貼り、本質的な情報が埋もれる。[コンテキスト設計](/prompting/context-design)参照。
- **検証なきループ**: 「もっと良くして」を繰り返すだけ。客観的な合否基準を毎回与える。
- **役割への過信**: 「あなたは天才ハッカー」で能力が増えると思い込む。役割は重み付けにすぎない。
:::

## まとめ

- 技法は**場面とセット**で覚える。万能なものはない。
- 複雑なものは分解、品質は自己レビュー、暴走は制約、捏造はリファレンス提示で対処。
- 客観的フィードバックによる**反復改善ループ**が実務の主戦場。
