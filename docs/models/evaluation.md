# AI出力の評価(Evals)

AI駆動開発で最も多い失敗は、**「評価せずに本番投入すること」** です。プロンプトを少し変えただけで出力品質がどう動いたか、新しいモデルに乗り換えて改善したのか劣化したのか——これらを **測る仕組み(Evals)** がなければ、改善はすべて勘と気分になります。

Evalは、LLMを使ったシステムにおける **テストスイートに相当するもの** です。従来のソフトウェアでテストなしのリファクタリングが無謀であるように、Evalなしのプロンプト変更・モデル変更も無謀です。

::: tip ひとことで言うと
Evalとは **「LLMシステムに対する自動テスト」** です。入力に対する出力を、ゴールデン正解・判定基準・回帰の観点から繰り返し測り、変更が改善か劣化かを定量的に判断できるようにします。
:::

## なぜEvalが必要か

LLMの出力は **確率的で非決定的** です。同じ入力でも出力が揺れ、プロンプトの些細な変更が品質を大きく動かします。この性質が、従来のソフトウェアと根本的に異なる難しさを生みます。

| 従来のソフトウェア | LLMシステム |
| --- | --- |
| 同じ入力→同じ出力(決定的) | 同じ入力→揺れる出力(確率的) |
| 正誤が明確 | 「良し悪し」が連続的・主観を含む |
| テストはpass/fail | スコア・分布で評価 |
| 変更の影響が局所的 | プロンプト1文字で全体が動く |

Evalが解く問題は具体的に次の通りです。

- **回帰の検出** — プロンプト/モデル変更で、直っていたケースが壊れていないか。
- **比較の客観化** — 2つのプロンプト、2つのモデルのどちらが良いかを数値で。
- **品質の可視化** — 「なんとなく良くなった」を「正解率がXからYへ」に変える。
- **意思決定の根拠** — モデル選定・リリース判断の証拠。

## 評価手法の全体像

評価手法は、対象タスクの「正解の測りやすさ」によって使い分けます。

| 手法 | 仕組み | 向く対象 | コスト |
| --- | --- | --- | --- |
| ゴールデンセット | 入力と期待出力のペアを用意し照合 | 正解が明確なタスク | 作成は重いが実行は安い |
| プログラム的検証 | コンパイル/テスト/正規表現等で機械判定 | コード・構造化出力 | 安い・客観的 |
| LLM-as-a-judge | 別のLLMに出力の良し悪しを採点させる | 自由文・主観的品質 | 中(API課金) |
| ルーブリック評価 | 採点基準を明文化して点数化 | 多面的な品質 | 中 |
| 人手評価 | 人間が採点・ランク付け | 最終確認・基準作り | 高い・遅い |
| 回帰テスト | 過去に直したケースを固定して再実行 | 劣化防止 | 安い(自動化前提) |

### ゴールデンセット

最も基本的な手法です。**入力と、期待される出力(または満たすべき条件)のペア** を集めたデータセットを用意し、システム出力をそれと照合します。

```text
# golden_set.jsonl の例(1行1ケース)
{"input": "1から100までの和を返すPython関数", "must_contain": ["def", "return", "5050"]}
{"input": "空配列のとき例外を投げる平均関数", "must_contain": ["raise", "def"]}
```

完全一致での照合は脆い(言い回しの違いで落ちる)ため、**「満たすべき条件」での部分照合**(キーワード包含、構造の検査、後述のプログラム的検証)を組み合わせるのが実用的です。

### プログラム的検証

コードや構造化出力(JSON等)では、**機械的に白黒つけられる** ことが多く、これが最も信頼できる評価軸になります。

- JSONとしてパースできるか / スキーマに合致するか
- コードがコンパイル(型チェック)を通るか
- 生成されたテストが通るか
- 正規表現・パーサで構造が満たされているか

主観が入らず、安く速く回せるのが最大の利点です。**評価可能な出力形式を設計段階で選ぶ**ことが、Evalを容易にします。

### LLM-as-a-judge

自由文の要約・説明・回答など、**機械的な正解照合が難しい** 出力には、別のLLMを審査員(judge)として使う手法が有効です。

```text
[判定プロンプトの骨子]
あなたは厳格な採点者です。以下の基準で回答を1〜5で採点してください。
基準: 正確さ、簡潔さ、指示の遵守
入力: {input}
回答: {output}
出力フォーマット: {"score": <1-5>, "reason": "<理由>"}
```

::: warning LLM-as-a-judge の落とし穴
審査員LLMにもバイアスがあります。**長い回答を高く評価しがち**、**自分と同系統のモデルの出力を好みがち**、**提示順で評価が変わる(位置バイアス)** などが知られています。対策として、採点基準を具体化する、ペア比較では順序を入れ替えて両方試す、重要な判定は人手評価と相関を取る、といった工夫が要ります。審査員の判定そのものも、人手評価をゴールデンとして検証すべきです。
:::

### ルーブリック評価

「良い出力」が多面的なとき、**採点基準(ルーブリック)を明文化** して各観点を点数化します。

| 観点 | 配点 | 判定方法 |
| --- | --- | --- |
| 正確さ | 0-3 | 事実誤りがないか |
| 指示遵守 | 0-2 | フォーマット・制約を守ったか |
| 簡潔さ | 0-2 | 冗長でないか |
| 安全性 | 0-1 | 不適切な内容を含まないか |

ルーブリックは人手評価とLLM-as-a-judgeの両方で使え、**「何を良しとするか」をチームで合意する** 文書としても機能します。

### 回帰テスト

一度直したバグが再発しないよう、**過去に失敗して修正したケースを固定** し、変更のたびに再実行します。プロンプト改善やモデル更新で「直したはずのものが壊れる」事故を防ぐ、Evalの中で最も費用対効果が高い部分です。

## コード生成の評価

コード生成はAI駆動開発の中心であり、評価の階層を明確に持つべきです。**下の層ほど客観的で安く、上の層ほど主観的で高い**。

| 層 | 評価 | 客観性 | コスト |
| --- | --- | --- | --- |
| 1 | パース/コンパイル(型チェック)を通るか | 高 | 低 |
| 2 | 既存テスト・生成テストが通るか | 高 | 低 |
| 3 | リンタ・静的解析・セキュリティスキャンを通るか | 高 | 低 |
| 4 | 仕様(ルーブリック)を満たすか(LLM-judge) | 中 | 中 |
| 5 | 人手レビュー(可読性・設計) | 低(主観) | 高 |

```python
def evaluate_code_generation(generated_code, test_suite):
    results = {}
    # 層1: 構文・型
    results["compiles"] = try_compile(generated_code)
    if not results["compiles"]:
        return results  # 通らなければ以降は無意味
    # 層2: テスト通過率
    passed, total = run_tests(generated_code, test_suite)
    results["test_pass_rate"] = passed / total
    # 層3: 静的解析
    results["lint_clean"] = run_linter(generated_code).error_count == 0
    # 層4/5は別パイプライン(judge / 人手)で
    return results
```

::: tip pass@k という指標
コード生成では **「k回生成してそのうち1回でも正解する確率」(pass@k)** がよく使われます。1回で正解できなくても、複数サンプルから検証して選べば実用になる、というエージェント的な使い方を反映した指標です。
:::

## オフライン評価とオンライン評価

評価には、本番投入の **前** に固定データセットで測るものと、**後** に実トラフィックで測るものがあります。両方が必要です。

| | オフライン評価 | オンライン評価 |
| --- | --- | --- |
| いつ | リリース前・CI | リリース後・本番 |
| データ | 固定のEvalセット | 実ユーザーの入力 |
| 測るもの | 正解率・回帰 | 採用率・修正率・満足度・コスト |
| 速度 | 速い(自動) | 遅い(蓄積に時間) |
| 代表性 | 作り手の想定に依存 | 現実そのもの |

オフラインは速く回せるが「想定の範囲」しか見られません。オンラインは現実を映すが反映が遅い。**オフラインで素早く回帰を防ぎ、オンラインで現実とのズレを学び、それをオフラインセットに還元する**という循環を作ります。

オンラインで使える代理指標(プロキシメトリクス)の例:

- **採用率** — 提案がそのまま受け入れられた割合(IDE補完など)。
- **編集距離** — 受け入れ後にユーザーがどれだけ手を入れたか(少ないほど良い)。
- **リトライ率** — 同じ要求で再生成された割合(高いと不満のサイン)。

## eval harness の擬似コード

評価を自動化する最小の骨組みです。CIに組み込み、プロンプト/モデル変更のたびに走らせます。

```python
def run_eval(system, eval_set, scorers):
    """
    system:  評価対象(プロンプト+モデルの組)
    eval_set: [{input, expected/criteria}, ...]
    scorers: 採点関数のリスト(プログラム的検証 / LLM-judge 等)
    """
    records = []
    for case in eval_set:
        output = system.run(case["input"])
        scores = {s.name: s(case, output) for s in scorers}
        records.append({"case": case, "output": output, "scores": scores})

    summary = aggregate(records)              # 平均スコア・合格率など
    summary["regressions"] = diff_against_baseline(records)  # 前回比の劣化
    return summary, records

def gate(summary, thresholds):
    """CIゲート: 基準を割ったら失敗させる"""
    for metric, min_value in thresholds.items():
        if summary[metric] < min_value:
            raise EvalFailure(f"{metric}={summary[metric]} < {min_value}")
    if summary["regressions"]:
        raise EvalFailure(f"回帰を検出: {summary['regressions']}")
```

ポイントは、**ベースラインとの差分(回帰)を必ず見ること**、そして **CIのゲート**にすることです。スコアを出すだけでは人は見なくなります。基準を割ったらマージを止める、までやって初めて機能します。

## 評価セットの作り方

良いEvalセットはツールでなく **資産** です。作り方の指針は次の通りです。

1. **実トラフィックから種を集める** — 想像で作らず、実際の入力ログ・失敗ケースから始める。
2. **失敗を優先的に取り込む** — うまくいったケースより、間違えたケース・エッジケースが価値を持つ。
3. **多様性を確保する** — 簡単/難しい、短い/長い、典型/例外をバランスよく。偏ると測れない領域ができる。
4. **期待値は「条件」で書く** — 完全一致でなく「満たすべき性質」で。脆さを避ける。
5. **継続的に育てる** — 本番で見つかった新しい失敗を、回帰ケースとして追加し続ける。

::: info サイズより質と代表性
数千件あっても偏っていれば無意味です。**実分布を代表する数十〜数百件**のほうが、無関係な大量データより役立ちます。まず小さく作って回し、失敗を取り込んで育てるのが現実的です。
:::

## 評価の落とし穴

- **Evalセットへの過剰適合** — プロンプトをEvalセットに合わせ込むと、見かけのスコアは上がるが本番で振るわない。**Evalセットは時々入れ替え**、隠しセット(holdout)を持つ。
- **指標の代理化** — 測りやすい指標を最適化した結果、本当に欲しい品質から乖離する(グッドハートの法則)。指標が目的の代理であることを忘れない。
- **平均値だけ見る** — 平均が上がっても、特定カテゴリで壊滅していることがある。**カテゴリ別・最悪ケース**も見る。
- **judgeを検証しない** — LLM-as-a-judge を盲信し、審査員自身の精度を測らない。人手との相関を定期的に確認する。
- **一度作って放置** — モデルもユーザーも動く。更新されないEvalは現実とずれていく。

## まとめ

- Evalは **LLMシステムの自動テスト**。確率的な出力の変更が改善か劣化かを定量化する。
- 正解が測れるものは **プログラム的検証**、自由文は **LLM-as-a-judge とルーブリック**、最終確認は **人手**。
- コード生成は **コンパイル→テスト→静的解析→仕様→人手** の階層で評価する。
- **オフラインで回帰を防ぎ、オンラインで現実を学ぶ**循環を作る。CIゲートにして初めて機能する。
- Evalセットは実トラフィックと失敗ケースから **育てる資産**。過剰適合とjudge盲信に注意。
