# 歴史と進化

AI駆動開発は突然現れたわけではありません。**「人間が書いたコードから次を予測する」** という発想は40年近く前から存在し、その精度と扱える文脈が段階的に拡張されてきた結果として、現在のエージェント協調に至っています。

このページでは、各転換点が技術的に **何を可能にしたか** に注目して進化を辿ります。重要なのは年号の暗記ではなく、「どの制約が外れた瞬間に、何が新しくできるようになったか」という因果です。

::: tip このページの読み方
歴史を「補完の精度向上」の単線で捉えると本質を見誤ります。実際には **(1)予測モデルの表現力**、**(2)扱えるコンテキストの広さ**、**(3)AIが取れる行動の種類** という3つの軸が、ほぼ独立に拡張されてきました。この3軸の交点が現在地です。
:::

## 全体年表

| 時期 | 出来事 | 技術的本質 | 可能になったこと |
| --- | --- | --- | --- |
| 1980s–1990s | IntelliSense / 静的補完 | シンボルテーブルと型情報の参照 | 識別子・メンバの想起補助 |
| 2000s | テンプレート / スニペット | 構文木と定型パターン | ボイラープレートの一括展開 |
| 2010s前半 | 統計的補完 | n-gram・コーパス頻度 | 「次に来やすいトークン」の提示 |
| 2017 | Transformer論文 | 自己注意機構 | 長距離依存を並列に学習 |
| 2018–2020 | GPT / GPT-2 / GPT-3 | 大規模事前学習 | few-shotでの汎化、自然言語からの生成 |
| 2021 | GitHub Copilot 公開 | コード特化LLM(Codex)のIDE統合 | 行〜関数単位のコード生成が日常化 |
| 2022 | ChatGPT 公開 | 指示調整(RLHF)+対話UI | 設計相談・デバッグの対話化 |
| 2023– | 大規模コンテキスト & ツール使用 | 長コンテキスト + function calling | リポジトリ規模の文脈、エージェント化 |
| 2024– | MCP と標準化 | ツール接続の共通プロトコル | エコシステムの相互運用 |
| 現在 | エージェント協調 | 計画・実行・観測のループ | タスク単位の委任とCI連携 |

## 前史: 補完は何を予測していたか

### IntelliSense 世代 — 「型が知っていること」を出す

Microsoftが1996年頃のVisual Studioで普及させた **IntelliSense** に代表される静的補完は、コンパイラやエディタが持つ **シンボルテーブルと型情報** を参照していました。`obj.` と打てばその型のメンバ一覧が出る——これは予測ではなく、**確定情報の検索**です。

この世代が解いたのは「人間の記憶の限界」であって、「コードの生成」ではありません。候補は常に正しい(型的に存在する)が、**文脈に対する適切さは判断できない**。100個のメンバを出せても、いま欲しい1個を選ぶのは人間でした。

### 統計的手法 — 「来やすさ」の導入

2010年代に入ると、大量のコードコーパスから **n-gram** などの統計モデルを学習し、「この文脈の次に来やすいトークン」を頻度順に並べる手法が現れます。EclipseのCode Recommendersや各種研究プロトタイプがこの系譜です。

ここで初めて **確率** が持ち込まれました。ただし扱える文脈は直前の数トークンに限られ、変数名のような未知語に弱く、意味的な一貫性も保てません。「ありがちな並び」は出せても「この関数がやりたいこと」は捉えられない、という壁がありました。

::: info 何が足りなかったか
前史の手法はいずれも **局所的** でした。直前の型、直前の数トークン——狭い窓しか見ていない。「この関数全体の意図」や「このファイルの設計方針」という広い文脈を予測に織り込む手段が、構造的に存在しなかったのです。この制約を外したのが次のTransformerです。
:::

## 転換点1: Transformer(2017)— 長距離依存を並列に

2017年の論文 *"Attention Is All You Need"* が提案した **Transformer** アーキテクチャは、**自己注意機構(self-attention)** によって、系列中の任意の位置どうしの関係を直接モデル化できるようにしました。

技術的に効いたのは2点です。

- **長距離依存の獲得** — 数十トークン前の変数定義と、いま書いている使用箇所を結びつけられる。RNNが苦手だった「離れた文脈」を扱える。
- **並列学習** — 系列を逐次処理せず一括で計算できるため、大規模コーパスでの学習が現実的なコストに収まった。

この「広い文脈を、大量データで」という組み合わせが、後続のすべてのモデルの土台になります。補完の予測窓が、数トークンから数百・数千トークンへと一気に広がる起点でした。

## 転換点2: GPT系列(2018–2020)— 生成と汎化

Transformerを **デコーダのみ** の構成で大規模に事前学習したのがGPT系列です。GPT(2018)、GPT-2(2019)、GPT-3(2020)と規模を上げる中で、質的な変化が起きました。

| モデル | 規模感 | 観測された質的変化 |
| --- | --- | --- |
| GPT | 小 | 事前学習+ファインチューニングの有効性 |
| GPT-2 | 中 | タスク横断のゼロショット傾向 |
| GPT-3 | 大 | **few-shot** での汎化、自然言語指示への追従 |

GPT-3で顕著になったのは、**プロンプトに数例を示すだけでタスクをこなす**(few-shot)振る舞いです。これは「タスクごとに専用モデルを訓練する」という前提を崩しました。**自然言語による指示が、そのままインターフェースになる**——AI駆動開発の前提条件がここで揃います。

## 転換点3: GitHub Copilot(2021)— 補完の日常化

2021年、GitHubとOpenAIは、コードに特化して訓練されたモデル(Codex)をエディタに統合した **GitHub Copilot** を公開しました。技術的には「コード特化LLM + IDE統合」という組み合わせです。

これが普及面で決定的だったのは、**既存のワークフローに溶け込んだ**点です。新しいツールを学ぶ必要はなく、いつものエディタでコメントや関数シグネチャを書けば、関数本体の候補が出る。前史の補完との違いは明確でした。

- 候補が **行を超えて関数単位** に達した
- 変数名やコメントといった **意図の手がかり** を読み取った
- 既存コードの **スタイルに追従** した

::: warning 普及と正しさは別物
Copilotは「もっともらしいコード」を高速に出しますが、それが正しい保証はありません。この時期から **生成物のレビューとテスト** が、補助機能ではなく必須工程として意識され始めます。詳細は[ハルシネーション対策](/quality/hallucination)を参照してください。
:::

## 転換点4: ChatGPT と対話(2022)— 設計相談ができる相手

2022年末の **ChatGPT** 公開は、**指示調整(instruction tuning)とRLHF**、そして対話UIの組み合わせによって、AIの使われ方そのものを変えました。

補完が「書いている途中を埋める」のに対し、対話は **往復** を可能にします。

- 「この設計の問題点は?」と **批評を求める**
- エラーログを貼って **原因を一緒に絞り込む**
- 「もっと簡潔に」と **反復で磨く**

これにより、AIの役割が「コードを出す道具」から「壁打ち相手」へ広がりました。[成熟度モデル](/guide/what-is-aidd)で言えば **L1(補完)からL2(対話)** への移行を、業界規模で引き起こした出来事です。

## 転換点5: 大規模コンテキストとエージェント化(2023–)

2023年以降、2つの能力が並行して伸びました。

### コンテキストウィンドウの拡大

数千トークンから **数十万トークン** へとコンテキストが拡大し、複数ファイルやリポジトリ規模の情報を一度に与えられるようになりました。「断片を見て推測する」から「関連コード全体を読んで答える」への移行です。

### ツール使用(function calling)

モデルが構造化された形式で **外部ツールの呼び出し** を出力できるようになりました。これが効いたのは、AIが取れる行動の種類が増えた点です。

```text
従来:  入力 → [モデル] → テキスト出力(終わり)

ツール使用:
  目標 → [モデル] → 行動(ファイル読む/コマンド実行)
            ↑                    │
            └──── 結果を観測 ←────┘   (ループ)
```

この **「計画→実行→観測→再計画」のループ** が、エージェントの中核です。単発生成からマルチステップの自律実行へ——Claude Code、Cline、Aider、Devinといったツールがこの系譜にあたります。エージェントの原理は[エージェントとは](/agents/what-is-agent)で詳述します。

## 転換点6: MCP と標準化(2024–)

エージェントが普及すると、新たな課題が生まれました。**「AIをどのツール・データソースに、どう繋ぐか」** が各製品でバラバラだったのです。同じ「GitHubに接続」を、ツールごとに別実装する非効率が顕在化しました。

これに対し、Anthropicが2024年に公開した **Model Context Protocol(MCP)** は、AIアプリケーションと外部ツール/データソースを繋ぐ **共通プロトコル** を提供します。USB-Cが周辺機器の接続を標準化したように、MCPは「AIへのツール接続」を標準化しようとする試みです。

| 標準化以前 | MCP以後 |
| --- | --- |
| ツールごとに独自の接続実装 | 共通プロトコルで接続を記述 |
| 連携先がツールにロックイン | サーバを書けば複数クライアントで再利用 |
| エコシステムが分断 | 相互運用可能なエコシステム |

標準化が意味するのは、機能の発明ではなく **再利用と相互運用** です。個々のチームが「車輪の再発明」をやめ、エコシステム全体で能力を共有できるようになりました。

## 現在地と、ここから

3つの軸で振り返ると、現在地が見えてきます。

| 軸 | 前史 | 現在 |
| --- | --- | --- |
| 予測の表現力 | n-gram(局所) | 大規模LLM(意味的) |
| コンテキストの広さ | 数トークン | 数十万トークン |
| 取れる行動 | テキスト出力のみ | ツール使用・マルチステップ実行 |

3軸すべてが揃った結果、AIは「スニペットを出す補助輪」から「タスクを委任できる協働者」へと役割を変えつつあります。一方で、**何を作るべきかの判断・トレードオフの評価・最終責任** は人間に残るという構図は、前史から一貫して変わっていません。

::: info 歴史から学べる実務的教訓
新しい能力は、古い能力を **置き換える** のではなく **積み重なる** 傾向があります。エージェントが使える今も、単純な補完は依然として有用です。タスクのリスクと可逆性に応じて、補完・対話・委任・協調を **使い分ける** のが熟練者の構えです。
:::
