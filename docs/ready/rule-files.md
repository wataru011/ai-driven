# ルールファイル雛形

AIコーディングツールの出力品質は、**毎回のプロンプトの巧拙よりも、常時読み込ませる「ルールファイル」の質**で決まります。ルールファイルとは、エージェントがセッション開始時に自動で読み込む、プロジェクト固有の規約・文脈を書いた設定ファイルのことです。これを整備しておけば、毎回「TypeScriptで、関数コンポーネントで、命名はこうで……」と書く必要がなくなります。

このページは**コピペ用テンプレート集**です。下のブロックをそのまま貼り、角括弧 `[...]` のプレースホルダを自分のプロジェクトの値で埋めてください。解説は最小限にとどめます。ルールファイルの設計思想を深掘りしたい場合は [プロンプトの基礎](/prompting/fundamentals) を参照してください。

::: tip 主要ツールとファイルの対応
| ツール | ルールファイル | 配置場所 |
| --- | --- | --- |
| Claude Code | `CLAUDE.md` | リポジトリ直下(サブディレクトリにも可) |
| 横断的な慣習 | `AGENTS.md` | リポジトリ直下 |
| GitHub Copilot | `.github/copilot-instructions.md` | `.github/` 配下 |
| Cursor | `.cursor/rules/*.mdc` または `.cursorrules` | リポジトリ直下 |
:::

::: info AGENTS.md という新しい慣習
`AGENTS.md` は、特定ベンダーに依存せず**複数のAIエージェントが共通で参照する**ことを意図した、比較的新しい慣習です。「どのツールを使うか未確定」「チームで複数ツールが混在する」場合は、`AGENTS.md` に本体を書き、各ツール固有ファイルからはそれを参照させる構成が管理しやすくなります。
:::

## CLAUDE.md(Claude Code 用)

最も充実させるべきファイルです。Claude Code はセッション開始時にリポジトリ直下の `CLAUDE.md` を自動で読み込みます。サブディレクトリに置けば、その配下で作業するときだけ追加で読み込まれます。

以下を `CLAUDE.md` という名前でリポジトリ直下に置き、角括弧を埋めてください。

````markdown
# [プロジェクト名]

## プロジェクト概要

[このプロジェクトが何を解決するか1〜3文で。例: 社内向けの在庫管理API。
 倉庫の入出庫を記録し、在庫数をリアルタイムに参照できるようにする。]

主要な利用者: [例: 倉庫オペレーター、社内の他システム]
ステータス: [例: 本番運用中 / 開発初期 / リプレイス進行中]

## 技術スタック

- 言語: [例: TypeScript 5.x / Node.js 20]
- フレームワーク: [例: NestJS 10]
- DB: [例: PostgreSQL 16(Prisma 経由)]
- テスト: [例: Vitest + Supertest]
- ビルド/パッケージ: [例: pnpm, tsup]
- インフラ: [例: AWS ECS, GitHub Actions でCI/CD]

## ディレクトリ構成

```
src/
  modules/        # ドメインごとのモジュール(各々 controller / service / repository)
  common/         # 横断的なユーティリティ、例外、ガード
  config/         # 環境変数の読み込みと検証
test/             # E2E テスト
prisma/           # スキーマとマイグレーション
```

新しいドメインは src/modules/[ドメイン名]/ に1モジュールとして追加する。

## コーディング規約

- [例: 公開APIには必ず明示的な型注釈をつける。戻り値の型推論に頼らない]
- [例: null と undefined を混在させない。「値がない」は null に統一]
- [例: 早期 return を優先し、ネストは2段までを目安にする]
- [例: 例外は common/exceptions の独自例外クラスを投げる。生の Error は使わない]
- [例: ログは common/logger を使う。console.log は禁止]
- 命名: [例: 変数・関数は camelCase、型・クラスは PascalCase、定数は UPPER_SNAKE]
- コメント: [例: 「何を」ではなく「なぜ」を書く。自明なコメントは付けない]

## テスト方針

- [例: 新規の service メソッドには必ず単体テストを追加する]
- [例: バグ修正は、まず失敗する再現テストを書いてから直す]
- [例: 外部I/O(DB・HTTP)はモックせず、testcontainers で実物に近い形で検証する]
- [例: カバレッジは目標値ではなく、分岐の網羅を重視する]
- テスト実行: `[例: pnpm test]`(単一ファイルは `[例: pnpm test src/modules/foo]`)

## やってはいけないこと

- [例: prisma/migrations を手で編集しない。必ず prisma migrate で生成する]
- [例: 環境変数を process.env から直接読まない。config/ 経由にする]
- [例: any 型を新規に追加しない。やむを得ない場合は理由をコメントで残す]
- [例: 依存ライブラリを勝手に追加しない。追加が必要なら提案だけして確認を取る]
- [例: 本番DBに対する破壊的なスクリプトを生成・実行しない]
- [例: .env や認証情報をコミット・出力しない]

## よく使うコマンド

| 目的 | コマンド |
| --- | --- |
| 開発サーバ起動 | `[例: pnpm dev]` |
| テスト | `[例: pnpm test]` |
| 型チェック | `[例: pnpm typecheck]` |
| Lint / 整形 | `[例: pnpm lint && pnpm format]` |
| マイグレーション生成 | `[例: pnpm prisma migrate dev]` |
| 本番ビルド | `[例: pnpm build]` |

変更後は必ず `[例: pnpm typecheck && pnpm test]` を通してから完了とすること。
````

::: warning 「やってはいけないこと」が効く
ルールファイルで最も効果が高いのは、肯定形の規約よりも**禁止事項の明示**です。AIは何かを生成しようとする方向に強くバイアスがかかっているため、「依存を勝手に増やすな」「any を使うな」のような禁止が、想定外の出力を防ぎます。
:::

## AGENTS.md

ベンダー非依存で、複数エージェントが参照する想定のファイルです。`CLAUDE.md` と内容が重複する場合は、**`AGENTS.md` を本体**にして、`CLAUDE.md` から参照させると二重管理を避けられます。

````markdown
# AGENTS.md

このファイルは、本リポジトリで作業するすべてのAIエージェントが
最初に読むべき共通ルールです。ツール固有の設定ファイルは、
このファイルを参照する形にしてください。

## このリポジトリについて

[プロジェクト名] — [1〜2文の概要]。
主要言語は [例: Go]、ビルドは [例: make] で行います。

## セットアップ

```
[例: make setup]   # 依存取得と git hooks の設定
[例: make dev]     # ローカル起動
```

## 作業の進め方

1. 変更前に関連するテストの場所を確認する([例: 各パッケージの *_test.go])。
2. 実装と同時にテストを追加・更新する。
3. 完了前に `[例: make check]`(lint + vet + test)を必ず通す。
4. コミットは小さく、1コミット1論点に保つ。

## 守るルール

- [例: 公開関数には必ず godoc コメントを付ける]
- [例: エラーは握りつぶさず、文脈を付けて返す(fmt.Errorf の %w を使う)]
- [例: パッケージ間の循環依存を作らない]

## やってはいけないこと

- [例: 生成コード(*.gen.go)を手で編集しない]
- [例: go.mod の依存を確認なく追加しない]
- [例: 秘密情報をログやコミットに含めない]

## 検証コマンド

- テスト: `[例: make test]`
- Lint: `[例: make lint]`
- 全チェック: `[例: make check]`
````

`CLAUDE.md` 側からは、先頭に次の1行を入れて参照させると管理が一本化できます。

```markdown
# このプロジェクトの共通ルールは AGENTS.md を参照してください。
# 以下は Claude Code 固有の補足のみを記載します。
```

## .github/copilot-instructions.md(GitHub Copilot 用)

GitHub Copilot は、リポジトリの `.github/copilot-instructions.md` を補完・チャットの文脈として読み込みます。Copilot は短く具体的な指示の方が効きやすいので、`CLAUDE.md` より**簡潔**にまとめるのがコツです。

```markdown
# Copilot Instructions

## プロジェクト

[プロジェクト名]。[言語/フレームワーク、例: TypeScript + React 18(関数コンポーネント + hooks)]。

## コード生成時の規約

- 言語は [例: TypeScript]。新規コードに any を使わない。
- [例: React コンポーネントは関数コンポーネントで書く。class は使わない]
- [例: 状態管理は zustand を使う。Redux は使わない]
- [例: スタイルは CSS Modules。インラインスタイルは避ける]
- [例: 非同期は async/await。.then チェーンは使わない]
- 命名: [例: コンポーネントは PascalCase、hooks は use から始める]

## テスト

- テストフレームワークは [例: Vitest + Testing Library]。
- [例: ユーザー操作の検証は role / label ベースのクエリで書く]

## 避けること

- [例: 非推奨APIや古い書き方を提案しない]
- [例: コンソールログを残さない]
- [例: 外部ライブラリの追加を伴う提案は最小限にする]
```

::: details パスごとに指示を分けたい場合(Copilot)
Copilot は `.github/instructions/*.instructions.md` というファイルに `applyTo` のフロントマターを付けることで、特定のパスにだけ適用する指示も書けます。フロントエンドとバックエンドで規約が大きく異なるモノレポなどで有効です。まずは `copilot-instructions.md` 1枚から始め、必要になったら分割するのが現実的です。
:::

## Cursor のルール(.cursor/rules/*.mdc / .cursorrules)

Cursor には2つの方式があります。新しい `.cursor/rules/*.mdc`(複数ファイルに分割でき、適用範囲を制御できる)を推奨します。旧来の `.cursorrules`(リポジトリ直下の単一ファイル)も引き続き動作します。

### .cursor/rules/*.mdc(推奨)

`.cursor/rules/` ディレクトリに、用途ごとに `.mdc` ファイルを置きます。`globs` で適用対象パスを、`alwaysApply` で常時適用かを制御します。

まずは全体に効く1枚を `.cursor/rules/000-project.mdc` として置きます。

```mdc
---
description: プロジェクト全体の共通ルール
globs:
alwaysApply: true
---

# [プロジェクト名] 共通ルール

- 言語: [例: TypeScript 5.x]。新規 any は禁止。
- [例: 早期 return を優先し、ネストは浅く保つ]
- [例: エラーは独自例外クラスで投げる]
- 命名: [例: 変数 camelCase、型 PascalCase、定数 UPPER_SNAKE]
- 変更後は必ず `[例: pnpm typecheck && pnpm test]` を通すこと。

## やってはいけないこと

- [例: 依存ライブラリを確認なく追加しない]
- [例: 秘密情報を出力・コミットしない]
```

特定の領域(例: フロントエンド)だけに効かせたい場合は、`globs` を指定した別ファイルを追加します。

```mdc
---
description: React コンポーネントの規約
globs: src/components/**/*.tsx
alwaysApply: false
---

# コンポーネント規約

- 関数コンポーネント + hooks のみ。class コンポーネントは使わない。
- [例: props は型エイリアスで明示。React.FC は使わない]
- [例: 副作用は useEffect に閉じ込め、依存配列を正確に書く]
```

### .cursorrules(旧来・単一ファイル)

シンプルに1枚で済ませたい場合は、リポジトリ直下に `.cursorrules` を置きます。

```text
# [プロジェクト名] のルール

技術スタック: [例: TypeScript + Next.js 14(App Router)]

コーディング規約:
- 新規コードに any を使わない
- [例: Server Component をデフォルトにし、必要な箇所だけ "use client"]
- [例: データ取得はサーバ側で行い、クライアントに最小限だけ渡す]
- 命名: 変数 camelCase、型 PascalCase

テスト:
- [例: 新規ロジックには Vitest で単体テストを追加する]

禁止事項:
- [例: 依存ライブラリを確認なく追加しない]
- [例: 環境変数を直接参照せず、config 経由にする]
- 秘密情報を出力・コミットしない

完了条件:
- `[例: pnpm typecheck && pnpm lint && pnpm test]` がすべて通ること
```

## 言語別の追記スニペット

共通テンプレートに、使用言語に応じて以下を追記します。「コーディング規約」セクションに足すことを想定しています。

### TypeScript 向け

```markdown
## TypeScript 規約

- tsconfig は strict: true 前提。新規コードに any / @ts-ignore を足さない。
  どうしても必要なら理由をコメントで残し unknown + 絞り込みを検討する。
- 「値がない」は null か undefined のどちらかに統一する(混在させない)。
- 公開関数の引数・戻り値には明示的な型を付ける。推論任せにしない。
- enum より union 型リテラル(例: type Status = "open" | "closed")を優先する。
- 非同期は async/await。Promise の握りつぶし(await 漏れ)に注意する。
- ランタイム境界(API入力・環境変数)は zod 等でパースして型を保証する。
```

### Python 向け

```markdown
## Python 規約

- 対応バージョンは [例: 3.12]。f-string を使い、% / .format は使わない。
- すべての関数・メソッドに型ヒントを付ける(mypy を通す前提)。
- 整形・Lint は ruff(format + lint)に従う。手で整形しない。
- 可変デフォルト引数(def f(x=[]))を使わない。None を既定にして内部で初期化する。
- 例外は握りつぶさない。except: の裸キャッチを禁止し、具体的な例外型を捕まえる。
- I/O とビジネスロジックを分離し、純粋関数を単体テストしやすく保つ。
- 依存管理は [例: uv / poetry]。requirements を手編集しない。
```

### Go 向け

```markdown
## Go 規約

- gofmt / goimports 済みであること。手で整形しない。
- エラーは必ず処理する。無視する場合は _ = で明示し、理由をコメントする。
- エラーは fmt.Errorf("...: %w", err) で文脈を付けてラップして返す。
- 公開された型・関数には godoc コメントを付ける(名前で始める)。
- context.Context は第1引数で受け取り、構造体に保持しない。
- goroutine を起動したら、終了とエラー伝播の経路を必ず設計する(リーク禁止)。
- panic はライブラリ境界で使わない。回復可能なエラーは error で返す。
```

## 良いルールファイルの3原則

テンプレートを埋めるときの指針です。長く書けばよいわけではありません。

::: tip 1. 具体的に
「きれいなコードを書く」「ベストプラクティスに従う」は情報量ゼロです。AIは既に「きれいに書こう」としています。効くのは**このプロジェクト固有の判断**です。「日付は必ず UTC で保存し、表示層でのみローカル変換する」のように、迷いを断つ一文を書きます。
:::

::: tip 2. 簡潔に
ルールファイルは毎回のコンテキストを消費します。冗長だと重要な指示が埋もれ、かえって守られなくなります。一般論(「テストは大事」)は削り、**守ってほしい具体的ルールだけ**を残します。目安として、本当に重要な禁止事項・規約を箇条書きで30〜60行に収めることを意識します。
:::

::: tip 3. 更新する
ルールファイルは「書いて終わり」ではありません。AIが同じ間違いを繰り返したら、それは**ルールファイルに書くべき内容が欠けている**サインです。レビューで何度も同じ指摘をしたら、その指摘をルールに昇格させます。逆に、もう守られている・自明になったルールは削ります。リポジトリの規約が変わったら真っ先に更新します。
:::

::: warning 矛盾と冗長を避ける
ファイルが複数(`AGENTS.md` と `CLAUDE.md` と Cursor rules)あると、内容が食い違いやすくなります。**本体を1つに決め、他は参照させる**のが鉄則です。同じことを別の言葉で繰り返すと、AIはどちらを優先すべきか判断できず、指示の効きが弱まります。
:::

## チェックリスト

ルールファイルを作ったら、以下を確認します。

- [ ] プロジェクト概要を読めば、何のリポジトリか1分で分かる
- [ ] 技術スタックにバージョンが入っている
- [ ] 「やってはいけないこと」が3つ以上、具体的に書いてある
- [ ] テストの実行コマンドと「完了条件」が明記されている
- [ ] 秘密情報の取り扱い(出力・コミット禁止)に触れている
- [ ] 一般論ではなく、このプロジェクト固有の判断が書かれている
- [ ] 複数ファイルがある場合、本体と参照の関係が明確で矛盾がない
