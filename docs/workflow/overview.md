# 開発ワークフローの全体像

プロンプトとコンテキストの技術([プロンプトの基礎](/prompting/fundamentals))は、それ単体では生産性に結びつきません。価値が生まれるのは、それらを**開発プロセスの各フェーズに正しく織り込んだ**ときです。このページは、設計から保守までの全体地図と、各フェーズ詳細への入り口を提供します。

::: tip このページの位置づけ
ここは"地図"です。各フェーズの具体的な進め方は、リンク先の専用ページで深掘りします。まず全体像と原則を掴んでください。
:::

## SDLC各フェーズへのAIの織り込み

ソフトウェア開発ライフサイクル(SDLC)の各フェーズで、AIの「得意」と「任せ方」は異なります。

| フェーズ | AIの主な役割 | 任せ方 | 人間が担う判断 | 詳細 |
| --- | --- | --- | --- | --- |
| 設計 | 案出し・比較・壁打ち | 複数案を出させ吟味 | 採用判断・トレードオフ | [設計フェーズ](/workflow/design) |
| 実装 | スキャフォールド・差分生成 | タスク分解して委任 | 分解の粒度・統合 | [実装フェーズ](/workflow/implementation) |
| テスト | テスト生成・エッジ洗い出し | 観点を指定して生成 | テストの妥当性 | [テストとレビュー](/workflow/testing-review) |
| レビュー | 一次レビュー・観点提示 | diffを渡し指摘させる | 最終承認・設計レビュー | [テストとレビュー](/workflow/testing-review) |
| 保守 | リファクタ・依存更新・移行 | テストで囲って委任 | 変更範囲・安全性 | [リファクタリングと保守](/workflow/refactoring) |

### 俯瞰図

```text
  要件 ──▶ 設計 ──▶ 実装 ──▶ テスト ──▶ レビュー ──▶ 保守
           │        │        │          │           │
   AI:   案の比較  差分生成  ケース網羅   一次指摘    リファクタ
   人:   採用判断  統合判断  妥当性確認   最終承認    範囲判断
           ▼        ▼        ▼          ▼           ▼
        ┌─────────────────────────────────────────────┐
        │        検証(テスト/型/レビュー)が全工程を貫く     │
        └─────────────────────────────────────────────┘
```

重要なのは、図の下段の**検証レイヤーが全フェーズを貫いている**点です。AIの出力は、どのフェーズでも検証を通して初めて信頼に足るものになります。

## 「AIファースト」ではなく「検証ファースト」

AI駆動開発で最も危険な発想は、「まずAIに全部書かせて、後で直す」という**AIファースト**です。生成は速いので大量のコードがすぐ手に入りますが、それを検証する仕組みがなければ、品質の低いコードを高速に量産するだけになります。

代わりに本サイトが推奨するのは**検証ファースト**です。

::: tip 検証ファーストの原則
**「どうやって正しさを確認するか」を、生成より先に決める。** 生成の速さではなく、検証の確実さを設計の起点にする。
:::

具体的には:

1. **生成の前に合否基準を決める** — テスト、型、受け入れ条件。何をもって「できた」とするか。
2. **検証可能な単位で生成させる** — 一度に大量ではなく、テストで囲える小さな差分で。
3. **客観的フィードバックで回す** — テスト結果・lint・型エラーをモデルに返して収束させる([反復改善ループ](/prompting/techniques))。
4. **人間は検証と判断に集中する** — 書く速さが上がった分、レビューと設計判断の比重を上げる。

::: warning 速さの錯覚
生成が速いと「生産性が上がった」と錯覚しがちですが、**手戻りとデバッグを含めた総コスト**で測るべきです。検証なき高速生成は、しばしば総コストを増やします。
:::

## タスクごとにAIの関与度を変える

[AI駆動開発とは](/guide/what-is-aidd)で示した成熟度モデル(L0〜L5)は、**プロジェクト全体ではなくタスクごとに適用**するのが熟練者の戦略です。

| タスクの性質 | 推奨する関与度 | 理由 |
| --- | --- | --- |
| 高リスク・不可逆(課金、認証、削除) | 低(L2: 対話) | 慎重なレビューが必須 |
| 定型・低リスク(ボイラープレート) | 高(L3〜L4: 委任) | 手戻りコストが低い |
| 探索的(技術調査、設計検討) | 中(L2: 壁打ち) | 判断は人間が握る |
| 機械的・反復的(依存更新、定型移行) | 高(L4〜L5) | 検証を自動化しやすい |

判断軸は **リスク × 可逆性 × 検証コスト** です。可逆で検証が安いタスクほど大胆に委任でき、不可逆で検証が高いタスクほど慎重になります。

## フェーズ横断で効くプラクティス

どのフェーズでも共通して効く、いわば"土台"となるプラクティスがあります。フェーズ固有のテクニックより、まずこれらを整えるほうが投資対効果は高いことが多いです。

- **規約の標準化** — `CLAUDE.md` 等で前提を固定し、全フェーズで一貫したコンテキストを与える([コンテキスト設計](/prompting/context-design))。手貼りの揺らぎをなくし、チームで再現性のある品質を出す土台になる。
- **小さな差分** — 設計でも実装でも保守でも、レビュー可能な大きさに刻む。差分が小さいほど誤りは見つけやすく、切り戻しも容易になる。
- **テストを資産化** — 検証ファーストの土台。テストが厚いほどAIに大胆に任せられる。逆にテストの薄い領域は、関与度を下げて慎重に進める。
- **判断の記録** — なぜその設計・実装を選んだかを残す。AIの提案を鵜呑みにせず、根拠を明示する。後から「なぜこうなっているか」を辿れることが、保守フェーズで効いてくる。

::: warning ツールを増やす前に土台を整える
新しいAIツールやエージェントを導入する前に、まず**テストの充実と規約の標準化**を進めるべきです。検証の土台が弱いままツールだけ高度化すると、速く生成されたコードを検証できず、かえって品質リスクが増えます。
:::

## 計測: 速さではなく総コストで見る

AI駆動開発の効果を測るとき、「生成にかかった時間」だけを見るのは誤りです。見るべきは、**着手から本番投入までの総リードタイム**と**手戻り・障害の発生率**です。

| 見るべき指標 | 避けるべき指標 |
| --- | --- |
| 着手〜マージまでの総時間 | 生成にかかった秒数 |
| レビュー指摘・手戻りの量 | 生成された行数 |
| 本番障害・ロールバック率 | 「速くなった気がする」体感 |
| 変更の切り戻し容易性 | カバレッジの数値だけ |

::: tip 体感に騙されない
生成が速いと「生産性が上がった」と感じやすいですが、それは錯覚のことがあります。**手戻りとデバッグを含めた総コスト**で評価し、検証ファーストが実際に総コストを下げているかを確認してください。
:::
