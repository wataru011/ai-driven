# モデルの選び方

AI駆動開発において「どのモデルを使うか」は、フレームワークやデータベースを選ぶのと同じ重みを持つ設計判断です。にもかかわらず、多くのチームは「いちばん賢いと言われているもの」を全タスクに使い、コストとレイテンシを無駄にしています。

本ページでは、モデル選定を **単一のランキング** ではなく **複数軸の最適化問題** として捉え直します。前提として[LLMの仕組み](/concepts/how-llms-work)と[コンテキストウィンドウ](/concepts/context-window)を理解しておくと、各軸の意味が腑に落ちます。

::: tip ひとことで言うと
**「タスクごとに最小十分なモデルを選ぶ」** のが熟練者の戦略です。全部にフラッグシップを使うのは、全部の処理をスーパーコンピュータで回すようなものです。
:::

## モデルを評価する5つの軸

モデルは一次元の「賢さ」では測れません。少なくとも次の5軸で評価します。

| 軸 | 問い | 高いほど良いか |
| --- | --- | --- |
| 能力(capability) | 難しい推論・長い依存関係・曖昧な要求を扱えるか | タスク依存 |
| コンテキスト長 | 一度にどれだけの情報を入力できるか | 大きいほど柔軟だがコスト増 |
| 速度・レイテンシ | 応答までの時間・スループット | 用途依存(対話は速さ重視) |
| コスト | 入力/出力トークン単価 | 安いほど良いが能力とトレードオフ |
| データ取扱い | データがどこで処理され、どう保持されるか | 要件依存(機密性) |

この5軸は互いにトレードオフの関係にあります。能力を上げればコストとレイテンシが上がり、ローカル実行でデータ主権を得れば能力で妥協する、というように。**「全軸で最良」のモデルは存在しない** という前提から始めるのが重要です。

### 軸1: 能力

能力は「難所をどれだけ越えられるか」です。具体的には次のような場面で差が出ます。

- 複数ファイル・長い依存関係をまたぐ推論
- 曖昧・不完全な要求からの妥当な補完
- 指示の正確な遵守(instruction following)
- ツール使用の的確さ(エージェント用途)

::: warning ベンチマーク数値を鵜呑みにしない
公開ベンチマーク(コーディング・推論など)はモデル選定の出発点にはなりますが、**あなたのタスク分布を代表しているとは限りません**。ベンチで高得点でも、自社の実タスクで振るわないことは珍しくありません。最終判断は必ず[自前の評価(Evals)](/models/evaluation)で行ってください。揮発しやすいスコアは執筆時点の各社一次情報を確認してください。
:::

### 軸2: コンテキスト長

コンテキスト長は「一度に渡せる情報量」の上限です。大きいほどリポジトリ規模の文脈を扱えますが、注意点があります。

- **長いほど高い** — 入力トークンに比例してコストとレイテンシが増えます。
- **長ければ精度が上がるとは限らない** — 関連の薄い情報で埋めると、かえって精度が落ちる("lost in the middle")。詳しくは[コンテキストウィンドウ](/concepts/context-window)を参照。

「とりあえず全部入れる」のではなく、[RAG](/concepts/rag)などで **関連する文脈だけを選別して渡す** のが定石です。

### 軸3: 速度・レイテンシ

速度には2つの側面があります。

- **レイテンシ(TTFT/応答完了まで)** — 対話的なIDE補完やチャットでは体感を直接左右します。
- **スループット(トークン/秒、並列度)** — バッチ処理や大量生成で効きます。

インライン補完のように「速さが正義」の用途では、能力を多少落としてでも軽量・高速なモデルを選ぶのが正解です。

### 軸4: コスト

コストは入力トークンと出力トークンで別単価が普通で、一般に **出力のほうが高い** 傾向があります。詳細な最適化は[コスト最適化](/models/cost-optimization)で扱いますが、選定段階では「このタスクにフラッグシップ級の単価を払う価値があるか」を問います。

### 軸5: データ取扱い(クラウド / ローカル)

見落とされがちですが、規制産業や機密コードを扱う組織では決定的な軸です。

- データがどの地域・どの事業者で処理されるか
- 入力がモデルの学習に使われないか(多くの事業者向けAPIは学習に使わない方針を明示しているが、**契約・設定を必ず確認**)
- ログ保持期間とアクセス制御
- オンプレ / VPC / ローカル実行の可否

機密度の高いコードベースでは、この軸が他の4軸より優先されることがあります。

## クローズド vs オープンウェイト

モデルの提供形態は大きく2つに分かれます。

| 観点 | クローズド(APIホスト型) | オープンウェイト(自己ホスト可) |
| --- | --- | --- |
| 入手形態 | APIエンドポイント経由 | 重みをダウンロードして実行 |
| 最高水準の能力 | フラッグシップが先行しやすい | 追随。一部用途では実用十分 |
| 運用負荷 | 事業者が運用(低い) | 自前でGPU・推論基盤を運用(高い) |
| データ主権 | 事業者の方針・契約に依存 | 自組織内で完結できる |
| カスタマイズ | 限定的(ファインチューン提供は事業者次第) | 重みに直接アクセス・微調整可能 |
| コスト構造 | 従量(トークン課金) | 固定(インフラ)+運用人件費 |

::: info 二者択一ではない
実務では **両方を併用** するのが現実的です。難所や対外品質が要る部分はクローズドのフラッグシップ、機密データの処理や大量の定型処理は自己ホストのオープンウェイト、というハイブリッド構成がよく採られます。
:::

### Claudeモデルファミリーの位置づけ

クローズド側の一例として、Anthropic の Claude ファミリーは能力と速度・コストの帯で複数のモデルを提供しています(執筆時点では Claude 4.X 系の **Opus 4.8 / Sonnet 4.6 / Haiku 4.5**、および **Fable 5**)。一般的な棲み分けの考え方は次の通りです。

- **最上位(Opus 級)** — 難所・複雑な推論・長い依存関係。能力最優先。
- **中位(Sonnet 級)** — 能力と速度・コストのバランス。日常の実装タスクの主力。
- **軽量(Haiku 級)** — 高速・低コスト。定型処理・分類・要約・大量バッチ。

具体的なコンテキスト長・単価・速度の数値は更新されるため、**各社の一次情報(モデルカード・料金ページ)を執筆時点で必ず確認**してください。

## ローカルLLMという選択肢

オープンウェイトモデルを手元のマシンや自組織のサーバーで動かす「ローカルLLM」には、明確な利点と制約があります。

**利点**

- データが外部に出ない(機密性・コンプライアンス)
- ネットワーク非依存・オフライン動作
- 従量課金がなく、大量実行で限界コストが下がる

**制約**

- 最高水準のフラッグシップに能力で及ばないことが多い
- GPU/メモリの確保と推論基盤の運用が必要
- 量子化で動かす場合、精度と速度のトレードオフが生じる

::: tip ローカルが効く典型
**「機密 × 定型 × 大量」** の交点。例えば社内コードのコメント生成、ログの一次分類、PRの要約など、能力よりも「外に出さない」「安く回す」が効くタスクです。難所の設計判断までローカルに任せようとすると、たいてい割に合いません。
:::

## タスク別の使い分け

モデル選定の核心は「タスクの性質に最小十分なモデルを割り当てる」ことです。

| タスク | 性質 | 推奨の方向性 |
| --- | --- | --- |
| 難しいバグの原因究明 | 高難度・低頻度 | フラッグシップ(能力最優先) |
| 新規機能の設計・実装 | 中難度・中頻度 | 中位モデルを主力に |
| インライン補完 | 低難度・超高頻度 | 軽量・高速モデル |
| コミットメッセージ生成 | 定型・高頻度 | 軽量モデル |
| 大量ログの分類・要約 | 定型・大量 | 軽量 or ローカル |
| 機密コードの処理 | データ主権が最優先 | 自己ホスト / VPC |

### ルーティングという考え方

一段進んだ運用では、入力の難易度を見て **動的にモデルを振り分ける**(ルーティング)構成を採ります。簡単な質問は軽量モデルへ、難所だけフラッグシップへエスカレーションする、という二段構えです。

```text
入力 → [難易度の見積もり]
        ├─ 易しい  → 軽量モデルで即応
        └─ 難しい  → フラッグシップへエスカレーション
                     (必要なら計画→実行に分解)
```

ルーティングはコストとレイテンシを大きく下げられますが、振り分けロジック自体の精度と保守が新たな関心事になります。詳細は[コスト最適化](/models/cost-optimization)で扱います。

## 選定のワークフロー

実務での選び方を手順に落とすと、おおむね次のようになります。

1. **タスクを分類する** — 難易度・頻度・リスク・データ機密度で棚卸しする。
2. **制約から絞る** — データ主権・予算・レイテンシ要件で候補を削る。これらは交渉不能なことが多い。
3. **候補を2〜3に絞る** — 軸を見て現実的な候補を残す。
4. **自前の評価で比べる** — [Evals](/models/evaluation) を使い、自社の実タスク分布で測る。ここが本丸。
5. **使い分けを設計する** — 単一モデルに固定せず、タスク別・ルーティングで最適化する。
6. **定期的に見直す** — モデルも価格も急速に動く。選定は一度きりでなく継続的に。

::: warning モデル選定はロックインに注意
特定モデルのクセ(プロンプトの効き方・出力フォーマット)に過剰適合すると、乗り換えコストが膨らみます。プロンプトとアプリケーションの境界を疎結合に保ち、**モデルを差し替え可能なコンポーネント**として設計しておくのが安全です。
:::

## まとめ

- モデルは「賢さ」の一次元でなく、**能力・コンテキスト長・速度・コスト・データ取扱い**の5軸で評価する。
- クローズドとオープンウェイトは二者択一でなく **併用** が現実的。難所はフラッグシップ、機密・定型はローカル。
- **タスクに最小十分なモデルを割り当て**、必要ならルーティングで動的に振り分ける。
- 最終判断は公開ベンチでなく **自前の評価** で。価格・スペックは揮発するので一次情報を確認する。
