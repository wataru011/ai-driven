# ベストプラクティス チェックリスト

実務でそのまま使える点検リストです。プロジェクト開始時、PRレビュー時、四半期の運用見直し時など、場面に応じて該当カテゴリを引いてください。印刷・コピーして使える密度で構成しています。

::: tip 使い方
すべてを一度に満たす必要はありません。**自分のタスクのリスクと可逆性に応じて** 必要な項目を選びます。重要な変更ほど厳格に、可逆な作業ほど軽量に。
:::

## 導入前(プロジェクト・タスクの準備)

- [ ] このタスクにAIを使う妥当性を判断した(リスク・可逆性・検証コストを考慮)
- [ ] 使うAIサービスが組織の承認済みリストにある
- [ ] 投入するデータの機密区分を確認した(機密・個人情報・認証情報を含まない)
- [ ] 利用契約のデータ取扱い条件(学習利用・保持)を把握している
- [ ] AIに渡すべきコンテキスト(規約・設計方針・関連コード)を準備した
- [ ] 成功の定義(完了条件・受け入れ基準)を自分の言葉で明確にした
- [ ] 生成結果を検証する手段(テスト・レビュー)が用意できている

## プロンプト

- [ ] 「何を・なぜ」を明確に伝え、制約と前提を明示した
- [ ] 期待する出力の形式・粒度を指定した
- [ ] 必要なコンテキスト(規約、型定義、関連コード)を渡した
- [ ] 複雑なタスクは段階に分解した(一度に欲張らない)
- [ ] 必要なら例(Few-shot)で期待挙動を示した
- [ ] 機密情報・認証情報・顧客データをプロンプトに含めていない
- [ ] AIの出力を鵜呑みにせず、前提や事実を確認する姿勢で受け取った
- [ ] 期待外れの出力は、責めるのではなくプロンプトを改善して再試行した

## 生成コードのレビュー

- [ ] 生成コードを **自分で読んで理解** した(理解できないコードはマージしない)
- [ ] 要件と制約を満たしているか確認した
- [ ] 存在しないAPI・ライブラリ・関数を使っていないか確認した(ハルシネーション検知)
- [ ] エッジケース・エラー処理・境界条件が妥当か確認した
- [ ] 既存のコード規約・設計方針に沿っているか確認した
- [ ] 冗長・不要な抽象・コピペの重複がないか確認した
- [ ] パフォーマンス上の明らかな問題がないか確認した
- [ ] 依存として追加されたライブラリの必要性・ライセンス・保守状況を確認した
- [ ] 「動くから良い」ではなく「正しく・保守可能か」で判断した

## セキュリティ

- [ ] 認証情報・APIキー・トークンがハードコードされていない
- [ ] 入力検証・出力エスケープが適切で、インジェクション耐性がある
- [ ] 認可・アクセス制御が要件どおり実装されている
- [ ] 機密情報がログ・エラーメッセージに漏れていない
- [ ] 既知の脆弱性を持つ依存を導入していない(依存スキャンを通した)
- [ ] AIが提案したセキュリティ実装(暗号・認証)を専門知識で検証した
- [ ] シークレットスキャン・SASTがCIで生成コードにも適用される
- [ ] 外部入力をプロンプトに渡す箇所でプロンプトインジェクションを考慮した

## テスト

- [ ] 変更に対応するテストが存在する(AI生成・人間作成いずれでも)
- [ ] AI生成テストが **意味のある検証** をしているか確認した(常に通るだけのテストでない)
- [ ] 正常系だけでなく異常系・境界値もカバーしている
- [ ] テストが実装の写像ではなく、仕様・振る舞いを検証している
- [ ] カバレッジの数値だけでなく、テストの質を確認した
- [ ] 重要な変更は、テストに加えて人間が動作を確認した

## チーム運用

- [ ] AI生成由来のコードを必要に応じて識別できる(コミット規約・PRラベル等)
- [ ] 効いたプロンプト・ワークフローをチームで共有している
- [ ] 「やらかし事例」を安全に共有できる(心理的安全性がある)
- [ ] AIの提案でも、最終的な品質責任はレビュー・マージした人間にあると合意している
- [ ] レビューがボトルネックになっていないか(生成量増の副作用)を観察している
- [ ] 新メンバーへのオンボーディングにAI活用の作法を含めている
- [ ] 評価軸を「速く書く」から「判断・設計・レビュー」へ調整している

## ガバナンス・測定

- [ ] AI利用ポリシー(許可・禁止・データ区分)が文書化され、周知されている
- [ ] 機密・規制対象データの外部投入を防ぐ技術的ガードレールがある
- [ ] 生成コードのライセンス適合性を点検する仕組みがある(ライセンススキャン)
- [ ] 監査ログで誰がどのAIサービスを業務利用したか把握できる
- [ ] インシデント時の連絡先とエスカレーション経路が明確である
- [ ] 導入効果のベースラインを取得し、定点観測している
- [ ] 測定はLOC・PR数でなく、DORA/SPACE等のバランス指標で行っている
- [ ] 生産性指標を個人査定に使っていない(システム改善に使う)
- [ ] ポリシーを技術・契約・規制の変化に応じて定期改訂している

::: warning このリストはスタート地点
チェックリストは思考の代替ではありません。項目を機械的に埋めるのではなく、**なぜその項目が必要か** を理解したうえで、自組織の文脈に合わせて取捨選択・追加してください。
:::
