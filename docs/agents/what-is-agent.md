# AIエージェントとは

**AIエージェント**とは、大規模言語モデル(LLM)を意思決定の中核に据え、**「観測(observe)→思考(reason)→行動(act)→観測」のループ**を回しながら、外部ツールを使って目標を達成するソフトウェアシステムです。

単発のテキスト生成と決定的に異なるのは、エージェントが **自分の行動の結果を見て次の一手を決める** という点にあります。コンパイルエラーを読んで修正し、テストの失敗を見て実装をやり直し、検索結果を踏まえて方針を変える。この「環境とのフィードバックループ」こそがエージェントの本質です。

::: tip ひとことで言うと
エージェントとは、**「LLM + ツール + ループ + 停止条件」** です。LLMが次の行動を選び、ツールがそれを実行し、結果を観測してまたLLMに戻す。これを目標達成(または打ち切り)まで繰り返します。
:::

## 単発生成との違い

従来のLLM利用は「プロンプトを入れたら答えが返る」一往復(single-shot)でした。エージェントはこれを **多段の自律実行(multi-step autonomy)** に拡張します。

| 観点 | 単発生成(チャット) | エージェント |
| --- | --- | --- |
| 実行回数 | 1往復 | 目標達成まで N 回ループ |
| 外部作用 | なし(テキストのみ) | ツールで読み書き・実行・検索 |
| 誤りの扱い | 人間が気づいて再指示 | 結果を観測し自己修正 |
| 状態 | なし(ステートレス) | メモリ・履歴を保持 |
| 制御主体 | 人間が各ステップを駆動 | モデルが次手を決定 |
| 典型例 | 「この関数を説明して」 | 「このバグを直してテストを通して」 |

単発生成は「考えを述べる」のに対し、エージェントは「**目標に向けて環境を変える**」。この差が、後述する得意/不得意と、品質・セキュリティ上の注意点をすべて規定します。

## エージェントの基本ループ

最小構成のエージェントは、驚くほど単純な制御フローで動きます。

```python
def agent(goal, tools, max_steps=20):
    memory = [system_prompt(goal, tools)]
    for step in range(max_steps):
        # 思考: 次の行動を決める(ツール呼び出しか、最終回答か)
        response = llm(memory, tools=tools)

        if response.is_final_answer:
            return response.text          # 停止条件1: 目標達成

        # 行動: モデルが選んだツールを実行
        result = execute(response.tool_call)

        # 観測: 結果をメモリに追加して次のループへ
        memory.append(response.tool_call)
        memory.append(observation(result))

    return "打ち切り(max_steps到達)"        # 停止条件2: 上限
```

このループには3つの設計ポイントがあります。

- **停止条件** — 「最終回答が出た」「ステップ上限に達した」「予算(トークン/時間)を超えた」のいずれか。停止条件の欠如はコスト暴走の主因です。
- **ツール集合** — エージェントが取れる行動の空間そのもの。ツールが貧弱なら何もできず、過剰なら誤用が増える([ツール使用とMCP](/agents/tools-and-mcp)で詳述)。
- **メモリ** — どこまでの履歴・観測を次の思考に渡すか。

## ReActパターン: 推論と行動の交互実行

エージェントの古典的かつ実用的な設計が **ReAct(Reasoning + Acting)** です。モデルに「**思考(Thought)→行動(Action)→観測(Observation)**」を明示的に交互生成させることで、行動の根拠を言語化させ、誤った行動を減らします。

```text
Thought: まずテストを実行して、どこが失敗しているか確認する必要がある
Action: run_tests()
Observation: FAILED tests/test_auth.py::test_login - AssertionError: 401 != 200

Thought: 401ということは認証で落ちている。auth.py のトークン検証を読む
Action: read_file("src/auth.py")
Observation: <ファイル内容...>

Thought: verify_token がトークンの有効期限を見ていない。ここを修正する
Action: edit_file("src/auth.py", ...)
Observation: 編集成功

Thought: 再度テストを実行して直ったか確認する
Action: run_tests()
Observation: PASSED (12 passed)

Thought: 全テスト通過。目標達成
Final Answer: トークン有効期限の検証漏れを修正し、テストを通しました
```

::: info なぜ「思考を書かせる」と精度が上がるのか
LLMは出力したトークンを次の入力の文脈として再利用します。推論過程を明示的に出力させることで、モデルは**自分の中間結論を参照しながら**次の行動を選べます。詳しくは[LLMの仕組み](/concepts/how-llms-work)を参照してください。
:::

## 計画と実行(Plan-and-Execute)

ReActは「一手ずつ考える」反応型ですが、タスクが大きいと近視眼的になり、迷走します。これを補うのが **計画と実行(Plan-and-Execute)** の分離です。

1. **計画フェーズ** — まずタスクをサブタスクへ分解し、実行順序・依存関係・成功条件を列挙する。
2. **実行フェーズ** — 各サブタスクをReact的に処理する。
3. **再計画(replan)** — 想定外の観測が出たら計画を更新する。

```text
Plan:
  1. 既存のユーザーモデルとDBスキーマを把握する
  2. パスワードリセットのエンドポイントを設計する
  3. トークン生成・検証ロジックを実装する
  4. メール送信を実装する(モック可)
  5. 統合テストを書いて通す

→ サブタスクごとに実行・観測・必要なら再計画
```

経験則として、**3〜4ステップを超えるタスクには明示的な計画を持たせる**と、成功率と可観測性が上がります。

## メモリ: 短期・長期・外部化

エージェントの賢さは「何を覚えているか」に大きく依存します。

| 種類 | 実体 | 役割 | 課題 |
| --- | --- | --- | --- |
| 短期メモリ | コンテキストウィンドウ内の会話・観測履歴 | 直近の作業文脈 | 長くなると溢れる・劣化する |
| 作業メモリ | スクラッチパッド(TODO・中間結論) | 計画の保持・自己整理 | 更新の一貫性 |
| 長期メモリ | ベクトルDB・ファイル・ナレッジベース | 過去の知見・規約・履歴の再利用 | 検索精度(RAG)に依存 |

::: warning コンテキストは無限ではない
コンテキストウィンドウが大きくても、無関係な観測で埋め尽くせば精度は落ちます(いわゆる "lost in the middle")。**要約・選別・外部化**でメモリを管理することが、長時間タスクを安定させる鍵です。
:::

## エージェントが得意なタスク/不得意なタスク

エージェント化が報われるかは、タスクの性質で決まります。

**得意(自律ループが効く)**

- **検証ループが回る作業** — テスト・型・リンタ・コンパイラという「正誤の機械判定」が存在する。失敗を観測して自己修正できる。
- **探索的だが可逆な作業** — コードベースの調査、ログ解析、リファクタリング案の試行。
- **定型的だが手数の多い作業** — 依存更新、機械的な移行、ボイラープレート生成。
- **分割統治できる作業** — 明確なサブゴールに分解でき、各々を独立に検証できる。

**不得意(ループが空回りしやすい)**

- **検証手段がない作業** — 「良いデザイン」「適切な命名規則」など、機械判定できない主観的判断。
- **不可逆・高リスクな作業** — 本番DBの破壊的変更、外部への課金、公開操作。人間の承認が必須。
- **長距離の整合性を要する作業** — 数十ファイルにまたがる一貫した設計判断。文脈が溢れて破綻しやすい。
- **要件が曖昧な作業** — 何が正解かが定義されていないと、ループの停止条件が立たない。

::: tip 設計の指針
**「機械的に検証できる目標」をエージェントに与えよ。** 成功条件が `テストが通る`・`型エラーがゼロ`・`ビルドが成功する` のように二値判定できるほど、エージェントは安定して機能します。
:::

## 自律度と制御のトレードオフ

エージェント設計の中心的な緊張関係が、**自律度(autonomy)と制御(control)**のバランスです。

| 自律度 | 人間の介入点 | 速度 | リスク | 向くタスク |
| --- | --- | --- | --- | --- |
| 低(Human-in-the-loop) | 各行動の前に承認 | 遅い | 低い | 破壊的・高リスク操作 |
| 中(Human-on-the-loop) | 要所(計画・PR)で確認 | 中 | 中 | 通常の実装タスク |
| 高(Full-auto) | 完了時のみ確認 | 速い | 高い | 可逆・定型・低リスク |

自律度を上げるほど速くなりますが、誤りが伝播する距離も伸びます。熟練者は **タスクのリスクと可逆性に応じて自律度を選び**、危険な行動(ファイル削除、外部送信、課金)にだけ承認ゲートを挟みます。

```text
低リスク・可逆(例: テスト追加)   → 高自律。完了後にまとめてレビュー
中リスク(例: 機能実装)          → 計画を承認 → 自律実行 → PRレビュー
高リスク・不可逆(例: マイグレ)  → 各ステップ承認、ドライラン必須
```

## まとめ

- エージェント = **LLM + ツール + ループ + 停止条件**。結果を観測して自己修正する点が単発生成と決定的に違う。
- **ReAct** で根拠を言語化し、**Plan-and-Execute** で大きなタスクを分解し、**メモリ管理** で長時間タスクを安定させる。
- 得意なのは**機械的に検証できるタスク**。不得意なのは検証手段のない主観的・不可逆な判断。
- **自律度はタスクのリスクと可逆性で選ぶ**。速度と制御はトレードオフ。
