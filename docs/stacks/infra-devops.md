# インフラ・DevOpsでのAI活用

インフラとDevOpsは、AIコード生成の**リスクとリターンが最も非対称**な領域です。リターンは大きい——YAMLの海、繰り返しの多いマニフェスト、定型的なCI設定は、まさに退屈で間違えやすく、AIが助けになります。一方でリスクは桁違いです。アプリのバグは1リクエストを壊しますが、**インフラのミスはクラスタごと、アカウントごと、データごと**を壊します。

しかもインフラコードは、AIが**もっともらしく書けてしまう**ぶん危険です。`*` のワイルドカード権限、`0.0.0.0/0` の全開放、`force_destroy = true`、`terraform apply` 相当の破壊的操作——どれも文法的には正しく、レビューを通り抜けやすい。本章は「便利さ」より「**事故をどう防ぐか**」に重心を置きます。

::: warning 大原則: 生成物は「適用」する前に「計画」を見る
インフラ領域でのAI活用の安全弁は、ただ一つに集約されます。**何を変更するかを `plan`(またはdry-run)で確認してから適用する**こと。AIが生成したコードを直接 `apply` / `kubectl apply` / 本番shellに流すのは、領域を問わず禁止と考えてください。
:::

## AI活用の勘所

### 1. IaC(Terraform等)は「差分」を必ず読む

Terraform/Pulumi/CloudFormationの生成は便利ですが、**生成されたコードの良し悪しは `plan` の差分でしか判断できません**。コードを眺めるだけでは「既存リソースが置換(destroy→create)される」ことに気づけません。

- 生成 → `terraform plan` → **差分を精読** → 必要なら修正依頼 → `apply`。
- とくに **`-/+`(置換)** と **`-`(削除)** の行を凝視する。DBやストレージの置換はデータ消失。
- `prevent_destroy` ライフサイクルを重要リソースに付け、AI生成コードの暴走を物理的に止める。

> このTerraformの `terraform plan` 出力を貼ります。destroy または replace されるリソースを
> 列挙し、それぞれデータ損失やダウンタイムの可能性を評価してください。
> 意図しない置換があれば、それを避ける書き方を提案して。

### 2. 最小権限を「明示的に」要求する

AIは権限まわりで**広く取りすぎる**傾向があります。IAMポリシーに `"Action": "*"` や `"Resource": "*"`、SecurityGroupに `0.0.0.0/0` を平然と書きます。

::: warning 過剰権限・過剰開放の典型
- IAM/RBACで `*` ワイルドカードや管理者権限を付与する
- `0.0.0.0/0` で全世界にSSH(22)やDB(5432等)を開放する
- S3バケットやストレージをpublic-readにする
- シークレットを環境変数やマニフェストに平文で埋め込む
- コンテナを `privileged: true` / root で実行する
:::

プロンプトで最小権限を不変条件として課します。

> 最小権限の原則で書いてください。
> - IAMは必要なActionとResourceのみ列挙。`*` のワイルドカードは使わない
> - インバウンドは必要なCIDR/SGからのみ。`0.0.0.0/0` は禁止(必要なら理由をコメントで)
> - ストレージは非公開がデフォルト
> - シークレットは直書きせず、Secrets ManagerやSealed Secrets等を参照

### 3. CI/CD設定は「権限」と「サプライチェーン」を疑う

CI/CDパイプライン(GitHub Actions等)は、AIが雛形を書きやすい一方で、**セキュリティ上の急所**です。

- **過剰なトークン権限** — `permissions` を絞らず `write-all` 相当にしがち。最小に絞らせる。
- **サードパーティActionの固定** — `uses: foo/bar@v3` のようなタグ参照は乗っ取りリスク。**コミットSHAでピン留め**させる。
- **シークレットのログ流出** — `echo` や `set -x` でシークレットがログに出る。
- **PRからのシークレットアクセス** — `pull_request_target` の誤用。

> このGitHub Actionsワークフローをセキュリティ観点でレビュー・修正してください。
> - `permissions` をジョブごとに最小化
> - サードパーティActionはコミットSHAでピン留め
> - シークレットがログに出る箇所がないか
> - 信頼できないコードに昇格権限が渡る経路がないか

### 4. Dockerfile / K8sマニフェストは「本番運用」を要求する

AIが書くDockerfileは「とりあえず動く」止まりになりがちです。本番品質を明示的に求めます。

| 観点 | 求めること |
| --- | --- |
| イメージ | 軽量・固定タグ(`:latest`禁止)、可能ならdistroless |
| ビルド | マルチステージで成果物のみ含める |
| 実行ユーザー | 非rootユーザーで実行 |
| K8s資源 | requests/limits を設定、liveness/readinessプローブ |
| K8sセキュリティ | `securityContext`(readOnlyRootFilesystem, runAsNonRoot, drop capabilities) |
| シークレット | マニフェスト直書き禁止。Secret参照 |

### 5. シェルスクリプトは「壊れ方」を設計させる

AIが生成するシェルスクリプトは、エラーで止まらず**間違ったまま走り続ける**ことがあります。

> bashスクリプトは堅牢に書いてください。
> - 先頭に `set -euo pipefail`
> - 変数は必ずクォート("$var")し、未定義変数で落ちるように
> - 破壊的操作(rm -rf 等)はパスを変数展開する前にガード(空文字での `rm -rf "$DIR/"` 事故防止)
> - 危険な操作の前に確認プロンプトまたは --dry-run オプション

::: warning 古典的事故: `rm -rf "$DIR/"`
`$DIR` が未定義だと `rm -rf /` 相当になります。`set -u` と、削除前のパス検証(空でないか、想定プレフィックス配下か)を必ず入れさせること。
:::

## 有効なプロンプト例

### plan前提でのIaC生成

```text
AWSにVPC + ECS Fargateサービスを構築するTerraformを書いてください。
前提: いきなり apply はしません。あなたのコードを plan で確認します。
- 最小権限: タスクロールは必要なActionのみ。`*` 禁止
- セキュリティグループ: インバウンドはALBからのみ。0.0.0.0/0 はALBの443のみ許容
- ステートフルなリソース(RDS等)には prevent_destroy を付け、誤削除を防ぐ
- 変数化し、ハードコードした値(ARN、CIDR)を残さない
最後に「plan で特に注視すべき差分」を箇条書きで添えて。
```

### 既存マニフェストの安全監査

```text
このKubernetesマニフェスト群を本番運用とセキュリティの観点で監査してください。
- requests/limits、プローブの有無
- securityContext(非root、readOnlyRootFilesystem、capabilities)
- 平文シークレット、過剰なRBAC
- イメージタグが latest や可変でないか
各指摘に重大度(高/中/低)と修正案を付けて。推測ではなくマニフェストの記述に基づいて。
```

## 落とし穴と対策

::: warning インフラ・DevOps特有の典型的な失敗
- **破壊的差分の見落とし** — `apply` で既存リソースが置換・削除される。
- **過剰権限/過剰開放** — `*` 権限、`0.0.0.0/0`、public ストレージ。
- **サプライチェーン** — タグ参照のAction/イメージ、未検証の外部モジュール。
- **シークレット直書き** — マニフェスト・env・ログへの平文混入。
- **冪等でないスクリプト** — 再実行で副作用が重複、または途中失敗で中途半端。
- **状態の不整合** — Terraform stateと実環境のドリフトを無視。
- **古い構文** — 非推奨のプロバイダ引数、廃止されたK8s APIバージョン。
:::

| 落とし穴 | 機械的に弾く手段 |
| --- | --- |
| 過剰権限/誤設定 | `tfsec` / `checkov` / `kube-score` / `kubesec` をCIに |
| 破壊的差分 | `plan` を必須化し、destroy/replaceを人間承認(またはCIで検出して停止) |
| サプライチェーン | ActionのSHAピン留めを強制、イメージのダイジェスト固定、依存スキャン |
| シークレット混入 | `gitleaks` / `trufflehog` でシークレットスキャン |
| Dockerfile品質 | `hadolint` でlint |
| ドリフト | 定期的な `plan`(drift detection)をCIで |

## 検証フロー

インフラのAI活用は、**検証フローを固定化**することで初めて安全になります。

1. **生成** — AIにコードを書かせる(最小権限・本番品質を明示)。
2. **静的解析** — `tfsec`/`checkov`/`hadolint`/`gitleaks` を通す。AIの見落としをツールで補う。
3. **plan / dry-run** — `terraform plan`、`kubectl apply --dry-run=server --diff`、shellは `--dry-run`。差分を**人間が精読**。
4. **隔離環境で適用** — まずstaging/サンドボックスへ。本番と同等構成で挙動確認。
5. **本番適用は承認制** — destroy/replace を含む変更は必ずレビュー承認。可能なら段階適用・ロールバック手順を用意。

::: tip 「不可逆操作」には特別扱いを
DBの削除・置換、ストレージの削除、本番への破壊的変更は、**人間の明示承認なしには実行されない**ガード(`prevent_destroy`、手動approvalステップ、保護ブランチ)を仕組みとして置きます。AIの判断やレビューの注意力だけに頼らないことが、この領域の鉄則です。セキュリティ全般は [セキュリティリスク](/quality/security) も参照してください。
:::

## 人間がやること / AIに任せること

| 人間が担う | AIに任せる |
| --- | --- |
| `plan` 差分の最終承認 | マニフェスト/IaCの初稿生成 |
| 不可逆操作の許可判断 | CI/CD設定の雛形 |
| 権限・公開範囲の妥当性確認 | Dockerfile/スクリプトの作成 |
| 本番適用の意思決定 | 静的解析指摘の修正 |
| ロールバック戦略の設計 | ドキュメント・diff説明の生成 |
