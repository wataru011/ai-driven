# 設計ドキュメントの作り方

[設計フェーズ](/workflow/design)では、AIを「思考を加速する相棒」として使う方法を扱いました。このページでは一歩進めて、その思考の成果物である**設計ドキュメントを、何を・どの順番で・どの粒度で作るか**を体系化します。

設計ドキュメントは、書くこと自体が目的ではありません。**チームの理解を揃え、判断の根拠を残し、未来の自分とAIに正しい文脈を渡すため**にあります。AI駆動開発ではこの最後の点が特に重要です。良い設計ドキュメントは、そのままAIへの上質なコンテキストになり、実装・テスト・レビューの全工程の精度を底上げします。

::: tip このページの前提
AIは設計ドキュメントの**初稿生成と壁打ちの相手**として極めて有用です。一方で、何を残し何を捨てるか、どのトレードオフを取るかという**設計判断と、その責任は常に人間**にあります。本ページの雛形はすべて「AIに叩き台を出させ、人がレビューして確定する」流れを前提にしています。
:::

## 全体像: ドキュメントの連鎖

設計ドキュメントは独立した文書の寄せ集めではなく、**上流の決定が下流の制約になる連鎖**です。上流ほど「なぜ作るか(Why)」、下流ほど「どう作るか(How)」に寄ります。まずこの依存関係と作成順序を押さえます。

| # | ドキュメント | 主に答える問い | 主な入力(依存元) | 主な読者 | AIの使いどころ |
| --- | --- | --- | --- | --- | --- |
| 1 | 課題/要求(背景) | なぜやるのか | ビジネス文脈・ユーザーの声 | 全員・意思決定者 | 要求の言語化・抜けの指摘 |
| 2 | 要件定義 | 何を満たすか | 1 | 開発者・QA・PO | 機能/非機能の構造化 |
| 3 | アーキテクチャ設計 | 全体をどう組むか | 2 | テックリード・開発者 | 複数案とトレードオフ |
| 4 | 基本設計(API/データモデル/画面) | 境界をどう切るか | 3 | 開発者・連携先 | インターフェースのたたき台 |
| 5 | 詳細設計 | 各部品をどう動かすか | 4 | 実装担当 | 処理フロー・エッジケース列挙 |
| 6 | ADR(意思決定記録) | なぜそう決めたか | 2〜5の各判断時点 | 将来の開発者 | 選択肢と背景の整理 |
| 7 | テスト計画 | どう検証するか | 2,4,5 | QA・開発者 | テスト観点の網羅 |
| 8 | 運用/Runbook | どう運用・復旧するか | 3,4,5 | 運用・オンコール | 手順の文書化・抜け検出 |

::: info ADRだけは「順番」ではなく「時点」で生まれる
1〜5,7,8はおおむね上から順に作りますが、**ADRは設計のどの段階でも、重要な判断を下した瞬間に**書きます。「PostgreSQLを選んだ」「イベントソーシングは採用しない」といった意思決定が発生したら、その都度1枚残すのが原則です。詳細は後述します。
:::

```mermaid
flowchart TD
  A[1. 課題/要求] --> B[2. 要件定義]
  B --> C[3. アーキテクチャ設計]
  C --> D[4. 基本設計]
  D --> E[5. 詳細設計]
  B --> G[7. テスト計画]
  D --> G
  C --> H[8. 運用/Runbook]
  D --> H
  ADR[6. ADR] -. 各判断時点で随時 .-> C
  ADR -. .-> D
  ADR -. .-> E
```

::: warning 連鎖は「下流から上流」へも検証する
上から順に作るのが基本ですが、下流を書いていると上流の穴に気づきます。詳細設計でエッジケースを詰めるうちに、要件の抜けが見つかるのはよくあることです。**矛盾を見つけたら上流に戻って直す**。ドキュメントは一度書いて終わりではなく、連鎖全体の整合を保つものです。
:::

## 最小セット vs フルセット

すべてのプロジェクトで8種すべてを書くのは過剰です。**プロジェクトの規模・リスク・寿命**に応じて出し分けます。

| ドキュメント | 軽量(個人/PoC/短命) | 標準(チーム開発) | フル(大規模/規制対象) |
| --- | --- | --- | --- |
| 課題/要求 | READMEに数行 | 1セクション | 独立文書 |
| 要件定義 | 箇条書き | 機能/非機能を分離 | トレーサビリティ付き |
| アーキテクチャ設計 | 図1枚 | 構成図+主要判断 | 複数ビュー+品質特性 |
| 基本設計 | コード/型で代替 | API/データモデル文書 | 画面・帳票含む全境界 |
| 詳細設計 | 省略(コードが設計) | 複雑な箇所のみ | 全モジュール |
| ADR | 重要判断のみ数枚 | 主要判断を継続記録 | 全判断を必須記録 |
| テスト計画 | テストコードで代替 | 観点リスト | 計画書+合否基準 |
| 運用/Runbook | 省略可 | 障害対応の要点 | 完全なRunbook+SLO |

::: tip 「コードが設計を兼ねる」範囲を見極める
型定義・スキーマ・テストコードは、それ自体が正確で最新の設計ドキュメントです。軽量プロジェクトでは、別文書を書くより**コードを設計が伝わる形に保つ**ほうが乖離が起きません。文書を増やすほど更新負債も増える、という前提でセットを選びます。
:::

## 各ドキュメントの作り方

ここからは個別に、**目的 / 含める項目 / AIの使いどころ / 人が必ず判断する点 / プロンプト雛形 / 完成の目安**を示します。

### 1. 課題/要求(背景)

- **目的**: なぜこれを作るのか、解こうとしている課題は何かを言語化する。後続すべての判断の拠り所。
- **含める項目**: 背景・現状の課題・対象ユーザー・達成したいこと(成功の定義)・スコープ外・制約(予算/期限/既存資産)。
- **AIの使いどころ**: 漠然とした要求を質問で掘り下げ、暗黙の前提を表面化させる。
- **人が必ず判断する点**: 何を「やらない」と決めるか(スコープ外)。成功の定義。
- **完成の目安**: チーム外の人が読んで「何を・なぜ作るか」を誤解なく説明できる。

```text
あなたはプロダクトマネージャー兼アーキテクトです。
次のざっくりした要求を「課題/背景ドキュメント」に整理してください。

要求:
「[ここに生の要求を貼る]」

出力構成:
- 背景 / 解こうとしている課題
- 対象ユーザーと、その人が今困っていること
- 成功の定義(何が起きれば成功か。可能なら計測可能な形で)
- スコープ外(あえてやらないこと)
- 制約・前提

ルール:
- 不明な点は確定させず「確認が必要な点」として箇条書きで質問する
- ビジネス価値が不明な機能は、勝手に正当化しない
```

### 2. 要件定義

- **目的**: 課題を、検証可能な要件に翻訳する。機能要件と非機能要件を分けるのが肝。
- **含める項目**: 機能要件(ユーザーストーリー/ユースケース)・非機能要件(性能/可用性/セキュリティ/運用)・前提・未確定事項。
- **AIの使いどころ**: 要件の構造化、抜けと矛盾の洗い出し、非機能要件の観点提示。
- **人が必ず判断する点**: 各要件の優先度。非機能の数値目標(レイテンシ・SLOなど)。
- **完成の目安**: 各要件が「どうなれば満たされたと言えるか」まで書けている。

```text
次の背景ドキュメントから、要件定義のたたき台を作ってください。

[背景ドキュメントを貼る]

出力:
## 機能要件
- ユーザーストーリー形式([誰が][何のために][何をする])
- 各ストーリーに受け入れ条件の案

## 非機能要件
- 性能 / 可用性 / セキュリティ / 保守性 / 運用 の各観点で、
  この種のシステムで通常検討すべき項目を列挙
- 数値が必要な箇所は [要決定] と明示し、勝手に値を埋めない

## 未確定事項 / リスク
- 要件間の矛盾や、前提が崩れると影響が大きい点
```

::: warning 非機能要件こそAI任せにしない
AIは「可用性99.9%」「レイテンシ200ms以内」といった数値を**もっともらしく埋めがち**です。これらはビジネス要求とコストのトレードオフで決まる経営判断です。AIには[要決定]として穴を空けさせ、値は人が入れます。

詳しくは[出力の評価](/quality/evaluating-output)も参照してください。
:::

### 3. アーキテクチャ設計

- **目的**: 要件を満たすシステム全体の構造を決める。下流すべての制約になる最重要文書。
- **含める項目**: 全体構成図・主要コンポーネントと責務・データの流れ・採用技術と理由・主要な品質特性への対応・代替案の却下理由。
- **AIの使いどころ**: 本質的に異なる複数案の生成、トレードオフの比較表、推奨が崩れる条件の提示。
- **人が必ず判断する点**: どの案を採るか。チームのスキル・既存資産との整合。可逆性の評価。
- **完成の目安**: 主要な技術選定すべてに「なぜ採ったか/なぜ他を捨てたか」がある。

```text
次の要件を満たすアーキテクチャを、本質的に異なる3案で出してください。

[要件定義を貼る]

各案について表で比較:
- 概要(主要コンポーネントとデータの流れ)
- メリット / デメリット
- 適する規模・前提
- 運用負荷と実装コストの目安

最後に:
- あなたの推奨と、その理由
- 「この推奨が成り立たなくなる前提条件」
- 私のチーム情報([人数・スキル・既存基盤を書く])を踏まえた注意点
```

却下した案とその理由は、そのままADRの材料になります。**選ばなかった道を記録する**のが、後から判断を見直すための鍵です。

### 4. 基本設計(API/データモデル/画面)

- **目的**: コンポーネント間・システム間の**境界(インターフェース)**を確定する。並行開発と連携の前提。
- **含める項目**: データモデル(テーブル/型・制約・状態遷移)・API(エンドポイント・入出力・エラー)・画面(主要画面と遷移)・整合性/同時実行の方針。
- **AIの使いどころ**: スキーマやAPI一覧のたたき台生成、命名やエラー設計の一貫性チェック。
- **人が必ず判断する点**: 境界の切り方(どこまでを1つのサービス/APIにするか)。後方互換の方針。
- **完成の目安**: 別担当者がこの境界だけ見て、内部実装を知らずに連携部分を作れる。

```text
[採用したアーキテクチャ]で、基本設計のたたき台を出してください。

## データモデル
- 主要エンティティ、カラム/型/制約、主要インデックス
- 状態を持つものは状態遷移を明示
- 監査要件: 履歴は追記のみ(更新・削除しない)

## API
- エンドポイント一覧(メソッド/パス/入出力概要/主なエラー)
- 同時編集・冪等性の方針

制約:
- DB は [PostgreSQL など]
- 設計上の判断が割れる箇所には「判断ポイント」コメントを付ける

たたき台でよい。決め打ちせず、選択肢があるなら併記すること。
```

::: tip 「判断ポイント」コメントを必ず求める
AIの設計は流暢ゆえに、判断が分かれる箇所も断定口調で書かれます。「ここはこう決めたが別案もある」という箇所を明示させると、**人がレビューすべき急所**がそのまま浮かび上がります。
:::

### 5. 詳細設計

- **目的**: 個々のコンポーネント内部の振る舞いを、実装に落とせる粒度で記述する。
- **含める項目**: 主要な処理フロー・アルゴリズム・エッジケースと例外処理・トランザクション境界・外部依存の扱い。
- **AIの使いどころ**: エッジケースの列挙、例外パスの洗い出し、処理フローの図化。
- **人が必ず判断する点**: 性能と複雑さのトレードオフ。どこまで作り込むか(過剰設計の抑制)。
- **完成の目安**: 実装者が迷わず書ける。ただし**コードで自明な箇所は文書化しない**。

```text
[対象機能]の詳細設計を詰めます。次の正常系フローに対し、
考慮漏れがちなエッジケース・例外・並行実行時の問題を洗い出してください。

正常系:
[ステップを箇条書き]

観点:
- 入力異常 / 外部依存の失敗 / タイムアウト / 部分的失敗
- 同時実行・再試行時の整合性(冪等性)
- 各ケースで「どう振る舞うべきか」は私が決めるので、
  あなたは選択肢と影響を示すに留める
```

::: warning 詳細設計は最も乖離しやすい
詳細設計はコードに最も近く、**コードが変わると真っ先に陳腐化**します。文書として残すのは「コードから読み取りにくい意図・前提・トレードオフ」に絞り、手続きそのものはテストコードに語らせるのが安全です。
:::

### 6. ADR(意思決定記録)

- **目的**: 「なぜその判断をしたか」を、判断した時点の文脈ごと残す。将来の自分とAIへの説明責任。
- **含める項目**: 文脈・決定・検討した代替案・結果(良い影響/悪い影響/トレードオフ)・ステータス。
- **AIの使いどころ**: 検討した選択肢の整理、トレードオフの言語化、過去ADRとの矛盾チェック。
- **人が必ず判断する点**: 決定そのもの。受け入れるトレードオフ。
- **完成の目安**: 数年後にこの決定を疑う人が、当時の文脈を理解し再評価できる。

ADRは1判断につき1ファイル、短く保つのが定石です。以下をテンプレートとして使えます。

```markdown
# ADR-0001: [決定を一言で。例: 状態管理にイベントソーシングを採用しない]

- ステータス: 提案中 | 承認済み | 廃止 | [ADR-0007]により置換
- 日付: 2026-06-30
- 決定者: [氏名/ロール]

## 文脈 (Context)
どんな状況・制約・要求の中でこの判断が必要になったか。
前提となる事実を、価値判断を交えずに書く。

## 決定 (Decision)
何を決めたかを能動態・断定で1〜数文。
「私たちは [X] を採用する」。

## 検討した代替案 (Alternatives)
- 案A: [概要] — 却下理由: [...]
- 案B: [概要] — 却下理由: [...]

## 結果 (Consequences)
- 良い影響: [...]
- 悪い影響 / 受け入れるトレードオフ: [...]
- この決定が崩れる条件(将来こうなれば再検討): [...]
```

::: tip ADRはAIへの最高のコンテキストになる
ADRは「なぜこうなっているか」を凝縮した文書です。実装やレビューをAIに依頼するとき、関連ADRを渡すと、**過去の判断に反する提案を抑えられます**。設計の意図を毎回説明し直す手間も消えます([プロンプトの基礎](/prompting/fundamentals))。
:::

### 7. テスト計画

- **目的**: 要件が満たされたことをどう検証するかを、実装前に決める。
- **含める項目**: テスト観点(機能/非機能)・テストレベル(単体/結合/E2E)・重点領域とリスクベースの優先度・合否基準。
- **AIの使いどころ**: 要件からのテスト観点の網羅的列挙、境界値・異常系の洗い出し。
- **人が必ず判断する点**: どこに労力を集中するか(リスクの高い領域)。自動化の範囲。
- **完成の目安**: 主要な受け入れ条件すべてに、対応する検証手段がある。

```text
次の要件定義と基本設計から、テスト観点を洗い出してください。

[要件定義 / 基本設計を貼る]

出力:
- 観点を「機能 / 境界値 / 異常系 / 非機能(性能・セキュリティ)」で分類
- 各観点をテストレベル(単体/結合/E2E)に割り当て
- リスクが高く重点的に検証すべき領域を指摘
- 受け入れ条件のうち、検証手段が対応していないものを指摘
```

詳しい進め方は[テストとレビュー](/workflow/testing-review)を参照してください。

### 8. 運用/Runbook

- **目的**: 本番で「いつもと違うこと」が起きたとき、誰でも復旧に動けるようにする。
- **含める項目**: デプロイ手順・監視項目とアラート・典型的な障害と対応手順・ロールバック・エスカレーション先・SLO。
- **AIの使いどころ**: 想定障害シナリオの列挙、手順の抜け検出、見やすい手順書への整形。
- **人が必ず判断する点**: 何を監視・アラートするか。エスカレーション体制。許容ダウンタイム。
- **完成の目安**: その場にいなかった人が、Runbookだけ見て一次対応を完了できる。

```text
[システム概要]のRunbook初稿を作ります。
このアーキテクチャで起こりうる障害シナリオを列挙し、
それぞれに一次対応手順の雛形を作ってください。

[アーキテクチャ / 基本設計を貼る]

各シナリオ:
- 検知方法(どの監視で気づくか)
- 影響範囲
- 一次対応の手順(コマンドや確認項目)
- エスカレーション条件

注意:
- 環境固有の値(URL・閾値・連絡先)は [要記入] とし、推測で埋めない
```

::: warning Runbookの推測値は事故のもと
AIは閾値や復旧コマンドを**もっともらしく生成**しますが、それが本番環境に正しい保証はありません。手順の骨格はAIに作らせ、実コマンド・閾値・連絡先は必ず人が実環境に照らして埋めます。
:::

## AIで設計書を書くときの落とし穴

設計ドキュメントは「文章がうまい」ほど信頼されがちで、AIはまさにその文章がうまい。だからこそ危険です。

| 落とし穴 | 何が起きるか | 対策 |
| --- | --- | --- |
| もっともらしい嘘 | 存在しないAPIや誤った前提を、自信たっぷりに書く | 根拠を問う。一次情報で裏取り |
| 過剰設計 | 今不要な抽象化・汎用性・マイクロサービス化を盛る | YAGNIを明示。最小構成から問う |
| 決め打ち | 判断が割れる箇所を断定で書き、選択肢を隠す | 「判断ポイント」コメントを必須化 |
| トレードオフの欠落 | メリットだけ並べ、代償を書かない | 「悪い影響/代償は?」と必ず追加質問 |
| 文脈無視 | チーム事情・既存資産・規制を知らずに一般解を出す | 制約を渡す。一般解と最適解の差を埋める |

### レビュー観点

AIが生成した設計書は、必ず以下の観点で人がレビューします。

- **根拠**: 主要な主張に「なぜそうなるか」があるか。なければ問い返す。
- **代替案**: 重要な判断に、検討して却下した案が示されているか。
- **トレードオフ**: メリットと対になる代償が書かれているか。
- **文脈整合**: 自チームの規模・スキル・既存資産・規制に合うか。
- **過不足**: 不要な複雑さがないか / 逆に非機能や異常系が抜けていないか。
- **可逆性**: 後から覆せる判断か。覆せないものは一段慎重にレビューする。

::: tip 設計書のレビューもAIと壁打ちできる
自分が書いた(あるいはAIが書いた)設計書を、別のAIセッションに「最も強い反論をして」「見落としているトレードオフを挙げて」と渡すと、盲点が出ます。ただし最終判断は人間が握る、という原則は変わりません([設計フェーズ](/workflow/design)のdevil's advocate参照)。
:::

## 仕様駆動開発との接続

ここまでは「人がレビューするためのドキュメント」を主に扱いました。これを一歩進め、**ドキュメントを実装と検証の正式な起点にする**のが仕様駆動開発です。

| 観点 | 本ページの設計ドキュメント | 仕様駆動開発 |
| --- | --- | --- |
| 主目的 | 理解・判断・記録の共有 | 実装の生成元・検証の基準 |
| 厳密さ | 人が読んで分かればよい | 機械可読・受け入れ条件が検証可能 |
| AIの役割 | 初稿生成・壁打ち | 仕様からのコード/テスト生成 |

要件定義と受け入れ条件を厳密に書けるなら、それを仕様としてAIに実装させ、仕様に対してテストする流れに発展できます。詳しくは[仕様駆動開発](/advanced/spec-driven)を参照してください。本ページのドキュメント群、とりわけ**要件定義(2)とテスト計画(7)**が、その仕様の土台になります。

## ドキュメントとコードの乖離を防ぐ

設計ドキュメント最大の敵は、**コードと食い違ったまま放置されること**です。古い設計書は、無いより有害です。AIがそれを文脈として読み、誤った前提で実装してしまうからです。

運用の原則は単純です。**生成 → 人がレビュー → コードと一緒に更新**を回し続けます。

- **更新をPRに含める**: 設計に影響する変更は、ドキュメント更新を同じPRに入れる。レビュー対象にする。
- **真実の源を一つに**: 同じ事実を二箇所に書かない。スキーマ・型・OpenAPI定義など**コードから生成できるものは生成**し、手書き文書はそれを重複させない。
- **AIで差分検出**: 実装変更後、変更diffと既存設計書をAIに渡し「この変更で陳腐化した記述は?」と聞くと、更新漏れを拾える。
- **ADRは書き換えず追記**: 過去のADRは履歴。覆すときは古いADRを消さず、新しいADRで「[ADR-0001]を置換」と記録する。

```text
次の設計ドキュメントと、今回の実装変更diffを照合してください。

[設計ドキュメントを貼る]
---
[git diff を貼る]

出力:
- diffによって事実と食い違うようになったドキュメントの記述
- 追記・修正すべき箇所の具体案
- 影響を受けそうな関連ドキュメント(ADR含む)
```

::: warning 文書を増やすほど乖離リスクは増える
「念のため」で書いた使われない設計書は、更新されずに腐ります。**読まれ、更新される文書だけを残す**。最小セットの判断(前掲)が、長期的な乖離リスクを下げる最も効果的な手段です。
:::

## まとめ

- 設計ドキュメントは**連鎖**。上流の決定が下流の制約になる。順序と依存を意識して作る。
- 各ドキュメントは「AIに叩き台、人がレビューして確定」。**判断と責任は人間**。
- 規模・リスク・寿命に応じて**最小セットとフルセットを出し分ける**。書きすぎは負債。
- AIの落とし穴(もっともらしい嘘・過剰設計・決め打ち・トレードオフ欠落)を前提に、根拠・代替案・代償をレビューする。
- ADRは「なぜ」を残す資産であり、AIへの最高のコンテキストになる。
- 最大の敵は**コードとの乖離**。生成→レビュー→更新を回し、真実の源を一つに保つ。
