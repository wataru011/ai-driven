# ツール使用とMCP

エージェントが「行動」できるのは、**ツール使用(tool use / function calling)** という仕組みのおかげです。本ページでは、ツール使用がどう動くか、良いツール定義をどう設計するか、そしてそれを標準化する **MCP(Model Context Protocol)** とは何かを掘り下げます。

::: tip 前提
本ページは[AIエージェントとは](/agents/what-is-agent)の続きです。エージェントのループ(観測→思考→行動)における「行動」の実体が、ここで説明するツール使用です。
:::

## ツール使用(function calling)の仕組み

LLM自体はテキストを生成するだけで、ファイルを読んだりAPIを叩いたりはできません。ツール使用とは、**「モデルに使える関数の一覧を渡し、モデルにどれをどの引数で呼ぶか選ばせ、実行はアプリ側が行う」** という協調プロトコルです。

重要な誤解の解消: **モデルがツールを実行するわけではありません。** モデルは「この関数をこの引数で呼びたい」という構造化された意図(JSON)を出力するだけで、実際の実行は呼び出し側のアプリケーションが担います。

```text
1. アプリ → モデル:  ユーザーの依頼 + 使えるツールの定義一覧を送る
2. モデル → アプリ:  「get_weather(city="Tokyo") を呼びたい」(tool_use)
3. アプリ:           実際に get_weather を実行(モデルは関与しない)
4. アプリ → モデル:  実行結果「晴れ, 24℃」を返す(tool_result)
5. モデル → アプリ:  結果を踏まえた最終回答、または次のツール呼び出し
```

この往復は1回とは限りません。モデルは結果を見て次のツールを呼び、また結果を観測する——これがエージェントループの実体です。

## ツール定義の設計

ツールは、**名前・説明・入力スキーマ**の3要素で定義します。モデルはこの定義「だけ」を頼りにツールを選ぶため、定義の質がエージェントの賢さを直接左右します。

```python
# Anthropic方式のツール定義(JSON Schemaで入力を記述)
tool = {
    "name": "search_orders",
    "description": (
        "顧客の注文を検索する。注文IDが分かっている場合はそれで、"
        "分からない場合はメールアドレスで検索する。"
        "返金や配送状況の問い合わせ対応に使う。"
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "order_id": {
                "type": "string",
                "description": "注文ID(例: ORD-12345)。不明なら省略",
            },
            "email": {
                "type": "string",
                "description": "顧客のメールアドレス。order_id がない場合に使う",
            },
            "status": {
                "type": "string",
                "enum": ["pending", "shipped", "delivered", "cancelled"],
                "description": "絞り込みたい注文ステータス(任意)",
            },
        },
        "required": [],
    },
}
```

良いツール定義の原則をチェックリストにまとめます。

- [ ] **説明は「何をするか」だけでなく「いつ使うか」を書く** — モデルは説明文を読んで選ぶ。使いどころが書かれていないと誤用する。
- [ ] **引数の型・制約を厳密にする** — `enum`・必須/任意・フォーマットを明示し、不正入力の余地を減らす。
- [ ] **ツールは少数精鋭に** — ツールが多すぎると選択を誤る。重複や似たツールは統合する。
- [ ] **副作用を名前で表す** — `get_*`(読み取り)と `delete_*`(破壊的)を区別できる命名にする。
- [ ] **エラーは構造化して返す** — 失敗時も「何が悪かったか」をモデルが読めるテキストで返し、自己修正させる。
- [ ] **冪等性を意識する** — リトライされても安全か。課金・送信系は特に注意。

::: warning ツールの説明は「モデルへのドキュメント」
ツールの `description` は人間ではなくモデルが読むAPIドキュメントです。曖昧な説明はそのまま誤呼び出しになります。実際に呼び間違いが起きたら、まず説明文を改善するのが定石です。
:::

## MCP(Model Context Protocol)とは

ツール使用には長らく標準がなく、AIアプリごとに独自方式でツールやデータ源を繋ぎ込んでいました。同じ「GitHubに繋ぐ」「DBに繋ぐ」連携を、ツールやモデルが変わるたびに作り直す——この M×N の組み合わせ爆発が課題でした。

**MCP(Model Context Protocol)** は、この問題を解くために **Anthropicが2024年に提唱・公開したオープン標準** です。AIアプリケーション(ホスト)と、外部のツール・データ・機能を提供する側(サーバー)とのあいだの接続方法を標準化します。

::: info MCPの位置づけ
MCPはしばしば **「AI連携におけるUSB-C / LSP」** にたとえられます。一度MCP対応のサーバーを作れば、MCPに対応した任意のホスト(AIアプリ)から再利用できる。M×Nの個別実装を、M+Nの標準実装に変えるのが狙いです。オープン標準として公開されており、特定ベンダー専用ではありません。
:::

MCPが標準化する主な概念は3つです。

| 概念 | 内容 | 例 |
| --- | --- | --- |
| **Tools** | モデルが呼び出せる関数(副作用あり) | `create_issue`、`run_query` |
| **Resources** | モデルが読み込めるデータ(文脈として) | ファイル、DBスキーマ、ドキュメント |
| **Prompts** | 再利用可能なプロンプトテンプレート | 「このPRをレビューして」定型 |

## MCPサーバーの概念

MCPは **ホスト / クライアント / サーバー** の構成で動きます。

```text
┌─────────────────────────────┐
│  ホスト(AIアプリ)           │   例: IDE拡張、チャットアプリ、エージェント基盤
│   └─ MCPクライアント ────────┼──┐
└─────────────────────────────┘  │  標準化されたプロトコル
                                  │  (stdio / HTTP など)
        ┌─────────────────────────▼──────────┐
        │  MCPサーバー(機能の提供側)         │
        │   ・Tools(操作)                    │
        │   ・Resources(データ)              │
        │   ・Prompts(定型)                  │
        └────────────┬────────────────────────┘
                     │
        ┌────────────▼─────┐  ┌──────────────┐  ┌──────────┐
        │  GitHub / DB     │  │ ファイルシステム│  │ 社内API  │
        └──────────────────┘  └──────────────┘  └──────────┘
```

- **ホスト** はAIアプリ本体。内部に **MCPクライアント** を持ち、1つ以上のMCPサーバーに接続する。
- **MCPサーバー** は、ある外部システム(GitHub、DB、ファイルシステム、社内ツールなど)への接続を、Tools/Resources/Promptsという統一インターフェースで公開する独立プロセス。
- 通信は標準化されており、ローカルなら **stdio**、リモートなら **HTTP** ベースのトランスポートが使われます。

意義は明確です。**一度GitHub用のMCPサーバーを作れば、MCP対応の任意のホストから使い回せる。** ツールエコシステムが「アプリ専用の作り込み」から「再利用可能な標準部品」へ移行します。

## セキュリティ上の注意

ツール使用とMCPは、エージェントに「現実世界へ作用する力」を与えます。力には危険が伴います。

::: warning ツールはエージェントの攻撃面そのもの
モデルは信頼できない入力(検索結果、ファイル内容、外部データ)を文脈として読みます。その入力に **悪意ある指示が混入** していれば、間接的プロンプトインジェクションによって意図しないツール呼び出しが誘発されえます。詳しくは[セキュリティリスク](/quality/security)を参照してください。
:::

実務での対策をチェックリストにまとめます。

- [ ] **最小権限の原則** — 各ツール/サーバーには必要最小限の権限だけを与える。読み取りで足りるなら書き込み権限を渡さない。
- [ ] **破壊的操作に承認ゲート** — 削除・送信・課金・公開などの不可逆操作は、自動実行せず人間の承認を挟む(human-in-the-loop)。
- [ ] **サンドボックス化** — コマンド実行・ファイル操作はコンテナや隔離環境で行い、ホストや本番環境への到達を遮断する。
- [ ] **入力/出力の検証** — ツールの引数をスキーマで検証し、戻り値もそのまま信用しない(SQLインジェクション、パストラバーサル等)。
- [ ] **MCPサーバーの出所を確認** — 信頼できないサードパーティのMCPサーバーは、ツール説明文に悪意ある指示を仕込める。導入前にコードと権限を精査する。
- [ ] **秘密情報の分離** — APIキーや認証情報はモデルの文脈に載せず、ツール実行層が安全に保持する。
- [ ] **監査ログ** — どのツールがどの引数で呼ばれたかを記録し、事後追跡できるようにする。
- [ ] **レート制限とコスト上限** — 暴走ループによる外部API濫用・課金爆発を防ぐ。

```python
# 承認ゲートの例: 破壊的ツールだけ人間の確認を挟む
DESTRUCTIVE = {"delete_file", "drop_table", "send_email", "charge_card"}

def execute(tool_call):
    if tool_call.name in DESTRUCTIVE:
        if not human_approves(tool_call):     # human-in-the-loop
            return "実行は承認されませんでした"
    return sandboxed_run(tool_call)           # 隔離環境で実行
```

## まとめ

- ツール使用は **「モデルが呼ぶ関数を選び、実行はアプリが行う」** 協調プロトコル。実行主体はモデルではない。
- ツール定義の質(名前・説明・スキーマ)がエージェントの賢さを決める。説明には「いつ使うか」を書く。
- **MCPはAnthropicが提唱したオープン標準** で、Tools/Resources/Promptsを統一インターフェースで公開し、連携の再利用を可能にする。
- ツールは攻撃面でもある。**最小権限・サンドボックス・承認ゲート・監査ログ** を前提に設計する。
