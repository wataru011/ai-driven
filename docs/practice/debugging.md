# デバッグ支援の実践

デバッグは、AIが「賢いラバーダック」として最も力を発揮する領域です。ただし、**渡す情報の質がそのまま回答の質を決める**点は、実装生成以上にシビアです。「動かない、直して」だけでは、AIは推測でコードを書き、見当違いの修正を提案します。

このページでは、**情報の渡し方 → 仮説生成 → 検証 → 絞り込み**というループを、具体的なバグを題材に実演します。原則は[プロンプトの基礎](/prompting/fundamentals)と共通ですが、デバッグ特有の「分からないことを分からないまま渡す」技術に踏み込みます。

::: tip デバッグの本質は仮説検証
デバッグとは「症状から原因への探索」です。AIは**仮説を素早く大量に出す**のが得意ですが、**どれが正しいかを決めるのは検証(=実行・観測)**です。AIの仮説を鵜呑みにせず、検証で潰していく姿勢が要になります。
:::

## 何を渡すかで決まる

デバッグプロンプトに含めるべき情報を、優先度順に並べます。

| 情報 | なぜ必要か |
| --- | --- |
| 症状(期待 vs 実際) | 何が「バグ」かの定義。これがないと修正の正解が決まらない |
| 再現手順 | 再現できないバグは直したか確認できない |
| スタックトレース/エラー全文 | 発生箇所と呼び出し経路の一次情報 |
| 関連コード | 推測でなく実物を見て判断させる |
| 環境(バージョン等) | 同じコードでも環境差で挙動が変わる |
| すでに試したこと | 同じ提案の堂々巡りを防ぐ |
| 変更履歴(直近のdiff) | 「いつから壊れたか」は強力な手がかり |

::: warning スタックトレースは「全文・原文」で渡す
要約したり一部だけ貼ると、肝心の行が落ちます。トレースは長くても**そのまま貼る**。ただし**機密情報(トークン、内部URL、個人情報、本番のクエリ内容)はマスクする**。ログをそのまま外部LLMに貼る行為は情報漏洩リスクです([セキュリティリスク](/quality/security))。
:::

## 題材のバグ

Node.js のジョブ処理で、「ときどき」注文確定メールが**二重送信**される、という報告です。再現性が低く、ログにもエラーは出ていません。厄介なやつです。

```text
// 報告された症状
- 注文確定メールが、ごく稀に2通届く(数百件に1件程度)
- エラーログは出ていない。500も返っていない
- いつからかは不明
```

## 段階1: 情報を構造化して渡す

最初のプロンプトは、修正を求めず**「原因の仮説を確率付きで」**出させます。情報を整理して渡すのがコツです。

```text
あなたは分散システムのデバッグに長けたエンジニアです。
断続的に発生する「メール二重送信」バグを調査しています。
今は修正案ではなく、原因の仮説を「ありそうな順」に挙げてください。
各仮説に「これが原因なら観測されるはずの兆候」と「確認方法」を添えて。

## 症状
- 注文確定メールが数百件に1件、2通送られる
- エラーログなし、HTTP 500 なし、再現手順は未確定

## アーキテクチャ
- 注文確定APIが BullMQ(Redis)にメール送信ジョブを enqueue
- ワーカーが3プロセス、各 concurrency=5 でジョブを消費
- メール送信は外部SaaS(SendGrid)をHTTPで呼ぶ
- ジョブの「送信済み」フラグは送信成功後に DB に書く

## 関連コード(ワーカー)
```ts
worker.process(async (job) => {
  const order = await db.order.findUnique({ where: { id: job.data.orderId } })
  if (order.mailSent) return            // 送信済みならスキップ
  await sendgrid.send(buildMail(order)) // 外部送信
  await db.order.update({               // 送信後にフラグ
    where: { id: order.id }, data: { mailSent: true },
  })
})
```
```

応答イメージ:

```text
仮説(ありそうな順):

1. [非冪等 + リトライ] 送信成功したが mailSent 更新前にワーカーが落ちる/
   ジョブがタイムアウト→ BullMQ がリトライ→ 再送信。
   兆候: 同一 orderId のジョブが2回 active になった形跡、
         送信〜フラグ更新の間にプロセス再起動やGCの長停止。
   確認: ジョブの attemptsMade とワーカーのrestartログを突き合わせる。

2. [競合(チェックと更新の間)] findUnique で mailSent=false を読んだ2つの
   ジョブ(別プロセス)が、両方とも送信に進む。read→send→write が
   アトミックでない典型的 TOCTOU。
   兆候: 同一 orderId が短時間に2ジョブ enqueue されている。
   確認: enqueue 側で重複投入がないか、ジョブ数を orderId で集計。

3. [二重 enqueue] そもそも注文確定APIがリトライされ、ジョブが2件入る。
   兆候: API のクライアント側リトライ、ロードバランサの再送。
   確認: queue 内の同一 orderId ジョブ件数。

4. [SendGrid 側の二重] HTTPリトライでSaaS側が二重受信。
   兆候: 自前ログでは1回送信なのに2通届く。
   確認: SendGrid の Activity ログで送信回数を確認。
```

これは良い出発点です。**4つの仮説それぞれに「観測されるはずの兆候」**が付いているのが重要で、これがそのまま次の調査計画になります。

## 段階2: 仮説を検証で絞る

人間が検証します。AIの仮説を**実データで潰していく**のがこの段階です。

- 仮説4(SendGrid二重): Activityログ確認 → **自前で2回POSTしていた**。SaaS側の問題ではない。仮説4を棄却。
- 仮説3(二重enqueue): キュー履歴を集計 → 同一orderIdのジョブは1件のみ。**棄却**。
- 仮説1と2が残る。どちらも「ジョブが2回実行された」か「2ジョブが競合した」かの違い。

ここでBullMQのジョブ履歴を確認すると、問題の注文では**同一ジョブの `attemptsMade` が 2**になっていました。つまり1件のジョブがリトライされている。仮説1(非冪等+リトライ)が濃厚です。

```text
検証結果を共有します:
- SendGrid側の二重ではない(自前で2回POSTしていた)
- キューには同一orderIdのジョブは1件だけ
- ただし問題のジョブは attemptsMade=2(リトライされている)

つまり「送信成功 → mailSent更新前にジョブが失敗扱い → リトライで再送信」が
起きているようです。この仮説を裏付ける/反証するために、次に何を確認・
計測すべきか、最小の手順で教えてください。
```

応答イメージ:

```text
仮説を確定させる確認手順:
1. ジョブの failedReason / lock 期限を確認。lock duration を超えて
   send が長引くと、BullMQ はジョブを stalled とみなし再投入する。
   → sendgrid.send の p99 レイテンシと lockDuration を比較。
2. 送信〜フラグ更新の間にエラーが握り潰されていないか
   (update が失敗してもジョブは success 扱いになっていないか)。
3. 再現実験: send 内に意図的な遅延を注入し、lockDuration を超えさせて
   stalled → リトライ が起きるか、ステージングで確認。
```

確認すると、`sendgrid.send` がときどき **lockDuration(30秒)を超えて遅延**し、BullMQがジョブを stalled と判断して**別ワーカーに再投入** → 送信済みフラグはまだ立っていないので再送信、という流れが確定しました。原因特定です。

::: tip 再現できないバグは「再現させる」のが先
Heisenbug(観測しようとすると消える、稀にしか出ないバグ)は、**まず確実に再現する方法を作る**ことが最大の山場です。今回のように「遅延を注入してタイムアウト経路を強制的に通す」など、AIに再現シナリオを設計させるのは有効な使い方です。
:::

## 段階3: 修正と再発防止

原因が「非冪等な処理 + リトライ」だと分かれば、修正方針は明確です。**送信を冪等にする**か、**送信とフラグ更新をアトミックに近づける**。AIに方針を比較させます。

```text
原因確定: 送信が冪等でなく、stalled リトライで再送される。
再発を防ぐ修正案を、堅牢性の順にトレードオフ付きで2〜3案出してください。
既存スタック(BullMQ + Postgres + SendGrid)の範囲で。
```

応答イメージ:

```text
案A 冪等キー(推奨): メール送信に idempotency key を付ける。
   SendGrid のような送信先が冪等キーに対応していれば、同一キーの
   再送はSaaS側で1回に集約される。最も堅牢。要: SaaSの対応確認。

案B 送信前にフラグを立てる(claim): update で mailSent=true を
   「まだfalseの場合のみ」アトミックに立て、更新できたジョブだけが送信。
   UPDATE ... WHERE mailSent=false RETURNING で claim する。
   ただし「フラグは立ったが送信失敗」だと未送信になるため、
   送信失敗時のリカバリ(再試行キュー)が別途必要。

案C lockDuration を伸ばす: 対症療法。遅延の上振れを超えられない保証はなく、
   根本解決にならない。単独採用は非推奨。
```

ここで人間が判断します。SendGridは冪等キーに対応しているので**案A**を主軸にしつつ、二重防御として**案B(claim)**も入れる、と決めました(送信は外部依存で何が起きるか分からないため、防御は多層に)。

```diff
 worker.process(async (job) => {
-  const order = await db.order.findUnique({ where: { id: job.data.orderId } })
-  if (order.mailSent) return
-  await sendgrid.send(buildMail(order))
-  await db.order.update({ where: { id: order.id }, data: { mailSent: true } })
+  // claim: まだ未送信のときだけアトミックに「送信中」を確定
+  const claimed = await db.order.updateMany({
+    where: { id: job.data.orderId, mailSent: false },
+    data: { mailSent: true },
+  })
+  if (claimed.count === 0) return  // 既に他ジョブが処理済み
+  const order = await db.order.findUnique({ where: { id: job.data.orderId } })
+  // 冪等キーで SaaS 側の二重も防ぐ(stalled リトライでも同一キー)
+  await sendgrid.send(buildMail(order), { idempotencyKey: `order-mail-${order.id}` })
 })
```

::: warning 修正にも新たな穴がある
案Bには「claim 後に送信が失敗すると `mailSent=true` のまま未送信になる」という新しい不具合の余地があります。レビューで必ず指摘されるべき点です。本番では**送信失敗時にフラグを戻すか、失敗ジョブを別キューで監視**する必要があります。AIの提案も人間の修正も、**新しい不変条件を壊していないか**を常に問うてください。
:::

## 二分探索的デバッグ

「いつからか分からない」バグには、**二分探索**が効きます。`git bisect` は機械的に犯人コミットを特定する道具で、AIはその過程を補助できます。

```text
このバグ(メール二重送信)は2か月前のリリースにはなかったと判明しました。
git bisect で犯人コミットを特定したい。
- good: v1.4.0(2か月前)、bad: HEAD
- 1回の判定は「ステージングで send に遅延注入し再送が起きるか」で行う
bisect の手順と、各ステップで何を確認すべきかを整理してください。
判定スクリプト(自動化できる形)も提案して。
```

AIは bisect の流れ、自動判定スクリプトの雛形、各ステップの注意(フレーキーな判定は bisect を狂わせる)を整理してくれます。**判定が不安定なバグでは bisect 自体が誤誘導する**ため、「判定を確定的にする」ことが先、という指摘まで引き出せると上出来です。

::: tip 二分探索はコードだけでなく入力にも使える
「どの入力でバグるか」も二分探索できます。巨大な入力データを半分にして再現するか試す、設定を半分ずつ無効化する、など。「最小再現ケース(minimal reproduction)」を作る作業そのものが二分探索です。AIに「この入力を、バグが消えない範囲で最小化する戦略を」と頼むと、削る順序を提案してくれます。
:::

## ログ解析

大量のログから異常を見つける作業もAIが補助できます。ただし**ログ全文を貼るのは非効率かつ危険**(トークン浪費・情報漏洩)。**抽出・集計してから渡す**のが鉄則です。

```text
（あらかじめ grep/集計したログのサマリを貼る）
以下は問題発生時刻 ±5分の、orderId=XXX に関するログ行を時系列で並べたものです。
個人情報はマスク済みです。
このシーケンスから読み取れる異常な点と、time gap が不自然な箇所を指摘し、
「この間に何が起きた可能性が高いか」を推論してください。

12:00:01 worker-2 job=987 picked up order=XXX
12:00:01 worker-2 sending mail order=XXX
12:00:33 worker-1 job=987 STALLED, re-queued      <- 32秒の沈黙
12:00:33 worker-3 job=987 picked up order=XXX
12:00:33 worker-3 sending mail order=XXX
12:00:34 worker-2 mail sent order=XXX
12:00:34 worker-3 mail sent order=XXX
```

AIは「worker-2 が送信を開始してから 32秒応答がなく、その間に stalled 判定で worker-3 に再投入され、結果2回送信されている」という解釈を返します。**人間が既に持っている仮説を、ログで裏付ける**のがこの使い方の本質です。

::: warning ログ解析でのハルシネーション
AIはログにない事象を「あったかのように」補完することがあります([ハルシネーション対策](/quality/hallucination))。AIの解釈は**必ずログの実際の行と突き合わせて**ください。「このログ行のどこからその結論を導いた?」と根拠を問い返すのが有効です。
:::

## デバッグのアンチパターン

| アンチパターン | 何が起きるか | 対処 |
| --- | --- | --- |
| 「動かない直して」 | 推測で的外れな修正 | 症状・再現・トレースを構造化して渡す |
| 最初の仮説に飛びつく | 真因でない箇所を直し再発 | 仮説を複数出させ検証で潰す |
| AIの仮説を検証せず採用 | 「もっともらしい嘘」で時間浪費 | 実行・観測で必ず裏付ける |
| ログ全文をそのまま貼る | トークン浪費・情報漏洩 | 抽出・集計・マスクしてから渡す |
| 再現できないまま修正 | 直ったか確認不能 | まず確実な再現を作る |
| 修正の副作用を見ない | 新しいバグを埋め込む | 修正が壊す不変条件を問う |

## まとめ

- デバッグは**仮説検証ループ**。AIは仮説を量産するが、決めるのは検証。
- 渡す情報の質がすべて。**症状・再現・トレース全文・関連コード・試したこと**を構造化。機密はマスク。
- 仮説には「これが原因なら見えるはずの兆候」を付けさせ、実データで潰す。
- 稀なバグ(Heisenbug)は**確実な再現を作る**のが先。
- `git bisect`・入力の最小化など**二分探索**を活用。判定を確定的にすることが前提。
- ログは**抽出・集計してから**。AIの解釈はログの実行と突き合わせる。
