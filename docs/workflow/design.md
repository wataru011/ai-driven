# 設計フェーズ

設計は、AI駆動開発で**最もレバレッジが効く**フェーズです。理由は単純で、設計の誤りは下流(実装・テスト・保守)のすべてに波及するため、ここでの質がプロジェクト全体のコストを左右するからです。

そしてAIは、設計フェーズでは**コードを書く道具ではなく、思考を加速する相棒**として最も力を発揮します。一人でホワイトボードに向かう代わりに、無限に付き合ってくれて、別の視点を出し、抜けを指摘してくれる相手だと考えてください。

::: tip このフェーズでのAIの本質
設計フェーズのAIは「答えを出す存在」ではなく「**選択肢を広げ、検討を深める存在**」です。最終的な設計判断は、文脈と責任を持つあなたが下します。
:::

## 1. 要件整理

曖昧な要求を、構造化された要件に翻訳する作業をAIに手伝わせます。**抜けや矛盾を洗い出す**のに有効です。

```text
あなたは経験豊富なソフトウェアアーキテクトです。
次のざっくりした要求から、機能要件・非機能要件・前提・未確定事項を
構造化して整理してください。

要求:
「社内向けに、申請を承認フローに回すツールがほしい。
 承認者は複数いて、誰か一人が承認すればOK。あとで履歴も見たい」

整理の際:
- 曖昧な点は「確認が必要な点」として明示的に質問してください
- 勝手に仕様を確定させず、解釈の分岐を示してください
```

「曖昧な点を質問させる」のがポイントです。AIに勝手な前提で進ませず、**意思決定すべき点を可視化**します。

## 2. アーキテクチャ案の比較

単一の案を出させるのではなく、**複数案とそのトレードオフ**を出させます。これがAIを使った設計の核心です。

```text
この承認フローツールのアーキテクチャを3案出してください。
それぞれについて以下を表で比較:
- 概要
- メリット / デメリット
- 適する規模・前提
- 実装コストの目安

案は本質的に異なるアプローチにすること(例: 状態をDBで持つ /
イベントソーシング / 外部ワークフローエンジン利用)。
最後に、あなたの推奨と「その推奨が崩れる条件」を述べてください。
```

::: tip 「推奨が崩れる条件」を問う
単に「おすすめは?」と聞くのではなく、**「その推奨が成り立たなくなる前提」**を問うことで、自分の状況に当てはまるかを判断できます。これは鵜呑み防止の実践的なテクニックです。
:::

## 3. API/データモデルのたたき台生成

設計の方向が決まったら、具体的なインターフェースのたたき台を出させます。**ゼロから書くより、叩き台を批評するほうが速い**からです。

```text
承認フロー(案2: DBで状態管理)で、以下のたたき台を出してください。

## データモデル
- 主要テーブルとカラム、型、制約、インデックス
- 状態遷移(申請 → 承認待ち → 承認/却下)をどう表現するか

## API
- REST エンドポイント一覧(メソッド、パス、リクエスト/レスポンス概要)
- 楽観ロックなど同時編集対策の方針

制約:
- DB は PostgreSQL
- 履歴は監査用に追記のみ(更新・削除しない)

たたき台でよいので、設計上の判断ポイントにはコメントを付けてください。
```

出力された設計には必ず**判断ポイントのコメント**を求めます。「ここはこう決めたが、別の選択肢もある」という箇所が、人間がレビューすべき急所だからです。

## 4. トレードオフ分析の壁打ち

設計判断で迷ったとき、AIを議論相手にします。**自分の案をぶつけ、反論させる**使い方が効果的です。

```text
承認の状態を1つのテーブルの status カラムで持つか、
状態遷移をイベントとして別テーブルに記録するかで迷っています。

私は前者(シンプル)に傾いていますが、この判断に対して
反対の立場から最も強い反論をしてください。
そのうえで、後者を選ぶべき具体的な兆候(将来こうなりそうなら後者)を
挙げてください。
```

::: tip 反論を求める(devil's advocate)
AIは同調しやすい傾向があります。「賛成して」ではなく「**最も強く反論して**」と頼むことで、自分の盲点を突く意見を引き出せます。
:::

## 鵜呑みにしないための判断軸

AIの設計提案は、もっともらしくても**最適とは限りません**。以下を常に自問します。

| 観点 | 問い |
| --- | --- |
| 文脈整合 | 自分のチームの規模・スキル・既存資産に合うか |
| 前提の妥当性 | AIが置いた暗黙の前提は、自分の状況で成り立つか |
| 過剰設計 | 今は不要な汎用性・抽象化を持ち込んでいないか |
| 既知の制約 | AIが知らない社内事情・運用制約に反していないか |
| 可逆性 | この判断は後から覆せるか。覆せないなら一段慎重に |

::: warning AIは"あなたの文脈"を知らない
AIの提案は一般論として妥当でも、あなたのチームの暗黙知(運用体制、過去の失敗、政治的制約)を反映していません。**一般解と最適解のギャップを埋めるのが人間の仕事**です。
:::

::: warning もっともらしさ ≠ 正しさ
整然とした図や流暢な比較表は、内容の正しさを保証しません。提案の根拠(なぜそのトレードオフになるか)を必ず確認し、疑わしければ「その主張の根拠は?」と問い返してください。
:::

## アンチパターン

::: warning 設計フェーズの失敗例
- **単一案の即採用**: 最初に出た案をそのまま実装に回す。複数案を出させ比較する。
- **質問させない**: 曖昧な要求のまま設計させ、誤った前提で進む。
- **過剰設計の追認**: AIが提案した不要な抽象化を「賢そう」と採用する。
- **判断の丸投げ**: 「どうすべき?」と聞いて答えに従う。判断軸は人間が握る。
- **文脈の未共有**: 既存システムや制約を渡さず、一般論で設計させる([コンテキスト設計](/prompting/context-design))。
:::

## まとめ

- 設計はAIのレバレッジが最大。だが役割は「相棒」であって「決裁者」ではない。
- 単一案ではなく**複数案とトレードオフ**を出させる。
- 「推奨が崩れる条件」「最も強い反論」を問い、盲点を突かせる。
- AIは**あなたの文脈を知らない**。一般解と最適解のギャップを人間が埋める。
