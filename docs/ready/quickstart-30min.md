# 30分クイックスタート

このページのゴールはただ一つ、**今日の30分で、AI駆動開発を「あなたの実務」に1回組み込む** ことです。学習用の練習ではなく、**いま抱えている本物のリポジトリ** に対して、安全な範囲で1ループ回します。

経験者にとっての本当の難所はツール操作ではなく、**「どのタスクを、どの粒度で任せ、どう検証するか」という運用の勘** です。これは読むだけでは身につきません。だからこのページは、最初から実物に手を当てます。

::: tip 進め方
タイマーを30分でセットしてください。各ブロックは時間が来たら**未完でも次へ進みます**。完璧を目指さず、まずループを1周通すことが目的です。詰まったら[つまずき表](#つまずいたら)へ。
:::

::: warning 始める前に1分だけ確認
- 触るのは **gitで管理され、作業ブランチを切れる** リポジトリにする(`git switch -c ai-quickstart`)。
- 機密(APIキー・認証情報・個人情報・未公開ソース)を投入しない。会社の環境なら**AI利用ポリシーを先に確認**する。
- 生成物は**無検証でmainにマージしない**。今日の成果物は使い捨ててもよい前提で進める。
:::

## タイムボックス全体像

| 時間 | ブロック | 成果物 |
| --- | --- | --- |
| 0–5分 | ツールを1つ選ぶ | 起動できるツール |
| 5–10分 | ルールファイルを置く | リポジトリ直下の最小ルール |
| 10–20分 | 小さな安全タスクで実装ループ | 検証可能な変更1件 |
| 20–25分 | レビュー&テスト | 通過したテスト/読めた差分 |
| 25–30分 | 振り返り | 任せる/自分でやるの線引き1行 |

---

## 0–5分: ツールを1つだけ選ぶ

比較に時間を溶かさないでください。**いまの環境で最も摩擦が少ないもの** を1つだけ選び、起動します。詳しい選定軸は[ツールの選び方](/tools/choosing)にありますが、今日の選択は下表で十分です。

| あなたの状況 | 最初の一手 | 形態 |
| --- | --- | --- |
| VS Code / JetBrains 中心 | GitHub Copilot / JetBrains AI | IDE統合(補完+チャット) |
| エディタを軽く乗り換えてよい | Cursor / Windsurf | AI特化エディタ |
| ターミナル中心で作業する | Claude Code / Aider / Cline | CLI・エージェント型 |
| まず対話だけ試したい | チャットUI(ブラウザ) | 対話のみ |

::: info モデルの目安
迷ったら高性能モデルから始めると、生成品質のブレで判断を惑わされずに済みます。Claude系なら最新の Claude 4.X 系(Opus 4.8 / Sonnet 4.6 / Haiku 4.5)が選択肢です。日常の軽い補完は軽量モデル、設計やレビューは高性能モデル、と後で使い分ければ十分です。
:::

**やること(5分)**

1. 上表からツールを1つ選ぶ。すでに入っているものがあればそれでよい。
2. インストール/サインインし、**起動だけ** 確認する。設定の作り込みはしない。
3. 作業ブランチを切る。

```bash
git switch -c ai-quickstart
```

::: warning 落とし穴
ツール比較・拡張機能あさり・設定沼は今日の敵です。「動く1つ」で先へ進みます。
:::

## 5–10分: ルールファイルを置く

ルールファイルは、ツールに毎回口頭で伝えていた前提(言語・規約・やってはいけないこと)を**常駐の文脈**として与える仕組みです。これがあるだけで出力の的中率が体感で変わります。詳しい書式とツール別の置き場所は[ルールファイル](/ready/rule-files)にまとめてあります。今日は**最小版**を1枚置きます。

**やること(5分)**

1. リポジトリ直下に、使うツールに対応したルールファイルを作る(例:汎用なら `AGENTS.md`、Cursorなら `.cursor/rules/`、Claude Codeなら `CLAUDE.md`。詳細は[ルールファイル](/ready/rule-files))。
2. 下のテンプレに、あなたのプロジェクトの実情を5行だけ埋める。
3. 保存してコミットする。

```markdown
# プロジェクトのルール(最小版)

## 技術スタック
- 言語/バージョン: [例: TypeScript 5.x / Node 20]
- フレームワーク: [例: React 関数コンポーネント + hooks]
- テスト: [例: vitest。テストは実装と同じ階層に *.test.ts]

## 守ること
- 既存の命名・ディレクトリ構成・エラー処理の流儀に合わせる
- 新しい依存は勝手に追加しない。必要なら理由とともに提案する
- 不明点は推測で埋めず、前提を明示して確認する

## やってはいけないこと
- [例: src/legacy/ 以下は変更しない]
- [例: 認証・課金まわりのコードは触らない]
```

::: tip なぜ最初にこれを置くのか
ルールファイルは「毎回同じ前置きを書く手間」を消し、**チーム全員の出力を底上げ** します。今日は最小で構いません。育て方は[ルールファイル](/ready/rule-files)で。
:::

## 10–20分: 小さな安全タスクで実装ループ

ここが本番です。**間違っても害がなく、正しさを自分で確認できる** タスクを1つ選び、生成→検証のループを回します。

**良い最初のタスク(どれか1つ)**

- 既存の純粋関数に**ユニットテストを書かせる**(テスト実行で正しさを確認できる)
- 小さな**バグ1件の修正**(再現手順が手元にあるもの)
- 既存コードの**読解・要約**(あなたが答えを知っている)
- 正規表現・SQLの**下書き**(手元で実行して確かめられる)

**避けるタスク**

- 本番に直接触れる変更、認証・課金・セキュリティ周り
- 検証手段が用意できない中核ロジック
- 「全体をいい感じに」という曖昧で大きな依頼

**やること(10分)**

1. 上から安全タスクを1つ選ぶ。題材ファイルを開いておく。
2. 下のプロンプトの角括弧 `[...]` を埋めて投げる。**文脈・タスク・制約・期待** の4点を必ず添える。
3. 出力を**そのまま受け入れない**。次のブロックで検証する。

「既存関数にテストを書かせる」場合のコピペプロンプト:

```text
あなたは[言語]に精通したエンジニアです。このリポジトリの規約はルールファイルに従ってください。

## 文脈
対象は [ファイルパス] の [関数名]。役割は [この関数が何をするか1行]。

## タスク
[関数名] のユニットテストを書いてください。

## 制約
- 既存の [テストフレームワーク] を使う
- このリポジトリの既存テスト([参考テストのパス])のスタイルに合わせる
- モックは最小限に

## 期待
- 正常系に加え、境界値・空入力・異常系を必ず含める
- 各テストが何を検証しているか、テスト名から読み取れること
- 出力は変更後のテストファイル全体
```

「小さなバグ1件を直す」場合のコピペプロンプト:

```text
あなたは[言語]に精通したエンジニアです。リポジトリの規約はルールファイルに従ってください。

## 状況
[ファイルパス] の [関数/箇所] で、次の不具合が起きます。
[再現手順 または エラーメッセージ]

## 期待する動作
[こうなってほしい、を具体的に。例: 入力が空のとき例外ではなく空配列を返す]

## 制約
- 既存のシグネチャは変えない(変える場合は理由を添えて提案)
- 新しい依存は追加しない

## 出力
1. 原因の説明(1〜3行)
2. 修正後のコード
3. このバグを再現し、修正で通るようになるテストを1つ
```

::: tip 一発で完璧を狙わない
出力は「同僚が出してきたドラフトPR」です。意図のない確率的なドラフトだと思って、**短いサイクルで磨く** のがコツ。レビューで見つけた問題は次のプロンプトに具体的に反映します。
:::

## 20–25分: レビュー&テスト — ここが本番

最も時間をかけるべき工程です。生成物は**外部検証**を通して初めて価値になります。下のチェックを通してください。

**やること(5分)**

1. まず**読んで理解できるか**。説明できないコードは採用しない。
2. **実際に動かす**。テストを実行し、変更を走らせる。
3. 嘘のAPI・ライブラリがないか確認する([ハルシネーション対策](/quality/hallucination))。
4. 落ちたら、原因を**1つだけ**指摘して再生成する(下のプロンプト)。

```bash
# 例: テストを走らせる(あなたのプロジェクトのコマンドに置き換え)
npm test            # または: pytest / go test ./... / cargo test
git diff            # 差分を自分の目で読む
```

再生成のためのフィードバックプロンプト:

```text
テストを実行したところ、次の結果になりました。

[失敗ログ または 期待と違った挙動]

この原因を1つに絞って説明し、最小の修正だけ提案してください。
無関係な箇所は変更しないでください。
```

::: warning 生成テストを盲信しない
テストもAIが書いたなら、**テスト自体の妥当性** を疑ってください。「常に成功する空テスト」「アサーションが弱い」などが紛れ込みます。1つは自分の手で値を確認します。
:::

レビュー通過チェックリスト:

- [ ] 差分を**読んで理解**できた(他人に説明できる)
- [ ] テスト/実行で**動作を確認**した
- [ ] 使われているAPI・ライブラリが**実在**する
- [ ] 空入力・境界値・異常系の**抜け** がない
- [ ] 「自分が書いた」と言える品質である

## 25–30分: 振り返り

30分の価値は、最後の5分の言語化で確定します。

**やること(5分)**

1. 今回のループで**どこに時間がかかったか** を一言メモする。多くの場合、ボトルネックは生成ではなく**検証**です。
2. 今日のタスクは「AIに任せて速かったか/自分でやった方が速かったか」を判定する。
3. 次の1行を埋める。これが最大の収穫です。

```text
このタスク([タスク種別])は → [任せる / 自分でやる]
理由: [検証が容易だった / 文脈が暗黙すぎた / 速かった など]
```

::: info これが本質
AI駆動開発の上達は知識量ではなく**回したループの回数**に比例します。そして熟練者の優位は「**このタスクは任せる/これは自分でやる** の線引き」にあります。今日の1行が、その地図の最初の1点です。
:::

## 個人版 まとめチェックリスト

今日の30分を終える前に、ここを満たしているか確認してください。

- [ ] ツールを1つ選んで**起動**できた
- [ ] リポジトリ直下に**最小ルールファイル**を置いた([ルールファイル](/ready/rule-files))
- [ ] **安全タスク1件**で生成→検証ループを1周した
- [ ] 差分を**読み、テストを実行**した
- [ ] 機密を投入せず、**無検証マージをしなかった**
- [ ] 「任せる/自分でやる」の**線引き1行**を書いた

## チーム版: 最初の1週間プラン

個人で1周回せたら、チーム導入は**1週間** で土台を作ります。いきなり全員・全タスクに広げず、**小さく勝ってから広げる** のが鉄則です。本格的な合意形成は[チーム運用ポリシー](/ready/team-policy)で扱います。

::: warning チーム導入で最初にやってはいけないこと
- ルール不在のまま全員に配る(出力品質と作法がバラつき、レビューが破綻する)
- 機密の扱い・利用範囲を決めずに使い始める([チーム運用ポリシー](/ready/team-policy)を先に)
- 評価指標を「行数」「生成量」にする(検証されない量産はむしろ負債)
:::

| 日 | テーマ | 具体アクション |
| --- | --- | --- |
| 1日目 | 旗振りと範囲決め | 推進担当を1人決める。**使ってよいデータ範囲**と禁止事項を仮で合意([チーム運用ポリシー](/ready/team-policy)) |
| 2日目 | 共有ルールファイル | リポジトリに**チーム共通のルールファイル**をコミットし、全員が同じ前提で生成できる状態にする([ルールファイル](/ready/rule-files)) |
| 3日目 | 各自が個人版を1周 | 全員がこの30分プランを実施。安全タスクで1ループ |
| 4日目 | プロンプトの共有 | 効いたプロンプトを**チームの定型集**に蓄積する([プロンプトテンプレート集](/ready/prompt-templates)) |
| 5日目 | レビュー基準の合意 | 「AI生成PRのレビューで必ず見る点」を明文化。生成テストの扱いも決める |
| 週末 | 仕分けと次の一手 | 「速くなったタスク/遅くなったタスク」を共有。次週は**CI連携**へ([CI連携](/ready/ci-integration)) |

::: tip チームの成果を測る指標
「生成量」ではなく、**レビューを通過してマージされ、後で問題を出さなかった変更の割合** で見ます。これが健全な導入かどうかの最良のシグナルです。
:::

## やってはいけないこと(個人・チーム共通)

::: warning これだけは守る
- **機密を投入しない** — APIキー・認証情報・個人情報・未公開ソースを、保持ポリシー未確認のまま送らない。ダミー値・最小再現で尋ねる。
- **無検証でマージしない** — 「動いたっぽい」でmainに入れない。確率的生成物は必ずテスト・実行で外部検証する。
- **理解できないコードを採用しない** — 説明できないコードはあなたの負債であって、AIの手柄ではない。
- **大きく曖昧なタスクを丸投げしない** — 検証可能な小タスクに割る。
- **生成テストを盲信しない** — テストもAIが書いたなら、テスト自体を疑う。
:::

## つまずいたら

| 症状 | ありがちな原因 | 対処 |
| --- | --- | --- |
| 出力が的外れ | 文脈・制約が不足 | 関連コードと前提を明示。ルールファイルを充実([ルールファイル](/ready/rule-files)) |
| 嘘のAPIを使う | ハルシネーション | 公式ドキュメントで存在確認([ハルシネーション対策](/quality/hallucination)) |
| 大きすぎて検証不能 | タスクが粗い | 小さく分割し1ステップずつ |
| かえって遅い | タスク不適合 | そのタスクは自分で書く判断も正解 |
| どのツールか決められない | 比較沼 | [ツールの選び方](/tools/choosing)の基準で1つに絞る |
