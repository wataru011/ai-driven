# 生成物を人が評価する方法

AIは大量の生成物を高速に吐き出します。しかし**その生成物が要件を満たし、安全で、保守できるかを最終判断するのは人間**です。本ページは、AIが生成したコード・設計・テスト・ドキュメントを、人がどう評価するかに焦点を当てます。

::: tip このページの立ち位置
自動Eval・LLM-as-a-judgeなどの**仕組みによる評価**は[モデルの評価](/models/evaluation)で扱います。本ページはそれと対をなす**人間による評価**に焦点を当てます。両者は代替ではなく補完です。機械でふるい、人が最終判断する分担を前提にしてください。
:::

## なぜ「人が評価する」技術が要るのか

人間が書いたコードのレビューと、AI生成物の評価は似て非なるものです。AI生成物には固有の癖があります。

- **もっともらしさが過剰** — 文章もコードも自信たっぷりで、誤りが目立ちません。人は「それらしさ」に流されます。
- **意図の説明が後付け** — 生成物に付随する説明は、生成後に作られた**もっともらしい物語**であって、実際の動作の証明ではありません。
- **量が多い** — 1人のレビュアーが処理できる量を超えがちです。だからこそ**評価の型**が要ります。

::: warning 「動くこと」と「正しいこと」は別物
AI生成物は**ハッピーパスを通すこと**に最適化されがちです。デモが動くことと、仕様を満たし異常系でも壊れないことは別問題です。評価は「動いたか」ではなく「**正しいか・妥当か・安全か**」を問います。
:::

## 評価の観点フレーム

何を見るかを最初に固定します。生成物の種類が変わっても、この8観点は共通の骨格になります。

| 観点 | 問い | 主に見る対象 |
| --- | --- | --- |
| 正しさ(仕様充足) | 仕様・受け入れ基準を満たすか。境界・異常系で壊れないか | コード・テスト |
| 妥当性(要件・意図) | そもそも要件・ユーザー意図に合っているか。問題を解いているか | 設計・コード |
| 安全性(セキュリティ) | 入力検証・認可・秘密情報・脆弱な依存はないか | コード・設計・依存 |
| 可読性・保守性 | 半年後に他人が読めるか。変更しやすいか | コード・ドキュメント |
| 規約適合 | プロジェクトの命名・構成・既存パターンに沿うか | コード・ドキュメント |
| 過不足 | 頼んでいない実装(過剰)や、抜けた要件(不足)はないか | すべて |
| テスト妥当性 | テストが**意味のある検証**をしているか。実装に合わせただけでないか | テスト |
| ハルシネーション | 実在しないAPI・パッケージ・設定・引用を使っていないか | すべて |

::: info 観点に優先順位をつける
8観点を毎回フルで見る必要はありません。**不可逆・高リスクな箇所(認証・課金・データ削除・外部公開API)は全観点を厚く**、影響の小さい箇所は軽く。リスクに応じた濃淡が、評価を持続可能にします。濃淡のつけ方は[大量生成物のレビュー戦略](/quality/reviewing-at-scale)で詳述します。
:::

### 正しさと妥当性の違い

混同しやすいので分けて押さえます。

- **正しさ** = 与えられた仕様を満たすか。「仕様どおりに動くか」。
- **妥当性** = その仕様・実装が**そもそも正しい問題を解いているか**。「作るべきものを作ったか」。

AIは指示に忠実に動くため、**間違った指示を正確に実装する**ことがよくあります。正しく動いていても、要件を取り違えていれば妥当性ゼロです。評価では必ず**元の要件・意図に立ち返って**妥当性を問い直します。

## レビュー時の認知バイアス

人間の評価を最も狂わせるのは、技術ではなく**バイアス**です。AI生成物の評価では以下が特に効きます。

| バイアス | 症状 | 対策 |
| --- | --- | --- |
| automation bias(自動化バイアス) | 「AIが出したから正しいだろう」と検証を省く | 「AIが書いた」を信頼の理由にしない。出力を常に被告人扱いする |
| もっともらしさへの追従 | 流暢で自信ある説明に納得し、裏取りを省く | 説明ではなく**動作・テスト・出典**で確認する |
| アンカリング | 最初の生成案に引きずられ、より良い設計を検討しない | 案を独立に評価。必要なら代替案を別生成させる |
| レビュー疲れ | 量が多く、後半ほど雑になる | 量を制限する。重要箇所を先に見る。サンプリングする |
| 確証バイアス | 「たぶん合っている」前提で、合う証拠だけ探す | 反証を探す。壊す入力を先に考える |

::: warning automation biasは最大の敵
AIの出力は流暢で構造が整っているため、人は無意識に信頼します。これがautomation biasです。**「もっともらしさ」は正しさの証拠ではありません。** 評価者は意識的に懐疑を維持し、流暢さに減点も加点もしないでください。
:::

バイアスへの最も実践的な防御は、**評価の入口で立場を固定する**ことです。生成物を「正しいと推定し、誤りを探す」のではなく、「**誤っていると推定し、正しさの証拠を要求する**」立場をとります。後者は automation bias と確証バイアスの両方を同時に弱めます。

具体的には、レビュー開始時に「この差分はどの入力で壊れるか」を先に列挙してから本文を読みます。壊れ方の仮説を先に持つと、もっともらしい説明に流されにくくなります。

## 受け入れ基準(Definition of Done)

評価をブレさせないために、**事前に合格条件を言語化**します。生成してから「なんとなく良さそう」で判断しないこと。

- [ ] 仕様・受け入れ基準が**文書化**されており、生成前から存在する。
- [ ] 機械的チェック(フォーマット・リント・型・テスト・静的解析・シークレットスキャン)がすべて緑。
- [ ] 8観点のうち、その変更のリスクに応じた観点を確認済み。
- [ ] 呼んでいるAPI・依存・設定が**実在し現行版で正しい**ことを確認済み。
- [ ] 過剰実装・要件の抜けがないことを確認済み。
- [ ] **マージする人間**が内容を理解し、品質に責任を負える状態。

::: tip DoDは生成の前に決める
受け入れ基準を生成後に作ると、生成物に都合よく基準を合わせてしまいます(確証バイアス)。基準は**生成より先**に固定し、評価はそれに照らすだけにします。これは[モデルの評価](/models/evaluation)で言う「評価基準の先決め」と同じ思想です。
:::

受け入れ基準は、AIへの指示文の一部としても機能します。生成前にDoDを言語化しておけば、それを**そのままプロンプトに渡す**ことで、生成物が最初から基準に寄ります。評価で弾く前に、生成側で満たさせるほうが安価です。逆に言えば、DoDが書けないタスクは**そもそも評価できないタスク**であり、生成させる前に要件を詰め直すべきサインです。

## 種類別チェックリスト

生成物の種類ごとに、固有の落とし穴があります。

### 生成コードを評価する

- [ ] 仕様・受け入れ基準を満たすか(正常系だけでなく異常系)。
- [ ] エッジケース — 空・null・0・負数・巨大入力・並行アクセスを処理しているか。
- [ ] エラー処理 — 例外を握りつぶしていないか。失敗が観測可能か。
- [ ] 呼び出しているAPI・ライブラリ・設定は実在し、現行版で正しいか([ハルシネーション対策](/quality/hallucination))。
- [ ] 規約適合 — 命名・構成・既存パターンに沿うか。車輪の再発明をしていないか。
- [ ] セキュリティ — 入力検証・認可・秘密情報のハードコードがないか([セキュリティリスク](/quality/security))。
- [ ] 過不足 — 頼んでいない抽象化・設定を盛り込んでいないか。要件の抜けはないか。

### 生成された設計・アーキテクチャを評価する

- [ ] 解こうとしている問題と要件に**妥当**か(正しい問題を解いているか)。
- [ ] 既存システムの制約・前提と整合するか。既存資産を無視して再発明していないか。
- [ ] 過剰設計 — 現状の規模に不相応な抽象化・分散・汎用化をしていないか。
- [ ] 障害時・スケール時の振る舞いが説明されているか(机上で都合よくないか)。
- [ ] トレードオフが明示され、却下した代替案の理由があるか。

### 生成テストを評価する

- [ ] テストが**実装と独立した仕様**から書かれているか。実装に合わせただけでないか。
- [ ] 失敗すべき入力でちゃんと失敗するか(常にtrueを返す空テストでないか)。
- [ ] 異常系・境界値・エラーパスをカバーしているか(正常系だけでないか)。
- [ ] アサーションが本質を検証しているか。表面的な値だけ見ていないか。
- [ ] テストを一度**わざと壊して**、赤くなることを確認したか。

::: warning テストもAIが書くと甘くなる
AIに「テストも書いて」と頼むと、**実装に合わせた当たり前のテスト**を書きがちで、本来検出すべきバグを見逃します。テストを評価するときは「このテストが赤くなる現実の不具合は何か」を自問してください。赤くならないテストは検証していません。
:::

### 生成ドキュメントを評価する

- [ ] 記述がコードの**現在の実際の動作**と一致するか(理想や過去でなく)。
- [ ] 例・コマンド・設定値が実在し、実際に動くか(コピペで再現したか)。
- [ ] 引用・参照したAPI・仕様・リンクが実在するか([ハルシネーション対策](/quality/hallucination))。
- [ ] 読者の前提に合っているか。冗長・空疎な定型文で水増しされていないか。

## 評価を速く正確にするコツ

観点とチェックリストを、実務速度で回すための具体策です。

| コツ | 内容 |
| --- | --- |
| 差分で見る | 全体ではなく**変更差分**に集中する。差分が大きすぎる生成物は、評価不能として突き返す |
| テストで裏取り | 目視で「正しそう」と判断せず、**壊す入力を含むテスト**を走らせて事実で確認する |
| 出典を確認する | 使われているAPI・パッケージ・設定を**一次情報(公式ドキュメント)**で照合する。AIの説明を出典扱いしない |
| AIに自己説明させて検証 | 「なぜこの実装か」「どの入力で壊れるか」をAIに説明させ、その**主張を人間が独立検証**する。説明をそのまま信じるのではなく、検証の足がかりにする |
| 小さく分割させる | レビュー可能な大きさに生成を分けさせる(下記) |

::: tip AIの自己説明は「証拠」ではなく「仮説」
AIに自己レビューや説明をさせると弱点の手がかりは得られますが、その説明自体もまた生成物であり、もっともらしいだけのことがあります。自己説明は**検証すべき仮説**として扱い、最終判断は人間がテストと出典で裏取りしてください。
:::

### 小さく分割させてレビュー可能性を上げる

評価精度は**差分の大きさに反比例**します。巨大な生成物は、人が正しく評価できません(レビュー疲れと automation bias が効く)。

- 巨大PRを生成させない。エージェントにも**1つの関心事=1つの小さな差分**で出力させる。
- 1差分で評価が完結する粒度に保つ(他ファイルを大量に追わないと判断できない差分は分割を要求)。
- 機械的チェックを先に通し、人間は**機械が見られないこと**に集中する。

分割の粒度・PRレビューの運用基準は[PRレビューの基準](/ready/pr-review-standards)に、量が増えたときの戦略は[大量生成物のレビュー戦略](/quality/reviewing-at-scale)に続きます。

::: info 評価できない生成物は不合格
「大きすぎて読めない」「説明が追えない」差分は、内容の良し悪し以前に**評価不能=不合格**です。評価可能性は生成側の責任。読める大きさで出させることを運用ルールにします。
:::

## まとめ

- 人間の評価は**正しさ・妥当性・安全性・可読性・規約適合・過不足・テスト妥当性・ハルシネーション**の8観点で行う。
- 「動くこと」と「正しいこと・妥当なこと」は別。要件・意図に立ち返って妥当性を問い直す。
- 最大の敵は技術ではなく**automation bias**。もっともらしさを信頼の理由にしない。
- 受け入れ基準(DoD)を**生成より先**に固定し、評価はそれに照らすだけにする。
- 差分で見る・テストで裏取り・出典確認・自己説明の独立検証・小さく分割、で速く正確に評価する。
