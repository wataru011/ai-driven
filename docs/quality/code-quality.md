# コード品質の担保

AI生成コードは速くて便利ですが、**人間が書いたコードとは違うパターンの欠陥**を持ち込みます。本ページでは、その品質特性を理解し、機械的なガードレールとレビュー設計でどう品質を担保するかを扱います。

::: tip 大前提
LLMは**もっともらしいテキスト**を生成するのであって、**正しいコード**を保証しません。生成物は常にレビューとテストの対象です。なぜそうなるかは[LLMの仕組み](/concepts/how-llms-work)を参照してください。
:::

## AI生成コードの品質特性

AI生成コードには、人間のコードとは異なる**典型的な失敗モード**があります。これを知っているかどうかでレビューの精度が変わります。

| 特性 | 内容 | なぜ起きるか |
| --- | --- | --- |
| もっともらしいが微妙にズレる | 一見正しく動きそうだが、境界条件・符号・off-by-oneを誤る | 統計的にそれらしい続きを生成するため |
| 過剰実装(over-engineering) | 頼んでいない抽象化・設定・例外処理を盛り込む | 「良いコード」のパターンを過剰に再現 |
| 古いAPI/非推奨手法 | 学習時点で一般的だった古い書き方を使う | 学習データの分布が最新でない |
| 規約不一致 | プロジェクト固有の命名・構成・スタイルに従わない | プロジェクト文脈を知らない |
| 存在しないAPI/パッケージ | それらしいが実在しない関数・ライブラリを呼ぶ | ハルシネーション(別ページで詳述) |
| 沈黙のエラー処理 | 例外を握りつぶす、ログだけ出して続行する | 「動くように見せる」方向に最適化 |

::: warning 「動いた」は「正しい」ではない
AI生成コードは**ハッピーパスは通るがエッジで壊れる**ことが多いです。レビューでは正常系より、**異常系・境界・並行性・リソース解放**を重点的に見ます。
:::

具体例として、もっともらしいズレを挙げます。

```python
# AIが生成しがちな「もっともらしいが惜しい」コード
def get_page(items, page, per_page):
    start = page * per_page          # page=1 が2ページ目になる(1始まり想定なら off-by-one)
    end = start + per_page
    return items[start:end]          # items が None のケース、per_page=0 のケース未考慮
```

## ガードレール: 機械で守る

レビューは貴重なので、**機械が判定できることは機械に任せ**、人間のレビューを本質的な判断に集中させます。ガードレールは多層で構成します。

```text
生成 → フォーマッタ → リンタ → 型チェック → テスト → セキュリティ走査 → CIゲート → 人間レビュー
       │            │       │           │       │                │            │
       自動整形      規約      型安全性     振る舞い  脆弱性・秘密情報   合否判定      設計・意図
```

| 層 | ツール例(言語非依存の概念) | 捕まえる欠陥 |
| --- | --- | --- |
| フォーマッタ | 自動整形ツール | スタイルのブレ |
| リンタ | 静的解析リンタ | アンチパターン、未使用、規約違反 |
| 型チェック | 静的型システム | 型の不整合、null安全性 |
| テスト | 単体・統合・プロパティテスト | 振る舞いの誤り、回帰 |
| セキュリティ | SAST・シークレットスキャン・依存監査 | 脆弱性、秘密情報の混入([詳細](/quality/security)) |
| CIゲート | CI上の必須チェック | 上記すべての合格を**マージ条件**にする |

::: tip 型とテストはAI時代に価値が上がる
**型システムとテストは、AIに対する最良のガードレール**です。エージェントは「型エラーゼロ」「テスト緑」という機械判定可能な目標があると自己修正できます([AIエージェントとは](/agents/what-is-agent)参照)。AI活用度を上げるほど、型とテストへの投資対効果は高まります。
:::

### CIゲートの設計

AI生成コードを前提にするなら、**人間の善意に頼らず、CIで強制**します。

- [ ] フォーマット・リント・型チェックを**必須**にし、失敗したらマージ不可。
- [ ] テストカバレッジの**低下を検知**する(新規コードに対するカバレッジを要求)。
- [ ] SAST・シークレットスキャン・依存脆弱性スキャンをパイプラインに組み込む。
- [ ] 生成コードであっても**例外を設けない**。「AIが書いたから」はスキップの理由にならない。

## レビュー観点

機械を通った後、人間は**機械が見られないこと**に集中します。AI生成コード特有のレビューチェックリストです。

- [ ] **意図との一致** — 要件・仕様どおりか。頼んでいない機能を勝手に足していないか(過剰実装)。
- [ ] **存在性の確認** — 呼んでいるAPI・ライブラリ・設定は**実在し、現行版で正しい**か([ハルシネーション対策](/quality/hallucination))。
- [ ] **エッジケース** — 空・null・0・負数・巨大入力・並行アクセスを処理しているか。
- [ ] **エラー処理** — 例外を握りつぶしていないか。失敗が**観測可能**か。
- [ ] **規約適合** — プロジェクトの命名・構成・既存パターンに沿っているか。車輪の再発明をしていないか。
- [ ] **セキュリティ** — 入力検証、認可、秘密情報のハードコードがないか。
- [ ] **テストの質** — テストが**本当に意味のある検証**をしているか(常にtrueを返すだけの空テストでないか)。
- [ ] **可読性・保守性** — 半年後に他人が読めるか。不要な抽象化で複雑化していないか。

::: warning テスト自体もAIが書くと甘くなる
AIに「テストも書いて」と頼むと、**実装に合わせた当たり前のテスト**を書きがちで、本来検出すべきバグを見逃します。テストは**実装と独立した仕様**から書く意識を持ち、テストの妥当性自体をレビューしてください。
:::

## 「レビューがボトルネック」問題

AI駆動開発の最大の構造的課題がこれです。**生成は桁違いに速くなる一方、レビューは速くならない。** むしろ生成量が増えるほど、レビュー負荷が線形以上に増え、ここがスループットの上限になります。

```text
従来:    [遅い実装] → [レビュー]          レビューは全体の一部
AI駆動:  [速い生成] → [  レビュー  ]      レビューが支配的なボトルネック
                       ▲ ここが詰まる
```

対処は「レビュー量を減らす」と「レビューの質・効率を上げる」の両面で行います。

| 戦略 | 具体策 |
| --- | --- |
| 検証を機械化し**レビュー対象を絞る** | 機械で判定できるものはCIで弾き、人間は設計・意図だけ見る |
| **小さく出させる** | 巨大PRは生成させない。エージェントにも小さい単位で出力させ、差分を読める大きさに保つ |
| **検証可能性を設計に織り込む** | テスト容易な構造にし、生成物の正しさを機械判定しやすくする |
| **生成と検証を分離** | 実装エージェントとレビューエージェントを分ける([マルチエージェント](/agents/multi-agent))。ただしAIレビューは人間レビューの代替でなく**前段フィルタ**として使う |
| **責任を明確化** | 「AIが書いた」ではなく、**マージした人間が品質に責任を負う**文化を維持する |

::: tip ボトルネックは移動する
AIは「書く」ボトルネックを解消しますが、それは**ボトルネックを"レビュー・検証・統合"へ移動させた**にすぎません。AI駆動開発の習熟とは、この新しいボトルネックを管理する技術の習得でもあります。
:::

## まとめ

- AI生成コードは**もっともらしいズレ・過剰実装・古いAPI・規約不一致**という固有の失敗モードを持つ。
- **フォーマッタ→リンタ→型→テスト→セキュリティ→CIゲート**の多層ガードレールで、機械が守れることは機械に任せる。
- 人間のレビューは**意図・存在性・エッジ・セキュリティ・テストの質**に集中する。
- 真のボトルネックは**レビュー**。小さく出させ、検証を機械化し、責任を人間に残すことで管理する。
