# コンテキスト設計

プロンプトの基礎とテクニックを押さえても、**渡す情報が間違っていれば**出力は的を外します。コンテキスト設計とは、限られたコンテキストウィンドウという予算の中で、「判断に必要な情報だけを、再現可能な形で渡す」設計作業です。

LLMにとってコンテキストは作業机です。机が小さすぎれば必要な資料が乗らず、散らかりすぎれば肝心の資料が埋もれます。前提として[コンテキストウィンドウ](/concepts/context-window)の仕組みを理解しておくと、ここでの判断の意味がよく分かります。

::: tip 中心となる問い
「このタスクの判断に**本当に必要な情報は何か**」「逆に、**渡すと害になる情報は何か**」。この2つを常に問うのがコンテキスト設計です。
:::

## なぜ「全部渡す」が間違いなのか

コンテキストウィンドウが広がった今、「関連ファイルを全部貼ればいい」と考えがちですが、これは3つの理由で逆効果です。

1. **ノイズによる希釈** — 無関係な情報が多いと、モデルの注意が分散し、重要な制約を見落とす(いわゆる "lost in the middle")。
2. **トークンコストと遅延** — 入力が増えれば料金も応答時間も増える。
3. **誤った手がかり** — 古いコードや無効化された設定を渡すと、それを正しい前提だと誤認する。

::: warning コンテキストは「多ければ良い」ではない
広いコンテキストウィンドウは「全部入れてよい」許可証ではなく、「**選別の手間を省いてよい**」許可証でもありません。選別はむしろ重要度を増します。
:::

## 何を渡し、何を渡さないか

判断の基本は「このタスクの出力に影響するか」です。

| 渡すべき | 渡さない(あるいは要約) |
| --- | --- |
| 変更対象のコードそのもの | 無関係なモジュール |
| 直接呼び出す/される関数の定義 | 推移的な依存の奥深く |
| 守るべき規約・スタイルの実例 | 全社規約の全文(該当箇所だけ抜く) |
| データ構造・型定義・スキーマ | 自動生成された冗長なコード |
| 再現条件・エラーログ | 関係ないログ行 |
| 関連するテストの一例 | テスト全件 |

### 関連コードの選別

「変更対象」とその**1ホップ先**(直接の呼び出し関係、依存する型)までを基本範囲とします。それ以上は、必要性が確認できたときに追加します。

```text
## 変更対象
（処理を変えるファイルAを貼る）

## 関連(参照のみ、変更不可)
（A が呼ぶヘルパー関数 B のシグネチャと型定義を貼る）

## 規約の見本
（プロジェクトで典型的なファイル C を1つ貼り「このスタイルに合わせて」）
```

### 要約という選択肢

長大なコードや仕様は、**全文ではなく要約**を渡すほうが効くことがあります。「このサービスは何を入力に何を返すか」の数行で足りるなら、実装全文は不要です。長い設計ドキュメントは、まずモデルに要約させ、その要約を以後のコンテキストとして使う二段構えも有効です。

## プロジェクト規約ファイル(CLAUDE.md など)

毎回同じ規約を手で貼るのは非効率で、貼り忘れも起きます。ツールによっては、**リポジトリ直下の規約ファイルを自動的にコンテキストへ読み込む**仕組みがあります(例: Claude Code の `CLAUDE.md`、各種エージェントの "rules" ファイル)。

ここに「常に効いてほしい前提」を集約します。

```markdown
# プロジェクト規約(CLAUDE.md の例)

## 技術スタック
- TypeScript 5.x / Node 20 / React 18(関数コンポーネント + hooks)
- テスト: Vitest。テストは対象と同階層に *.test.ts

## コーディング規約
- 例外は投げず Result 型で返す(src/lib/result.ts)
- 公開関数には必ず JSDoc を付ける
- any 禁止。型が不明なら unknown + 絞り込み

## やってはいけないこと
- 既存の公開 API シグネチャを断りなく変更しない
- node_modules を直接編集しない
- 新規依存の追加は事前に提案し、承認を得てから
```

::: tip 規約ファイルの設計指針
- **短く、効くものだけ** — 守られない理想論を並べると全体の信頼度が下がる。
- **"やってはいけない"を明記** — 禁止事項は暴走防止に直結する。
- **実例を1つ添える** — 「Result 型で返す」だけでなく、その型の場所を示す。
- **更新を運用に組み込む** — 規約が変わったらファイルも直す。陳腐化した規約は害。
:::

## トークン予算の配分

コンテキストウィンドウは有限の予算です。タスクごとに、何にトークンを割くかを意識します。ざっくりした配分のイメージは次の通りです。

| 用途 | 目安 | 補足 |
| --- | --- | --- |
| 指示(役割・目的・制約) | 小 | 簡潔に。冗長な指示は逆効果 |
| 規約・スタイルの実例 | 中 | 該当箇所のみ抜粋 |
| 変更対象コード | 大 | ここが主役 |
| 関連コード(参照) | 中 | 1ホップ先まで |
| 出力のための余白 | 確保必須 | 長い出力を求めるなら入力を削る |

::: warning 出力の余白を食いつぶさない
入力でウィンドウを埋め尽くすと、肝心の生成結果が途中で打ち切られます。長いコード生成を求めるなら、**入力側を要約・削減して出力の余地を残す**こと。原理は[コンテキストウィンドウ](/concepts/context-window)で詳述します。
:::

## 配置の順序も効く

同じ情報でも、**置く位置**で効き方が変わります。経験則として:

- **最重要の制約は冒頭か末尾**に置く(中央は埋もれやすい)。
- **長い参考資料は先**に、**具体的な指示は後**に置くと、指示が直近の文脈として強く効く。
- リファレンス資料には明確な見出し(`--- API ドキュメント ---` など)を付け、境界を示す。

## 再現性のあるコンテキスト運用

その場限りの手貼りは、品質がブレ、他人と共有できません。**コンテキストを資産として管理**します。

- **規約ファイルをバージョン管理** — `CLAUDE.md` やルールファイルをリポジトリにコミットし、チームで共有・レビューする。
- **プロンプトテンプレート化** — 頻出タスク(レビュー依頼、テスト生成)はテンプレ化し、可変部分だけ差し替える。
- **コンテキストの差分管理** — 「うまくいったとき/いかなかったとき」で渡した情報の差を記録し、改善する。
- **生成のたびに同じ前提** — 同じ規約・同じ形式で渡すことで、出力の再現性が上がる。

::: details なぜ再現性が重要なのか
LLMは確率的なので出力は完全には固定できませんが、**入力(コンテキスト)を固定すれば品質の分散は小さくできます**。属人的な「プロンプトの上手い人頼み」を脱し、チームで安定した品質を出すには、コンテキストの標準化が不可欠です。
:::

## アンチパターン

::: warning コンテキスト設計の失敗例
- **全リポジトリ貼り付け**: ノイズで希釈され、コストも増える。
- **古い情報の混入**: 廃止済みコードや旧仕様を渡し、それを前提にされる。
- **規約の二重管理**: 規約ファイルとプロンプトに矛盾する指示が混在する。
- **要約の丸投げ**: 重要な制約まで要約で削ぎ落とし、判断材料を失う。
- **出力余白の枯渇**: 入力で埋め尽くし、生成が途中で切れる。
:::

## まとめ

- コンテキスト設計とは「必要な情報だけを再現可能に渡す」設計作業。
- **多ければ良いではない**。ノイズは出力を希釈する。
- 変更対象 + 1ホップ先 + 規約の実例、を基本範囲に。
- 規約ファイルで「常に効く前提」を標準化し、バージョン管理する。
- トークン予算を意識し、**出力の余白を残す**。
