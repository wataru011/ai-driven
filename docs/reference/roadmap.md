# 学習ロードマップ

AI駆動開発は、ツールを触れば身につくものではありません。**「仕組みの理解」→「個人の実践」→「チームへの展開」→「高度な自律化」** という順で積み上げると、つまずきにくく定着します。このページは本ハンドブックを使った段階的な学習パスです。

::: tip 使い方
全レベルを一気に読む必要はありません。今の自分のレベルから始め、各ステップのチェックポイントを満たしてから次へ進んでください。チェックボックスは進捗管理に使えます。
:::

## 全体像

```
Level 1 入門          Level 2 実践          Level 3 チーム展開      Level 4 高度
─────────────       ─────────────        ──────────────        ─────────────
仕組みを知る    →    個人で使い倒す   →     チームで標準化     →   自律化・仕様駆動
(L1〜L2)             (L2〜L3)              (L3)                  (L4〜L5)

土台:               道具と型:             統制と展開:            自律と検証:
LLMの仕組み          ツール選定            導入戦略               エージェント
プロンプト基礎       ワークフロー          セキュリティ           仕様駆動開発
```

成熟度レベル(L0〜L5)の定義は[AI駆動開発とは](/guide/what-is-aidd)を参照してください。本ロードマップの4レベルは、その成熟度を学習者の視点で並べ直したものです。

## Level 1 — 入門:仕組みを知る

**目的**: AIが「なぜそう振る舞うか」を理解し、過信も拒絶もしない地に足のついた感覚を作る。

| ステップ | 読むページ | 得られるもの |
| --- | --- | --- |
| 1 | [AI駆動開発とは](/guide/what-is-aidd) | 全体像・成熟度モデル・誤解の整理 |
| 2 | [はじめての一歩](/guide/getting-started) | まず手を動かす最初の成功体験 |
| 3 | [LLMの仕組み](/concepts/how-llms-work) | 確率的生成という土台のメンタルモデル |
| 4 | [プロンプトの基礎](/prompting/fundamentals) | 意図・文脈・制約を伝える基本作法 |

**習得目安**: 1〜2週間(毎日少しずつ触れる前提)。

::: details Level 1 チェックポイント
- [ ] AIが「正しいコード」ではなく「もっともらしいテキスト」を出すと説明できる
- [ ] トークン・コンテキストウィンドウが何かを言える([用語集](/reference/glossary)で確認)
- [ ] 簡単なタスクをAIに頼み、出力をレビューして取り込めた
- [ ] 曖昧な指示と具体的な指示で出力が変わることを体験した
:::

## Level 2 — 実践:個人で使い倒す

**目的**: 日々の開発でAIを「委任できる協働者」として使い、生産性を実感する。

| ステップ | 読むページ | 得られるもの |
| --- | --- | --- |
| 1 | [ツールの全体像](/tools/landscape) | 補完型/対話型/エージェント型の地図 |
| 2 | [モデルの選び方](/models/choosing-models) | タスク別のモデル使い分け |
| 3 | [プロンプトカタログ](/patterns/prompt-catalog) | 再利用できるプロンプトの型 |
| 4 | [開発ワークフロー](/workflow/overview) | 開発フローへの組み込み方 |
| 5 | [機能開発の実践](/practice/feature-development) | 1機能をAIと作り切る流れ |

**習得目安**: 3〜6週間。

::: details Level 2 チェックポイント
- [ ] タスクの難度に応じてモデルを使い分けている
- [ ] よく使うプロンプトを自分のテンプレートとして持っている([チートシート](/reference/cheatsheet)を起点に)
- [ ] 大きなタスクを小さく割ってAIに渡せる
- [ ] 生成物に対し、テストとレビューを習慣として通している
- [ ] AIに頼るべきタスク/自分で書くべきタスクを判断できる
:::

::: warning 「速いが雑」に陥らない
Level 2 の罠は、生成量だけ増えてレビューが追いつかなくなることです。PRを小さく保ち、検証を省略しないこと。失敗パターンと対策は[FAQ](/reference/faq)に整理しています。
:::

## Level 3 — チーム展開:標準化する

**目的**: 個人の成功をチームの再現性に変える。ガイドライン・レビュー基準・データ取り扱いを整える。

| ステップ | 読むページ | 得られるもの |
| --- | --- | --- |
| 1 | [導入戦略](/adoption/strategy) | 小さく始めて広げる進め方 |
| 2 | [セキュリティ](/quality/security) | データ取り扱いとインジェクション対策 |
| 3 | [チートシート](/reference/cheatsheet) | レビュー観点・禁止操作の共有早見表 |
| 4 | [生産性の測定](/adoption/metrics) | 効果測定の正しいやり方と落とし穴 |

**習得目安**: 1〜3か月(組織規模に依存)。

::: details Level 3 チェックポイント
- [ ] チームのAI利用ガイドライン(入れてよいデータ/レビュー必須範囲)がある
- [ ] 依存追加時の実在性確認手順がチームに浸透している
- [ ] レビュー観点が共有され、人間のレビューを設計・セキュリティに集中できている
- [ ] 効果を単一指標で測る誘惑に陥っていない([生産性の測定](/adoption/metrics))
- [ ] 新メンバーが学習ループに乗れるオンボーディングがある
:::

## Level 4 — 高度:自律化と仕様駆動

**目的**: エージェントに複数ステップを任せ、検証可能な仕様を起点に開発を駆動する。

| ステップ | 読むページ | 得られるもの |
| --- | --- | --- |
| 1 | [エージェントとは](/agents/what-is-agent) | 自律実行の仕組みと委任の線引き |
| 2 | [仕様駆動開発](/advanced/spec-driven) | 仕様を基準に実装・受け入れする進め方 |
| 3 | [RAG](/concepts/rag) | コードベース・ドキュメントへのグラウンディング |

**習得目安**: 継続的(到達点ではなく運用フェーズ)。

::: details Level 4 チェックポイント
- [ ] タスクのリスク・可逆性で委任範囲を判断できる
- [ ] エージェントに最小権限・禁止操作・承認ゲートを設定している
- [ ] 仕様を先に固め、それを受け入れ基準として使えている
- [ ] 検証(テスト・eval)を自動化し、自律実行の安全網にしている
- [ ] 不可逆・高リスク操作には必ず人間の承認を挟んでいる
:::

## 学び続けるために

技術は速く変わりますが、**原理は陳腐化しにくい**です。製品名やフラグではなく、仕組み・型・検証手順に投資してください。次に何を読むか迷ったら[リソース集](/reference/resources)から一次情報をたどれます。

| 困ったとき | 行き先 |
| --- | --- |
| 用語が分からない | [用語集](/reference/glossary) |
| 素朴な疑問に答えがほしい | [FAQ](/reference/faq) |
| すぐ使える型がほしい | [チートシート](/reference/cheatsheet) |
| 一次情報を当たりたい | [リソース集](/reference/resources) |
