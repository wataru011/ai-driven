# ドキュメント生成

ドキュメントは「書かれないか、書かれても腐る」運命にあります。AIはこの両方に効きます。**書く労力を下げ**、**コードとの乖離を検出する**。ただし、ドキュメント生成にはコード生成とは別種の落とし穴があります — **もっともらしいが間違ったドキュメント**は、コードのバグより発見が遅れ、長く人を惑わせます。

このページでは、README・API doc・ADR・コードコメント・チュートリアルそれぞれについて、**コードを根拠に生成し、誤りを検出する**実践を扱います。

::: warning ドキュメントのハルシネーションは特に危険
コードは実行すれば間違いが分かりますが、ドキュメントの誤りは**読者が信じてしまう**まで気づかれません。AIが生成したドキュメントは「コードという一次情報に照らして検証する」のが大原則です([ハルシネーション対策](/quality/hallucination))。
:::

## 大原則: コードを根拠に、推測を排す

ドキュメント生成プロンプトの良し悪しは、**「コードに書いてあることだけを書かせ、推測を禁じる」**かどうかで決まります。

```text
# ❌ 推測を許すプロンプト
このプロジェクトのREADMEを書いて
```

```text
# ✅ 根拠を縛るプロンプト
以下のコードと設定ファイルだけを根拠にREADMEを書いてください。
コードから読み取れない情報(作者の意図、未実装の機能、外部の前提)は
推測で書かず、「(要確認)」のプレースホルダにしてください。
事実とコードの行が対応するように。
```

この「**読み取れないものは推測しない**」という制約が、ドキュメント生成の品質を決定づけます。

## README の生成

READMEは「初見の人が動かせる」ことがゴールです。コード・`package.json`・既存の設定を根拠に生成させます。

```text
あなたはOSSのドキュメントに長けたエンジニアです。
以下を根拠に README.md を生成してください。
- package.json(scripts, dependencies)
- src/index.ts(エントリポイント)
- .env.example(必要な環境変数)

## 構成
1. プロジェクトの一行説明(package.json の description を起点に)
2. 前提環境(Node バージョンは engines から)
3. セットアップ手順(scripts の実際のコマンドを使う。創作しない)
4. 環境変数の表(.env.example のキーと用途)
5. 主要なnpm scripts の一覧と説明

## 厳守
- コマンドは package.json の scripts に実在するものだけ
- 環境変数の「用途」がコードから判断できない場合は (要確認) と書く
（各ファイルを貼る)
```

応答イメージ(抜粋):

````markdown
## セットアップ

```bash
npm install
cp .env.example .env   # 値を埋める
npm run dev            # 開発サーバ起動(scripts.dev: tsx watch src/index.ts)
```

## 環境変数

| 変数 | 用途 | 必須 |
| --- | --- | --- |
| `DATABASE_URL` | PostgreSQL 接続文字列 | 必須 |
| `JWT_SECRET` | JWT 署名鍵 | 必須 |
| `SENDGRID_KEY` | メール送信(要確認: 未使用の可能性) | (要確認) |
````

::: tip 「(要確認)」が品質のシグナル
AIが堂々と埋めた箇所より、**「(要確認)」と正直に保留した箇所**のほうが信頼できます。プレースホルダが残ることを「失敗」と捉えず、「人間が確認すべき点を可視化してくれた」と捉えてください。
:::

## API ドキュメントの逆生成

既存のルート定義から、API仕様を逆生成します。コードが一次情報なので、推測が入りにくく相性が良い用途です。

```text
以下の Express ルート定義から、API リファレンスを生成してください。
各エンドポイントについて:
- メソッドとパス
- 認証の要否(ミドルウェアから判断)
- リクエスト(パスパラメータ/クエリ/ボディ)とその型
- レスポンス(成功/エラーのステータスと形)
コードから読み取れるものだけ。バリデーションスキーマ(zod)があれば
それを正とすること。例レスポンスは型から構成し、値は代表的なダミーで。
（ルートとzodスキーマを貼る)
```

応答イメージ(抜粋):

````markdown
### POST /products/:id/reviews

レビューを投稿する(認証必須)。

| 区分 | 内容 |
| --- | --- |
| 認証 | 必須(Bearer JWT) |
| パスパラメータ | `id`: string(商品ID) |
| ボディ | `{ rating: number(1..5), comment?: string(<=1000) }`(zod: reviewSchema) |

成功: `201 Created`
```json
{ "id": "ckxyz...", "rating": 5, "comment": "良い商品", "createdAt": "2026-06-30T12:00:00Z" }
```
エラー: `401`(未認証) / `403`(未購入) / `409`(重複) / `422`(検証失敗)
````

::: warning 生成APIドキュメントの検証
逆生成でも誤りは混入します。特に「エラーレスポンスの網羅」はコードを正確に追わないと漏れます。可能なら **OpenAPI スキーマを生成 → 実際のレスポンスとの突き合わせテスト**(契約テスト)まで繋げると、ドキュメントが「検証された仕様」になります。手書きのコメントより、**zodスキーマやルート定義から機械的に生成**するパイプラインのほうが鮮度を保ちやすい点も重要です。
:::

## ADR(設計判断の記録)

ADR(Architecture Decision Record)は「なぜそう決めたか」を残す文書です。**判断の背景は人間の頭の中にしかない**ため、AIには「叩き台の構造化」をさせ、中身は人間が埋めます。

```text
設計判断を ADR 形式で文書化したい。以下のメモを整形してください。
創作はせず、メモにない理由は付け足さないこと。

## メモ
- 商品レビューの平均評価を「都度集計」にした
- 非正規化(Productに平均を保持)も検討したが見送った
- 理由: 現状レビュー数は商品あたり最大数千、aggregateで十分速い
- 整合性管理のコストを避けたい
- 将来 p95 が劣化したら非正規化を再検討

## ADR 構成
- タイトル / ステータス / コンテキスト / 決定 / 結果(トレードオフ) /
  再検討の条件
```

応答イメージ:

```markdown
# ADR-007: レビュー平均評価は都度集計とする

- ステータス: Accepted (2026-06-30)

## コンテキスト
商品ごとのレビュー平均評価を表示する。算出方法として
(A) 参照時に都度 aggregate と (B) Product に平均を非正規化保持 を検討した。

## 決定
都度集計(A)を採用する。

## 結果(トレードオフ)
- 利点: 実装が単純、書き込み時の整合性管理が不要
- 欠点: 参照のたびに集計コスト。商品あたりレビューが膨大だと劣化しうる
- 現状の規模(商品あたり最大数千件)では aggregate で十分高速と判断

## 再検討の条件
- レビュー一覧/平均取得の p95 レイテンシが許容を超えたら (B) を再検討
```

::: tip ADRは「決めた瞬間」に書く
ADRの価値は鮮度です。判断の文脈が頭にあるうちに、AIに**口頭メモを構造化**させるのが最も効率的。後から思い出して書くと、肝心の「なぜ却下したか」が抜けます。[機能開発のエンドツーエンド](/practice/feature-development)で下した設計判断は、その場でADRにすると将来の自分を助けます。
:::

## コードコメント

コメント生成は便利ですが、**「何をしているか(what)」を冗長に説明するだけのコメント**は害悪です。良いコメントは**「なぜ(why)」**を説明します。

```text
以下の関数に docstring/コメントを追加してください。
## 方針
- 「何をしているか」のコードをなぞるだけのコメントは書かない
- 「なぜそうしているか」「非自明な前提」「注意すべき副作用」を中心に
- 公開関数には JSDoc(引数/戻り値/throw する例外)を付ける
- コードから読み取れない意図は (要確認) とし、推測で断定しない
```

```ts
// ❌ what をなぞるだけ(ノイズ)
// i を 1 増やす
i++

// ✅ why を説明(価値がある)
// SendGrid は lockDuration(30s)を超えると stalled 判定で再投入するため、
// 送信前に claim してから送ることで二重送信を防ぐ。詳細は ADR-009。
```

::: warning コメントとコードの乖離を生まない
AIに大量のコメントを生成させると、**コードが変わってもコメントが残り、嘘になる**未来を増やします。コメントは「自明でない部分に絞る」。乖離を防ぐ最善は、**コメントを減らしてコードを自己説明的にする**ことです。生成したコメントは「これは本当に必要か?」とふるいにかけてください。
:::

## チュートリアルの生成と検証

チュートリアルは「手順通りにやれば動く」ことが命。AIが生成したチュートリアルの最大のリスクは、**実際には動かない手順**(存在しないオプション、順序の誤り)です。

```text
このライブラリの「最小のはじめ方」チュートリアルを書いてください。
## 厳守
- コードは src/index.ts の実際のエクスポートと型に基づくこと
- 各ステップの後に「ここで何が起きるはずか(期待される出力)」を書く
- 存在しない関数・オプションを使わない。不明なら (要確認) と明記
（公開API(index.ts の export)と型定義を貼る)
```

::: tip チュートリアルは「生成したら必ず実行する」
チュートリアルの検証は1つだけ: **書かれた手順を実際にその通り実行する**。AIに「このチュートリアル通りに進めて、各ステップで詰まる点・誤りを指摘して」と**追検証**させるのも有効ですが、最終的には人間(または CI 上のスクリプト)が実行して確認します。「動くコード例」を **doctest や CI のサンプル実行**に組み込めば、チュートリアルの腐敗を機械的に検出できます。
:::

## ドキュメントの鮮度維持

ドキュメントの本当の課題は「書くこと」より「**腐らせないこと**」です。AIを鮮度維持に使います。

### 乖離の検出

コード変更時に、関連ドキュメントとの矛盾をAIに検出させます。

```text
以下は今回のコード差分(diff)と、現在の README/API doc です。
この変更によって、ドキュメントの記述で「古くなった・矛盾する」箇所を
列挙してください。修正案も添えて。該当なしなら「該当なし」と。
（diff とドキュメントを貼る)
```

応答イメージ:

```text
矛盾箇所:
1. README「環境変数」: 今回 REDIS_URL が追加されたが表に未記載。
   修正案: | REDIS_URL | ジョブキュー接続 | 必須 | を追加。
2. API doc「POST /reviews」: comment の最大長が 500 と書かれているが、
   差分で 1000 に変更されている。500 → 1000 に修正。
```

::: tip CIに組み込む
この「diff vs docs の乖離検出」は、PR時のCIに組み込めます([AIコードレビューbotの構築](/practice/code-review-bot)と同じ仕組み)。ドキュメント更新を**人間の善意に頼らず、機械的に促す**のが鮮度維持の現実解です。ただしAIの指摘は**提案**であり、誤検知もあるため、人間が採否を判断します。
:::

### 誤りの検出(レビュー用途)

既存ドキュメントをコードに照らして「事実確認」させます。

```text
以下のドキュメントの各記述を、添付したコードに照らして
「正しい / 古い / コードに根拠なし(要確認)」に分類してください。
推測で「正しい」としないこと。根拠にしたコードの箇所を示して。
```

これは特に、**古いドキュメントを引き継いだとき**に有効です。何が信用でき何が腐っているかの地図ができます。

## アンチパターン集

| アンチパターン | 何が起きるか | 対処 |
| --- | --- | --- |
| 推測を許して生成 | もっともらしい嘘が紛れる | 「根拠コードのみ・不明は(要確認)」 |
| what をなぞるコメント乱造 | ノイズ増、乖離の温床 | why に絞る、コードを自己説明的に |
| チュートリアルを未実行で公開 | 動かない手順で離脱 | 必ず実行、CIでサンプル検証 |
| 一度書いて放置 | 確実に腐る | diff vs docs の乖離検出をCI化 |
| 生成docを無検証で信頼 | 誤りを読者が信じる | コードに照らして事実確認 |

## まとめ

- ドキュメント生成は **コードを根拠に、推測を禁じる** のが大原則。
- 不明点を **(要確認)** で残させることが、むしろ品質のシグナル。
- API doc は **スキーマ/ルートから逆生成** すると鮮度を保ちやすい。
- ADR は **決めた瞬間に**、口頭メモをAIで構造化。
- コメントは **why に絞る**。乖離を生むコメント乱造は避ける。
- チュートリアルは **必ず実行**して検証。
- 最大の課題は鮮度。**diff vs docs の乖離検出をCI化**し、機械的に促す。
