# チーム利用ポリシー雛形

このページは、[ガバナンスと倫理](/adoption/governance)で示した原則を、**社内にそのまま配布できる文書** に落とし込んだ実践版です。下のテンプレートをコピーし、角括弧 `[会社名]` のプレースホルダを自社の値に置き換えるだけで、最初のドラフトが完成します。

::: tip このページの使い方
ポリシーは「完璧な最終版」を待つより、**たたき台を素早く配布して回す** ほうがうまくいきます。まずはテンプレートをコピーして埋め、関係部門のレビューを経て社内Wikiやドキュメントツールに掲載してください。3〜6か月ごとの改訂を前提に運用します。
:::

::: warning 法務・セキュリティ部門の確認は必須
本テンプレートは一般的な雛形であり、法的助言ではありません。**データ取扱い・ライセンス・規制対応の最終判断は、必ず自社の法務/セキュリティ部門の確認を経てください。** 業界規制(医療・金融・公共など)や所在地の法令により、追加・変更すべき項目があります。
:::

## ポリシー本文テンプレート(コピペ用)

以下のコードブロックを丸ごとコピーして使ってください。`[会社名]` `[ツール名]` `[窓口]` などのプレースホルダを置き換えます。

```markdown
# [会社名] AI駆動開発 利用ポリシー

- バージョン: v1.0
- 最終改訂日: [YYYY-MM-DD]
- 管理責任者: [部門/担当者]
- 次回見直し予定: [YYYY-MM-DD]

## 1. 目的

本ポリシーは、[会社名] の従業員および業務委託者が、ソフトウェア開発業務において
AIコーディング支援ツール(以下「AIツール」)を安全かつ効果的に利用するための
ルールを定めるものです。目的は次の3点です。

1. 開発生産性と品質の向上を、組織として再現可能な形で実現する。
2. 機密情報・個人情報・顧客データの漏洩、ライセンス事故、セキュリティ事故を防ぐ。
3. 「迷ったときに各自が正しく判断できる原則」を共有し、現場の判断を支える。

本ポリシーは禁止事項の列挙ではなく、安心して活用するための土台です。

## 2. 適用範囲

- 対象者: [会社名] の全従業員、契約社員、業務委託者、インターン。
- 対象行為: 業務に関わるコード・設計・ドキュメント・テストの生成、補助、レビューに
  AIツールを利用するすべての行為。
- 対象ツール: 第3章の承認リストに記載されたAIツール、および今後追加されるもの。
- 個人デバイス・個人アカウントでの業務利用も本ポリシーの対象とします。

## 3. 使ってよいツール(承認リスト)

業務で利用してよいAIツールは、以下の「承認済みリスト」に限ります。
リスト外のツールを業務データで使う場合は、第10章の窓口に事前申請してください。

| ツール | 用途 | 契約形態 | 備考 |
| --- | --- | --- | --- |
| [ツール名] | コード生成・補助 | 法人契約(データ学習なし設定) | [備考] |
| [ツール名] | レビュー補助・設計相談 | エンタープライズ契約 | [備考] |
| [ツール名] | ドキュメント作成 | [契約形態] | [備考] |

承認の前提条件:
- 入力データを学習に利用しない設定、またはゼロデータ保持(zero data retention)が
  契約・設定で担保されていること。
- 無料プラン・個人アカウントは、明示的に許可された範囲を除き業務利用しない。
- 管理者による利用状況の把握(監査ログ)が可能な契約であること。

## 4. 入力してよい情報・してはいけない情報

AIツールへの入力(プロンプト)には、以下のデータ区分のルールを適用します。

| データ区分 | 例 | 外部AIツールへの入力 |
| --- | --- | --- |
| 公開情報 | OSSコード、公開ドキュメント、一般技術知識 | 原則OK |
| 社内一般 | 一般的な社内コード、設計メモ | 承認ツール・データ保護設定下で条件付きOK |
| 機密 | 認証情報、APIキー、トークン、顧客データ、個人情報、契約情報 | 禁止 |
| 規制対象 | 医療・金融・個人データなど法令が定めるもの | 禁止 / 専用環境のみ |

具体的な禁止入力(必ず守る):
- 認証情報(パスワード、APIキー、シークレット、トークン、秘密鍵)。
- 個人情報(氏名、メール、住所、電話番号、マイナンバー等)および顧客データ。
- 第三者から預かった情報、NDA配下の情報、他社の機密。
- 未公開の事業計画、財務情報、M&A等の重要事実。

ログ・スタックトレース・設定ファイルには機密が紛れ込みやすいため、
貼り付け前に必ず内容を確認し、マスキング・匿名化してから入力してください。

## 5. 生成コードの責任と必須レビュー

- 承認者責任の原則: AIが生成したコードであっても、レビューしマージした人間が
  品質・セキュリティ・ライセンス適合性の責任を負う。「AIが書いた」は免責にならない。
- 理解できないコードはマージしない。生成コードは自分で読んで理解したうえで取り込む。
- 必須レビュー: 本番・共有ブランチへ入るすべての生成コードは、第三者の人間レビューと
  CI(テスト・静的解析・セキュリティスキャン)を通過させる。
- AI生成だからといってレビューを軽くしない。ハルシネーション(存在しないAPI等)の
  リスクがあるため、むしろ検証を厚くする。

レビュー観点と運用基準の詳細は社内の「PR・レビュー基準」を参照してください。

## 6. ライセンス・著作権

- 生成物を取り込んだ瞬間、それは [会社名] のコードベースの一部であり、
  ライセンス適合性とIPの責任は当社が負う。「AI製だから免責」は通用しない。
- ライセンススキャン/コードクローン検出をCIに組み込み、特定ライセンス
  (コピーレフト等)への酷似・丸ごとコピーを検知する。
- AIが提案したライブラリのライセンスを鵜呑みにしない。AIはライセンス情報を
  誤ることがあるため、導入前に必ず確認する。
- 生成物の権利帰属・商用利用・再配布の可否は、利用ツールの契約・利用規約に従う。

## 7. セキュリティ要件

- 生成コードにも、人間が書いたコードと同等以上のセキュリティ基準を適用する。
- シークレットスキャン・SAST(静的解析)・依存脆弱性スキャンをCIで必須とし、
  生成コードにも適用する。
- 認証・認可・暗号など、セキュリティ上重要な実装は専門知識で検証する。
  AIの提案をそのまま信用しない。
- 外部入力をプロンプトに渡す箇所では、プロンプトインジェクションを考慮する。
- 機密がログ・エラーメッセージに出力されていないか確認する。

## 8. 禁止事項

- 認証情報・APIキー・顧客データ・個人情報を外部AIツールに入力すること。
- 承認リスト外のツール(個人アカウント等)を業務データで利用すること。
- 生成コードをレビューなしで本番・共有ブランチへマージすること。
- ライセンス・出典が不明な大量コードを、検証せず取り込むこと。
- AIの出力を、事実確認なしに対外的な保証・回答として用いること。
- 規制対象データを、許可された専用環境以外で扱うこと。

## 9. 違反時の対応

- 違反または違反の疑い(特に機密入力・漏洩)を認識したら、ただちに第10章の窓口へ
  報告する。報告は遅延させない。早期報告は被害を最小化する。
- 初動として、対象データの追加拡散を止め、関係するアクセス・認証情報を必要に応じて
  失効・ローテーションする。
- 当社は報告内容に基づき、影響範囲の調査・是正・再発防止を行う。
- 故意・重大な過失による違反は、就業規則に基づく対応の対象となる場合がある。
- 善意の報告者を不利益に扱わない(心理的安全性を確保し、隠蔽を防ぐ)。

## 10. 問い合わせ先

- 一般的な相談・ツール申請: [窓口/チャンネル]
- セキュリティインシデント・機密入力の疑い: [セキュリティ窓口/緊急連絡先]
- ライセンス・法務に関する判断: [法務窓口]
- 本ポリシーの改訂提案: [管理責任者]

判断に迷う場合は、自己判断で進めず必ず窓口に相談してください。
「迷ったら聞く」がもっとも安全な選択です。
```

::: details テンプレートを自社用に置き換えるときの注意
- `[会社名]` `[ツール名]` `[窓口]` などのプレースホルダは検索置換ですべて埋める。埋め残しがないか公開前に確認する。
- 承認ツール表は、調達・法務が確認した契約条件と一致させる。契約が変わったら表も更新する。
- 規制業界では第4章・第7章に業界固有の要件(データレジデンシー、監査要件等)を追記する。
:::

## 導入チェックリスト

ポリシーを「作って終わり」にしないための、配布前後の点検リストです。

**配布前**

- [ ] テンプレートのプレースホルダ(`[会社名]` 等)をすべて置き換えた
- [ ] 承認ツールリストを調達・法務が確認した契約条件と一致させた
- [ ] データ区分の定義が自社の情報分類規程と整合している
- [ ] 法務/セキュリティ部門のレビューを受け、承認を得た
- [ ] 業界規制(該当する場合)の追加要件を反映した
- [ ] 問い合わせ窓口・緊急連絡先が実在し、対応者が決まっている
- [ ] バージョン・改訂日・次回見直し予定を記入した

**配布・浸透**

- [ ] 全対象者に周知し、いつでも参照できる場所(Wiki等)に掲載した
- [ ] 新メンバーのオンボーディングに本ポリシーの説明を組み込んだ
- [ ] 承認ツールの推奨設定(データ保護設定)の手順を併せて案内した
- [ ] 機密入力を防ぐ技術的ガードレール(DLP・シークレットスキャン)を併用している

**運用・改訂**

- [ ] 監査ログで利用状況を把握できる状態にある
- [ ] 違反・ヒヤリハットを安全に報告できる経路が機能している
- [ ] 3〜6か月ごと、または契約・規制の変化時に改訂する担当を決めた
- [ ] 改訂履歴(バージョンと変更点)を記録している

## 軽量版 / 厳格版の出し分け

組織の規模・リスク許容度に応じて、テンプレートの厳格さを調整します。**重すぎるポリシーは現場が迂回し、軽すぎるポリシーは事故を防げません。** 自社の段階に合わせてください。

| 観点 | 軽量版(スタートアップ向け) | 厳格版(規制業界向け) |
| --- | --- | --- |
| 分量 | 1ページに凝縮。原則中心 | 詳細規定+別紙(データ分類規程等)を整備 |
| 承認ツール | 少数を許可、申請は軽量 | 承認プロセスを正式化、調達・法務の事前審査必須 |
| データ入力 | 「機密を入れない」を徹底 | データ区分を細分化し、規制データは専用環境限定 |
| レビュー | 第三者レビュー+CI必須 | レビュー+監査証跡+承認記録の保存を義務化 |
| 監査ログ | 法人契約の標準機能を活用 | 利用ログの長期保存・定期監査を制度化 |
| 改訂頻度 | 半年ごと、軽量に見直し | 四半期ごと、規制変化を継続トラッキング |

::: tip まず軽量版から始める
立ち上げ初期は軽量版で素早く配布し、利用が広がるにつれて厳格版の要素を足していくのが現実的です。最初から厳格版を作り込もうとして配布が遅れるより、**守れる粒度のものを早く回す** ほうが組織を守れます。導入の進め方は[導入戦略](/adoption/strategy)を参照してください。
:::

::: warning 技術的ガードレールでポリシーを補完する
人間の判断ミスは必ず起きます。機密入力を防ぐDLP、シークレットスキャン、ライセンススキャンといった**技術的ガードレール**を併用し、ポリシー(人の判断)と二重の防御を構成してください。実装の詳細は[セキュリティ](/quality/security)と[セキュリティ深掘り](/advanced/security-deep-dive)を参照。
:::

## まとめ

- ポリシーは完璧を待たず、**たたき台を素早く配布して回す**。3〜6か月ごとに改訂する。
- テンプレートをコピーし、プレースホルダを埋め、法務/セキュリティの確認を経て配布する。
- 入力可否は契約条件に基づくデータ区分で明文化し、機密・個人情報・顧客データは入力禁止。
- 生成コードの責任は **承認しマージした人間** にある。必須レビューとCIを通す。
- 軽量版/厳格版を組織の段階に合わせて出し分け、技術的ガードレールで補完する。
