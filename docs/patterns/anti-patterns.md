# アンチパターン集

良いやり方を学ぶのと同じくらい、**典型的な失敗を名前で知っておく**ことには価値があります。アンチパターンは「一見うまくいきそうで、繰り返し痛い目を見る型」です。名前があれば、レビューやペアプロで「それ無限リトライになってない?」と早期に指摘できます。

各アンチパターンを **症状 → なぜ悪い → 対処** で記述します。まず全体像を一覧で示し、その後に詳説します。

::: tip 共通する根っこ
ほとんどのアンチパターンは、たった2つの誤解に行き着きます。(1) AIの出力を**検証せず信頼する**、(2) AIに**人間の判断責任を委ねる**。この2点を疑う癖があれば、多くは未然に防げます。
:::

## 一覧

| # | アンチパターン | 症状(ひとこと) | 対処の核 |
| --- | --- | --- | --- |
| 1 | 丸投げ巨大タスク | 「アプリ全部作って」 | タスクを分解する |
| 2 | 無検証マージ | 動作確認せず取り込む | テスト・実行で検証 |
| 3 | 文脈不足 | 背景を渡さず推測させる | コンテキスト設計 |
| 4 | 過信(自動化バイアス) | 出力を疑わない | 批判的レビュー前提 |
| 5 | 無限リトライ | 同じ指示を投げ続ける | 立ち止まって診断 |
| 6 | 機密投入 | 秘密情報を貼る | 投入前にマスク |
| 7 | レビュー崩壊 | 生成量が多すぎて見切れない | 小さく出させる |
| 8 | 古いAPI鵜呑み | 存在しないAPIを使う | 公式で裏取り |
| 9 | テストもAI任せ | 実装者がテストも書く | テストは独立検証に |
| 10 | プロンプト沼 | プロンプトを磨き続ける | 撤退ラインを引く |
| 11 | ロックイン軽視 | 1ツール前提で固める | 抽象化と移植性 |
| 12 | 幻覚の見逃し | もっともらしい嘘を信じる | 事実は外部で確認 |

## 1. 丸投げ巨大タスク

**症状** — 「ECサイトを一式作って」「この機能、全部よろしく」のように、巨大で曖昧なタスクを一度に渡す。

**なぜ悪い** — タスクが大きいほど、仕様の取りこぼし・誤解・暴走が増え、出力も巨大になってレビュー不能になります。途中で方向がずれると、手戻りも巨大化します。検証可能な単位になっていないため、どこが正しくどこが間違いかも切り分けられません。

**対処** — [ステップ分解](/patterns/prompt-catalog#_5-ステップ分解-decomposition)で、独立してレビュー・テストできる単位に割る。まず**計画だけ**を出させ、承認してから着手する。委任の粒度は[実装フェーズ](/workflow/implementation)を参照。

::: warning
「大きいタスクを賢いモデルなら一発で」は幻想です。モデルが賢くても、**人間がレビューできる粒度**でないと品質は担保できません。
:::

## 2. 無検証マージ

**症状** — 生成されたコードを、実行もテストもせずにそのままコミット/マージする。

**なぜ悪い** — LLMの出力は**確率的**で、もっともらしく見えて動かないコードが普通に混じります。コンパイルが通る/読んで正しそう、は動作する保証になりません。無検証マージは技術的負債とバグを高速に量産します。

**対処** — 生成物は必ず**実行・テストで検証**してから取り込む。[テスト先行](/patterns/prompt-catalog#_10-テスト先行-test-first)で先にテストを固定しておくと、検証が機械化できる。CIを通過の門にする。

| やりがち | あるべき |
| --- | --- |
| 読んで「良さそう」→ マージ | テスト通過 + 動作確認 → マージ |
| 「AIが書いたから大丈夫」 | 「AIが書いたからこそ検証する」 |

## 3. 文脈不足

**症状** — 既存コード・規約・前提を渡さず、「いい感じに」実装させる。

**なぜ悪い** — モデルはあなたのリポジトリを知りません。文脈がなければ**推測で埋める**しかなく、既存規約と衝突する独自実装が生まれます。再作業の最大の原因の1つです。

**対処** — [コンテキストエンジニアリング](/patterns/context-engineering)に従い、関連コード・規約の見本・制約を選別して渡す。「このファイルに合わせて」([リファレンス駆動](/patterns/prompt-catalog#_9-リファレンス駆動))が最も確実。

::: tip
文脈不足の逆である「**過剰な文脈**」も別の失敗です。全部貼ると重要情報が埋もれます。詰め込みではなく**選別**が答えです。
:::

## 4. 過信(自動化バイアス)

**症状** — 自分でゼロから書くときより、AIの出力を甘く評価する。流暢な説明に納得して検証を省く。

**なぜ悪い** — 人間には、機械の出力を過度に信頼する**自動化バイアス**があります。AIの文章は自信満々で流暢なので、誤りでも説得力を持ってしまいます。結果、人間が書いたコードより**チェックが甘くなる**という逆転が起きます。

**対処** — 「AIの出力は、信頼できない優秀な新人のPR」と位置づけ、**人間のコード以上に**批判的にレビューする。[自己レビューパターン](/patterns/prompt-catalog#_6-自己レビュー-批評)で一次足切りはできるが、最終判断は人間が負う。

## 5. 無限リトライ

**症状** — 期待通りにならないとき、原因を考えずに「もう一回」「やっぱり違う」と同じ指示を投げ続ける。

**なぜ悪い** — モデルが間違える原因(文脈不足・仕様の曖昧さ・そもそも無理な要求)を放置したまま再生成しても、確率的に少し違うだけで本質は直りません。時間とトークンを溶かし、運良く通った出力を「正しい」と誤認する危険もあります。

**対処** — 2〜3回で期待に届かなければ**立ち止まる**。失敗は指示の問題か、文脈の問題か、要求自体の問題かを切り分ける。[エラー解決パターン](/patterns/prompt-catalog#_19-エラー解決)で原因の仮説を立ててから、プロンプトかコンテキストを**設計し直す**。

::: warning
「リトライで偶然通った」コードは、再現性のない当たりです。なぜ直ったか説明できないなら、まだ理解できていません。
:::

## 6. 機密投入

**症状** — APIキー、本番の認証情報、個人情報、社外秘コードを、無頓着にプロンプトへ貼る。

**なぜ悪い** — 投入した情報は、ツールやモデル提供者の取り扱いポリシー次第でログ・学習・第三者送信の対象になり得ます。一度漏れた秘密は取り消せません。コンプライアンス違反や情報漏洩に直結します。

**対処** — 投入前に**マスク/ダミー化**を徹底する。秘密はプレースホルダに置換する。組織のデータ取り扱いポリシーと、使うツールのデータ保持/学習方針を事前に確認する。秘密検知をCIやpre-commitに入れる。

```text
# ❌ そのまま貼る
DB_PASSWORD=Sup3rSecr3t!  に接続できない

# ✅ マスクして相談
DB_PASSWORD=<REDACTED> に接続できない。接続文字列の組み立ては下記。
```

## 7. レビュー崩壊(生成量過多)

**症状** — 一度に大量のコードを生成させ、差分が大きすぎて誰もまともにレビューできない。

**なぜ悪い** — AIは人間より圧倒的に速く大量のコードを吐けます。生成速度がレビュー速度を超えると、**レビューが形骸化**します。「読みきれないからLGTM」は、無検証マージ(#2)と同じ結末です。

**対処** — 出力を**小さい差分**に分割させる([ステップ分解](/patterns/prompt-catalog#_5-ステップ分解-decomposition))。PRは1つの関心事に絞る。「レビューできる量しか生成しない」を規律にする。生成が速いことは、**レビュー不要**を意味しません。

| 指標 | 危険サイン |
| --- | --- |
| 1PRの変更行数 | 数百行を超え、誰も全部読まない |
| レビュー時間 | 生成時間より明らかに短い |

## 8. 古いAPI鵜呑み

**症状** — モデルが提案したライブラリのメソッドやオプションを、存在確認せずに使う。

**なぜ悪い** — モデルの知識には**学習時点のカットオフ**があり、その後に変わった/廃止されたAPIを使うことがあります。また、存在しないメソッドを**それらしく捏造**することもあります(幻覚)。バージョン不一致で動かない、非推奨APIで将来壊れる、といった事故になります。

**対処** — 外部API・ライブラリの使い方は**公式ドキュメントで裏取り**する。使用バージョンをプロンプトで明示する([制約付き生成](/patterns/prompt-catalog#_7-制約付き生成))。型チェック・実行で存在を機械的に確認する。

::: warning
特に変化の速いフレームワークやSDKでは要注意です。「動くはず」のコードが、実在しないAPIを呼んでいることがあります。
:::

## 9. テストもAI任せ

**症状** — 実装を書かせたAIに、同じ流れでそのままテストも書かせ、それで検証完了とする。

**なぜ悪い** — 実装とテストが**同じ誤解を共有**していると、間違った実装に対して間違ったテストが書かれ、両方が仲良く通ってしまいます。テストは実装から**独立した検証**であってこそ意味があります。

**対処** — テストを実装の**前**に、独立した観点で固める([テスト先行](/patterns/prompt-catalog#_10-テスト先行-test-first))。少なくとも、テストの**仕様(期待値)は人間がレビュー**する。AIにテストを書かせること自体は有用だが、「実装者がテストも兼ねる」構造を避けるのが要点。

::: tip
理想は「人間が期待する振る舞い(テスト)を合意 → AIが実装 → テストで検証」。テストが仕様の番人になります。
:::

## 10. プロンプト沼

**症状** — 出力が不満なたびにプロンプトへ条件を足し続け、巨大化した呪文を延々と磨く。

**なぜ悪い** — プロンプトを盛るほど焦点がぼやけ、互いに矛盾する指示が混ざり、かえって悪化することがあります。本来は**コードを直接書いたほうが速い**場面で、プロンプト調整に時間を溶かします。

**対処** — **撤退ラインを引く**。「2〜3回試して届かなければ自分で書く/設計を見直す」。問題がプロンプトでなく**コンテキストや要求の側**にないか疑う([無限リトライ](#_5-無限リトライ)と地続き)。プロンプトは必要最小から足す。

::: warning
プロンプトエンジニアリングは目的ではなく手段です。**完成したソフトウェア**が目的。プロンプトを磨くこと自体が仕事になっていたら危険信号です。
:::

## 11. ベンダーロックイン軽視

**症状** — 特定のAIツール/モデル/プラットフォーム固有の機能やプロンプト前提で、ワークフローやコードを深く固める。

**なぜ悪い** — モデルや料金、提供条件は変わります。1つのベンダーに不可分に依存すると、価格改定・仕様変更・サービス終了のリスクをまるごと負います。移行コストが高すぎて、不利な条件でも乗り換えられなくなります。

**対処** — モデル呼び出しを**抽象化レイヤ**の背後に隔離し、差し替え可能にする。プロンプトやパイプラインを過度に1モデルへ最適化しない。重要なロジックはツール外の**通常のコード/テスト**として残す。複数モデルで動くかを時々確認する。

::: tip
AIは強力な道具ですが、**プロジェクトの資産はコードとテストとドメイン知識**であって、特定ツールへの依存ではありません。道具は替えが効く状態を保ちます。
:::

## 12. 幻覚の見逃し

**症状** — もっともらしいが事実でない出力(存在しない関数、誤った仕様、捏造された根拠)を、流暢さに釣られて信じる。

**なぜ悪い** — LLMは「正しいこと」ではなく「もっともらしいこと」を生成します。幻覚は**自信満々で出てくる**ため、知らない領域ほど見抜けません。誤った前提が下流の判断を汚染します。

**対処** — 検証できる事実(API・数値・仕様・引用)は**外部の一次情報で確認**する。[説明生成](/patterns/prompt-catalog#_14-説明生成)では「推測は推測と明記」を要求する。**自分が検証できる範囲**でのみ出力を信用する、を原則にする。

::: warning
「知らない領域だからAIに聞く」が最も危険です。知らないからこそ、その出力の真偽を判定できません。未知の領域では、AIの出力は**仮説**として扱い、必ず裏を取ってください。
:::

## チェックリスト

レビューやふりかえりで使える、アンチパターン検知の質問です。

- [ ] このタスク、人間がレビューできる粒度に割れているか?(#1, #7)
- [ ] このコード、実行とテストで検証したか?(#2, #9)
- [ ] モデルに必要な文脈を渡したか / 過剰に渡していないか?(#3)
- [ ] 自分でゼロから書くときと同じ厳しさでレビューしたか?(#4, #12)
- [ ] 直らないとき、原因を診断せず再生成していないか?(#5, #10)
- [ ] 機密情報を投入していないか?(#6)
- [ ] 使ったAPIは公式で裏取りしたか?(#8, #12)
- [ ] 1つのツールに不可分に依存していないか?(#11)

## まとめ

- アンチパターンの根は、**無検証の信頼**と**判断責任の委譲**の2つ。
- AIの出力は「信頼できない優秀な新人のPR」。人間のコード以上に検証する。
- 速く大量に作れることは、**レビュー不要を意味しない**。むしろレビューが律速になる。
- 道具は替えが効く状態を保つ。資産はコード・テスト・ドメイン知識。
