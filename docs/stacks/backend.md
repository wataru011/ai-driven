# バックエンド開発でのAI活用

バックエンドは、フロントエンドと違って成果物が**目に見えません**。レイアウトの崩れのように一目でわかる失敗がなく、不具合は「特定の入力」「特定の同時実行」「特定のデータ状態」でのみ顔を出します。だからこそ、AIが生成するバックエンドコードは**もっともらしく動いて見えるが、境界で壊れる**という事故が起きやすい領域です。

AIに任せて効果が高いのは、**形が決まった層**(DTO変換、バリデーション、CRUDハンドラ、シリアライズ)です。逆に、**正しさの判断が必要な層**(トランザクション境界、認可ルール、整合性制約)は人間が設計を握り、AIには実装の手を貸させる、という分担が安全です。

::: tip この章の前提
セキュリティの詳細(SQLインジェクション、認証の脆弱性、シークレット管理など)は [セキュリティリスク](/quality/security) で扱います。本章ではバックエンド設計・実装の勘所に集中し、セキュリティは要所で参照します。
:::

## AI活用の勘所

### 1. API設計は「契約」を先に固める

エンドポイント実装をいきなり書かせると、命名・ステータスコード・エラー形式がバラバラになります。先に**API契約(スキーマ)**を確定させ、それを文脈にして実装させるのが鉄則です。

- OpenAPI / gRPC proto / GraphQLスキーマを**先に**AIと一緒に設計する。
- 契約が決まったら「このスキーマに厳密に従って実装して」と指示する。
- リクエスト/レスポンスのDTOはスキーマから生成し、手書きしない。

これにより、フロントとの認識ずれ、ステータスコードの揺れ、エラー形式の不統一を一掃できます。

### 2. ビジネスロジックは「仕様」を言語化してから

ビジネスロジックはプロジェクト固有で、LLMの一般知識では補えません。AIに「いい感じに」と頼むと、**ありがちな一般論**を書いてしまい、ドメインの制約を無視します。

有効なのは、**仕様を箇条書きにして渡す**ことです。

> 在庫引当のロジックを実装してください。仕様:
> - 引当は注文確定時に行う。在庫が不足する場合は注文全体を失敗させる(部分引当は不可)
> - 同一商品への同時注文は在庫の二重引当を起こしてはならない(行ロックで防ぐ)
> - 取り消し時は引当を戻す。冪等であること(同じ取消が2回来ても在庫は1回分しか戻らない)

この粒度で渡すと、AIは**ロックの取り方や冪等性の実装**まで踏み込んだコードを出せます。曖昧なまま頼むと、ここが必ず抜けます。

### 3. 認証・認可は「テンプレを疑う」

AIは認証コードのテンプレを大量に知っていますが、**あなたの認可モデルは知りません**。とくに認可(誰が何にアクセスできるか)は、生成物をそのまま信じてはいけません。

::: warning 認可で起きがちな生成事故
- **認可チェックの欠落** — 認証(ログイン済みか)は書くが、認可(その人がそのリソースを操作してよいか)を忘れる。いわゆるIDOR(オブジェクトレベル認可の不備)を量産する。
- **フロントでの制御に依存** — 「UIで隠しているから」とサーバー側チェックを省く。
- **権限の昇格経路** — 一括更新APIで他人のレコードまで触れてしまう。
:::

対策は「**すべてのリソースアクセスで所有者・権限を必ず検証する**」をプロジェクトの不変条件として明文化し、AIに毎回守らせること。詳細は [セキュリティリスク](/quality/security) を参照してください。

### 4. エラー処理は「握りつぶし」を許さない

LLMは、エラーを`catch`して**ログだけ出して握りつぶす**コードを書きがちです。あるいは全例外を500で返してしまい、クライアントが回復できません。

依頼時に方針を明示します。

> エラーは以下の方針で:
> - 業務エラー(在庫不足など)は明示的な型で表現し、適切なHTTPステータス(409など)へマップ
> - 予期せぬ例外のみ500。スタックトレースはクライアントに漏らさない
> - エラーレスポンスは契約のエラースキーマに統一(code, message, details)
> - catch して何もしない(握りつぶし)は禁止

### 5. トランザクション境界は人間が引く

「どこからどこまでを1トランザクションにするか」は設計判断であり、AIに委ねると**境界が曖昧**になります。複数の集約をまたぐ更新、外部API呼び出しを含む処理などでは、境界の引き方が整合性を左右します。

- トランザクション内で外部API(メール送信、決済)を呼ばせない(ロールバックできない副作用)。
- 「ここで commit、ここまでが1単位」を人間が指定し、AIに実装させる。
- 結果整合でよい部分は、アウトボックスパターンなどを明示的に指示する。

## 有効なプロンプト例

### 契約からのエンドポイント実装

```text
添付の OpenAPI スキーマの POST /orders を実装してください(Node.js + Fastify + Prisma)。
- リクエスト/レスポンスのDTOはスキーマから生成済みの型を使う
- 在庫引当は別添の仕様に従い、トランザクション + 行ロックで二重引当を防ぐ
- 認可: 注文を作成できるのは認証ユーザー本人のみ。userId はトークンから取得し、
  リクエストボディの userId は信用しない
- エラー: 在庫不足は 409 + エラースキーマ。バリデーションエラーは 400。
- まず実装方針(トランザクション境界とロック対象)を箇条書きで提示してから実装して
```

### 既存コードのN+1検出

```text
このリポジトリ層のコードを読み、N+1クエリが発生する箇所を指摘してください。
- 各指摘について「どのループでどのクエリが繰り返されるか」を具体的に
- 修正案は eager loading / バッチ取得 / JOIN のどれが適切かを理由とともに
- 推測ではなく、実際に発行されるクエリの回数で説明して
```

## 落とし穴と対策

::: warning バックエンド特有の典型的な失敗
- **N+1クエリ** — ループ内でリレーションを引く。動くがスケールで破綻する。
- **境界条件の見落とし** — 空配列、0件、null、上限値、ページ末尾、タイムゾーン境界。
- **同時実行の無視** — 単一リクエストでは正しいが、競合で二重実行・二重引当が起きる。
- **冪等性の欠如** — リトライ前提のAPIなのに、二重実行で副作用が二重になる。
- **ページネーションの罠** — OFFSET依存で大きなページが遅い、件数がずれる。
- **時刻・タイムゾーン** — サーバーローカル時刻の使用、UTC統一の欠如。
- **N+1ならぬ過剰フェッチ** — 必要ない列まで`SELECT *`で取得。
:::

これらは「動くデモ」では露見せず、本番の負荷とデータ分布で初めて顔を出します。**生成コードを負荷・境界の観点でレビューする**のが人間の役割です。

| 落とし穴 | 検出・予防の手段 |
| --- | --- |
| N+1 | ORMのクエリログ、`EXPLAIN`、専用リンタ、テストでのクエリ回数アサート |
| 境界条件 | プロパティベーステスト、境界値を明示したテストケース生成依頼 |
| 同時実行 | 並行実行テスト、DB制約(ユニーク制約)で最後の砦を作る |
| 冪等性 | 冪等キーの導入、同一リクエスト2回送信テスト |

## テストと検証

### 契約テスト(Contract Testing)

API契約をスキーマで固めたら、**実装が契約から逸脱していないか**を機械的に検証します。これはマイクロサービス間の認識ずれを防ぐ最も費用対効果の高い検証です。

- スキーマ(OpenAPI等)に対するリクエスト/レスポンスの検証をテストに組み込む。
- 消費者駆動契約テスト(Pact等)で、サービス間の互換性を担保する。
- AIには「この契約に違反する入出力を検出するテストを書いて」と依頼できる。

### 境界とエラー経路のテスト

AIにテストを依頼するときは、**ハッピーパスだけでなく境界とエラー経路**を明示的に要求します。放っておくと正常系ばかり書かれます。

```text
このサービスメソッドのテストを書いてください。正常系に加えて以下を必ず網羅:
- 在庫0/在庫1で同時に2件引当(競合)
- 取消を2回(冪等性)
- 不正なuserIdでのアクセス(認可拒否を確認)
- 外部API呼び出しが失敗したときのロールバック
モックは外部依存のみ。DBは実DB(テスト用)に対して実行して整合性を確認して。
```

### 「動いた」を信用しない

LLMは「テストが通った」を成功とみなしますが、**テスト自体が甘ければ無意味**です。生成されたテストは、上記の境界・同時実行・認可を本当に検証しているかを人間が確認します。詳しくは [テストとレビュー](/workflow/testing-review) を参照してください。

## 人間がやること / AIに任せること

| 人間が担う | AIに任せる |
| --- | --- |
| API契約・ドメイン仕様の確定 | 契約に従ったハンドラ実装 |
| トランザクション境界の決定 | DTO変換・バリデーション・CRUD |
| 認可モデルの設計と最終確認 | エラーマッピングの実装 |
| 同時実行・整合性の責任 | N+1検出とリファクタ案 |
| テスト観点(境界・競合)の選定 | テストコードの雛形化 |
