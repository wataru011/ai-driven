# 新規開発 vs 既存改修の違い

AI駆動開発の進め方を考えるとき、最初に分けるべき軸は「使うツール」でも「採用するモデル」でもありません。**そのコードベースが、白紙(グリーンフィールド)なのか、すでに動いているもの(ブラウンフィールド)なのか**です。この一点で、AIの効きどころ、リスクの形、最初に踏み出す一歩がまるで変わります。

同じ「AIに実装させる」でも、新規開発では「AIの速度をいかに制御するか」が問題になり、既存改修では「AIにいかに既存の文脈を正しく理解させるか」が問題になります。両者を区別せずに同じ進め方を当てはめると、新規では暴走し、既存では的外れな提案に振り回されます。

::: tip このページの位置づけ
ここは**分岐点の地図**です。両アプローチの違いを俯瞰し、自分のプロジェクトがどちらに寄るかを判断するためのページです。具体的な進め方は、それぞれ[新規開発(0→1)のアプローチ](/approach/greenfield)と[既存改修(ブラウンフィールド)のアプローチ](/approach/brownfield)で詳しく扱います。
:::

## 2つのアプローチの本質的な違い

まず言葉を揃えます。

- **グリーンフィールド(Greenfield / 新規開発)**: ゼロから作る。既存コードの制約がなく、技術選定もアーキテクチャも自由に決められる。0→1。
- **ブラウンフィールド(Brownfield / 既存改修)**: すでに動いているシステムに手を入れる。既存コード・データ・利用者・運用が制約として存在する。1→N。

この2つは、AIにとって**制約とコンテキストの所在が正反対**です。

新規開発では、制約は「まだ存在しない」。だからAIは自由に書けますが、自由すぎて方向を見失います。コンテキストは人間の頭の中(要件・意図)にしかなく、それを明示的にAIへ渡さない限り、AIは流行りの一般解を勝手に選びます。

既存改修では、制約は「すでにコードとして存在する」。AIはその制約を読み取れれば的確に振る舞えますが、読み取れなければ既存の規約・前提・暗黙のルールを平気で破壊します。コンテキストはコードベースの中に大量にあり、それをいかにAIへ正しく渡すかが勝負になります。

## 比較表: どこがどう違うか

| 観点 | グリーンフィールド(新規) | ブラウンフィールド(既存改修) |
| --- | --- | --- |
| 制約の所在 | まだ存在しない。人間が能動的に与える必要がある | すでにコード・データ・運用に埋め込まれている |
| コンテキストの源 | 人間の頭の中(要件・意図)。明示しないと伝わらない | 既存コードベース。読み取って渡す必要がある |
| AIの主な効きどころ | スキャフォールド生成・複数案の壁打ち・定型の高速生成 | 既存コードの読解・仕様復元・影響範囲の洗い出し |
| 主なリスク | 暴走・過剰設計・技術選定バイアス・一貫性崩壊 | 既存規約の破壊・影響範囲の見落とし・退行(regression) |
| 速度の出し方 | AIが速い。ただし制御しないと速く間違う | 慎重さが先。理解にコストをかけてから着手 |
| 最初の一歩 | 要件・仕様を固め、規約とルールファイルを先に置く | 既存コードを読ませ、現状仕様と影響範囲を把握する |
| 必要なドキュメント | 要件定義・アーキテクチャ・規約(これから作る) | 復元した仕様・依存関係図・変更の影響範囲(これから抽出する) |
| 典型的な失敗モード | 「動くが、誰も意図していない設計」になる | 「一見動くが、既存機能を静かに壊している」 |
| 検証の重心 | テスト先行で振る舞いを定義していく | 既存テスト・回帰テストで「壊していないこと」を担保する |
| 人間の判断の急所 | 何を作らないか・どの制約を自分に課すか | どこまで触ってよいか・どこが触ると危険か |

::: info 同じ「AIに任せる」でも、任せ方が逆になる
新規では「AIに自由に書かせて、人が削る・絞る」方向の制御になりがちです。既存では「AIに既存を理解させてから、狭く・浅く触らせる」方向の制御になります。前者は引き算、後者は足し算の慎重さ、と捉えると整理しやすいでしょう。
:::

## なぜ違いが生まれるのか — 3つの軸で読み解く

### 軸1: 制約は「与えるもの」か「読み取るもの」か

LLMは確率的にもっともらしい続きを生成します。制約が与えられなければ、訓練データ上で多数派だった選択(流行りのフレームワーク、定番のディレクトリ構成、一般的なパターン)に流れます。

新規開発では、この性質が両刃の剣になります。白紙に対してAIは即座に「それらしい全体像」を出せますが、それはあなたのチームやドメインに最適化されたものではなく、**一般的に無難なもの**です。だから新規では、人間が制約を能動的に与えることがすべての出発点になります。

既存改修では逆に、制約はすでにコードとして大量に存在します。問題は、AIがそれを**読み取れているか**です。コンテキストとして既存の規約・型・テスト・隣接コードを渡せていれば、AIはその制約に沿って振る舞います。渡せていなければ、既存と矛盾する一般解を提案してきます。

### 軸2: リスクの形が違う — 「速い暴走」か「静かな破壊」か

新規開発の失敗は、たいてい**目に見えて派手**です。過剰なマイクロサービス化、使いもしない抽象化レイヤ、流行りだけで選んだ技術スタック。動きはするが、誰も意図していない設計が積み上がります。幸い、白紙ゆえに作り直しのコストは比較的低い段階で気づけます。

既存改修の失敗は、**静かで気づきにくい**のが厄介です。AIが提案した変更が、一見正しく動いているのに、既存の別機能の前提を壊している。影響範囲を見落とした退行は、テストが薄ければ本番で初めて顕在化します。ここでのコストは新規よりはるかに高くつきます。

::: warning 既存改修ほど「動いた」を信用しない
新規では「動かない」がすぐ分かります。既存改修の本当の怖さは「**触った箇所は動くが、触っていないつもりの箇所が壊れる**」ことです。AIの提案がローカルで動いたことは、既存機能を壊していない証明には一切なりません。回帰テストと影響範囲の確認が、新規以上に重みを持ちます。
:::

### 軸3: AIの効きどころが違う

AIが最も力を発揮する場所が、両者で異なります。

- **新規での効きどころ**: ゼロからの立ち上げ。スキャフォールド生成、ボイラープレートの一括生成、アーキテクチャ案の複数比較(壁打ち)、規約ファイルの初稿。白紙に対する「最初の形」を高速に出すのが得意です。
- **既存での効きどころ**: 大量のコードの読解。既存実装からの仕様復元、依存関係の可視化、影響範囲の推定、既存パターンに沿った変更案。**人間が読むのに時間がかかる量のコードを、要約・構造化する**のが得意です。詳しくは[リバースエンジニアリング](/approach/reverse-engineering)で扱います。

## 現実はどちらが多いか

ここで誤解を解いておきます。AI駆動開発の入門記事や派手なデモは、ほぼ例外なく**新規開発(0→1)**です。白紙から数分でアプリの骨格が立ち上がる様子は見栄えがよく、AIの速度を最も分かりやすく示せるからです。

しかし、実務のソフトウェア開発時間の大半は、**既存改修(ブラウンフィールド)**に費やされます。新機能の追加、バグ修正、リファクタリング、依存更新、移行作業 — そのほとんどは、すでに動いているコードへの変更です。完全な新規プロジェクトに着手する機会は、キャリア全体で見ればごく一部です。

::: warning デモの印象と実務のギャップに注意
「AIで0→1が速い」という体験は本物ですが、それは**あなたが日々向き合う仕事の少数派**かもしれません。多くのエンジニアにとって投資対効果が高いのは、既存改修でAIをいかに使いこなすか — つまり**既存コードの理解とリバースエンジニアリングの技術**です。新規のスピード感だけを見て「AIは既存改修では使えない」と結論づけるのは、もったいない誤解です。
:::

この非対称性は、学習の優先順位にも影響します。派手な新規デモに惹かれる気持ちは自然ですが、日々の生産性を上げたいなら、既存改修でのAI活用にこそ時間を投資する価値があります。

## 同じタスクで、最初のプロンプトはこう変わる

両者の違いは、最初にAIへ投げるプロンプトの形に最も鮮明に現れます。「ユーザー認証を実装する」という同一のタスクを、それぞれのアプローチで始めるとどうなるかを比べます。

新規(グリーンフィールド)では、**制約と意図を与えること**から始めます。

```text
新規プロジェクトに認証機能を実装します。まだコードはほぼありません。

前提・制約:
- 言語/フレームワーク: [採用したもの]
- 想定ユーザー規模・運用体制: [書く]
- 守りたい方針: [例: 外部IdP連携は将来、まずはシンプルに]

まず、本質的に異なる認証方式の選択肢を2〜3案、
トレードオフ付きで出してください。実装はまだしないでください。
過剰な汎用化はせず、最小構成から始められる案を優先します。
```

既存(ブラウンフィールド)では、**既存を読み取らせ、影響範囲を把握すること**から始めます。

```text
既存システムに認証まわりの変更を加えたいです。まず現状を理解させてください。

関連すると思われるコード:
[認証・セッション・ユーザー周辺のファイルやディレクトリを貼る/指す]

出力:
- 現在の認証フローを、実装から読み取れる範囲で要約
- 既存の規約・パターン(命名・エラー処理・テストの書き方)
- この変更が影響しそうな箇所と、壊すと危険な前提
- 不明な点・前提が曖昧な点

この段階では変更案を出さず、現状把握に徹してください。
```

::: tip 出だしで「生成」を急がせない
どちらのアプローチも、最初のプロンプトは**生成ではなく整理**を求めています。新規は選択肢の整理、既存は現状の整理。いきなり実装させないことが、暴走と的外れの両方を防ぎます。
:::

## どちらに寄るかを見極める

実際のプロジェクトは、純粋な新規でも純粋な既存でもなく、**スペクトラム上のどこか**に位置します。次の問いで自分の位置を確かめます。

| 問い | 新規寄り | 既存寄り |
| --- | --- | --- |
| 技術スタックは自由に選べるか | 選べる | 既存に縛られる |
| 既存の利用者・データはあるか | ない | ある(壊せない) |
| 参照すべき既存コードはどれだけあるか | ほぼない | 大量にある |
| 後方互換を守る必要があるか | 不要 | 必須 |
| 失敗時の作り直しコストは | 低い | 高い |

::: tip 「新規モジュールを既存システムに足す」は混合ケース
よくあるのは、既存の大きなシステムに、比較的独立した新規モジュールを追加するケースです。これは**モジュール内部は新規のアプローチ、外部との境界は既存改修のアプローチ**という混合になります。境界(API・データ契約)では既存の制約を厳密に読み取り、内部実装では新規の自由度を活かす、と切り分けます。判断の単位はプロジェクト全体ではなく、**変更のスコープごと**です。
:::

## アプローチを取り違えると何が起きるか

最も多い事故は、ツールやモデルの選択ミスではなく、**アプローチの取り違え**です。それぞれの方向で失敗が起きます。

### 既存改修を新規のように扱う

これが圧倒的に多い失敗です。すでに動いているシステムに対して、AIに「この機能を実装して」と新規同様に丸投げすると、AIは既存の規約・前提を知らないまま、一般解で新しいコードを書きます。結果、既存と矛盾する命名・エラー処理・データの扱いが混入し、コードベースの一貫性が静かに崩れていきます。

既存改修では、**実装を頼む前に必ず理解のフェーズを挟む**。既存コードを読ませ、現状仕様と既存パターンを把握させてから、それに沿った変更案を出させます。この一手間を飛ばすと、AIの速度はそのまま技術的負債の生産速度になります。

### 新規を既存のように扱う

逆もあります。本来自由に決められる新規プロジェクトで、AIが「一般的にこうします」と出した最初の構成を、まるで既存の制約のように受け入れてしまうケースです。AIの一般解を、検討すべき選択肢ではなく動かせない前提として扱うと、自チームに最適でない構成に早々と固定されます。

新規では、AIの推奨は**動かせる提案**です。複数案を出させ、自チームの事情で選び直す自由を、最初から手放さないことが大切です。

::: warning 「どちらのアプローチか」をタスク開始時に明言する
AIへの最初のプロンプトで、「これは既存システムへの変更だ」「これは白紙からの新規だ」を明示するだけで、生成の方向が大きく変わります。既存だと伝えれば理解を優先し、新規だと伝えれば選択肢を広げます。スコープごとにこの宣言を最初に置く習慣が、取り違えを防ぐ最も簡単な対策です。
:::

## 共通する原則は変わらない

アプローチは違っても、AI駆動開発の根本原則は両者で同じです。

- **AIは初稿生成と壁打ちの相手**。設計判断と最終的な責任は常に人間が持つ。
- **検証可能性を先に設計する**。新規ならテスト先行で振る舞いを定義し、既存なら回帰テストで現状を固定してから触る。
- **コンテキストを明示的に渡す**。新規は意図を、既存は既存コードを。渡さなければAIは一般解に流れる。
- **可逆性を意識する**。覆せる判断は素早く、覆せない判断は慎重に。

違うのは「**どの制約を、どこから持ってくるか**」だけです。この一点さえ押さえれば、あとは共通の原則を当てはめられます。

## どちらのガイドへ進むか

- 白紙から作る、技術選定が自由 → **[新規開発(0→1)のアプローチ](/approach/greenfield)** へ。AIの速度を制御し、白紙だからこそ制約を自分で与える進め方を扱います。
- すでに動くシステムに手を入れる → **[既存改修(ブラウンフィールド)のアプローチ](/approach/brownfield)** へ。既存コードの理解と影響範囲の見極めを中心に扱います。
- まず既存コードをAIに理解させたい → **[リバースエンジニアリング](/approach/reverse-engineering)** へ。実装からの仕様復元の技術を扱います。

## まとめ

- AI駆動開発の最初の分岐は、コードベースが**白紙か、すでに動いているか**。
- **新規**: 制約は存在しない。人間が能動的に与える。AIは速いが暴走もしやすい。引き算の制御。
- **既存**: 制約はコードに埋め込まれている。読み取って渡す。文脈理解とリバエンが鍵。静かな破壊に注意。
- 実務の大半は**既存改修**。派手な新規デモと、日々の仕事の比率は一致しない。
- 共通原則(AIは初稿・人が判断、検証可能性を先に、文脈を明示)は変わらない。違うのは制約の出どころだけ。
