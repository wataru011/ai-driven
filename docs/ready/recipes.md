# 逆引きレシピ集

「やりたいこと」から引ける、実務頻出タスクのレシピ集です。各レシピは **ゴール / 前提 / 手順 / コピペプロンプト / 検証方法 / 注意** の6点セット。プロンプト内の角括弧 `[...]` は、あなたの状況に置き換えて使ってください。

::: tip 使い方
すべてのレシピは**検証可能性**を前提にしています。出力をそのまま信じず、各レシピの「検証方法」を必ず通してください。プロンプトの土台を理解したい場合は[プロンプトの基礎](/prompting/fundamentals)、定型化したい場合は[プロンプトテンプレート集](/ready/prompt-templates)へ。
:::

::: warning 全レシピ共通の前提
- 機密(APIキー・認証情報・個人情報・未公開ソース)を投入しない。ダミー値・最小再現で尋ねる。
- 生成物は無検証でマージしない。テスト・実行・レビューを通す。
- 言語/バージョン/フレームワークは毎回明示する(プロンプトに含める、またはルールファイルに常駐させる→[ルールファイル](/ready/rule-files))。
:::

## 1. 既存バグを直したい {#recipe-1}

**ゴール**: 再現するバグを、原因特定とともに最小修正する。

**前提**: 再現手順またはエラーメッセージがある。対象ファイルが特定できている。

**手順**

1. 再現手順またはエラーを正確に控える。
2. 対象コードを文脈として渡し、下のプロンプトを投げる。
3. 「原因の説明」を読んで納得できなければ、納得できるまで往復する。
4. 提案された再現テストを先に走らせ、**赤になること**を確認してから修正を適用する。

```text
あなたは[言語]に精通したエンジニアです。

## 状況
[ファイルパス] の [関数/箇所] で次の不具合が起きます。
[再現手順 または エラーメッセージ全文]

## 期待する動作
[こうなってほしい、を具体的に]

## 制約
- 既存シグネチャは変えない(変える場合は理由を添えて提案)
- 新しい依存は追加しない
- 無関係な箇所は変更しない

## 出力
1. 原因の説明(1〜3行)
2. このバグを再現し、修正後に通るテストを1つ
3. 修正後のコード(差分が分かる形で)
```

**検証方法**: 再現テストが修正前は赤・修正後は緑になること。既存テストが緑のままであること。

**注意**: 「原因の説明」が曖昧なまま修正だけ受け取らない。原因を理解できない修正は採用しない。

## 2. 新機能を足したい {#recipe-2}

**ゴール**: 既存規約に沿って、検証可能な単位で機能を追加する。

**前提**: 受け入れ条件を言語化できる。挿入先のファイル/層が分かる。

**手順**

1. 機能を**1PRで検証できる粒度**まで小さくする(大きければ分割)。
2. 既存の類似実装を1つ見本として渡す。
3. 概要→コード→テストの順で出させる。
4. 概要だけ先にレビューし、方向が合ってからコードを受け取る。

```text
## 目的
[機能の1行説明]。

## 既存
- 関連エンドポイント/モジュール: [パス]
- 現在の挙動: [現状]
- 参考にすべき類似実装: [パス](このスタイルに合わせる)

## 要件
- [入力/パラメータと制約]
- [出力/レスポンス形式]
- [エッジケース: 例 空・最大件数・権限なし]

## 制約
- 既存の[層/クエリビルダ/規約]を踏襲する
- 新規依存は追加しない

## 出力
1. 変更点の概要(箇条書き)
2. 実装コード
3. 境界値を含むテスト
```

**検証方法**: 受け入れ条件をテストに落として実行。手動でも主要パスを1回通す。

詳しい進め方は[機能開発の実践](/practice/feature-development)を参照。

**注意**: 「全部いい感じに」は禁物。受け入れ条件が書けないなら、まだ分割が足りない。

## 3. テストを増やしたい {#recipe-3}

**ゴール**: 重要関数の振る舞いを、読めるテストで固める。

**前提**: 対象関数の仕様を自分が説明できる。テストフレームワークが決まっている。

**手順**

1. テストしたい関数と、その仕様を1〜2行で書く。
2. 既存テストを1つ見本として渡し、スタイルを揃える。
3. 正常系・境界・異常系を網羅させる。
4. **アサーションの中身**を自分の目で確認する。

```text
あなたは[言語]のテストに精通したエンジニアです。

## 対象
[ファイルパス] の [関数名]。仕様: [1〜2行]。

## 指示
[テストフレームワーク]で単体テストを書いてください。
[参考テストのパス] のスタイル・命名に合わせてください。

## 網羅してほしいケース
- 正常系(代表的な入力)
- 境界値([具体: 0件・最大・空文字 など])
- 異常系([具体: null・不正型・例外])

## 出力
- テストファイル全体
- 各テスト名は「何を検証するか」が読み取れる命名にする
```

**検証方法**: テストを実行して緑。あえて実装に1行バグを入れて**赤くなる**ことを確認(テストが効いている証拠)。

**注意**: カバレッジ数値だけ追わない。アサーションが弱いと「通るだけのテスト」が量産される。

## 4. 失敗しているテストを直したい {#recipe-4}

**ゴール**: 落ちているテストを、原因を見極めて正しく直す。

**前提**: 失敗ログがある。テストが「仕様」なのか「実装」なのかを判断できる。

**手順**

1. 失敗ログ全文と、対象テスト・対象実装を渡す。
2. **「テストが間違っているのか、実装が間違っているのか」を先に判定**させる。
3. 判定に同意できたら、その側だけ直す。

```text
次のテストが失敗しています。

## 失敗ログ
[ログ全文]

## テストコード
[該当テスト]

## 実装コード
[該当実装]

## 指示
まず、失敗の原因が「テスト側の誤り」か「実装側の誤り」かを判定し、
根拠を述べてください。そのうえで、誤っている側だけを最小修正してください。
両方を都合よく書き換えないでください。
```

**検証方法**: 当該テストが緑になり、かつ**他のテストが赤くならない**こと。

**注意**: テストを通すために実装の期待値をこっそり緩める「テスト改ざん」を見逃さない。意図した仕様変更なのかを必ず確認する。

## 5. リファクタしたい {#recipe-5}

**ゴール**: 外部から見た振る舞いを変えずに、内部構造を改善する。

**前提**: **対象を覆うテストがある**(なければレシピ3で先に用意)。

**手順**

1. リファクタの目的(可読性/重複排除/責務分離など)を1つに絞る。
2. 「振る舞いを変えない」を制約として明示する。
3. 変更前後でテストが緑のままであることを確認する。

```text
## 目的
[ファイル/関数] を [可読性向上 / 重複排除 / 責務分離] のためにリファクタしたい。

## 厳守
- 外部から見た振る舞い(入出力・公開API)は一切変えない
- 既存テストはすべて緑のまま
- 1コミットで説明できる範囲に変更をとどめる

## 指示
1. 変更方針を箇条書きで先に示す
2. リファクタ後のコード
3. 振る舞いが変わっていないと言える根拠
```

**検証方法**: 既存テストが**変更前後で同じ結果**。差分が「構造変更だけ」で振る舞いに踏み込んでいないか目視。

**注意**: テストがない状態でのリファクタは禁物。安全網なしの構造変更は、静かなバグの温床。

## 6. コードをレビューしてほしい {#recipe-6}

**ゴール**: 自分のレビュー前に、観点を漏らさず一次チェックする。

**前提**: レビュー対象の差分がある。AIの指摘は**候補**であって最終判断は自分が下す。

**手順**

1. 差分(またはファイル)と、レビューで重視する観点を渡す。
2. 重大度付きで指摘させる。
3. 各指摘を自分で検証し、誤検知を捨てる。

```text
あなたは経験豊富なコードレビュアーです。

## レビュー対象
[差分 または ファイル]

## 重視する観点
- 正しさ(ロジック・境界条件・例外処理)
- セキュリティ(入力検証・権限・機密の扱い)
- 可読性・命名・既存規約との整合
- テストの十分性

## 出力
指摘を「重大 / 中 / 軽微」に分類し、各指摘に
「該当箇所」「理由」「修正案」を添えてください。
問題がない箇所は列挙しないでください。
```

**検証方法**: 各指摘が実際に問題かを**自分で再確認**。少なくとも「重大」は手元で再現/確認する。

**注意**: AIレビューは見落とし削減の補助であり、責任の移譲ではない。マージ判断はあなたの仕事。

## 7. エラーの原因を知りたい {#recipe-7}

**ゴール**: スタックトレースから原因の仮説を素早く立てる。

**前提**: エラーメッセージ/スタックトレースと、周辺コードがある。

**手順**

1. エラー全文と、関係しそうなコードを渡す。
2. **複数の仮説を確度順**に挙げさせる。
3. 上位仮説から手元で再現・確認する。

```text
次のエラーの原因を特定したいです。

## エラー
[スタックトレース全文]

## 関連コード
[該当箇所]

## 環境
[言語/バージョン, ランタイム, 関連ライブラリのバージョン]

## 指示
考えられる原因を確度の高い順に最大3つ挙げ、
それぞれ「根拠」と「確認方法(手元で再現する手順)」を示してください。
推測の場合は推測と明記してください。
```

**検証方法**: 提示された確認手順で**実際に再現**できるか。再現して初めて原因確定。

**注意**: 最初の仮説に飛びつかない。確認手順で再現できなければ次の仮説へ。

## 8. ログから障害を追いたい {#recipe-8}

**ゴール**: 大量ログから障害の時系列と要因を絞り込む。

**前提**: 該当時間帯のログ抜粋がある(**機密・個人情報はマスクする**)。

**手順**

1. ログを時間帯で絞り、機密をマスクして渡す。
2. タイムラインの再構成と異常点の指摘を求める。
3. 指摘された時刻を実ログ・メトリクスと突き合わせる。

```text
障害のログを解析したいです。

## ログ(機密はマスク済み)
[ログ抜粋]

## 分かっている事象
[いつ何が起きたか, 影響範囲]

## 指示
1. ログから時系列のタイムラインを再構成
2. 異常の起点と思われる箇所を指摘(根拠付き)
3. 次に確認すべきログ/メトリクスを提案

ログにない事実を断定しないでください。不明点は不明と書いてください。
```

**検証方法**: 再構成タイムラインを**生ログ・監視メトリクス**と照合。憶測で埋まった箇所がないか確認。

**注意**: ログを貼る前にトークン・PII・認証情報をマスク。外部送信ポリシーを確認([チーム運用ポリシー](/ready/team-policy))。

## 9. 正規表現を作りたい {#recipe-9}

**ゴール**: マッチ/非マッチ要件を満たす正規表現を、説明付きで得る。

**前提**: マッチさせたい例・させたくない例を複数用意できる。

**手順**

1. 正例・反例を箇条書きで渡す。言語の正規表現方言を指定する。
2. パターンと**各部の説明**を出させる。
3. 全例を手元で実行して確認する。

```text
正規表現を作りたいです。方言は [例: JavaScript / PCRE / RE2] です。

## マッチさせたい(正例)
- [例1]
- [例2]

## マッチさせたくない(反例)
- [例3]
- [例4]

## 指示
1. 上記をすべて満たす正規表現
2. パターンを部分ごとに分解した説明
3. 想定される取りこぼし/誤マッチがあれば指摘
```

実行確認用のテストコードも一緒に頼むと早いです。

```text
さらに、上の正例・反例をすべて検証する [言語] のテストコードも出してください。
```

**検証方法**: 正例が全マッチ・反例が全非マッチになることを実行で確認。

**注意**: 過度に複雑な正規表現は保守不能。可読性が破綻するなら、コードでの分割処理を選ぶ。ReDoS(壊滅的バックトラッキング)に注意。

## 10. SQLを書きたい {#recipe-10}

**ゴール**: 要件を満たすクエリを、対象RDBMSの方言で得る。

**前提**: テーブル定義(スキーマ)とDBの種類・バージョンを渡せる。

**手順**

1. スキーマ(`CREATE TABLE` か列の説明)とDB種別を渡す。
2. 取得/更新したい条件を明確に書く。
3. `EXPLAIN` と少量データで検証する。

```text
SQLを書きたいです。DBは [例: PostgreSQL 16] です。

## スキーマ
[CREATE TABLE 文 または 列と型の一覧]

## やりたいこと
[取得/集計/更新したい条件を具体的に]

## 制約
- [例: インデックスは (created_at desc) がある]
- [例: 1クエリで完結させたい / N+1を避けたい]

## 出力
1. クエリ
2. 何をしているかの説明
3. 想定される性能上の注意(インデックス利用など)
```

**検証方法**: 開発DBで実行し結果を確認。重いクエリは `EXPLAIN`(または `EXPLAIN ANALYZE`)で実行計画を確認。

**注意**: 破壊的クエリ(`UPDATE`/`DELETE`)は必ずトランザクション内かつ `WHERE` を二重確認。本番データで直接試さない。

## 11. APIクライアントを生成したい {#recipe-11}

**ゴール**: API仕様から型付きクライアントコードを生成する。

**前提**: APIのエンドポイント仕様(OpenAPI/サンプルレスポンス)がある。

**手順**

1. 仕様(またはサンプルのリクエスト/レスポンス)を渡す。
2. 言語・HTTPクライアント・エラー処理方針を指定する。
3. 実エンドポイントに対して疎通確認する。

```text
APIクライアントを生成したいです。

## 言語/環境
[言語/バージョン], HTTPクライアント: [例: fetch / axios / requests]

## API仕様
[OpenAPIの抜粋 または サンプルのリクエスト・レスポンス]

## 要件
- レスポンスは型付けする
- エラー(4xx/5xx/タイムアウト)を呼び出し側が扱える形にする
- 認証は [方式]、機密はコードに直書きしない(環境変数から)

## 出力
1. クライアントコード
2. 使用例
3. 簡単な疎通テスト
```

**検証方法**: 実エンドポイント(またはモックサーバ)に対して疎通テストを実行。エラー時の挙動も確認。

**注意**: 仕様にないフィールドを勝手に補完していないか確認。APIキーは環境変数化し、プロンプトに**実キーを貼らない**。

## 12. 型を付けたい {#recipe-12}

**ゴール**: 型のないコードに、実態に合った型を付与する。

**前提**: 型チェッカ(tsc / mypy など)を実行できる。

**手順**

1. 対象コードと、実際に流れる値の例を渡す。
2. `any`/`Any` の濫用を禁止する。
3. 型チェッカを通し、赤を潰すまで往復する。

```text
このコードに型を付けたいです。言語は [例: TypeScript / Python(型ヒント)]。

## 対象コード
[コード]

## 実際に流れる値の例
[代表的な入力/出力のサンプル]

## 制約
- any / Any への逃げは禁止(本当に不可能な箇所のみ理由を添えて)
- 既存の挙動は変えない(型注釈の追加のみ)
- [既存の型定義/スキーマ] と整合させる

## 出力
- 型を付けたコード
- 型チェッカで警告が出そうな箇所があれば指摘
```

**検証方法**: 型チェッカ(`tsc --noEmit` / `mypy`)が通る。実行時の挙動が変わっていないことをテストで確認。

**注意**: 型を通すためにロジックを書き換えていないか確認。型注釈の追加に閉じているかが肝。

## 13. ドキュメントを書きたい {#recipe-13}

**ゴール**: コードの実態に即したドキュメント(README/関数コメント等)を書く。

**前提**: ドキュメント化する対象コードを渡せる。

**手順**

1. 対象コードと、読者(利用者/保守者)を指定する。
2. 出力形式(README節/docstring/JSDoc)を指定する。
3. 記述が実装と一致しているか確認する。

```text
[対象] のドキュメントを書きたいです。

## 対象コード
[コード]

## 読者
[例: 初めて使う社内開発者 / 将来の保守担当]

## 形式
[例: READMEの「使い方」節 / 各公開関数のJSDoc]

## 指示
- 実際の引数・戻り値・例外に厳密に基づくこと
- 使用例を1つ含める
- コードに書かれていない動作を推測で書かない
```

**検証方法**: 記述された引数・戻り値・例外が**実装と一致**するか1項目ずつ照合。使用例を実際に動かす。

**注意**: AIは「あるべき仕様」を書きがち。**実装の事実**とずれていないかが最重要。

## 14. コミットメッセージを作りたい {#recipe-14}

**ゴール**: 差分の意図が伝わるコミットメッセージを書く。

**前提**: ステージ済みの差分(`git diff --staged`)がある。

**手順**

1. 差分を渡す(機密が含まれないこと)。
2. チームの規約(Conventional Commits等)を指定する。
3. 生成文が**差分の実態**と一致するか確認する。

```text
次の差分のコミットメッセージを書いてください。

## 規約
[例: Conventional Commits / 1行要約+本文 / 日本語]

## 差分
[git diff --staged の出力]

## 指示
- 件名は[文字数目安]以内、命令形/[規約に従う]
- 本文に「何を・なぜ」を簡潔に
- 差分に存在しない変更を書かない
```

ターミナル派は差分を直接渡すワンライナーも便利です。

```bash
git diff --staged | <あなたのAI CLI>  # 差分を貼り付けてメッセージ生成を依頼
```

**検証方法**: 件名・本文が差分の内容と一致するか確認。誇張・捏造がないか。

**注意**: 差分に映り込んだ機密(`.env` 等)をそのまま渡さない。ステージ内容を先に確認。

## 15. PR説明文を作りたい {#recipe-15}

**ゴール**: レビュアーが短時間で意図を把握できるPR説明を書く。

**前提**: ブランチの差分やコミット履歴がある。

**手順**

1. 変更概要・背景・差分(または主要コミット)を渡す。
2. レビュー観点・テスト方法・影響範囲を含めさせる。
3. レビュアー視点で過不足を点検する。

```text
プルリクエストの説明文を書いてください。

## 背景/目的
[なぜこの変更が必要か]

## 変更概要
[何を変えたか / 主要コミット or 差分]

## テンプレ項目
- 概要(3行以内)
- 変更点(箇条書き)
- テスト方法(レビュアーが再現できる手順)
- 影響範囲・リスク
- レビューで特に見てほしい点

実際の差分にない内容は書かないでください。
```

**検証方法**: レビュアーの立場で読み、「これで再現・判断できるか」を確認。テスト手順を自分でなぞる。

**注意**: 「レビューで見てほしい点」を具体に。曖昧なPR説明はレビュー品質を下げる。GitHub運用は[CI連携](/ready/ci-integration)も参照。

## 16. 依存を更新したい {#recipe-16}

**ゴール**: ライブラリ更新に伴う破壊的変更を把握し、安全に追従する。

**前提**: 現バージョンと目標バージョン、変更履歴(CHANGELOG/リリースノート)がある。

**手順**

1. 対象ライブラリの現/目標バージョンと、自分の使用箇所を渡す。
2. 破壊的変更と必要な移行作業を洗い出させる。
3. **公式リリースノート**で裏取りし、テストを全実行する。

```text
依存を更新したいです。

## 対象
[ライブラリ名] を [現バージョン] → [目標バージョン] に上げる。

## 自分のコードでの使用箇所
[該当コード]

## 指示
1. このバージョン間の破壊的変更のうち、上記コードに影響するものを列挙
2. 必要な移行作業を手順化
3. 不確実な点は「公式リリースノートで要確認」と明示

知識が古い可能性があるため、断定せず確認ポイントを示してください。
```

**検証方法**: 指摘を**公式CHANGELOG/リリースノート**で裏取り。更新後にテスト全実行・主要動線を手動確認。

**注意**: AIの学習時点が古いと、最新版の破壊的変更を知らない。必ず一次情報で確認([ハルシネーション対策](/quality/hallucination))。

## 17. 別言語へ移植したい {#recipe-17}

**ゴール**: あるコードを、別言語へ振る舞いを保って移植する。

**前提**: 元コードと、移植先の言語/規約がある。入出力の同値を確認できる。

**手順**

1. 元コードと、移植先言語・利用ライブラリ方針を渡す。
2. 言語固有のイディオムに直すよう指示する。
3. 同じ入力で**同じ出力**になることをテストで確認する。

```text
このコードを [元言語] から [移植先言語/バージョン] へ移植してください。

## 元コード
[コード]

## 指示
- 1対1の逐語訳ではなく、移植先言語の慣用的な書き方にする
- 標準ライブラリを優先。外部依存は最小限(追加時は理由を添える)
- 元コードの入出力・例外挙動を保つ

## 出力
1. 移植後のコード
2. 元と挙動が変わりうる箇所(数値精度・null扱いなど)の注意
```

**検証方法**: 同一の入力セットに対し、元と移植先で**出力が一致**するか比較テスト(できればゴールデンテスト)。

**注意**: 数値精度・文字コード・null/undefined・例外の扱いは言語間でずれやすい。境界を重点的に検証。

## 18. 設定ファイルを作りたい {#recipe-18}

**ゴール**: ツール/CIの設定ファイルを、目的に合った形で用意する。

**前提**: 対象ツールとバージョン、達成したい構成が言える。

**手順**

1. ツール名・バージョンと、やりたい構成を渡す。
2. 各設定項目の意味を説明させる(コピペして終わりにしない)。
3. バリデータ/dry-runで構文と挙動を確認する。

```text
[ツール名/バージョン] の設定ファイルを作りたいです。

## やりたいこと
[達成したい構成。例: PRごとにlintとtestを走らせる]

## 環境
[OS / ランタイム / 既存の制約]

## 出力
1. 設定ファイルの中身
2. 主要な項目が何をしているかの説明
3. よくある落とし穴があれば注意

存在しない設定キーを推測で書かないでください。不確実なら明示してください。
```

**検証方法**: ツールのバリデータ(あれば)や dry-run で確認。CI設定は小さなブランチで実走させる。

**注意**: 設定キーはハルシネーションの温床。存在しないオプションを書かれていないか公式ドキュメントで確認。CI全体像は[CI連携](/ready/ci-integration)へ。

## 19. パフォーマンスを改善したい {#recipe-19}

**ゴール**: 計測に基づいて、ボトルネックを具体的に改善する。

**前提**: **計測データ**(プロファイル/ベンチ結果)がある。当て推量から始めない。

**手順**

1. まず計測する。遅い箇所のプロファイル結果を渡す。
2. 計算量・I/O・メモリの観点で改善案を出させる。
3. 改善前後でベンチを取り、**実測で効果を確認**する。

```text
パフォーマンスを改善したいです。

## 対象
[ファイル/関数]

## 計測結果
[プロファイル/ベンチの抜粋。どこが遅いか]

## 制約
- 振る舞いと出力は変えない
- [メモリ上限 / 互換性 などの制約]

## 出力
1. ボトルネックの分析(計算量・I/O・メモリの観点)
2. 改善案を効果が大きい順に
3. 各案のトレードオフ(可読性・メモリとのバーター等)
```

**検証方法**: 改善前後で**同じ条件のベンチ**を取り、実測で速くなったことを確認。出力が変わらないことをテストで保証。

**注意**: 推測での「最適化」は禁物。まず計測。可読性を犠牲にする最適化は、効果が実測で見合うときだけ。

## 20. セキュリティを点検したい {#recipe-20}

**ゴール**: 一般的な脆弱性の観点で、コードの危険箇所を洗い出す。

**前提**: 点検対象コードがある。AIの指摘は**候補**で、最終判断と専用ツールが必要。

**手順**

1. 対象コードと、扱うデータ・信頼境界を渡す。
2. 代表的な脆弱性カテゴリで点検させる。
3. 指摘を再現確認し、専用ツール(SAST/依存スキャン)も併用する。

```text
あなたはセキュリティに厳格なレビュアーです。

## 対象コード
[コード]

## 文脈
扱うデータ: [例: ユーザー入力, 認証情報]、信頼境界: [どこから外部入力か]

## 点検観点
- インジェクション(SQL/コマンド/XSS等)
- 認証・認可の不備
- 入力検証・出力エスケープの漏れ
- 機密の取り扱い(ログ出力・ハードコード)
- 安全でない依存・既知の危険関数

## 出力
各指摘に「該当箇所」「想定される攻撃」「重大度」「修正案」を付けてください。
推測の場合は明記してください。
```

**検証方法**: 重大指摘は**実際に再現**を試みる。SAST・依存脆弱性スキャン・シークレットスキャンを併用([CI連携](/ready/ci-integration))。

**注意**: AIのセキュリティ点検は補助であり保証ではない。見落としも誤検知もある。専用ツールと人のレビューを置き換えない。機密は投入しない。

## 21. コードを読解・要約したい {#recipe-21}

**ゴール**: 不慣れな既存コードの構造と要点を素早くつかむ。

**前提**: 対象コードがある。一部は自分が答えを知っている(答え合わせのため)。

**手順**

1. 対象コード/モジュールを渡す。
2. 全体の役割→主要な流れ→注意点、の順で要約させる。
3. 自分が知っている部分と照合し、ずれがないか確認する。

```text
このコードを読解したいです。

## 対象
[コード または モジュールの説明とファイル一覧]

## 知りたいこと
1. 全体の役割(3行で)
2. 主要な処理の流れ(入口から出口まで)
3. 注意すべき副作用・状態・外部依存
4. 読み解く際に確認すべき箇所

コードから読み取れない事実は推測と明記してください。
```

**検証方法**: 自分が既に知っている部分の説明が正しいか照合。誤りがあれば全体の信頼度を割り引く。

**注意**: 要約は理解の入口であって結論ではない。重要箇所は必ず原典コードに当たる。

## 22. CLIスクリプトを作りたい {#recipe-22}

**ゴール**: 反復作業を自動化する小さなCLIスクリプトを得る。

**前提**: 入力・出力・実行環境が言える。破壊的操作はdry-runで守れる。

**手順**

1. やりたい処理・入出力・実行環境(シェル/言語)を渡す。
2. **dry-run と確認プロンプト**を入れさせる(破壊的操作の場合)。
3. まず安全な対象で試す。

```text
次の作業を自動化するスクリプトを作りたいです。

## やること
[処理内容。入力と出力を具体的に]

## 環境
[例: bash on Linux / Python 3.12]

## 要件
- 破壊的操作(削除・上書き)には --dry-run を用意し、デフォルトはdry-run
- エラー時は分かるメッセージで停止する
- 使い方(--help)を含める

## 出力
1. スクリプト
2. 使用例(dry-runと実行)
```

**検証方法**: まず `--dry-run` で出力を確認 → 安全なサンプルで実行 → 本番対象へ。

**注意**: ファイル削除・一括置換・本番接続を伴うスクリプトは、必ずdry-runと小規模試行を経てから使う。

## 23. エラーメッセージを改善したい {#recipe-23}

**ゴール**: 利用者・開発者が原因と対処に辿り着けるエラーメッセージにする。

**前提**: 現状のエラーメッセージと、発生文脈がある。

**手順**

1. 現状の文言と、それが出る状況を渡す。
2. 「何が・なぜ・どうすれば」を含む改善案を出させる。
3. 機密や内部実装を漏らしていないか確認する。

```text
このエラーメッセージを改善したいです。

## 現状のメッセージ
[現状の文言]

## 発生状況
[いつ出るか / 想定読者は利用者か開発者か]

## 指示
- 「何が起きたか・なぜか・どうすればよいか」を含める
- 内部実装の詳細や機密(パス・トークン・スタック)を露出しない
- [トーン: 簡潔/丁寧 など] に合わせる

改善案を2〜3パターン出してください。
```

**検証方法**: 改善案を実際の文脈に置いて読み、原因と対処に辿り着けるか確認。機密露出がないか点検。

**注意**: 利用者向けと開発者向け(ログ)でメッセージの粒度を分ける。利用者にスタックトレースを見せない。
