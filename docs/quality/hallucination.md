# ハルシネーション対策

**ハルシネーション(hallucination)** とは、LLMが事実に反する、あるいは存在しない情報を、**もっともらしく自信たっぷりに**生成してしまう現象です。コードの文脈では「存在しない関数を呼ぶ」「実在しないパッケージをインストールさせる」「APIの使い方をそれらしく間違える」といった形で現れ、品質とセキュリティの両面で問題を起こします。

::: tip ハルシネーションはバグではなく性質
ハルシネーションはモデルの欠陥というより、**動作原理の必然的な副産物**です。完全には消せません。だからこそ「ゼロにする」のではなく「検出して緩和する」発想で対処します。
:::

## なぜ起きるのか

LLMは「事実を検索して答える」のではなく、**文脈に続く統計的にもっともらしいトークンを生成**しています。詳細は[LLMの仕組み](/concepts/how-llms-work)に譲りますが、要点は次のとおりです。

- モデルは**真偽の区別を持たない**。「正しい答え」も「もっともらしい嘘」も、生成過程は同じ。
- 学習データに無いことを問われても、**「分かりません」より、それらしい続きを埋める**方向に最適化されている。
- **自信(口調)と正しさは無相関**。流暢で断定的でも、内容が正しい保証はない。
- 学習データの**カットオフ以降の情報**や、希少な事実は特に外しやすい。

::: warning 流暢さに騙されない
ハルシネーションが危険なのは、**間違いが間違いに見えない**からです。自然な変数名、正しそうな引数、それらしい説明文を伴うため、レビューで見逃されやすい。流暢さを正しさの証拠と取り違えないことが第一歩です。
:::

## コード文脈での典型パターン

| パターン | 内容 | 危険度 |
| --- | --- | --- |
| 存在しないAPI/メソッド | それらしいが実在しない関数・メソッドを呼ぶ | 高(実行で露呈するが、稀に別物にヒット) |
| 存在しないパッケージ | 実在しないライブラリを `import` / インストールさせる | 最高(供給網攻撃の二次リスク) |
| 誤った引数/シグネチャ | 実在するAPIだが引数の数・順序・名前を間違える | 高 |
| 古い/廃止されたAPI | 過去には存在したが現行版で削除された機能を使う | 中 |
| それらしい誤説明 | 動作・仕様の説明が自信満々に間違っている | 中(コードより気づきにくい) |
| 捏造された設定/フラグ | 実在しない設定キーやCLIフラグを提示する | 中 |

```python
# ハルシネーションの典型例
import pandas as pd

# pandas に存在しないメソッドをそれらしく呼ぶ
df.drop_duplicates_by(["id"])     # 実在しない(正しくは drop_duplicates(subset=["id"]))

# 実在するが引数が間違っている
df.merge(other, how="outer", on="id", validate="one_to_many")  # validate値が不正かもしれない
```

## 二次リスク: スラップスクワッティング

「存在しないパッケージ」のハルシネーションは、単なるエラーでは済まない**セキュリティ問題**に発展します。

**スラップスクワッティング(slopsquatting)** とは、LLMが繰り返し**幻覚する架空のパッケージ名**を攻撃者が先回りして登録し、その名前に**悪意あるコードを仕込む**手口です。

```text
1. LLMが「便利そうな」存在しないパッケージ名を繰り返し提案する
2. 攻撃者がその名前を実在のパッケージとして公開リポジトリに登録(悪意あるコード入り)
3. 開発者がAIの提案どおり `install その名前` を実行
4. マルウェアが実行環境に侵入
```

::: warning AIの提案するパッケージは必ず実在を確認
`install` や `import` をAIに提案されたら、**そのパッケージが本当に正規のものか**を、公式リポジトリ・ダウンロード数・メンテナンス状況・リポジトリの実在で確認してください。タイポスクワッティング(typosquatting)と同様の警戒が必要です。これは[セキュリティリスク](/quality/security)の供給網リスクと地続きです。
:::

## 検出と緩和

ハルシネーションは消せませんが、**多くは機械的に検出できます**。「実在するか」「動くか」「型が合うか」は機械が判定できるからです。

| 手段 | 何を捕まえるか | 自動化 |
| --- | --- | --- |
| 実行(run) | 存在しないAPI/構文エラー(実行時に落ちる) | 高 |
| テスト | 誤った振る舞い・引数違い | 高 |
| 型チェック | 存在しないメソッド・型不整合 | 高 |
| リンタ/コンパイラ | 未定義シンボル・未解決import | 高 |
| ドキュメント照合 | API名・引数・仕様の正しさ | 中(公式ドキュメントと突合) |
| パッケージ実在確認 | 架空パッケージ(slopsquatting) | 中 |
| RAGによる裏取り | カットオフ以降・希少な事実の正確性 | 中 |
| 人間検証 | 説明の誤り・設計判断・文脈依存の誤り | 低(but必須) |

::: tip 型とテストはハルシネーションキラー
**存在しないAPIや誤った引数の多くは、型チェック・コンパイル・テスト実行で即座に露呈します。** 「動かして確認できる」環境を整えるほど、ハルシネーションは無害化されます。エージェントにこの検証ループを与えると、自己修正で勝手に直すことも多い([AIエージェントとは](/agents/what-is-agent)参照)。
:::

### RAGによる裏取り

**RAG(Retrieval-Augmented Generation)** は、モデルに回答させる前に**信頼できる情報源を検索して文脈に与える**手法です。最新ドキュメント・社内仕様・正確なAPIリファレンスを根拠として渡すことで、記憶頼みの幻覚を減らせます。

```text
質問 → 関連ドキュメントを検索 → 取得した一次情報を文脈に注入 → その根拠に基づき生成
                                                              └→ 出典を提示させ、検証可能にする
```

ただしRAGも万能ではありません。検索が的外れなら効果はなく、取得した情報の**裏取り**は依然必要です。

## 実務フロー

ハルシネーション対策を、生成から統合までのフローに落とし込みます。

```text
┌─ 生成 ────────────────────────────────────────────┐
│ 1. 根拠を要求する: 「出典・公式ドキュメントを示して」 │
│ 2. RAGで一次情報を文脈に与える(可能なら)          │
└────────────────────┬───────────────────────────────┘
                     ▼
┌─ 機械検証(自動・必須) ──────────────────────────┐
│ 3. コンパイル/型チェック → 未定義シンボルを検出     │
│ 4. リンタ → 未解決importを検出                      │
│ 5. テスト実行 → 誤った振る舞い・引数違いを検出       │
│ 6. パッケージ実在確認 → slopsquatting を排除         │
└────────────────────┬───────────────────────────────┘
                     ▼
┌─ 人間検証(機械が見られない部分) ────────────────┐
│ 7. API名・引数を公式ドキュメントと照合              │
│ 8. 説明文の正しさを確認(コードと別に検証)          │
│ 9. 設計・文脈依存の妥当性を判断 → 統合(マージ)     │
└────────────────────────────────────────────────────┘
```

チェックリストにすると次のとおりです。

- [ ] AIが提案した**API・メソッドが実在し、現行版で正しい**かを確認した。
- [ ] AIが提案した**パッケージが正規に実在する**ことを確認した(slopsquatting対策)。
- [ ] **コンパイル・型チェック・リンタ・テスト**を通した。
- [ ] AIの**説明文**を、コードとは別に公式情報と突き合わせた。
- [ ] カットオフ以降・希少な事実は、**一次情報で裏取り**した(RAGや手動検索)。
- [ ] 「自信のある口調」を正しさの証拠にしなかった。

## まとめ

- ハルシネーションは**動作原理の必然**で、ゼロにはできない。「検出して緩和する」発想で臨む。
- コード文脈の典型は**存在しないAPI/パッケージ・誤った引数・それらしい誤説明**。
- 架空パッケージは**スラップスクワッティング**という供給網攻撃の二次リスクを生む。提案パッケージは必ず実在確認。
- **実行・型・テスト・リンタ**で多くは機械的に検出できる。RAGで裏取りし、説明と設計判断は人間が検証する。
